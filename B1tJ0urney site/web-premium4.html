📖 Chapter 1 — JavaScript Foundations Revisited

Before diving deep, we solidify the core pillars that everything else rests on: execution model, scope, types, and modern syntax.

🚀 1.1 How JavaScript Actually Runs

JavaScript is a single-threaded, non-blocking, asynchronous language powered by an event-driven runtime. Understanding this model is the #1 key to writing good JS.

💡 Mental model: Think of JavaScript as a chef with one stove (single thread) but a kitchen timer (async). The chef doesn't stand idle waiting for water to boil — they start other tasks and come back when the timer rings.

The JavaScript Engine

V8 (Chrome/Node), SpiderMonkey (Firefox), and JavaScriptCore (Safari) all follow the same process:

1. Parsing

Source code → Tokens → Abstract Syntax Tree (AST). Syntax errors are caught here.

2. Compilation (JIT)

AST → Bytecode → Machine Code. Modern engines compile "hot" code paths for speed.

3. Execution

Code runs inside an Execution Context on the Call Stack.

4. Garbage Collection

The engine automatically frees memory no longer referenced (mark-and-sweep algorithm).

🧬 1.2 Execution Context & the Call Stack

Every time JavaScript runs code, it creates an Execution Context. There are two main types:

  • Global Execution Context (GEC): Created once when the script starts. Contains the global object (window in browsers, global in Node).
  • Function Execution Context (FEC): Created every time a function is called. Each has its own this, arguments, and variable environment.

Each Execution Context goes through two phases:

Creation Phase (Hoisting)

var declarations are hoisted and set to undefined. Function declarations are fully hoisted. let and const exist in a "Temporal Dead Zone" until their line.

Execution Phase

Code runs line by line. Variables are assigned values. Functions are called, pushing new contexts onto the stack.

JavaScript 
console.log(a); // undefined (hoisted)
// console.log(b); // ❌ ReferenceError (TDZ)

var a = 10;
let b = 20;

function greet(name) {
    // New FEC created here
    let message = "Hello, " + name;
    return message;
}

greet("Alice"); // FEC pushed, then popped after return
The Call Stack is a data structure that records where in the program we are. When a function is called, it's pushed. When it returns, it's popped. If the stack overflows (infinite recursion), you get a RangeError: Maximum call stack size exceeded.

🔭 1.3 Scope: Lexical, Block, and Module

Scope determines where variables are accessible. JavaScript uses lexical (static) scoping — meaning scope is defined by where code is written, not where it's called.

The Scope Chain

When a variable isn't found in the current scope, the engine walks up the scope chain until it reaches the global scope.

Scope Chain Example 
const globalVar = "I'm global";

function outer() {
    const outerVar = "I'm outer";
    
    function inner() {
        const innerVar = "I'm inner";
        // Can access all three:
        console.log(innerVar);  // ✅
        console.log(outerVar);  // ✅ (scope chain)
        console.log(globalVar); // ✅ (scope chain)
    }
    
    // console.log(innerVar); // ❌ innerVar not accessible here
    inner();
}

outer();

var vs let vs const — The Full Picture

Variable Comparison 
// var — function-scoped, hoisted, re-declarable (avoid in modern JS)
if (true) {
    var x = 10;
}
console.log(x); // 10 (leaks out of block!)

// let — block-scoped, not hoisted (TDZ), re-assignable
if (true) {
    let y = 20;
}
// console.log(y); // ❌ ReferenceError

// const — block-scoped, not hoisted (TDZ), NOT re-assignable
const PI = 3.14159;
// PI = 3; // ❌ TypeError

// ⚠️ const with objects — the binding is constant, not the value
const user = { name: "Alice" };
user.name = "Bob"; // ✅ This is fine
// user = {}; // ❌ This fails
⚠️ Rule of thumb: Always prefer const. Use let only when you need to reassign. Never use var in modern code.

🧠 1.4 Closures — Deep Understanding

A closure is when a function "remembers" variables from its outer scope even after the outer function has finished executing. This is one of JavaScript's most powerful — and most misunderstood — features.

Basic Closure 
function makeCounter() {
    let count = 0; // This variable is "closed over"
    
    return function() {
        count++;
        return count;
    };
}

const counter = makeCounter();
console.log(counter()); // 1
console.log(counter()); // 2
console.log(counter()); // 3

// count is private — nobody outside can access it directly!
// console.log(count); // ❌ ReferenceError

Practical Uses of Closures

Real-world Patterns 
// 1. Data Privacy / Encapsulation
function createBankAccount(initialBalance) {
    let balance = initialBalance; // private
    
    return {
        deposit(amount) { balance += amount; },
        withdraw(amount) {
            if (amount > balance) throw new Error("Insufficient funds");
            balance -= amount;
        },
        getBalance() { return balance; }
    };
}

const account = createBankAccount(100);
account.deposit(50);
console.log(account.getBalance()); // 150

// 2. Function Factories
function multiplier(factor) {
    return (number) => number * factor;
}

const double = multiplier(2);
const triple = multiplier(3);
console.log(double(5));  // 10
console.log(triple(5));  // 15

// 3. Memoization (caching results)
function memoize(fn) {
    const cache = new Map();
    return function(...args) {
        const key = JSON.stringify(args);
        if (cache.has(key)) return cache.get(key);
        const result = fn.apply(this, args);
        cache.set(key, result);
        return result;
    };
}

const expensiveCalc = memoize((n) => {
    // Simulate heavy computation
    return n * n;
});

console.log(expensiveCalc(10)); // Computed: 100
console.log(expensiveCalc(10)); // From cache: 100

🔗 1.5 Prototypes & Prototype Chain

JavaScript is a prototype-based language. Every object has an internal link to another object called its prototype. This chain continues until null is reached.

Prototype Chain 
const animal = {
    breathe() { return "breathing..."; }
};

const dog = Object.create(animal); // dog's prototype is animal
dog.bark = function() { return "Woof!"; };

console.log(dog.bark());    // "Woof!" — own property
console.log(dog.breathe()); // "breathing..." — found on prototype
console.log(dog.toString()); // found on Object.prototype

// Check the chain
console.log(Object.getPrototypeOf(dog) === animal); // true
console.log(Object.getPrototypeOf(animal) === Object.prototype); // true
console.log(Object.getPrototypeOf(Object.prototype)); // null

Constructor Functions (Old-style OOP)

Constructor Pattern 
function Person(name, age) {
    this.name = name;
    this.age = age;
}

// Methods on prototype (shared across all instances — memory efficient!)
Person.prototype.greet = function() {
    return `Hi, I'm ${this.name} and I'm ${this.age} years old.`;
};

Person.prototype.birthday = function() {
    this.age++;
};

const alice = new Person("Alice", 30);
const bob = new Person("Bob", 25);

console.log(alice.greet()); // "Hi, I'm Alice..."
alice.birthday();
console.log(alice.age); // 31

// Both share the same greet function in memory
console.log(alice.greet === bob.greet); // true ✅

🏛️ 1.6 Classes — Modern OOP in JavaScript

ES6 Classes are syntactic sugar over prototypes. They make OOP patterns cleaner and more readable.

Full Class Example 
class Animal {
    // Private field (ES2022)
    #energy = 100;
    
    constructor(name, species) {
        this.name = name;
        this.species = species;
    }
    
    // Instance method
    speak() {
        return `${this.name} makes a sound.`;
    }
    
    // Getter
    get info() {
        return `${this.name} (${this.species})`;
    }
    
    // Static method (called on class, not instance)
    static kingdom() {
        return "Animalia";
    }
    
    // Private method
    #recharge() {
        this.#energy = 100;
    }
}

class Dog extends Animal {
    #tricks = [];
    
    constructor(name) {
        super(name, "Canis lupus familiaris"); // Call parent constructor
        this.type = "dog";
    }
    
    // Override parent method
    speak() {
        return `${this.name} barks: Woof!`;
    }
    
    learn(trick) {
        this.#tricks.push(trick);
    }
    
    get knownTricks() {
        return [...this.#tricks]; // Return a copy
    }
}

const rex = new Dog("Rex");
console.log(rex.speak());           // "Rex barks: Woof!"
console.log(rex.info);              // "Rex (Canis lupus familiaris)"
console.log(Animal.kingdom());      // "Animalia"
rex.learn("sit");
rex.learn("shake");
console.log(rex.knownTricks);       // ["sit", "shake"]
console.log(rex instanceof Animal); // true
Private fields (#field) are truly private in JavaScript — they cannot be accessed or modified from outside the class, not even by subclasses.

⚡ Chapter 2 — Asynchronous JavaScript (Complete Guide)

Async is where JavaScript truly shines — and where most bugs hide. This chapter covers every tool from callbacks to async/await, plus the Event Loop internals.

⏳ 2.1 The Event Loop — In Depth

The Event Loop is JavaScript's mechanism for handling async operations while remaining single-threaded. It coordinates three components:

Call Stack

Where synchronous code executes. LIFO (Last In, First Out). When it's empty, the Event Loop checks the queues.

Web APIs / Node APIs

Browsers/Node handle async work here (timers, fetch, file I/O). When done, callbacks are sent to queues.

Microtask Queue

High priority. Contains Promise callbacks (.then, .catch) and queueMicrotask(). Emptied completely before macro tasks.

Task Queue (Macrotasks)

Lower priority. Contains setTimeout, setInterval, I/O callbacks. Processed one at a time per Event Loop cycle.

Event Loop — Execution Order 
console.log("1: Script start");

setTimeout(() => console.log("5: setTimeout"), 0); // Macrotask

Promise.resolve()
    .then(() => console.log("3: Promise .then #1")) // Microtask
    .then(() => console.log("4: Promise .then #2")); // Microtask

queueMicrotask(() => console.log("2.5: queueMicrotask")); // Microtask

console.log("2: Script end");

// Output:
// 1: Script start
// 2: Script end
// 2.5: queueMicrotask   ← Microtask queue (before macrotasks)
// 3: Promise .then #1   ← Microtask queue
// 4: Promise .then #2   ← Microtask queue (chained)
// 5: setTimeout          ← Macrotask queue (after ALL microtasks)
⚠️ Important: The microtask queue must be completely empty before the next macrotask runs. This means a loop that keeps adding microtasks can starve the macrotask queue (and the UI rendering)!

📜 2.2 Callbacks — The Foundation

A callback is a function passed as an argument to another function, to be called later. It's the original async pattern in JavaScript.

Callbacks 
// Simple callback
function fetchUser(id, callback) {
    setTimeout(() => {
        const user = { id, name: "Alice" }; // simulated fetch
        callback(null, user); // (error, data) convention
    }, 1000);
}

fetchUser(1, (err, user) => {
    if (err) return console.error(err);
    console.log(user); // { id: 1, name: "Alice" }
});

// ❌ "Callback Hell" — nested callbacks become unmaintainable
fetchUser(1, (err, user) => {
    fetchPosts(user.id, (err, posts) => {
        fetchComments(posts[0].id, (err, comments) => {
            // Deeply nested, hard to read, error-prone
        });
    });
});
💡 Node.js Error-First Convention: Callbacks follow the pattern callback(error, result). Always check the error first before using the result.

🤝 2.3 Promises — Complete Guide

A Promise is an object representing the eventual completion or failure of an async operation. It's a more structured alternative to callbacks.

Promise States

⏳ Pending

Initial state. The operation is still in progress.

✅ Fulfilled

Operation completed successfully. The .then() handler runs.

❌ Rejected

Operation failed. The .catch() handler runs.

🔒 Settled

Either fulfilled or rejected. The .finally() handler always runs.

Creating and Chaining Promises 
// Creating a Promise
function fetchUser(id) {
    return new Promise((resolve, reject) => {
        setTimeout(() => {
            if (id > 0) {
                resolve({ id, name: "Alice", role: "admin" });
            } else {
                reject(new Error("Invalid user ID"));
            }
        }, 500);
    });
}

// Consuming with .then() / .catch() / .finally()
fetchUser(1)
    .then(user => {
        console.log("Got user:", user.name); // "Alice"
        return user.role; // Pass data to next .then()
    })
    .then(role => {
        console.log("Role:", role); // "admin"
    })
    .catch(error => {
        console.error("Error:", error.message); // Catches ALL errors above
    })
    .finally(() => {
        console.log("Done"); // Always runs
    });

Promise Static Methods

Promise.all, allSettled, race, any 
const p1 = fetch("/api/users");
const p2 = fetch("/api/posts");
const p3 = fetch("/api/comments");

// Promise.all — resolves when ALL succeed, rejects if ANY fail
const [users, posts, comments] = await Promise.all([p1, p2, p3]);

// Promise.allSettled — waits for ALL, never rejects
const results = await Promise.allSettled([p1, p2, p3]);
results.forEach(result => {
    if (result.status === "fulfilled") {
        console.log("Success:", result.value);
    } else {
        console.log("Failed:", result.reason);
    }
});

// Promise.race — resolves/rejects with the FIRST settled promise
const fastest = await Promise.race([p1, p2, p3]);

// Promise.any — resolves with the FIRST fulfilled (ignores rejections)
// Only rejects if ALL are rejected (AggregateError)
const first = await Promise.any([p1, p2, p3]);

✨ 2.4 Async / Await — Modern Async Code

async/await is syntactic sugar over Promises that makes async code look and behave more like synchronous code. It's the preferred way to write async code today.

async/await Basics 
// An async function ALWAYS returns a Promise
async function getUser(id) {
    // await pauses execution until the Promise resolves
    const response = await fetch(`/api/users/${id}`);
    
    if (!response.ok) {
        throw new Error(`HTTP Error: ${response.status}`);
    }
    
    const user = await response.json();
    return user; // Auto-wrapped in Promise.resolve()
}

// Error handling with try/catch
async function loadProfile(id) {
    try {
        const user = await getUser(id);
        const posts = await fetch(`/api/posts?userId=${user.id}`);
        const postsData = await posts.json();
        
        return { user, posts: postsData };
    } catch (error) {
        console.error("Failed to load profile:", error.message);
        return null; // Graceful fallback
    } finally {
        console.log("Profile load attempt complete");
    }
}

Parallel vs Sequential Execution

Avoiding the Sequential Trap 
// ❌ Sequential (slow) — each waits for the previous
async function slowLoad() {
    const user = await fetchUser(1);    // 500ms
    const posts = await fetchPosts(1);  // 500ms
    const tags = await fetchTags(1);    // 500ms
    // Total: ~1500ms
}

// ✅ Parallel (fast) — all start at the same time
async function fastLoad() {
    const [user, posts, tags] = await Promise.all([
        fetchUser(1),
        fetchPosts(1),
        fetchTags(1)
    ]);
    // Total: ~500ms (limited by the slowest)
}

// ✅ Parallel with individual handling
async function flexibleLoad() {
    const userPromise = fetchUser(1);
    const postsPromise = fetchPosts(1);
    
    const user = await userPromise;   // Start both, await separately
    const posts = await postsPromise; // Already running!
}
⚠️ Common Mistake: Using await inside .forEach() does NOT work as expected. Use for...of for sequential async loops, or Promise.all(arr.map(...)) for parallel ones.
Async Loops 
const ids = [1, 2, 3, 4, 5];

// ❌ forEach doesn't await properly
ids.forEach(async (id) => {
    const user = await fetchUser(id); // These run concurrently but forEach doesn't wait
    console.log(user); // Order unpredictable, errors swallowed
});

// ✅ Sequential with for...of
for (const id of ids) {
    const user = await fetchUser(id); // Truly sequential
    console.log(user);
}

// ✅ Parallel with Promise.all + map
const users = await Promise.all(ids.map(id => fetchUser(id)));
console.log(users); // All users, order preserved

🌊 2.5 Generators & Async Generators

Generator functions can pause and resume execution, yielding multiple values over time. They're the foundation of many async patterns.

Generators 
// Generator function (note the *)
function* numberGenerator() {
    yield 1;
    yield 2;
    yield 3;
    return "done"; // Final value
}

const gen = numberGenerator();
console.log(gen.next()); // { value: 1, done: false }
console.log(gen.next()); // { value: 2, done: false }
console.log(gen.next()); // { value: 3, done: false }
console.log(gen.next()); // { value: "done", done: true }

// Generators are iterable!
for (const num of numberGenerator()) {
    console.log(num); // 1, 2, 3 (return value skipped)
}

// Infinite sequence generator
function* fibonacci() {
    let [a, b] = [0, 1];
    while (true) {
        yield a;
        [a, b] = [b, a + b];
    }
}

const fib = fibonacci();
for (let i = 0; i < 8; i++) {
    process.stdout.write(fib.next().value + " "); // 0 1 1 2 3 5 8 13
}

// Async Generator
async function* fetchPages(url) {
    let page = 1;
    while (true) {
        const res = await fetch(`${url}?page=${page}`);
        const data = await res.json();
        if (data.length === 0) return;
        yield data;
        page++;
    }
}

for await (const page of fetchPages("/api/items")) {
    process(page); // Handle each page as it arrives
}

🧩 Chapter 3 — Functional Programming in JavaScript

FP techniques make code more predictable, testable, and maintainable. JavaScript supports first-class functions, making it ideal for FP patterns.

🎯 3.1 Core FP Principles

Pure Functions

Same input always produces same output. No side effects (no modifying external state, no I/O).

Immutability

Don't mutate data. Create new values instead. Prevents subtle bugs from shared mutable state.

Higher-Order Functions

Functions that take other functions as arguments or return functions. Examples: map, filter, reduce.

Function Composition

Combine small functions into larger ones. Output of one becomes input of the next.

Pure Functions vs Impure 
// ❌ Impure — modifies external state
let total = 0;
function addImpure(n) {
    total += n; // Side effect!
    return total;
}

// ✅ Pure — same input = same output, no side effects
function addPure(currentTotal, n) {
    return currentTotal + n;
}

🔧 3.2 Array Methods — The Core Toolkit

Master these and you'll write cleaner, more expressive code. These are the workhorses of modern JavaScript.

map, filter, reduce — Complete Examples 
const products = [
    { name: "Laptop", price: 1200, category: "Electronics", inStock: true },
    { name: "Shirt", price: 30, category: "Clothing", inStock: false },
    { name: "Phone", price: 800, category: "Electronics", inStock: true },
    { name: "Book", price: 20, category: "Education", inStock: true },
    { name: "Headphones", price: 150, category: "Electronics", inStock: true },
];

// .map() — transform each element
const names = products.map(p => p.name);
// ["Laptop", "Shirt", "Phone", "Book", "Headphones"]

const withTax = products.map(p => ({
    ...p,
    priceWithTax: +(p.price * 1.23).toFixed(2)
}));

// .filter() — keep elements matching condition
const inStock = products.filter(p => p.inStock);
const electronics = products.filter(p => p.category === "Electronics");
const affordable = products.filter(p => p.price < 500 && p.inStock);

const totalValue = products.reduce((sum, p) => sum + p.price, 0); // 2200

const byCategory = products.reduce((groups, product) => {

    return groups;
}, {});

// Chain methods for complex queries
const result = products

🔀 3.3 Currying & Partial Application

Currying 
// Currying: transform f(a, b, c) into f(a)(b)(c)
const add = a => b => a + b;
const add5 = add(5);   // Returns a new function
console.log(add5(3));  // 8
console.log(add5(10)); // 15

// Generic curry function
function curry(fn) {
    return function curried(...args) {
        if (args.length >= fn.length) {
            return fn.apply(this, args);
        }
        return function(...args2) {
            return curried.apply(this, args.concat(args2));
        };
    };
}

const multiply = curry((a, b, c) => a * b * c);
console.log(multiply(2)(3)(4)); // 24
console.log(multiply(2, 3)(4)); // 24
console.log(multiply(2)(3, 4)); // 24

// Practical use: reusable validators
const isLongerThan = minLength => str => str.length > minLength;
const isValidEmail = email => email.includes("@") && email.includes(".");

const isValidUsername = isLongerThan(3);
console.log(isValidUsername("Jo"));     // false
console.log(isValidUsername("Alice"));  // true

🧩 3.4 Function Composition

Compose and Pipe 
// compose: right-to-left execution
const compose = (...fns) => x => fns.reduceRight((acc, fn) => fn(acc), x);

// pipe: left-to-right execution (more intuitive)
const pipe = (...fns) => x => fns.reduce((acc, fn) => fn(acc), x);

const trim = str => str.trim();
const toLowerCase = str => str.toLowerCase();
const removeSpaces = str => str.replace(/\s+/g, "-");
const addPrefix = str => `slug-${str}`;

const createSlug = pipe(trim, toLowerCase, removeSpaces, addPrefix);

console.log(createSlug("  Hello World  ")); // "slug-hello-world"

// Point-free style
const users = [
    { name: "Alice", score: 85 },
    { name: "Bob", score: 92 },
    { name: "Charlie", score: 78 }
];

const getScore = user => user.score;
const isHighScore = score => score > 80;
const formatScore = score => `${score}%`;

const getHighScores = pipe(
    arr => arr.map(getScore),
    arr => arr.filter(isHighScore),
    arr => arr.map(formatScore)
);

console.log(getHighScores(users)); // ["85%", "92%"]

🎯 4.2 Events — Deep Dive

Events, Delegation & Custom Events 
// addEventListener — preferred over onclick="..."
const btn = document.querySelector("#myBtn");

function handleClick(event) {
    event.preventDefault();  // Prevent default action (form submit, link)
    event.stopPropagation(); // Stop bubbling up the DOM
    
    console.log(event.target);   // The element that was clicked
    console.log(event.currentTarget); // The element the listener is on

📡 4.3 Fetch API & HTTP — Production Patterns

API Service Layer 
// Production-grade API wrapper
class ApiService {
    #baseUrl;
    #headers;
    
    constructor(baseUrl, token = null) {
        this.#baseUrl = baseUrl;
        this.#headers = {
            "Content-Type": "application/json",
            ...(token && { "Authorization": `Bearer ${token}` })
        };
    }
    
    async #request(method, endpoint, body = null, options = {}) {
        const controller = new AbortController();
        const timeout = setTimeout(() => controller.abort(), 10000); // 10s timeout
        
        try {
            const response = await fetch(`${this.#baseUrl}${endpoint}`, {
                method,
                headers: { ...this.#headers, ...options.headers },
                body: body ? JSON.stringify(body) : null,
                signal: controller.signal
            });
            
            clearTimeout(timeout);
            
            if (!response.ok) {
                const error = await response.json().catch(() => ({}));
                throw new ApiError(response.status, error.message || "Request failed", error);
            }
            
            // Handle 204 No Content
            if (response.status === 204) return null;
            return response.json();
            
        } catch (error) {
            clearTimeout(timeout);
            if (error.name === "AbortError") throw new Error("Request timeout");
            throw error;
        }
    }
    
    get(endpoint, options) { return this.#request("GET", endpoint, null, options); }
    post(endpoint, body, options) { return this.#request("POST", endpoint, body, options); }
    put(endpoint, body, options) { return this.#request("PUT", endpoint, body, options); }
    patch(endpoint, body, options) { return this.#request("PATCH", endpoint, body, options); }
    delete(endpoint, options) { return this.#request("DELETE", endpoint, null, options); }
}

class ApiError extends Error {
    constructor(status, message, data) {
        super(message);
        this.status = status;
        this.data = data;
    }
}

// Usage
const api = new ApiService("https://api.example.com", "my-token");

try {
    const user = await api.get("/users/1");
    const updated = await api.patch("/users/1", { name: "New Name" });
} catch (error) {
    if (error instanceof ApiError && error.status === 401) {
        // Redirect to login
    }
}

const imageObserver = new IntersectionObserver((entries, observer) => {
    entries.forEach(entry => {
        if (!entry.isIntersecting) return;

⚡ 6.1 Performance Optimization

Debounce & Throttle 
// Debounce — run after user STOPS doing something
// Use for: search input, resize, scroll-end
function debounce(fn, delay) {
    let timer;
    return function(...args) {
        clearTimeout(timer);
        timer = setTimeout(() => fn.apply(this, args), delay);
    };
}

const searchHandler = debounce(async (query) => {
    const results = await searchAPI(query);
    renderResults(results);
}, 300); // Wait 300ms after user stops typing

input.addEventListener("input", (e) => searchHandler(e.target.value));

// Throttle — run at MOST once per interval
// Use for: scroll, mousemove, rate-limiting
function throttle(fn, interval) {
    let lastTime = 0;
    return function(...args) {
        const now = Date.now();
        if (now - lastTime >= interval) {
            lastTime = now;
            fn.apply(this, args);
        }
    };
}

const trackScroll = throttle(() => {
    updateProgressBar(window.scrollY);
}, 100); // At most once every 100ms

window.addEventListener("scroll", trackScroll);

🎯 6.2 Design Patterns in JavaScript

Observer Pattern 
// Observer / Event Emitter Pattern
class EventEmitter {
    #listeners = new Map();
    
    on(event, callback) {
        if (!this.#listeners.has(event)) {
            this.#listeners.set(event, new Set());
        }
        this.#listeners.get(event).add(callback);
        return () => this.off(event, callback); // Returns unsubscribe fn
    }
    
    off(event, callback) {
        this.#listeners.get(event)?.delete(callback);
    }
    
    emit(event, data) {
        this.#listeners.get(event)?.forEach(cb => cb(data));
    }
    
    once(event, callback) {
        const wrapper = (data) => {
            callback(data);
            this.off(event, wrapper);
        };
        this.on(event, wrapper);
    }
}

// Usage

    updateUserUI(user);
});

🔒 6.3 Security Best Practices

  • XSS (Cross-Site Scripting): Never use innerHTML with user input. Use textContent or DOM methods. Sanitize with DOMPurify if HTML is needed.
  • CSRF: Use SameSite cookie attribute and CSRF tokens for state-changing requests.
  • Content Security Policy: Set CSP headers to control which scripts can run.
  • Sensitive Data: Never store tokens in localStorage (vulnerable to XSS). Use HttpOnly cookies or sessionStorage with care.
  • Dependencies: Regularly audit with npm audit. Use Snyk or Dependabot.
Safe DOM Manipulation 
// ❌ DANGEROUS — executes any script in the string
element.innerHTML = userInput; // XSS vulnerability!

// ✅ SAFE — treats content as text only
element.textContent = userInput;

// ✅ SAFE — building DOM programmatically
const p = document.createElement("p");
p.textContent = userInput; // Automatically escaped
container.appendChild(p);

// ✅ If you MUST insert HTML — sanitize first
import DOMPurify from "dompurify";
element.innerHTML = DOMPurify.sanitize(userInput);
