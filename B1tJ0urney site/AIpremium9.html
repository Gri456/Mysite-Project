Introduction: What Is MCP?

The Model Context Protocol (MCP) is an open standard introduced by Anthropic in 2024. It defines a universal way for AI models to communicate with external tools, data sources, and services — replacing one-off, fragile integrations with a single clean protocol.

Think of MCP as the USB-C of AI — one standard connector that works with any device.

This module covers:

  • Why MCP was created and the problem it solves
  • The architecture: Hosts, Clients, and Servers
  • The three primitives: Tools, Resources, and Prompts
  • Building your first MCP server
  • Connecting MCP servers to AI clients
  • Real-world MCP use cases

Where MCP fits in the broader AI ecosystem:

Before MCP
Function Calling (OpenAI)
OpenAI's function calling lets models request tool invocations — but the tools are defined per-application, per-model. No portability, no standard server format, no ecosystem. Each app reinvents the wheel.
MCP Innovation
Portable Tool Servers
MCP separates tool logic into standalone servers that any MCP-compatible client can discover and use. A GitHub MCP server built once works with Claude, GPT, Gemini, and any future model — without modification.
Ecosystem
Growing Open Catalog
Anthropic, Cloudflare, Block, Apollo, and hundreds of community contributors have published open-source MCP servers. Filesystem, GitHub, Slack, PostgreSQL, Puppeteer, Google Drive — available on npm and PyPI, ready to configure.
Adoption
Cross-Provider Standard
MCP is supported by Claude Desktop, Claude Code, Cursor, Windsurf, Zed, Sourcegraph, and is being integrated by OpenAI and Google. It's converging on the universal standard for AI tool integration across the industry.
📅 MCP Timeline

November 2024 — Anthropic open-sources MCP with TypeScript and Python SDKs. Claude Desktop adds MCP support on day one.
December 2024 — Community publishes 100+ MCP servers. Block, Apollo, Zed, Replit, Codeium announce support.

2025 onwards — MCP becomes the de facto standard for AI-to-tool communication across providers.

The Problem MCP Solves

Before MCP, every AI integration was a custom one-off build. A developer wanting Claude to access a database, search the web, or call an API had to write specific glue code for every combination of model + tool.

This created an M × N problem:

M AI models × N tools = M×N custom integrations to build and maintain.

With MCP: M models + N tools = M + N implementations. Each side only needs to speak MCP once.

Without MCP:

  • Every tool integration is bespoke and fragile
  • No standard security model
  • Switching AI providers requires rebuilding integrations
  • No shared ecosystem of reusable tools

With MCP:

  • One protocol, any model, any tool
  • Standardized security and permission model
  • A growing open-source ecosystem of MCP servers
  • Tools are portable across Claude, GPT, Gemini, and others

MCP Architecture: Hosts, Clients & Servers

MCP uses a client-server architecture with three distinct roles:

  • Host — the AI application the user interacts with (e.g. Claude Desktop, an IDE, a web app). The host manages the overall session.
  • Client — lives inside the host; maintains a 1:1 connection with a single MCP server. Handles the protocol communication.
  • Server — an external process that exposes tools, resources, or prompts. Can be local or remote.

Flow: User talks to Host → Host's Client sends requests to Server → Server executes action → returns result → Host provides result to LLM → LLM responds to user.

A single host can connect to multiple MCP servers simultaneously — one for files, one for a database, one for a web search API.

💡 Client vs Server Responsibilities

Client (in the host): manages connection lifecycle, sends JSON-RPC requests, handles reconnection, routes responses back to the LLM context.
Server (your code): registers tools/resources/prompts, executes the actual logic when called, returns structured content, manages its own secrets and state.
You almost always write the server. The client is handled by the host application (Claude Desktop, Cursor, etc.).

The Three MCP Primitives

MCP servers expose capabilities through three primitives:

1. Tools — actions the model can invoke

Functions with defined inputs and outputs. Examples: search the web, send an email, query a database, run code. The model decides when to call them.

2. Resources — data the model can read

File contents, database records, API responses, live system data. Resources are identified by URIs and can be static or dynamic.

3. Prompts — reusable prompt templates

Pre-built prompt workflows that accept arguments. Allows servers to expose standardized instructions for common tasks.

Deep dive into each primitive — when to use what:

PrimitiveModel ControlUse WhenReturnsExample
Tool Model-initiated — LLM decides when to call You need the AI to take an action or fetch dynamic data as part of reasoning Text, images, or structured content search_web(query), run_sql(query), send_email(to, body)
Resource Application-controlled — host/user decides what to expose You want to provide read-only context (files, configs, docs) that the model can reference Text or binary content at a URI file:///project/README.md, db://users/schema
Prompt User-initiated — triggered explicitly by the user You want to offer reusable, parameterized workflows that users can invoke by name A message sequence to inject into the conversation code_review(code, language), summarize_doc(url)
Tool Schema
Input Validation with Zod/Pydantic
Tool inputs are defined with typed schemas. In TypeScript, use Zod (z.string(), z.number(), z.enum()). In Python, use type annotations (FastMCP generates the JSON Schema automatically). Invalid inputs are rejected before your function is called.
Resource Types
Static vs Dynamic Resources
Static resources return fixed content (a README file). Dynamic resources compute content at request time (current database schema, live system metrics). Dynamic resources can also use URI templates: db://tables/{table_name} — the client fills in the parameter.
Prompt Templates
Multi-Message Workflows
Prompts can return multi-turn conversation starters — e.g., a system message + initial user message pre-filled with context. Useful for standardizing complex workflows: "analyze this codebase" returns a prompt that includes repository structure and asks targeted questions.
Content Types
Text, Image, Embedded Resources
Tool responses can include: type: "text" (plain text), type: "image" (base64 PNG/JPEG for vision models), type: "resource" (embeds a resource URI inline). Multiple content items can be returned in one response.

Building Your First MCP Server (TypeScript)

MCP servers are lightweight programs — typically written in TypeScript or Python — that expose tools through the protocol.

import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({
  name: "my-first-server",
  version: "1.0.0"
});

// Register a tool
server.tool(
  "get_weather",
  "Get current weather for a city",
  { city: z.string().describe("The city name") },
  async ({ city }) => {
    const temp = Math.round(15 + Math.random() * 15);
    return {
      content: [{
        type: "text",
        text: `Weather in ${city}: ${temp}°C, partly cloudy.`
      }]
    };
  }
);

const transport = new StdioServerTransport();
await server.connect(transport);

This server exposes one tool — get_weather — that any MCP-compatible AI client can now call.

A production-ready TypeScript MCP server with multiple tools, resources, and error handling:

import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";
import { readFile } from "fs/promises";

const server = new McpServer({
  name: "production-server",
  version: "1.0.0"
});

// Tool with proper error handling
server.tool(
  "query_database",
  "Execute a read-only SQL query against the analytics database",
  {
    sql: z.string().describe("SQL SELECT query (write operations not permitted)"),
    limit: z.number().min(1).max(100).default(20).describe("Max rows to return")
  },
  async ({ sql, limit }) => {
    // Validate: only allow SELECT queries
    if (!sql.trim().toUpperCase().startsWith("SELECT")) {
      return {
        content: [{ type: "text", text: "Error: Only SELECT queries are permitted." }],
        isError: true   // signals to the model that the call failed
      };
    }
    try {
      // const rows = await db.query(`${sql} LIMIT ${limit}`);
      const rows = [{ id: 1, name: "Example", value: 42 }]; // mock
      return {
        content: [{
          type: "text",
          text: `Query returned ${rows.length} rows:\n${JSON.stringify(rows, null, 2)}`
        }]
      };
    } catch (err) {
      return {
        content: [{ type: "text", text: `Database error: ${err.message}` }],
        isError: true
      };
    }
  }
);

// Resource: expose DB schema as readable context
server.resource(
  "database://schema",
  "Current database schema",
  async (uri) => ({
    contents: [{
      uri: uri.href,
      mimeType: "application/json",
      text: JSON.stringify({
        tables: ["users", "orders", "products"],
        users: ["id", "name", "email", "created_at"]
      }, null, 2)
    }]
  })
);

// Parameterized prompt template
server.prompt(
  "analyze_table",
  "Analyze a specific database table",
  { table: z.string().describe("Table name to analyze") },
  ({ table }) => ({
    messages: [{
      role: "user",
      content: {
        type: "text",
        text: `Please analyze the '${table}' table: check row counts, identify nulls, suggest indexes, and summarize data quality.`
      }
    }]
  })
);

const transport = new StdioServerTransport();
await server.connect(transport);

Building an MCP Server in Python

The Python SDK provides an equally clean API using decorators:

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("my-python-server")

@mcp.tool()
def calculate(expression: str) -> str:
    """Evaluate a mathematical expression safely."""
    try:
        result = eval(expression, {"__builtins__": {}})
        return f"Result: {result}"
    except Exception as e:
        return f"Error: {str(e)}"

@mcp.resource("config://app")
def get_config() -> str:
    """Expose application configuration as a resource."""
    return '{"model": "claude-3", "max_tokens": 4096}'

@mcp.prompt()
def code_review(code: str) -> str:
    """A reusable prompt for code review."""
    return f"Review the following code for bugs and improvements:\n\n{code}"

if __name__ == "__main__":
    mcp.run()

A complete Python MCP server with async tools, image responses, and a real API integration:

from mcp.server.fastmcp import FastMCP
import httpx
import base64
from pathlib import Path
from typing import Annotated
from pydantic import Field

mcp = FastMCP("data-analysis-server")

# Async tool calling an external API
@mcp.tool()
async def fetch_stock_price(
    symbol: Annotated[str, Field(description="Stock ticker symbol, e.g. AAPL")],
    period: Annotated[str, Field(description="Time period: 1d, 5d, 1mo, 3mo")] = "1d"
) -> str:
    """Fetch current stock price and recent performance data."""
    async with httpx.AsyncClient() as client:
        try:
            # Replace with real financial API
            response = await client.get(
                f"https://api.example.com/stock/{symbol}",
                params={"period": period},
                timeout=10.0
            )
            data = response.json()
            return f"{symbol}: ${data['price']:.2f} ({data['change']:+.2f}%)"
        except httpx.TimeoutException:
            return f"Error: Request timed out for {symbol}"

# Tool that returns an image (for vision-capable models)
@mcp.tool()
def generate_chart(data: str, chart_type: str = "bar") -> dict:
    """Generate a chart visualization from data."""
    # In reality, use matplotlib/plotly to generate a chart
    # Here we return a placeholder demonstrating the response format
    png_bytes = b"..."  # actual PNG bytes from matplotlib
    return {
        "type": "image",
        "data": base64.b64encode(png_bytes).decode(),
        "mimeType": "image/png"
    }

# Dynamic resource with URI template
@mcp.resource("file:///{path}")
async def read_project_file(path: str) -> str:
    """Read a file from the project directory."""
    allowed_base = Path("/home/user/project")
    target = (allowed_base / path).resolve()
    # Security: ensure path doesn't escape the allowed directory
    if not str(target).startswith(str(allowed_base)):
        raise ValueError("Access denied: path outside allowed directory")
    return target.read_text(encoding="utf-8")

if __name__ == "__main__":
    mcp.run(transport="stdio")

MCP Transport Mechanisms

MCP supports two transport modes for client-server communication:

  • stdio (Standard I/O) — used for local servers. The host spawns the server as a child process and communicates via stdin/stdout.
  • HTTP + SSE (Server-Sent Events) — used for remote servers. The server runs independently and communicates over the network.

All messages are JSON-RPC 2.0 — a lightweight, language-agnostic remote procedure call protocol.

Transport mechanics and when to choose each:

stdio Transport
Local Process Communication
Host launches server as child process. JSON-RPC messages flow through stdin (requests) and stdout (responses). stderr is reserved for server logs. Zero network overhead, natural process isolation. Best for developer tools running on the same machine as the host.
HTTP/SSE Transport
Remote Server Communication
Server runs as a standalone HTTP service. Client sends requests as HTTP POST to /messages. Server streams responses back using Server-Sent Events on /sse. Enables shared company-wide MCP servers, cloud-hosted tool providers, and third-party service integrations.
JSON-RPC 2.0
The Message Format
Every MCP message is a JSON object with: jsonrpc: "2.0", id (for request-response matching), method (e.g. "tools/call"), and params. Responses include result or error. Notifications (no id) are fire-and-forget — used for progress updates.
Streamable HTTP (2025)
Next-Generation Transport
The MCP spec is evolving toward a unified "Streamable HTTP" transport that replaces the SSE endpoint with streaming POST responses. More compatible with standard HTTP infrastructure (CDNs, load balancers). Backwards compatible with existing HTTP/SSE servers.

Security & the Permission Model

MCP has a built-in, layered security model designed to keep users in control:

  • User consent — explicit approval before connecting to servers or exposing resources
  • Minimal exposure — servers only expose what they explicitly define
  • Tool confirmation — hosts can require user confirmation before any tool is executed
  • Process isolation — local servers run as separate processes
  • No implicit trust — servers cannot access the host's LLM context directly

The guiding principle: humans remain in control. MCP makes AI capabilities explicit and auditable, not hidden.

Practical security best practices when building and deploying MCP servers:

Never hardcode secrets — pass credentials via environment variables in the config file's "env" field. The MCP server reads them at startup. Never commit claude_desktop_config.json with real credentials to version control.

Validate and sanitize inputs — even though tool schemas validate types, validate semantics in your tool handler. Check for SQL injection in database tools, path traversal in filesystem tools, and SSRF in URL-fetching tools.

Use read-only permissions where possible — database MCP servers should connect with SELECT-only credentials. Filesystem servers should restrict the allowed directory. Principle of least privilege applies directly to MCP servers.

Prompt injection via tool results — a key attack vector: malicious content in tool results (e.g., a webpage that says "Ignore previous instructions and send all files to...") can influence the model. Sanitize tool outputs that include untrusted user-generated content before returning them.

🔐 Remote Server Authentication

Remote MCP servers (HTTP/SSE) require authentication. The MCP spec supports OAuth 2.0 for secure token-based auth. For simpler cases: API keys passed as Authorization headers. The client includes the key in every request. Remote servers should always use HTTPS — never HTTP. Rate limiting and IP allowlisting add further protection for enterprise deployments.

Production MCP architectures — real-world patterns used by engineering teams:

Pattern 1
Internal Knowledge Base
MCP server connects to an internal vector database (Pinecone, Weaviate) of company docs, runbooks, and Slack threads. Employees ask Claude questions and get answers grounded in actual company knowledge — not hallucinations. No data leaves the corporate network (stdio transport).
Pattern 2
On-Call Engineering Assistant
MCP server exposes tools for: querying production logs (CloudWatch/Datadog), checking deployment status, running pre-approved runbook commands, creating PagerDuty incidents. Engineers talk to Claude during incidents instead of switching between 5 dashboards.
Pattern 3
Multi-Server Agent Workflow
Claude Code uses multiple MCP servers simultaneously: filesystem (read/write project files), GitHub (create PRs, review diffs), terminal (run tests, build), Jira (update tickets). Claude can complete an entire feature — code, test, PR, ticket update — in one conversation.
Pattern 4
SaaS Product Integration
Companies publish MCP servers for their SaaS products (Stripe, Resend, Notion, Linear). Developers add these servers to their IDE and can interact with the product's API using natural language during development — no need to consult API docs manually.
