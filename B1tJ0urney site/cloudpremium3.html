⚙️ 3.1 Why Container Orchestration?

Containers solve the "it works on my machine" problem. But in production you need to run hundreds of containers reliably, at scale, with automatic recovery from failures.

Problems orchestration solves:
• Scheduling — which node should run this container?
• Self-healing — restart failed containers automatically.
• Scaling — add/remove instances based on load.
• Rolling updates — deploy new versions with zero downtime.
• Service discovery — how do services find each other?

☸️ 3.2 Kubernetes Architecture

Kubernetes (K8s) is an open-source container orchestration platform, originally developed by Google. It manages containerised workloads across a cluster of machines.

┌─────────────────────────────────────────────────────┐ │ CONTROL PLANE │ │ ┌──────────┐ ┌──────────┐ ┌────────────────────┐ │ │ │ API │ │etcd │ │ Scheduler + │ │ │ │ Server │ │(state) │ │ Controller Manager │ │ │ └──────────┘ └──────────┘ └────────────────────┘ │ └─────────────────────────────────────────────────────┘ │ │ │ ┌───────────▼──┐ ┌────────▼──┐ ┌───────▼──────────┐ │ Worker 1 │ │ Worker 2 │ │ Worker 3 │ │ ┌─────────┐ │ │ ┌───────┐ │ │ ┌─────────────┐ │ │ │ Pod A │ │ │ │ Pod B │ │ │ │ Pod C Pod D │ │ │ │ Pod E │ │ │ │ Pod F │ │ │ │ │ │ │ └─────────┘ │ │ └───────┘ │ │ └─────────────┘ │ │ kubelet │ │ kubelet │ │ kubelet │ └──────────────┘ └───────────┘ └───────────────────┘
  • API Server: Entry point — all kubectl commands hit the API server.
  • etcd: Distributed key-value store holding the entire cluster state.
  • Scheduler: Assigns pods to nodes based on resource availability.
  • Controller Manager: Ensures desired state matches actual state.
  • kubelet: Agent on each worker node — runs and monitors pods.
  • kube-proxy: Maintains network rules for pod communication.

📦 3.3 Core Kubernetes Resources

  • Pod: One or more containers sharing a network namespace and storage. The smallest deployable unit.
  • Deployment: Manages a ReplicaSet to maintain the desired number of pod replicas. Handles rolling updates and rollbacks.
  • Service: Stable DNS name and virtual IP for a set of pods. Types: ClusterIP, NodePort, LoadBalancer.
  • Ingress: HTTP/HTTPS routing rules — exposes services externally with path and host-based routing.
  • ConfigMap: Inject non-sensitive configuration into pods as env vars or files.
  • Secret: Like ConfigMap but base64-encoded — for passwords, tokens, certificates.
  • PersistentVolume (PV) / PVC: Decouple storage lifecycle from pod lifecycle.
  • Namespace: Logical cluster partitioning — isolate environments (dev, staging, prod).
  • DaemonSet: Run one pod per node — useful for log agents, monitoring agents.
  • StatefulSet: For stateful apps — stable network IDs and persistent storage per pod.
# Deployment manifest
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
  namespace: production
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api-server
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:
    metadata:
      labels:
        app: api-server
    spec:
      containers:
      - name: api
        image: myregistry/api:v2.1.0
        ports:
        - containerPort: 8080
        resources:
          requests:
            cpu: "250m"
            memory: "256Mi"
          limits:
            cpu: "500m"
            memory: "512Mi"
        readinessProbe:
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 5

📈 3.5 Autoscaling in Kubernetes

  • HPA (Horizontal Pod Autoscaler): Scales pod count based on CPU, memory, or custom metrics.
  • VPA (Vertical Pod Autoscaler): Adjusts CPU/memory requests automatically.
  • KEDA (Kubernetes Event-Driven Autoscaling): Scale based on queue depth, HTTP requests, cron — any external metric.
  • Cluster Autoscaler: Adds/removes worker nodes based on pending pods.
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-server-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80

⛵ 3.6 Helm — The Kubernetes Package Manager

Helm packages Kubernetes manifests into reusable, versioned charts. Think of it as apt or npm for Kubernetes.

# Add a chart repository
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update

# Search and inspect charts
helm search repo nginx
helm show values bitnami/nginx > nginx-values.yaml

# Install a release
helm install my-nginx bitnami/nginx \
  --namespace production \
  --create-namespace \
  --set replicaCount=3 \
  --set service.type=LoadBalancer \
  --values custom-values.yaml

# Upgrade a release
helm upgrade my-nginx bitnami/nginx \
  --values custom-values.yaml \
  --atomic \
  --timeout 5m

# Rollback to a previous revision
helm rollback my-nginx 2

# List and remove
helm list -A
helm uninstall my-nginx -n production
Helm chart structure:
my-chart/
├── Chart.yaml # metadata (name, version, description)
├── values.yaml # default configuration
├── templates/ # Kubernetes manifests with Go templating
│ ├── deployment.yaml
│ ├── service.yaml
│ └── _helpers.tpl
└── charts/ # dependencies (sub-charts)

🔄 3.7 ArgoCD — GitOps Continuous Delivery

ArgoCD implements GitOps for Kubernetes — it continuously watches a Git repository and automatically reconciles the cluster state to match the desired state defined in Git.

  • Application CRD: Defines the source (Git repo + path) and destination (cluster + namespace).
  • Sync policies: Manual (you approve) or automated (auto-sync on every commit).
  • Health checks: ArgoCD understands Kubernetes resource health natively.
  • RBAC: Teams only see and manage their own applications.
  • App of Apps pattern: One root ArgoCD application manages other applications — bootstrap an entire cluster from a single repo.
  • Multi-cluster: Manage multiple clusters from one ArgoCD installation.
# ArgoCD Application manifest
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-api
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/k8s-manifests
    targetRevision: main
    path: apps/my-api
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
    automated:
      prune: true       # delete removed resources
      selfHeal: true    # revert manual changes
    syncOptions:
    - CreateNamespace=true

🕸️ 3.8 What is a Service Mesh and Why?

As you decompose applications into microservices, service-to-service communication becomes a major concern. A service mesh addresses this at the infrastructure level — without changing your application code.

What a service mesh gives you for free:
• mTLS encryption — every service-to-service call is encrypted and authenticated.
• Traffic management — canary deployments, A/B testing, retries, timeouts.
• Observability — automatic metrics, traces, and logs for all service calls.
• Circuit breaking — stop cascading failures when a downstream service degrades.
• Rate limiting — protect services from overload.
Without mesh: [Service A] ──HTTP──▶ [Service B] no auth, no encryption, no retry With mesh: [Service A]──[Proxy]──mTLS──[Proxy]──[Service B] automatic encryption, retries, metrics

🔷 3.9 Istio — Full-Featured Service Mesh

Istio is the most feature-rich service mesh. It uses the Envoy proxy as a sidecar injected into every pod to intercept all network traffic.

  • Control Plane (Istiod): Manages proxy configuration, certificate issuance, and service discovery.
  • Data Plane: Envoy sidecars in every pod — handle all inbound and outbound traffic.
  • VirtualService: Define routing rules — canary, header-based, weighted splits.
  • DestinationRule: Define policies for a destination — load balancing, circuit breaking, TLS settings.
  • Gateway: Manage ingress and egress traffic at the mesh boundary.
  • PeerAuthentication: Enforce mTLS across the mesh.
# Canary release: 90% v1, 10% v2
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: checkout
spec:
  hosts:
  - checkout
  http:
  - route:
    - destination:
        host: checkout
        subset: v1
      weight: 90
    - destination:
        host: checkout
        subset: v2
      weight: 10
---
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: checkout
spec:
  host: checkout
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 10s
      baseEjectionTime: 30s
  subsets:
  - name: v1
    labels: { version: v1 }
  - name: v2
    labels: { version: v2 }

🔗 3.10 Linkerd — Lightweight Service Mesh

Linkerd is a CNCF-graduated project focused on simplicity and performance. Unlike Istio, it uses purpose-built micro-proxies written in Rust instead of Envoy.

  • Ultralight proxies: Linkerd2-proxy is tiny (~10MB) and has minimal CPU overhead vs Envoy.
  • Zero-config mTLS: Automatic mutual TLS with no configuration needed.
  • Golden metrics: Success rate, request volume, and latency for every service — automatic.
  • Traffic splitting: Weighted routing using the SMI (Service Mesh Interface) spec.
  • When to choose Linkerd: When you need a service mesh but Istio's complexity is too high — ideal for smaller teams or simpler requirements.
# Install Linkerd
linkerd install --crds | kubectl apply -f -
linkerd install | kubectl apply -f -
linkerd check

# Inject Linkerd into a namespace
kubectl annotate namespace production \
  linkerd.io/inject=enabled

# View traffic stats
linkerd viz stat deployments -n production
linkerd viz top deployment/api-server -n production

🔀 3.11 Advanced Traffic Management with Istio

# Header-based routing (user segmentation)
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: frontend
spec:
  hosts:
  - frontend
  http:
  - match:
    - headers:
        x-user-segment:
          exact: "beta"
    route:
    - destination:
        host: frontend
        subset: v2
  - route:
    - destination:
        host: frontend
        subset: v1
---
# Circuit breaker via DestinationRule
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: payment-service
spec:
  host: payment-service
  trafficPolicy:
    outlierDetection:
      consecutive5xxErrors: 3
      interval: 5s
      baseEjectionTime: 60s
      maxEjectionPercent: 50

🔐 3.12 Security in Kubernetes

  • RBAC (Role-Based Access Control): Control who can do what in the cluster. Use ServiceAccounts for workloads.
  • NetworkPolicies: Restrict pod-to-pod communication — default deny, explicit allow.
  • Pod Security Standards: Enforce security contexts at namespace level (privileged, baseline, restricted).
  • OPA/Gatekeeper: Policy engine — enforce custom rules (e.g., all containers must have resource limits).
  • Image scanning: Trivy, Snyk — scan container images in CI before deployment.
# NetworkPolicy — default deny all ingress
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Ingress
---
# Allow only frontend to reach api-server
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-api
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api-server
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend

🏗️ 3.13 Deployment Strategies

  • Rolling Update (default): Gradually replace old pods with new ones. Zero downtime. Built into Deployments.
  • Blue/Green: Two identical environments. Switch traffic all at once. Instant rollback.
  • Canary: Route small % of traffic to new version. Validate before full rollout. Requires service mesh or Argo Rollouts.
  • Argo Rollouts: Kubernetes controller for progressive delivery — canary, blue/green with automated analysis.
# Argo Rollouts — Canary with analysis
apiVersion: argoproj.io/v1alpha1
kind: Rollout
spec:
  strategy:
    canary:
      steps:
      - setWeight: 10     # 10% traffic to new version
      - pause: {duration: 5m}
      - setWeight: 30
      - pause: {duration: 10m}
      - setWeight: 100
      analysis:
        templates:
        - templateName: error-rate-check

📊 3.14 Observability in the Mesh

One of the biggest benefits of a service mesh is automatic observability — without any code changes.

  • Metrics: Istio exposes Prometheus metrics for every service: request rate, error rate, p50/p99 latency.
  • Distributed Tracing: Envoy adds trace headers (Jaeger, Zipkin, Tempo) automatically.
  • Kiali: Istio's service graph dashboard — visualise traffic flow, health, and configuration.
  • Grafana dashboards: Pre-built Istio dashboards for service-level metrics.
