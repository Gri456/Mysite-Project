🏗 1.1 What is Infrastructure as Code?

Infrastructure as Code (IaC) is the practice of managing and provisioning computing infrastructure through machine-readable configuration files, rather than manual processes or interactive configuration tools.

Core IaC principles:
• Declarative vs Imperative — describe what you want (declarative) vs how to get there (imperative).
• Idempotency — applying the same config multiple times produces the same result.
• Version Control — infrastructure changes tracked in Git, enabling rollback and full audit trail.
• Immutability — replace infrastructure instead of patching it in place.
• Automation — eliminate manual, error-prone configuration steps entirely.

⚙ 1.2 IaC Tool Categories

  • Provisioning tools: Create and manage cloud resources (VMs, networks, databases). Examples: Terraform, Pulumi, AWS CloudFormation, Azure Bicep.
  • Configuration management: Install and configure software on existing servers. Examples: Ansible, Chef, Puppet, SaltStack.
  • Container orchestration: Manage containerised workloads declaratively. Examples: Kubernetes manifests, Helm charts, Kustomize.
  • Policy as Code: Enforce governance rules on infrastructure. Examples: OPA (Open Policy Agent), Sentinel, Checkov.
Most teams combine tools: Terraform to provision cloud resources, Ansible to configure instances, and Kubernetes manifests for application workloads.

🌎 1.3 Terraform: Core Concepts

Terraform by HashiCorp is the most widely adopted IaC tool. It uses HCL (HashiCorp Configuration Language) to describe infrastructure declaratively.

# main.tf — provision an AWS EC2 instance
provider "aws" {
  region = "eu-west-1"
}

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"

  tags = {
    Name        = "web-server"
    Environment = "production"
  }
}
  • Provider: Plugin that connects Terraform to a platform (AWS, Azure, GCP, Kubernetes).
  • Resource: A managed infrastructure object (VM, bucket, database).
  • State file: Terraform tracks real-world resource state in terraform.tfstate.
  • Plan / Apply: terraform plan previews changes; terraform apply executes them.

📂 1.4 Terraform State Management

Terraform state is the source of truth for your infrastructure. Improper state management is one of the most common IaC failure points.

# backend.tf — remote state in S3 with DynamoDB locking
terraform {
  backend "s3" {
    bucket         = "my-tf-state-bucket"
    key            = "prod/terraform.tfstate"
    region         = "eu-west-1"
    dynamodb_table = "terraform-state-lock"
    encrypt        = true
  }
}
  • Local state: Stored on disk — fine for solo development, dangerous in teams.
  • Remote state: S3 + DynamoDB (AWS), Azure Storage, Terraform Cloud — enables collaboration and locking.
  • State locking: Prevents concurrent runs from corrupting state.
  • State commands: terraform state list, terraform state mv, terraform import.

📦 1.5 Terraform Modules

Modules are reusable, self-contained packages of Terraform configurations. They are the primary mechanism for code reuse and abstraction.

# Calling a reusable VPC module from the registry
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.1.0"

  name = "production-vpc"
  cidr = "10.0.0.0/16"

  azs             = ["eu-west-1a", "eu-west-1b", "eu-west-1c"]
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24", "10.0.103.0/24"]

  enable_nat_gateway = true
}
Best practice: Use the Terraform Registry for well-tested community modules. Build internal modules to enforce organisational standards across teams.

🔄 1.6 Pulumi: IaC with Real Programming Languages

Pulumi lets you write infrastructure using TypeScript, Python, Go, or C# — bringing the full power of general-purpose languages to IaC.

# Python Pulumi — provision an S3 bucket
import pulumi
import pulumi_aws as aws

bucket = aws.s3.Bucket("my-bucket",
    acl="private",
    versioning=aws.s3.BucketVersioningArgs(enabled=True),
    tags={"Environment": "production"}
)

pulumi.export("bucket_name", bucket.id)
  • Advantage over Terraform: Real loops, conditionals, and functions — no DSL limitations.
  • Pulumi Cloud: Managed state backend, secrets management, and audit logs.
  • Same concepts: Stack (= Terraform workspace), Project (= root module).

🤖 1.7 Ansible: Configuration Management

Ansible uses YAML playbooks to automate configuration, deployment, and orchestration. It is agentless — connecting via SSH.

# playbook.yml — install and configure Nginx
---
- name: Configure web servers
  hosts: webservers
  become: yes
  tasks:
    - name: Install Nginx
      ansible.builtin.package:
        name: nginx
        state: present
    - name: Ensure Nginx is running
      ansible.builtin.service:
        name: nginx
        state: started
        enabled: yes
    - name: Deploy config
      ansible.builtin.template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf
      notify: Restart Nginx
  handlers:
    - name: Restart Nginx
      ansible.builtin.service:
        name: nginx
        state: restarted
  • Inventory: Defines the hosts Ansible manages (static file or dynamic cloud inventory).
  • Roles: Reusable, structured units of Ansible content.
  • Idempotency: Tasks check current state before making changes.

🚀 1.8 GitOps and IaC Pipelines

GitOps applies Git as the single source of truth for infrastructure. Changes are made via pull requests, and automation applies them to the target environment.

GitOps principles:
• The desired state of the system is declared in Git.
• An automated agent continuously reconciles actual state with desired state.
• All changes go through Git — no manual interventions in production. 
# .github/workflows/terraform.yml
name: Terraform CI
on:
  pull_request:
    branches: [main]
jobs:
  terraform:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: hashicorp/setup-terraform@v3
      - run: terraform init
      - run: terraform fmt -check
      - run: terraform validate
      - run: terraform plan
  • Atlantis: Runs Terraform plan/apply triggered by pull request comments.
  • Flux / ArgoCD: Kubernetes GitOps operators that sync cluster state from Git.
  • GitHub Actions / GitLab CI: Run IaC pipelines on push and merge events.

🔒 1.9 Secrets and Security in IaC

Never store secrets in IaC code or state files. Use dedicated secrets management solutions.

  • HashiCorp Vault: Centralised secrets management with dynamic short-lived credentials.
  • AWS Secrets Manager / Parameter Store: Native AWS secret storage integrated with IAM.
  • SOPS: Encrypt secrets files committed to Git — works with Terraform and Ansible.
  • Environment variables: Pass secrets at runtime; never hardcode in configuration files.
  • State encryption: Always encrypt remote state — it can contain sensitive resource attributes in plaintext.
Rule: If it is a secret, it must not appear in plaintext in your repository or state file. Ever.

✅ 1.10 IaC Testing and Validation

  • terraform validate: Checks syntax and internal consistency of configurations.
  • terraform plan: Dry-run showing exactly what would change before applying.
  • tflint: Linter for Terraform code and provider-specific rule enforcement.
  • Checkov / tfsec: Static analysis scanning for security misconfigurations.
  • Terratest: Go library for writing automated tests against real provisioned infrastructure.
  • OPA / Sentinel: Policy-as-code to enforce compliance rules before any apply runs.

🏢 1.11 IaC at Scale: Workspaces and Environments

Managing multiple environments (dev, staging, production) requires a disciplined and consistent structure.

# Directory-based structure (preferred for large teams)
infrastructure/
├── modules/
│   ├── vpc/
│   ├── eks/
│   └── rds/
└── environments/
    ├── dev/
    ├── staging/
    └── production/
  • Workspaces: Isolated state within one config — useful for feature branches.
  • Directory-based: Separate folder per environment — more explicit, easier to audit.
  • Terragrunt: DRY wrapper for Terraform that reduces repetition across environments and modules.

☁ 1.12 Cloud-Native IaC Tools

  • AWS CloudFormation: Native AWS IaC in JSON/YAML. Tight integration but verbose and AWS-only.
  • AWS CDK: Define CloudFormation stacks using TypeScript, Python, or Java with higher abstraction.
  • Azure Bicep: Simplified DSL that compiles to ARM templates. Azure-native with excellent IDE support.
  • Google Deployment Manager: GCP-native IaC using YAML or Python templates.
Terraform and Pulumi are cloud-agnostic and preferred for multi-cloud or portable teams. Native tools offer deeper integration but create vendor dependency.
