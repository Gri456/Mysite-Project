Why Testing & Publishing Matter

Building an app is only half the work — the other half is making sure it:

  • Works correctly across devices and OS versions
  • Has no crashes or critical bugs
  • Performs well under real usage
  • Passes store review validation
  • Provides a smooth, consistent user experience

Topics covered: QA · Manual testing · Automated testing · Debugging · Build preparation · Store publishing · Post-release monitoring

Types of Mobile Testing

Unit Testing
Tests small, isolated pieces of logic. Fastest to write and run.
UI / UX Testing
Validates the interface is smooth, intuitive, and consistent.
Performance Testing
App load time, scroll smoothness, CPU & memory usage.
Compatibility Testing
iOS 14–17, Android 10–14, various screen sizes and hardware.
Security Testing
Authentication, encryption, permissions, data at rest.
Regression Testing
Ensures new features don't break existing functionality.

Manual Testing – Human Validation

Manual QA is essential, especially before store submission. Manual testing catches things automated tests can't.

Manual QA checklist:

  • Does navigation work on all screens?
  • Do buttons and gestures respond correctly?
  • Are animations smooth?
  • Does the app handle no-internet mode gracefully?
  • Are error messages shown correctly?
  • Is the UI responsive across all screen sizes?
  • Does dark mode work correctly?
  • Are permissions requested appropriately?

Automated Testing – Making QA Faster

Automation is critical for large or long-term apps with frequent updates.

1. Unit Tests — Test small pieces of logic in isolation.

// Swift – XCTest
XCTAssertEqual(add(2, 3), 5)
XCTAssertNotNil(user.email)

// Kotlin – JUnit
assert(result == 5)
assertEquals("Nuno", user.name)

2. UI Tests — Simulate user actions like tapping, swiping, and typing.

3. Integration Tests — Verify that components work correctly together.

Testing frameworks by platform:

  • iOS → XCTest, XCUITest
  • Android → Espresso (UI), JUnit (logic)
  • Flutter → flutter_test, WidgetTest
  • React Native → Detox, Jest

Debugging – Finding & Fixing Issues

Debugging tools help identify crashes, UI layout problems, memory leaks, API errors, and performance bottlenecks.

iOS Debugging (Xcode): Xcode Debugger · View Hierarchy Inspector · Instruments (memory, CPU, leaks) · Console logs

Android Debugging (Android Studio): Logcat · Android Profiler · Layout Inspector · ADB shell

Common debugging techniques:

  • Breakpoints — pause execution at a specific line
  • Logging — print statements to trace execution flow
  • Crash reports — stack trace analysis
  • Simulating poor network conditions
  • Testing on multiple emulators / real devices

Preparing an App for Release

Before submitting, apps must be optimized, bug-free, secure, and properly signed.

iOS Requirements: Archive build in Xcode · Provisioning profiles · Code signing certificates · App Store Connect metadata · App icon & screenshots for all device sizes

Android Requirements: Release build (.aab) · Signing keystore · Version code increment · Play Console metadata · Screenshots + feature graphic

Why Stores Reject Apps

🍎 Apple App Store

  • Crashes or bugs
  • Unfinished features
  • Poor UI or inaccessible elements
  • Inaccurate metadata
  • Privacy issues
  • No explanation for permissions
  • Broken links
  • Misuse of Apple brand

🤖 Google Play

  • Malware or suspicious behavior
  • Excessive permissions
  • Non-functional features
  • Policy violations
  • Sensitive content without warnings
  • Deceptive metadata
  • Targeting wrong age group
  • Data safety violations

Release Channels

iOS: TestFlight (beta distribution with up to 10,000 testers) → App Store (public release)

Android: Internal testing → Closed testing → Open beta → Production release

Apps should always go through TestFlight or Play Store internal tracks before public release — never skip beta testing.

Monitoring After Release

Publishing is not the finish line — production monitoring is critical.

Crash Analytics: Firebase Crashlytics · Sentry · Bugsnag

Performance: Slow screens · Device-specific behavior · Network failures · Memory pressure

User Feedback: App store reviews · Support tickets · In-app feedback forms

New bugs always appear when real users with real devices use the app at scale. Iteration and updates are a permanent part of the mobile development lifecycle.

Test-Driven Development (TDD)

TDD is a development methodology where tests are written before the actual code. The cycle is: Red → Green → Refactor.

1. Red – Write a failing test for a feature that doesn't exist yet.

2. Green – Write the minimum code needed to make that test pass.

3. Refactor – Clean up the code while keeping the test green.

// Swift TDD example
// Step 1: Write the failing test first
func testPasswordTooShort() {
    let validator = PasswordValidator()
    XCTAssertFalse(validator.isValid("abc"))
}

// Step 2: Implement just enough to pass
func isValid(_ password: String) -> Bool {
    return password.count >= 8
}

Benefits of TDD:

  • Forces clear thinking about requirements before coding
  • Results in naturally testable, modular code
  • Provides a safety net for future refactoring
  • Reduces time spent debugging later in development

UI Automation Testing in Depth

UI automation tests simulate real user interactions — tapping buttons, typing text, scrolling — without human involvement. They are slower than unit tests but catch a different class of bugs.

XCUITest (iOS): Apple's built-in framework for UI automation. Tests run on simulator or real device.

// XCUITest – tap login and verify result
let app = XCUIApplication()
app.launch()

app.textFields["Email"].tap()
app.textFields["Email"].typeText("user@test.com")
app.buttons["Login"].tap()

XCTAssertTrue(app.staticTexts["Welcome"].exists)

Espresso (Android): Google's UI testing framework. Automatically synchronizes with the UI thread, making tests stable and reliable.

// Espresso – type in EditText and click button
onView(withId(R.id.emailField))
    .perform(typeText("user@test.com"))
onView(withId(R.id.loginButton))
    .perform(click())
onView(withText("Welcome"))
    .check(matches(isDisplayed()))

Detox (React Native): End-to-end testing framework that runs full app flows on a real simulator or device. Designed to handle async operations gracefully.

Performance Profiling Tools

Slow apps lose users. Profiling tools help identify exactly where bottlenecks occur — whether in rendering, memory, CPU, or network.

Xcode Instruments (iOS):

  • Time Profiler – Shows which functions consume the most CPU time
  • Allocations – Tracks all memory objects created and retained
  • Leaks – Detects objects that are allocated but never freed
  • Core Animation – Measures frame rate and rendering performance
  • Network – Profiles all HTTP requests and response times

Android Profiler (Android Studio):

  • CPU Profiler – Record method traces and identify slow functions
  • Memory Profiler – Monitor heap usage and detect memory leaks
  • Network Profiler – Inspect API call timing and payload sizes
  • Energy Profiler – Understand how the app affects battery life

Key performance targets: App launch under 2 seconds · 60fps scroll (16ms per frame) · No more than 50MB RAM for typical apps · Battery usage should be minimal in the background

Accessibility Testing

Accessibility ensures that apps are usable by people with visual, motor, hearing, or cognitive disabilities. Both app stores now penalize apps with poor accessibility. In many regions, accessibility compliance is also a legal requirement.

iOS Accessibility: Enable VoiceOver to verify all interactive elements have meaningful labels. Use the Accessibility Inspector in Xcode to audit issues automatically.

Android Accessibility: Enable TalkBack and navigate your app purely by touch. Use the Accessibility Scanner app from Google to detect unlabelled buttons and low-contrast text.

Common accessibility issues to check:

  • All buttons and images have descriptive accessibility labels
  • Text contrast ratio meets WCAG 2.1 AA standard (4.5:1 minimum)
  • Interactive elements are large enough to tap (at least 44×44pt on iOS)
  • The app works without relying on color alone to convey information
  • Dynamic text sizes are supported (users who increase system font size)
  • No flashing content that could trigger photosensitive seizures

App Size Optimization

Large app sizes reduce download rates, especially in markets with limited storage or slow connections. Reducing app size is a key pre-release optimization step.

iOS – Reducing IPA size:

  • Use App Thinning — Apple delivers device-specific assets only (slicing)
  • Enable Bitcode (where applicable) for future optimizations by Apple
  • Compress images and use vector assets where possible
  • Remove unused frameworks and dead code
  • Use On-Demand Resources to load large assets only when needed

Android – Reducing AAB size:

  • Use Android App Bundle (.aab) — Play Store delivers only the APK splits needed per device
  • Enable R8/ProGuard to shrink and obfuscate code
  • Use WebP format for images instead of PNG/JPG
  • Enable resource shrinking to remove unused resources
  • Analyze with APK Analyzer in Android Studio to identify the largest contributors

App Store Optimization (ASO)

ASO is the process of improving an app's visibility and conversion rate in the App Store and Google Play — the equivalent of SEO for mobile apps.

Key ASO factors:

  • App title – Include the main keyword; limit is 30 chars (iOS) / 50 chars (Android)
  • Subtitle / Short description – Second most important keyword placement
  • Keywords field (iOS) – 100 character limit; no spaces, comma-separated
  • Long description – First 3 lines visible before "more" — make them count
  • Screenshots – The biggest driver of conversion; show value immediately
  • App icon – Recognizable, distinct from competitors, simple at small sizes
  • Ratings & reviews – Prompt users at the right moment using native review APIs
  • Update frequency – Regular updates signal an active, maintained app

Post-Release Update Strategy

Shipping v1.0 is the beginning of the lifecycle, not the end. A structured update strategy keeps users happy and the app growing.

Hotfix releases – For critical crashes or security issues. Should be submitted as quickly as possible, ideally within 24–48 hours of discovery.

Patch releases (x.x.1) – Bug fixes and small improvements. No new features. Can be shipped frequently without major review risk.

Minor releases (x.1.0) – New features, UX improvements, new content. Requires full QA cycle and may need updated store screenshots.

Major releases (2.0.0) – Significant redesigns or architecture changes. Treat like a new product launch — phased rollout recommended.

Phased rollouts (Google Play): Release to 1% → 5% → 20% → 50% → 100% of users. Allows you to catch widespread issues before they affect everyone.

iOS phased release: Apple also supports 7-day phased rollouts. You can pause a release at any stage if issues are detected.
