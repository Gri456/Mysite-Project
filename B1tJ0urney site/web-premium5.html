📘 5.1 What is a Frontend Framework?

Before frameworks existed, developers had to manually write JavaScript to find elements in the DOM, update them, listen to events, and keep everything in sync. This worked for small pages, but as applications grew, it became a nightmare to maintain.


A Frontend Framework is a structured toolset that handles this repetitive work for you. It provides a mental model and a set of conventions so that teams can build complex applications in a consistent, scalable, and efficient way.

🚫 Without a Framework (Vanilla JS): 
// To update a counter you had to:
const btn = document.getElementById('btn');
const display = document.getElementById('count');
let count = 0;
btn.addEventListener('click', () => {
    count++;
    display.innerText = count; // Manual DOM update every time
});
✅ With React (Framework): 
// React updates the DOM automatically when state changes
function Counter() {
    const [count, setCount] = useState(0);
    return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
📈 Scalability

Organizes massive applications across hundreds of files in a sustainable way without losing track of logic.

⚡ Reactivity

The UI automatically reflects changes in data — no manual DOM manipulation needed.

🌍 Ecosystem

Thousands of community libraries, tools, and plugins ready to integrate (routing, forms, animations, etc.).

🤝 Team Work

Enforces consistent patterns and conventions so multiple developers can collaborate cleanly.

🧩 5.2 Component Architecture

Think of a web app like a newspaper. The newspaper is made up of sections (header, articles, sidebar, footer). Each section can be reused and updated independently. This is the idea behind Components.


A component is a self-contained, reusable piece of UI. It has its own structure (HTML/JSX), style, and behavior (JavaScript). Components can be as small as a button or as large as an entire page.

REACT COMPONENT

// Functional Component
function UserCard({ name, role }) {
  return (
    <div className="card">
      <h2>{name}</h2>
      <p>{role}</p>
    </div>
  );
}

// Using the component
<UserCard name="Ana" role="Dev" />
<UserCard name="João" role="Designer" />

VUE COMPONENT (SFC)

<template>
  <div class="card">
    <h2>{{ name }}</h2>
    <p>{{ role }}</p>
  </div>
</template>

<script setup>
defineProps(['name', 'role']);
</script>

<style scoped>
.card { padding: 1rem; }
</style>
🌳 Component Tree — How apps are structured:
App (Root)
  ├── Header
  │    ├── Logo
  │    └── NavMenu
  ├── ProductList
  │    ├── ProductCard
  │    ├── ProductCard
  │    └── ProductCard
  └── Footer

Each component only knows about itself and its direct children. This isolation makes debugging and testing much easier.

📦 5.3 Props vs State — The Two Core Concepts

Understanding the difference between Props and State is the single most important concept in component-based frameworks.

🔌 Props (Properties)

  • → Passed from parent to child
  • → Read-only — child cannot change them
  • → Like arguments to a function
  • → Make components reusable and configurable
// Parent sends data
<Button label="Click me" color="blue" />

// Child receives it
function Button({ label, color }) {
  return <button style={{color}}>{label}</button>;
}

📦 State

  • → Internal data of a component
  • → Can change over time (user interaction, API response)
  • → When state changes, the component re-renders
  • → Local by default (stays inside the component)
// React — useState hook
const [isOpen, setIsOpen] = useState(false);

// Vue — ref
const isOpen = ref(false);

// Toggle on click
function toggle() { isOpen.value = !isOpen.value; }
💡 Golden Rule:

If data needs to be shared between components, lift it up to the closest common parent. If it's only used inside one component, keep it local with state. This is called "Lifting State Up".

♻️ 5.4 Component Lifecycle

Every component goes through a lifecycle — from the moment it appears on screen to the moment it disappears. Understanding this allows you to run code at the right moment (e.g., fetch data when a page loads, or clean up timers when a component is removed).

🐣
Mount

Component is created and inserted into the DOM. Ideal for fetching data or setting subscriptions.

🔄
Update

Component re-renders because props or state changed. React compares old vs new (Virtual DOM diffing).

🪦
Unmount

Component is removed from the DOM. Use this to clear timers, cancel requests, or remove listeners.

REACT — useEffect

useEffect(() => {
  // Runs on Mount
  console.log("Component mounted!");
  fetchData();

  return () => {
    // Runs on Unmount (cleanup)
    console.log("Component removed!");
  };
}, []); // [] = only on mount

// Runs every time 'userId' changes
useEffect(() => {
  fetchUser(userId);
}, [userId]);

VUE — Lifecycle Hooks

import { onMounted, onUnmounted } from 'vue';

onMounted(() => {
  // Runs when component mounts
  console.log("Component mounted!");
  fetchData();
});

onUnmounted(() => {
  // Cleanup on unmount
  console.log("Component removed!");
});

🖱️ 5.5 Events and User Interaction

Frameworks provide clean and consistent ways to handle user interactions: clicks, form inputs, keyboard events, and more. Events in components are declared inline and connected to handler functions.

REACT EVENTS

function LoginForm() {
  const [name, setName] = useState("");

  function handleSubmit(e) {
    e.preventDefault(); // stop page reload
    alert("Hello, " + name);
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={name}
        onChange={(e) => setName(e.target.value)}
        placeholder="Your name"
      />
      <button type="submit">Login</button>
    </form>
  );
}

VUE EVENTS

<template>
  <form @submit.prevent="handleSubmit">
    <input v-model="name" placeholder="Your name" />
    <button type="submit">Login</button>
  </form>
</template>

<script setup>
import { ref } from 'vue';
const name = ref('');

function handleSubmit() {
  alert('Hello, ' + name.value);
}
</script>

🔀 5.6 Conditional Rendering and Lists

Frameworks make it easy to show or hide parts of the UI based on data, and to render lists of items dynamically — no more writing repetitive HTML by hand.

Conditional Rendering

// React — using ternary and &&
function Alert({ isError }) {
  return (
    <div>
      {isError 
        ? <p className="error">Something went wrong!</p>
        : <p className="success">All good!</p>
      }
      {isError && <button>Retry</button>}
    </div>
  );
}

// Vue — v-if / v-else
// <p v-if="isError">Error!</p>
// <p v-else>All good!</p>

Rendering Lists

// React — .map() to render lists
const products = ["Laptop", "Phone", "Tablet"];

function ProductList() {
  return (
    <ul>
      {products.map((item, index) => (
        <li key={index}>{item}</li>
        // 'key' is required for performance!
      ))}
    </ul>
  );
}

// Vue — v-for
// <li v-for="item in products" :key="item">
//   {{ item }}
// </li>
⚠️ Why is the key prop important?

When rendering lists, React and Vue need a unique key for each item to efficiently update the DOM. Without it, the framework cannot tell which items changed, leading to bugs and poor performance. Always use a unique ID — not just the index if the list order can change.

🌐 5.7 SPA and Client-Side Routing

A Single Page Application (SPA) loads one HTML file. All navigation between "pages" happens in the browser without a full page reload — JavaScript swaps the components on screen instead.

🐌 Traditional (MPA)
  • → Every link causes a full page reload
  • → Server sends a new HTML file each time
  • → Slower, more server load
  • → State is lost on navigation
⚡ SPA
  • → Navigation happens instantly (JS updates DOM)
  • → Server only provides data (via API/JSON)
  • → Faster, feels like a native app
  • → State is preserved between views

REACT ROUTER — Basic Setup

import { BrowserRouter, Routes, Route, Link } from 'react-router-dom';

function App() {
  return (
    <BrowserRouter>
      <nav>
        <Link to="/">Home</Link>
        <Link to="/about">About</Link>
        <Link to="/contact">Contact</Link>
      </nav>

      <Routes>
        <Route path="/" element={<HomePage />} />
        <Route path="/about" element={<AboutPage />} />
        <Route path="/contact" element={<ContactPage />} />
        <Route path="/user/:id" element={<UserPage />} /> {/* Dynamic route */}
      </Routes>
    </BrowserRouter>
  );
}

VUE ROUTER — Basic Setup

// router/index.js
import { createRouter, createWebHistory } from 'vue-router';

const router = createRouter({
  history: createWebHistory(),
  routes: [
    { path: '/', component: HomePage },
    { path: '/about', component: AboutPage },
    { path: '/user/:id', component: UserPage }, // Dynamic
  ]
});

// In template: <RouterLink to="/">Home</RouterLink>
// View outlet: <RouterView />

🌍 5.8 Communicating with APIs

Most modern applications need to communicate with a backend server to fetch or send data — user profiles, products, messages, etc. This is done using HTTP requests inside components, typically on mount.

REACT — Fetch with useEffect

function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetch(`https://api.example.com/users/${userId}`)
      .then(res => res.json())
      .then(data => {
        setUser(data);
        setLoading(false);
      })
      .catch(err => {
        setError(err.message);
        setLoading(false);
      });
  }, [userId]);

  if (loading) return <p>Loading...</p>;
  if (error) return <p>Error: {error}</p>;
  return <h2>Hello, {user.name}!</h2>;
}

VUE — Fetch on Mount

const user = ref(null);
const loading = ref(true);

onMounted(async () => {
  try {
    const res = await fetch('/api/user/1');
    user.value = await res.json();
  } catch (e) {
    console.error(e);
  } finally {
    loading.value = false;
  }
});
📦 Axios — Easier than Fetch 
// npm install axios
import axios from 'axios';

const { data } = await axios.get('/api/users');
// Automatic JSON parsing
// Better error handling
// Interceptors for auth headers

🏪 5.9 Global State Management

Local state works great for one component. But what happens when multiple distant components need access to the same data? For example: the user's authentication status, shopping cart contents, or UI theme.

Passing props down through many layers ("prop drilling") becomes messy. The solution is a Global Store — a single source of truth accessible from anywhere in the app.

React — Context API (Built-in)

// 1. Create context
const AuthContext = createContext(null);

// 2. Provide it high in the tree
function App() {
  const [user, setUser] = useState(null);
  return (
    <AuthContext.Provider value={{ user, setUser }}>
      <AppContent />
    </AuthContext.Provider>
  );
}

// 3. Use anywhere in the tree
function Header() {
  const { user } = useContext(AuthContext);
  return <p>Welcome, {user?.name}</p>;
}

Vue — Pinia (Recommended)

// stores/auth.js
import { defineStore } from 'pinia';

export const useAuthStore = defineStore('auth', {
  state: () => ({
    user: null,
    isLoggedIn: false
  }),
  actions: {
    login(userData) {
      this.user = userData;
      this.isLoggedIn = true;
    },
    logout() {
      this.user = null;
      this.isLoggedIn = false;
    }
  }
});

// In any component:
// const auth = useAuthStore();
// auth.login({ name: 'Ana' });
🗺️ When to use what:
Solution Use when...
Local state Data used only inside one component (input values, modal open/close)
Context API / Provide-Inject Simple global data: theme, language, user session
Pinia / Zustand / Redux Complex shared state: cart, notifications, large feature modules
React Query / SWR Server state: caching API data, automatic refetching, loading states

📋 5.10 Forms and Validation

Forms are one of the most common UI patterns. Frameworks offer powerful ways to handle controlled inputs, validation, and submission.

REACT — Controlled Form + Validation

function RegisterForm() {
  const [form, setForm] = useState({ email: '', pass: '' });
  const [errors, setErrors] = useState({});

  function validate() {
    const newErrors = {};
    if (!form.email.includes('@'))
      newErrors.email = 'Invalid email';
    if (form.pass.length < 6)
      newErrors.pass = 'Min 6 characters';
    return newErrors;
  }

  function handleSubmit(e) {
    e.preventDefault();
    const errs = validate();
    if (Object.keys(errs).length > 0) {
      setErrors(errs);
    } else {
      console.log('Form submitted!', form);
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={form.email}
        onChange={e => setForm({...form, email: e.target.value})}
      />
      {errors.email && <p style={{color:'red'}}>{errors.email}</p>}
      <button type="submit">Register</button>
    </form>
  );
}
🔑 Controlled vs Uncontrolled

A controlled input has its value managed by React/Vue state. An uncontrolled input manages itself via the DOM (using ref). Controlled is recommended — it gives you full control over the value at all times.

📦 Form Libraries

For complex forms, use dedicated libraries:

  • → React Hook Form — lightweight, fast
  • → Formik — popular, full-featured
  • → Vee-Validate — Vue forms
  • → Zod / Yup — schema validation

⚡ 5.11 Performance Optimization

A fast application is not just a technical achievement — it directly impacts user experience and business metrics. Even 100ms of delay can reduce conversions. Here are the main techniques:

🦥 Lazy Loading

Load components only when they are needed (e.g., when the user navigates to that route). Reduces the initial bundle size.

// React
const Dashboard = React.lazy(() =>
  import('./Dashboard')
);
🧠 Memoization

Prevent unnecessary re-renders by remembering previous results of expensive calculations or components.

// React
const result = useMemo(() =>
  heavyCalculation(data), [data]
);
const MemoComp = React.memo(MyComponent);
✂️ Code Splitting

Split your JavaScript bundle into smaller chunks so the browser only downloads what it needs for the current page.

// Vite does this automatically
// per route when you use
// dynamic imports
🖼️ Virtual DOM

React keeps a lightweight copy of the DOM in memory. On state change, it compares old vs new (diffing) and only updates what changed — minimizing expensive real DOM operations.

🖥️ 5.12 Server-Side Rendering (SSR) and Static Generation (SSG)

SPAs are powerful, but they have a downside: the browser receives an almost empty HTML file and JavaScript builds the page. This hurts SEO and causes a blank screen while loading. That's where SSR and SSG come in.

📄
CSR (SPA)

Browser downloads JS, runs it, builds the page. Fast after load but slow first paint.

❌ Bad SEO

⚡
SSR

Server renders HTML for each request. User sees content immediately. Great for dynamic pages.

✅ Good SEO

🏗️
SSG

HTML is pre-built at compile time. Super fast. Great for blogs, documentation, marketing pages.

✅ Best performance

⚛️ Next.js (React)

The most popular React framework. Supports CSR, SSR, SSG and more in the same project. Used by Netflix, TikTok, Vercel.

npx create-next-app@latest
🟩 Nuxt.js (Vue)

The Vue equivalent of Next.js. File-based routing, SSR, SSG, and auto-imports. Elegant and powerful.

npx nuxi@latest init my-app

🧪 5.14 Testing Your Application

Writing tests ensures your app works correctly and keeps working as it grows. There are three main types of tests in frontend development:

🔬 Unit Tests

Test individual functions and components in isolation. Fast and specific. Use Jest or Vitest.

test('adds 1+2 to equal 3', () => {
  expect(add(1, 2)).toBe(3);
});
🖼️ Component Tests

Test how components render and behave when interacted with. Use React Testing Library or Vue Test Utils.

render(<Button label="Click" />);
fireEvent.click(screen.getByText('Click'));
expect(mockFn).toHaveBeenCalled();
🌐 E2E Tests

Simulate real user behavior in a real browser. Test the full flow from login to checkout. Use Cypress or Playwright.

cy.visit('/login');
cy.get('input').type('user@x.com');
cy.get('button').click();
cy.contains('Dashboard');

♿ 5.15 Accessibility (a11y)

Accessibility means building apps that everyone can use — including people with visual, motor, auditory, or cognitive impairments. It's not just ethical — it's required by law in many countries and improves SEO.

Key Principles:

  • → Use semantic HTML elements (<button> not <div>)
  • → All images need descriptive alt attributes
  • → Every interactive element must be keyboard-focusable
  • → Color contrast ratio must be at least 4.5:1
  • → Use ARIA attributes for complex widgets
  • → Form inputs must have visible labels

ARIA ATTRIBUTES EXAMPLE

<!-- Bad -->
<div onClick={toggle}>Menu</div>

<!-- Good -->
<button
  onClick={toggle}
  aria-expanded={isOpen}
  aria-label="Toggle navigation menu"
>
  Menu
</button>

<!-- Announce dynamic content -->
<div aria-live="polite">
  {statusMessage}
</div>
