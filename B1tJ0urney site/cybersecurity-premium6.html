Cryptography, Authentication & Cloud Security Fundamentals

Cryptography is the foundation of all modern digital security — it protects data in transit, at rest, and guarantees the identity of communicating parties.

In this module you will learn how symmetric and asymmetric ciphers work, hashing, TLS, multi-factor authentication, IAM and best practices for cloud security in environments like AWS, Azure and GCP.

🔐 Symmetric Cryptography

In symmetric cryptography, the same key is used to both encrypt and decrypt the message. It is fast and efficient for large volumes of data, but the challenge is sharing the key securely.

AES — Advanced Encryption Standard

The most widely used symmetric algorithm in the world. Supports 128, 192 or 256-bit keys. Used in HTTPS, VPNs, encrypted disks and much more. AES-256 is considered secure against brute-force attacks with current hardware.

Operating Modes — AES-GCM vs AES-CBC

AES-GCM (Galois/Counter Mode) is the currently recommended mode — it guarantees confidentiality and integrity (authentication). AES-CBC (Cipher Block Chaining) is older, still used but vulnerable to padding oracle attacks if poorly implemented.

# Encrypt a file with AES-256 using OpenSSL
openssl enc -aes-256-cbc -salt -in file.txt -out file.enc -k "password123"

# Decrypt
openssl enc -aes-256-cbc -d -in file.enc -out file_decrypted.txt -k "password123"

# Encrypt with AES-256-GCM (more secure)
openssl enc -aes-256-gcm -salt -in file.txt -out file.enc -k "password123"

# List available algorithms
openssl enc -list

🔑 Asymmetric Cryptography

Asymmetric cryptography uses a key pair: the public key (shared with everyone) and the private key (kept secret). What one key encrypts, only the other can decrypt.

RSA — Rivest–Shamir–Adleman

The best-known asymmetric algorithm. Based on the difficulty of factoring large prime numbers. Used to share symmetric keys (at the start of a TLS session) and for digital signatures. Minimum recommended size: 2048 bits (4096 for highly sensitive data).

ECC — Elliptic Curve Cryptography

A modern alternative to RSA — much smaller keys with equivalent security. A 256-bit ECC key is equivalent to a 3072-bit RSA key. Widely used in modern SSH and TLS 1.3. Common algorithms: Ed25519, ECDSA.

# Generate an RSA 2048-bit key pair
openssl genrsa -out private_key.pem 2048
openssl rsa -in private_key.pem -pubout -out public_key.pem

# Encrypt with public key (only the owner of the private key can decrypt)
openssl rsautl -encrypt -inkey public_key.pem -pubin -in message.txt -out message.enc

# Decrypt with private key
openssl rsautl -decrypt -inkey private_key.pem -in message.enc -out message_decrypted.txt

# Generate SSH key with Ed25519 (more modern and secure than RSA)
ssh-keygen -t ed25519 -C "your_email@example.com"

# Generate SSH RSA 4096 key (alternative)
ssh-keygen -t rsa -b 4096 -C "your_email@example.com"

# View the generated SSH public key
cat ~/.ssh/id_ed25519.pub

# Copy public key to a remote server
ssh-copy-id user@192.168.1.10
How it works in practice — TLS Session

At the start of an HTTPS connection, the server sends its certificate (with public key). The client verifies its validity, generates a symmetric session key, encrypts it with the server's public key and sends it. Only the server (with its private key) can decrypt it. From that point, communication uses AES — much faster than RSA for large data volumes.

🧮 Hashing — Hash Functions

A hash function transforms any data into a fixed-length value (digest). It is one-way — it is not possible to reverse the hash to obtain the original data. Even a small change in the input produces a completely different hash.

Common algorithms

MD5 — 128 bits, considered insecure for security purposes (known collisions). Still used for integrity checks on non-critical files. SHA-1 — 160 bits, also deprecated for security. SHA-256 / SHA-512 — SHA-2 family, secure and widely used. bcrypt / Argon2 — specifically designed for password hashing — intentionally slow to hinder brute force.

# Generate hashes with different algorithms
echo "password123" | sha256sum
echo "password123" | sha512sum
echo "password123" | md5sum

# Hash a file
sha256sum file.txt
sha256sum file.txt > checksums.txt   # save checksum
sha256sum -c checksums.txt           # verify integrity

# Check if two files are identical
sha256sum file1.txt file2.txt

# OpenSSL — more algorithms
openssl dgst -sha256 file.txt
openssl dgst -sha512 file.txt
openssl dgst -md5 file.txt           # avoid for security purposes
Salt — Protection against Rainbow Tables

A rainbow table is a pre-computed list of hashes for common passwords. To render them useless, we add a salt — a unique random value concatenated with the password before hashing. Even two identical passwords generate different hashes. bcrypt and Argon2 include salt automatically.

# Conceptual example: hash without salt vs with salt
# Without salt (vulnerable to rainbow tables):
SHA256("password123") = ef92b778...  # always the same

# With salt (resistant to rainbow tables):
SHA256("password123" + "xK9mR2pQ") = a7f3c1d2...  # unique per user
SHA256("password123" + "qW5nL8vY") = 9b2e4f71...  # different with another salt

# Python — example with bcrypt (install: pip install bcrypt)
# import bcrypt
# password = b"password123"
# salt = bcrypt.gensalt()         # generates random salt
# hashed = bcrypt.hashpw(password, salt)
# bcrypt.checkpw(password, hashed)  # verifies password

🌐 Digital Certificates & TLS/SSL

Digital certificates link a public key to an identity (person, server, organisation), verified by a trusted Certificate Authority (CA).

How an SSL/TLS Certificate Works

A certificate contains: the server's public key, the domain name, the validity period, and the CA's digital signature. When you access https://, your browser verifies that the certificate is valid and was signed by a trusted CA (such as DigiCert, Let's Encrypt, etc.).

# Inspect a website's certificate
openssl s_client -connect google.com:443 -showcerts

# View certificate details
echo | openssl s_client -connect google.com:443 2>/dev/null | openssl x509 -noout -text

# View expiry date
echo | openssl s_client -connect google.com:443 2>/dev/null | openssl x509 -noout -dates

# Create a self-signed certificate (for testing/labs)
openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem -days 365 -nodes

# Verify a local certificate
openssl x509 -in cert.pem -noout -text

# Let's Encrypt — free certificate for production (uses certbot)
# certbot --nginx -d domain.com
TLS 1.2 vs TLS 1.3

TLS 1.3 (2018) is faster and more secure — it removes legacy algorithms (RC4, DES, SHA-1), reduces the handshake from 2 to 1 round-trip, and uses forward secrecy by default. Avoid using TLS 1.0 and 1.1, which are considered insecure.

🔏 GPG — File and Email Encryption

GPG (GNU Privacy Guard) implements the OpenPGP standard and allows you to encrypt files, digitally sign documents and verify the identity of senders.

# Generate a GPG key pair
gpg --full-generate-key

# List keys in the keyring
gpg --list-keys               # public keys
gpg --list-secret-keys        # private keys

# Encrypt a file with a symmetric key (password)
gpg -c file.txt               # generates file.txt.gpg

# Encrypt for a specific recipient (public key)
gpg --encrypt --recipient email@example.com file.txt

# Decrypt
gpg --decrypt file.txt.gpg

# Digitally sign a file
gpg --sign file.txt

# Verify a signature
gpg --verify file.txt.gpg

# Export public key (to share)
gpg --export --armor email@example.com > public_key.asc

# Import another person's public key
gpg --import public_key.asc

🛡️ Authentication & MFA

Authentication is the process of verifying who you are. There are three possible factors: something you know (password), something you have (phone, token), something you are (biometrics).

MFA — Multi-Factor Authentication

Combines two or more different factors. Example: password + TOTP code on your phone. Even if the password is compromised, the attacker still needs the second factor. MFA reduces compromise risk by more than 99% according to Microsoft studies.

TOTP — Time-based One-Time Password

Generates a 6-digit code that changes every 30 seconds, based on a shared secret and the current time. Used by apps like Google Authenticator, Authy and Microsoft Authenticator. The algorithm is defined by RFC 6238.

Passkeys — The Future of Authentication

Passkeys replace passwords entirely. They use asymmetric cryptography — the private key stays on the device and never leaves. Authentication is done with local biometrics (fingerprint, face). Supported by Google, Apple, Microsoft and most modern browsers.

# Configure MFA for SSH with Google Authenticator
sudo apt install libpam-google-authenticator
google-authenticator          # configure for the current user

# Edit /etc/pam.d/sshd to enable MFA
# auth required pam_google_authenticator.so

# Edit /etc/ssh/sshd_config
# ChallengeResponseAuthentication yes
# AuthenticationMethods publickey,keyboard-interactive

# Restart SSH
sudo systemctl restart sshd

# Check login attempts and MFA in the logs
grep "Accepted\|Failed" /var/log/auth.log

👤 IAM — Identity and Access Management

IAM is the set of policies, processes and technologies that control who has access to what in an organisation or cloud system.

Principle of Least Privilege (PoLP)

Every user, service or process must have only the permissions strictly necessary for their function — nothing more. Dramatically reduces the impact of a compromised account. In AWS, for example, never use the root account for daily operations.

RBAC — Role-Based Access Control

Instead of assigning permissions directly to users, you create roles with specific permissions and assign users to roles. This simplifies management: changing the permissions of a role affects all users with that role.

# AWS CLI — IAM Management
# List IAM users
aws iam list-users

# Create IAM user
aws iam create-user --user-name alice

# List policies attached to a user
aws iam list-attached-user-policies --user-name alice

# Attach S3 read-only policy to user
aws iam attach-user-policy --user-name alice --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess

# Check effective permissions of a user
aws iam simulate-principal-policy --policy-source-arn arn:aws:iam::ACCOUNT:user/alice --action-names s3:GetObject

# List IAM roles
aws iam list-roles

# View policies of a role
aws iam list-attached-role-policies --role-name role-name
Tokens vs Passwords

Tokens (such as JWT or OAuth tokens) are temporary credentials with limited validity. They are preferable to passwords in APIs and cloud services because: they expire automatically, can be revoked without changing passwords, and contain only the permissions needed for that session.

☁️ Cloud Security — AWS, Azure & GCP

Cloud security follows the shared responsibility model — the provider (AWS, Azure, GCP) protects the infrastructure, you protect the data, access and configurations.

Common Cloud Security Mistakes

Public S3 buckets — one of the most common mistakes, already caused massive data breaches at well-known companies. Hard-coded credentials in source code — AWS tokens exposed on GitHub are found within minutes by bots. Overly permissive Security Groups — port 22 or 3389 open to 0.0.0.0/0 (the entire internet).

# AWS S3 — check if a bucket is public
aws s3api get-bucket-acl --bucket bucket-name
aws s3api get-bucket-policy --bucket bucket-name

# Block public access on an S3 bucket
aws s3api put-public-access-block \
  --bucket bucket-name \
  --public-access-block-configuration \
  "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"

# Check for permissive Security Groups (port 22 open to 0.0.0.0/0)
aws ec2 describe-security-groups --query "SecurityGroups[?contains(IpPermissions[].IpRanges[].CidrIp, '0.0.0.0/0')]"

# AWS CloudTrail — audit actions in the account
aws cloudtrail describe-trails
aws cloudtrail lookup-events --lookup-attributes AttributeKey=EventName,AttributeValue=DeleteBucket

# Check if MFA is enabled on the AWS root account
aws iam get-account-summary | grep MFADevices
Encryption in Transit vs at Rest

In transit — data protected while travelling over the network, using TLS/HTTPS. At rest — data encrypted when stored (disk, database, backups). In AWS, KMS (Key Management Service) manages encryption keys for data at rest (S3, RDS, EBS).

🗝️ Secrets & Credential Management

Passwords, API tokens, private keys and database connection strings are secrets — they must never be in source code, unprotected config files, or environment variables in production.

# NEVER DO THIS — credentials in code
# db_password = "supersecret123"  ← exposed on GitHub!
# AWS_SECRET_KEY = "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"  ← danger!

# Better practice — environment variables (development)
export DB_PASSWORD="supersecret123"
python app.py  # access via os.environ.get('DB_PASSWORD')

# AWS Secrets Manager — cloud solution
aws secretsmanager create-secret --name prod/db/password --secret-string "supersecret123"
aws secretsmanager get-secret-value --secret-id prod/db/password

# Automatic secret rotation (AWS)
aws secretsmanager rotate-secret --secret-id prod/db/password

# HashiCorp Vault — open source secrets management solution
vault kv put secret/db password="supersecret123"
vault kv get secret/db

# Check for exposed credentials in git history
git log --all --full-history -- "*.env"
git log -p | grep -i "password\|secret\|key\|token"
Credential Rotation

Credentials must be rotated regularly — especially after an employee leaves or a compromise is suspected. Tools like AWS Secrets Manager and HashiCorp Vault automate rotation. API tokens should have short validity periods (hours or days, not years).

✍️ Digital Signatures

A digital signature guarantees authenticity (who signed it) and integrity (the content has not been altered). It is like a handwritten signature, but mathematically verifiable.

How it works

The sender calculates the hash of the document and encrypts that hash with their private key — that is the signature. The receiver calculates the hash of the received document, decrypts the signature with the sender's public key, and compares the two hashes. If they match, the document is authentic and has not been altered.

# Sign a file with an RSA private key
openssl dgst -sha256 -sign private_key.pem -out signature.sig document.txt

# Verify signature with public key
openssl dgst -sha256 -verify public_key.pem -signature signature.sig document.txt

# Sign with GPG
gpg --detach-sign --armor document.txt      # generates document.txt.asc

# Verify GPG signature
gpg --verify document.txt.asc document.txt

# Verify signature of a Linux package (Debian example)
gpg --verify file.deb.sig file.deb
