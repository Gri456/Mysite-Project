💰 7.1 What is FinOps?

FinOps (Financial Operations) is a cultural practice that brings financial accountability to the variable-spend model of cloud. It unites Engineering, Finance, and Business so teams can make informed cost/velocity trade-offs.

The FinOps Foundation defines three phases:
• Inform — gain visibility. You cannot optimise what you cannot see. Tagging, dashboards, anomaly detection.
• Optimise — act on the data. Right-sizing, Reserved Instances, Spot, waste elimination.
• Operate — embed cost awareness into culture. Budget alerts, showback/chargeback, FinOps champions per team.
FinOps Maturity Model: CRAWL --> WALK --> RUN Basic Consistent Continuous tagging allocation optimisation Crawl: Some tags, manual cost reviews, reactive to bills Walk: Full tagging, showback reports, automated anomaly alerts Run: Chargeback, cost per feature, unit economics, RI automation

🏷️ 7.2 Tagging Strategy — The Foundation of Cost Visibility

Without tags, you cannot allocate costs to teams, products, or environments. Tags are key-value pairs on every resource — the backbone of showback and chargeback.

  • Mandatory tags: Enforce via SCP/Azure Policy. Common set: env, team, product, cost-centre, owner.
  • Consistency: Agree naming conventions upfront. Environment=prod vs env=production are two different tags to a billing tool.
  • Enforce at creation: Block resource creation without required tags using AWS Config rules or SCPs.
  • Tag coverage target: 95%+ of spend tagged. Use Cost Explorer untagged spend report as a hygiene KPI.
  • Inherited tags: Tags on an AWS Organization OU or Azure Resource Group propagate automatically — reducing manual effort.
# AWS SCP -- deny EC2 launch without required tags
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "DenyUntaggedEC2",
    "Effect": "Deny",
    "Action": ["ec2:RunInstances"],
    "Resource": ["arn:aws:ec2:*:*:instance/*"],
    "Condition": {
      "Null": {
        "aws:RequestTag/team":        "true",
        "aws:RequestTag/env":         "true",
        "aws:RequestTag/cost-centre": "true"
      }
    }
  }]
}

📊 7.3 AWS Cost Explorer and Cost Allocation

AWS Cost Explorer is the native tool for visualising, understanding, and forecasting AWS spend. It turns raw billing data into actionable insights.

  • Cost and Usage Reports (CUR): Most granular AWS billing data — hourly line-item per resource. Store in S3, query with Athena for custom analysis.
  • Cost Explorer filters: Slice by service, region, account, tag, instance type, and purchase option. Identify top cost drivers instantly.
  • RI/SP coverage: Shows what % of eligible usage is covered by Reserved Instances or Savings Plans — and unused commitment.
  • Rightsizing recommendations: AWS analyses CloudWatch CPU/memory metrics and suggests downsizing over-provisioned EC2 instances.
  • Anomaly Detection: ML-based alerts when spend deviates significantly from the predicted pattern — catches billing surprises before month end.
  • Forecasting: Predicts end-of-month spend based on current usage trends — feeds budget planning and finance reviews.
# AWS CLI -- top services spend this month
aws ce get-cost-and-usage \
  --time-period Start=2025-03-01,End=2025-04-01 \
  --granularity MONTHLY \
  --metrics "BlendedCost" \
  --group-by Type=DIMENSION,Key=SERVICE \
  --query 'ResultsByTime[].Groups[].{Service:Keys[0],Cost:Metrics.BlendedCost.Amount}' \
  --output table

# Enable anomaly detection for EC2 service
aws ce create-anomaly-monitor \
  --anomaly-monitor '{
    "MonitorName": "EC2-anomaly",
    "MonitorType": "DIMENSIONAL",
    "MonitorDimension": "SERVICE"
  }'

📦 7.4 Reserved Instances and Savings Plans

On-demand is the most expensive option. Committing to usage in advance unlocks up to 72% discount. Choose the right commitment model for each workload type.

Relative cost comparison (same baseline workload):

On-Demand100% — full price, no commitment
1-yr Savings Plan (No Upfront)~60% — moderate saving
3-yr Reserved (All Upfront)~38% — maximum saving
Spot Instance~10-30% — cheapest, interruptible

📋 Reserved Instances (RI)

Commit to a specific instance family, size, OS, and region (Standard) or any size/OS in a family (Convertible). 1-year or 3-year terms. All Upfront / Partial / No Upfront. Unused RIs can be sold on the RI Marketplace.

💳 Compute Savings Plans

Commit to a dollar amount of compute per hour. Applies automatically to EC2, Fargate, and Lambda — regardless of region, family, OS, or size. More flexible than RIs. Recommended default for most steady-state workloads.

Decision rule: Savings Plans for 24/7 baseline (web tier, databases). Spot for fault-tolerant batch jobs and CI/CD runners. On-Demand for variable peak traffic above the committed baseline.

⚡ 7.5 Spot Instances — Deep Dive

Spot Instances use spare EC2 capacity at up to 90% discount. AWS can reclaim them with a 2-minute warning. The key to Spot success is designing workloads that tolerate interruption.

  • 2-minute warning: AWS sends a termination notice via instance metadata and EventBridge. Your app must checkpoint state, drain in-flight work, and re-enqueue jobs.
  • Instance diversification: Request 5+ instance families simultaneously. More options = lower interruption probability. Never depend on a single Spot pool.
  • price-capacity-optimized strategy: Allocates from the Spot pool with the lowest interruption risk. Preferred over lowest-price for production workloads.
  • Best Spot use cases: CI/CD runners, batch ETL, ML training, rendering, stateless auto-scaling workers. Avoid: databases, stateful apps, single-instance critical services.
  • Karpenter (Kubernetes): Node autoprovisioner that natively provisions Spot and handles interruption — replaces the Spot interruption handler DaemonSet.
# EC2 Auto Scaling mixed on-demand + Spot (Terraform)
resource "aws_autoscaling_group" "app" {
  name                = "app-asg"
  vpc_zone_identifier = module.vpc.private_subnets
  min_size            = 2
  max_size            = 20

  mixed_instances_policy {
    instances_distribution {
      on_demand_base_capacity                  = 2
      on_demand_percentage_above_base_capacity = 0
      spot_allocation_strategy                 = "price-capacity-optimized"
    }
    launch_template {
      launch_template_specification {
        launch_template_id = aws_launch_template.app.id
        version            = "$Latest"
      }
      override { instance_type = "m5.xlarge"  }
      override { instance_type = "m5a.xlarge" }
      override { instance_type = "m6i.xlarge" }
      override { instance_type = "m6a.xlarge" }
      override { instance_type = "m5n.xlarge" }
    }
  }
}

📉 7.6 Right-Sizing and Waste Elimination

The most common finding in any cloud cost audit: instances running at 5-15% CPU utilisation. Right-sizing matches resource allocation to actual workload requirements.

Common waste patterns and fixes: Idle EC2/RDS instances --> Terminate or schedule shutdown Over-provisioned instance sizes --> Downsize based on metrics Unattached EBS volumes --> Delete or snapshot+delete Old EBS snapshots --> Lifecycle policy: expire after 30d Unused Elastic IPs --> Release (charged when unattached) NAT Gateway data transfer --> Use VPC endpoints / PrivateLink Dev environments running 24/7 --> Auto-shutdown nights+weekends (~70% saving) Oversized RDS --> Aurora Serverless v2 for variable load Unused RIs/SPs --> Sell on RI Marketplace / modify
# AWS CLI -- find unattached EBS volumes (immediate cost saving)
aws ec2 describe-volumes \
  --filters Name=status,Values=available \
  --query 'Volumes[*].{ID:VolumeId,Size:Size,Type:VolumeType,AZ:AvailabilityZone}' \
  --output table

# Find EC2 instances with < 5% average CPU over 14 days
aws cloudwatch get-metric-statistics \
  --namespace AWS/EC2 \
  --metric-name CPUUtilization \
  --dimensions Name=InstanceId,Value=i-0123456789abcdef0 \
  --start-time 2025-03-01T00:00:00Z \
  --end-time   2025-03-15T00:00:00Z \
  --period 1209600 \
  --statistics Average

# Schedule dev environment shutdown (Lambda + EventBridge)
# Stop all EC2 with env=dev tag at 20:00 Mon-Fri
resource "aws_cloudwatch_event_rule" "dev_stop" {
  name                = "stop-dev-instances"
  schedule_expression = "cron(0 20 ? * MON-FRI *)"
}

resource "aws_cloudwatch_event_rule" "dev_start" {
  name                = "start-dev-instances"
  schedule_expression = "cron(0 8 ? * MON-FRI *)"
}

👥 7.7 Showback and Chargeback

Showback and chargeback are mechanisms to allocate cloud costs to the teams or business units that generated them — creating financial accountability.

👁️ Showback

Show each team their cost without actually charging them. Raises awareness and creates accountability without budget friction. Good starting point for organisations new to FinOps. "You spent $12,400 last month on EC2 in the checkout team."

💷 Chargeback

Actually deduct cloud costs from each team's budget. Full financial accountability — teams internalise the cost of their decisions. Requires mature tagging and clear cost allocation rules. More powerful but needs cultural readiness.

  • Shared costs: Platform, networking, monitoring, and shared services must be allocated. Common methods: even split, proportional to usage, or direct attribution.
  • Unit economics: The most mature FinOps metric — cost per API call, cost per active user, cost per transaction. Connects engineering decisions to business outcomes.
  • FinOps dashboards: AWS Cost Explorer tags, Grafana with CUR data in Athena, CloudHealth, Apptio Cloudability, Spot.io Eco.
# Athena query -- monthly cost per team from CUR data
SELECT
  resource_tags_user_team         AS team,
  line_item_product_code          AS service,
  SUM(line_item_blended_cost)     AS total_cost_usd,
  SUM(line_item_usage_amount)     AS usage_amount
FROM
  cost_and_usage_report
WHERE
  line_item_line_item_type = 'Usage'
  AND year = '2025'
  AND month = '3'
GROUP BY
  resource_tags_user_team,
  line_item_product_code
ORDER BY
  total_cost_usd DESC
LIMIT 50;

💰 7.8 Infracost — Cost in the CI/CD Pipeline

Infracost is an open-source tool that adds cost estimates to Terraform pull requests — so engineers see the cost impact of their changes before they deploy, not when the bill arrives.

  • PR comments: Infracost posts a cost diff directly on GitHub/GitLab/Bitbucket PRs: "+$347/month" or "-$52/month" with a resource breakdown.
  • Supported providers: AWS, Azure, GCP — covers EC2, RDS, EBS, Lambda, CloudFront, Load Balancers, and 250+ resources.
  • Policy enforcement: Block PRs that exceed a monthly cost threshold using Infracost Cloud or custom CI steps.
  • Usage-based costs: Define expected usage (requests/month, GB transferred) in a usage YAML file for accurate Lambda and data transfer estimates.
# Install and run Infracost locally
brew install infracost
infracost auth login

# Generate cost estimate for a Terraform project
infracost breakdown --path . --format table

# Output:
# Name                              Monthly Qty  Unit      Monthly Cost
# aws_instance.web_server
#  Instance usage (Linux/UNIX, m5.xlarge)  730  hours         $145.16
#  root_block_device
#   Storage (general purpose SSD, gp3)     50  GB               $4.00
# aws_rds_instance.postgres
#  Database instance (db.m5.large)         730  hours         $138.70
#  Storage                                100  GB               $11.50
# OVERALL TOTAL                                               $299.36/month

# CI integration (GitHub Actions)
- name: Infracost diff
  uses: infracost/actions/setup@v3
  with:
    api-key: ${{ secrets.INFRACOST_API_KEY }}

- name: Post Infracost comment
  run: |
    infracost diff \
      --path . \
      --format json \
      --compare-to infracost-base.json \
      --out-file infracost-diff.json
    infracost comment github \
      --path infracost-diff.json \
      --repo $GITHUB_REPOSITORY \
      --github-token ${{ secrets.GITHUB_TOKEN }} \
      --pull-request ${{ github.event.pull_request.number }}

🛡️ 7.9 Cloud Custodian — Policy-as-Code for Cost Governance

Cloud Custodian is an open-source rules engine for cloud governance. Write YAML policies that query, filter, and act on cloud resources — automatically enforcing cost and compliance standards.

  • Declarative policies: Define what resources to find (filters) and what to do with them (actions) — tag, stop, terminate, notify, or quarantine.
  • Event-driven: Run on a schedule (CloudWatch Events) or react to real-time events (CloudTrail via Lambda). Detect and remediate immediately.
  • Multi-cloud: AWS, Azure, and GCP support with a unified policy language.
  • Dry-run mode: Test what would happen without taking action — audit before enforce.
# Cloud Custodian -- auto-stop untagged EC2 instances after 7 days
policies:
  - name: enforce-required-tags
    resource: ec2
    filters:
      - "tag:team": absent
      - type: instance-age
        days: 7
    actions:
      - type: stop
      - type: notify
        template: default
        subject: "[FinOps] Instance stopped - missing team tag"
        to: ["finops@company.com"]
        transport:
          type: sqs
          queue: https://sqs.eu-west-1.amazonaws.com/123/finops-alerts

  # Delete EBS volumes unattached for 30+ days
  - name: delete-unattached-volumes
    resource: ebs
    filters:
      - State: available
      - type: age
        days: 30
        op: ge
    actions:
      - type: snapshot
      - type: delete

  # Auto-shutdown dev EC2 instances outside business hours
  - name: stop-dev-after-hours
    resource: ec2
    filters:
      - "tag:env": dev
      - State.Name: running
    actions:
      - stop
    mode:
      type: periodic
      schedule: "rate(1 hour)"
      # Only runs outside Mon-Fri 08:00-20:00 UTC
      # (implement with Lambda time check)

📈 7.10 Budget Alerts and Forecasting

  • AWS Budgets: Set cost or usage budgets with alert thresholds (e.g., alert at 80% and 100% of monthly budget). Supports per-service, per-tag, and per-account budgets. Can trigger Lambda or SNS for auto-remediation.
  • Budget actions: Automatically apply an IAM policy (deny all) when a budget threshold is exceeded — enforce spending limits programmatically.
  • Cost anomaly alerts: Configure to alert on absolute ($100 threshold) or percentage (10% above expected) anomalies — avoids alert fatigue on small accounts.
  • Rightsizing forecasting: Use Compute Optimizer to model the impact of instance type changes on cost and performance before committing.
# AWS Budget with alert + SNS notification (Terraform)
resource "aws_budgets_budget" "monthly" {
  name         = "production-monthly-budget"
  budget_type  = "COST"
  limit_amount = "5000"
  limit_unit   = "USD"
  time_unit    = "MONTHLY"

  cost_filter {
    name   = "TagKeyValue"
    values = ["user:env$production"]
  }

  notification {
    comparison_operator        = "GREATER_THAN"
    threshold                  = 80
    threshold_type             = "PERCENTAGE"
    notification_type          = "ACTUAL"
    subscriber_email_addresses = ["finops@company.com"]
  }

  notification {
    comparison_operator        = "GREATER_THAN"
    threshold                  = 100
    threshold_type             = "PERCENTAGE"
    notification_type          = "FORECASTED"
    subscriber_sns_arns        = [aws_sns_topic.finops_alerts.arn]
  }
}

🔩 7.11 Multi-Cloud and Kubernetes Cost Management

  • Kubecost: Kubernetes-native cost monitoring. Allocates pod, namespace, and deployment costs based on actual CPU, memory, and network usage. Integrates with AWS, Azure, and GCP billing.
  • OpenCost: CNCF-backed open-source Kubernetes cost standard — the open-source alternative to Kubecost Cloud.
  • Karpenter + Spot: Karpenter provisions the cheapest node for each pod's requirements, consolidates under-utilised nodes, and handles Spot interruption — often reducing K8s node costs by 50-70%.
  • Finout / CloudHealth / Apptio: Multi-cloud FinOps platforms — unified billing view across AWS, Azure, and GCP with showback/chargeback automation.
# Kubecost -- query namespace costs via API
curl "http://kubecost.monitoring.svc/model/allocation?window=30d&aggregate=namespace" \
  | jq '.data[].sets[].allocations | to_entries[] | {
      namespace: .key,
      cpuCost: .value.cpuCost,
      memoryCost: .value.memoryCost,
      totalCost: .value.totalCost
    }'

# OpenCost Helm install
helm repo add opencost https://opencost.github.io/opencost-helm-chart
helm install opencost opencost/opencost \
  --namespace opencost \
  --create-namespace \
  --set opencost.exporter.cloudProviderApiKey=$AWS_PRICING_KEY

📋 7.12 FinOps Culture and Organisational Patterns

  • FinOps champions: Embed one FinOps-aware engineer per team responsible for reviewing cost dashboards weekly and flagging anomalies.
  • Weekly cost reviews: 15-minute review of last week's spend vs forecast. Highlight new top spenders, anomalies, and idle resources.
  • Cost in sprint planning: Include cost impact as a story attribute. "This feature will add ~$800/month in DynamoDB costs" — visible before it ships.
  • Unit economics KPIs: Track cost per transaction, cost per API call, cost per active user. Connect cloud spend to business outcomes in executive dashboards.
  • Golden path templates: Pre-approved Terraform modules with cost-optimal defaults — right-sized instances, Spot mixed, auto-shutdown for non-prod. Make the cheapest option the easiest option.

🎗 7.13 Data Transfer Costs — The Hidden Billing Trap

Data transfer charges are one of the most common billing surprises. Understanding the billing model prevents costly architecture mistakes.

AWS Data Transfer Pricing (approximate): Within same AZ: FREE Cross-AZ (same region): $0.01/GB each direction Cross-region: $0.02/GB (varies by region pair) To internet: $0.09/GB first 10TB/month Expensive patterns: App in us-east-1 --> RDS in eu-west-1 = $0.02/GB x ALL queries ECS task --> S3 via internet = $0.09/GB (use VPC endpoint!) Cross-AZ microservices (100GB/day) = $3/day = $90/month Fixes: Use VPC endpoints for S3/DynamoDB/KMS = FREE Co-locate services in same AZ (stateless) = FREE CloudFront in front of S3/API = cheaper egress rate Use S3 Transfer Acceleration only if needed

🔍 7.14 FinOps Maturity Assessment

Use this checklist to assess where your organisation is in the FinOps maturity journey:

CRAWL (basic visibility):
  [ ] All resources have at minimum env + team tags
  [ ] Monthly cost review meeting exists
  [ ] AWS Budgets alerts configured for each account
  [ ] Cost Explorer accessed regularly by engineering leads

WALK (consistent practice):
  [ ] 95%+ spend tagged and allocated to teams
  [ ] Showback reports sent to team leads monthly
  [ ] Rightsizing recommendations reviewed quarterly
  [ ] Anomaly detection configured with thresholds
  [ ] Infracost running on all Terraform PRs
  [ ] Cloud Custodian policies enforcing tag compliance
  [ ] Dev/staging environments auto-shutdown nights/weekends

RUN (continuous optimisation):
  [ ] Chargeback integrated with team budgets
  [ ] Unit economics tracked per product feature
  [ ] RI/SP coverage > 70% for stable workloads
  [ ] Spot usage > 40% of eligible compute
  [ ] Cost per team reviewed in sprint ceremonies
  [ ] Kubernetes costs tracked per namespace (Kubecost)
  [ ] FinOps champion role defined per engineering team
