🌐 5.1 VPC — Virtual Private Cloud

A VPC is a logically isolated section of the cloud provider's network where you launch resources. It is your private address space in the cloud.

┌──────────────────────────────────────────────────────────────┐ │ VPC: 10.0.0.0/16 (eu-west-1) │ │ │ │ ┌─────────────────────┐ ┌─────────────────────┐ │ │ │ AZ-1a │ │ AZ-1b │ │ │ │ ┌───────────────┐ │ │ ┌───────────────┐ │ │ │ │ │ Public Subnet │ │ │ │ Public Subnet │ │ │ │ │ │ 10.0.1.0/24 │ │ │ │ 10.0.2.0/24 │ │ │ │ │ │ [ALB] [NAT GW]│ │ │ │ [ALB] [NAT GW]│ │ │ │ │ └───────┬───────┘ │ │ └───────┬───────┘ │ │ │ │ │ │ │ │ │ │ │ │ ┌───────▼───────┐ │ │ ┌───────▼───────┐ │ │ │ │ │Private Subnet │ │ │ │Private Subnet │ │ │ │ │ │ 10.0.11.0/24 │ │ │ │ 10.0.12.0/24 │ │ │ │ │ │ [EC2] [RDS] │ │ │ │ [EC2] [RDS] │ │ │ │ │ └───────────────┘ │ │ └───────────────┘ │ │ │ └─────────────────────┘ └─────────────────────┘ │ │ │ │ Internet Gateway ←→ Public Subnets │ │ NAT Gateway → Private subnets outbound only │ └──────────────────────────────────────────────────────────────┘
  • CIDR block: The IP address range of the VPC (e.g., 10.0.0.0/16 = 65,536 addresses). Plan carefully — it cannot be shrunk after creation.
  • Public subnets: Directly reachable from the internet via an Internet Gateway. Used for load balancers, NAT Gateways, bastion hosts.
  • Private subnets: No direct internet access. Used for application servers, databases, and internal services.
  • Internet Gateway (IGW): The door between your VPC and the public internet. Attached to the VPC, not a subnet.
  • NAT Gateway: Allows private subnet resources to initiate outbound internet traffic (e.g., download packages) without being reachable inbound.
  • Route Tables: Define where traffic is routed. Each subnet is associated with one route table.
# Full VPC with public + private subnets (Terraform)
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.1.0"

  name = "prod-vpc"
  cidr = "10.0.0.0/16"

  azs             = ["eu-west-1a", "eu-west-1b", "eu-west-1c"]
  private_subnets = ["10.0.11.0/24", "10.0.12.0/24", "10.0.13.0/24"]
  public_subnets  = ["10.0.1.0/24",  "10.0.2.0/24",  "10.0.3.0/24"]

  enable_nat_gateway     = true
  single_nat_gateway     = false  # one per AZ for HA
  enable_dns_hostnames   = true
  enable_dns_support     = true

  tags = {
    Environment = "production"
    Team        = "platform"
  }
}

🔒 5.2 Security Groups and NACLs

AWS provides two layers of network access control. Understanding the difference is critical for both security and troubleshooting.

🛡️ Security Groups (stateful)

Act as virtual firewalls at the instance (ENI) level. Stateful — return traffic is automatically allowed. Rules are allow-only (no explicit deny). Evaluated as a union of all rules.

🧱 Network ACLs (stateless)

Act at the subnet level. Stateless — you must explicitly allow both inbound AND outbound directions. Rules are evaluated in order by number. Supports explicit deny.

# Security Group — allow HTTPS inbound from anywhere, all outbound
resource "aws_security_group" "alb_sg" {
  name   = "alb-sg"
  vpc_id = module.vpc.vpc_id

  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
    description = "HTTPS from internet"
  }

  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
    description = "HTTP — redirect to HTTPS"
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# App SG — only allow traffic FROM the ALB security group
resource "aws_security_group" "app_sg" {
  name   = "app-sg"
  vpc_id = module.vpc.vpc_id

  ingress {
    from_port       = 8080
    to_port         = 8080
    protocol        = "tcp"
    security_groups = [aws_security_group.alb_sg.id]
    description     = "App port — from ALB only"
  }
}

⚖️ 5.3 Load Balancers — ALB, NLB, and GLB

Load balancers distribute incoming traffic across multiple targets, improving availability and scalability.

  • ALB (Application Load Balancer) — Layer 7: HTTP/HTTPS aware. Routes by path (/api/*), hostname (api.example.com), headers, query strings. Supports WebSockets, gRPC, and sticky sessions.
  • NLB (Network Load Balancer) — Layer 4: Ultra-low latency TCP/UDP/TLS routing. Preserves source IP. Static IP / Elastic IP support. Best for real-time gaming, financial apps, IoT.
  • GLB (Gateway Load Balancer) — Layer 3: Deploy and scale third-party virtual appliances (firewalls, IDS/IPS) transparently.
  • Health checks: All load balancers perform health checks and remove unhealthy targets automatically. Configure path, interval, thresholds.
  • Target groups: Logical grouping of targets — EC2 instances, Lambda, IPs, or K8s pods via AWS Load Balancer Controller.
# ALB with HTTPS listener, path-based routing (Terraform)
resource "aws_lb" "main" {
  name               = "prod-alb"
  internal           = false
  load_balancer_type = "application"
  security_groups    = [aws_security_group.alb_sg.id]
  subnets            = module.vpc.public_subnets

  access_logs {
    bucket  = aws_s3_bucket.lb_logs.id
    enabled = true
  }
}

resource "aws_lb_listener_rule" "api_rule" {
  listener_arn = aws_lb_listener.https.arn
  priority     = 100

  action {
    type             = "forward"
    target_group_arn = aws_lb_target_group.api.arn
  }

  condition {
    path_pattern { values = ["/api/*"] }
  }
}

resource "aws_lb_listener_rule" "frontend_rule" {
  listener_arn = aws_lb_listener.https.arn
  priority     = 200

  action {
    type             = "forward"
    target_group_arn = aws_lb_target_group.frontend.arn
  }

  condition {
    path_pattern { values = ["/*"] }
  }
}

🌍 5.4 DNS — Route 53 and Resolution

DNS (Domain Name System) translates human-readable domain names into IP addresses. Route 53 is AWS's highly available, globally distributed DNS service.

  • A record: Maps a domain to an IPv4 address. Alias A records in Route 53 can point to AWS resources (ALB, CloudFront, S3) with no extra charge.
  • AAAA record: Maps a domain to an IPv6 address.
  • CNAME: Alias one domain to another. Cannot be used at the zone apex (root domain).
  • MX record: Directs email to mail servers.
  • TXT record: Arbitrary text — used for SPF, DKIM, domain ownership verification.
  • TTL (Time to Live): How long resolvers cache the record. Low TTL before migrations; higher TTL for stable records reduces cost and latency.
Route 53 Routing Policies:
• Simple — single value, no health checking.
• Weighted — split traffic by percentage (canary, A/B testing).
• Latency-based — route to the region with lowest latency for the user.
• Geolocation — route by user's geographic location (GDPR data residency).
• Geoproximity — route based on resource location with a bias setting.
• Failover — primary/secondary with health check-driven automatic failover.
• Multivalue Answer — return up to 8 healthy records, simple client-side load balancing. 
# Route 53 — weighted routing for canary (Terraform)
resource "aws_route53_record" "api_v1" {
  zone_id = aws_route53_zone.main.zone_id
  name    = "api.example.com"
  type    = "A"

  weighted_routing_policy { weight = 90 }
  set_identifier = "v1"

  alias {
    name                   = aws_lb.v1.dns_name
    zone_id                = aws_lb.v1.zone_id
    evaluate_target_health = true
  }
}

resource "aws_route53_record" "api_v2" {
  zone_id = aws_route53_zone.main.zone_id
  name    = "api.example.com"
  type    = "A"

  weighted_routing_policy { weight = 10 }
  set_identifier = "v2"

  alias {
    name                   = aws_lb.v2.dns_name
    zone_id                = aws_lb.v2.zone_id
    evaluate_target_health = true
  }
}

⚡ 5.5 CDN — CloudFront and Content Delivery

A CDN caches content at geographically distributed edge locations, serving users from the closest point of presence (PoP) to dramatically reduce latency.

Without CDN: User (Tokyo) ──────────────────────▶ Origin (Dublin) Round-trip: ~250ms With CDN: User (Tokyo) ──▶ Edge PoP (Tokyo) ──▶ Cache HIT Round-trip: ~5ms (cache hit) or Edge PoP ──▶ Origin (Dublin) on cache MISS
  • CloudFront (AWS): 450+ PoPs globally. Integrates natively with S3, ALB, API Gateway, Lambda@Edge, and CloudFront Functions.
  • Behaviours: Define caching rules per path pattern — cache static assets for 1 year, forward API requests to origin with no cache.
  • Cache-Control headers: Origin controls caching via Cache-Control: max-age=31536000, immutable (static) or no-store (API).
  • Origin Shield: Additional caching layer between edge and origin — reduces origin load by 60–90%.
  • Lambda@Edge / CloudFront Functions: Run code at the edge — A/B testing, request rewrites, auth header injection.
  • DDoS protection: AWS Shield Standard is included with CloudFront at no extra cost.
  • Alternatives: Cloudflare, Fastly, Azure CDN, Akamai — evaluate on PoP count, DDoS protection, and edge compute capability.
# CloudFront distribution with S3 origin + ALB origin (Terraform)
resource "aws_cloudfront_distribution" "main" {
  enabled             = true
  default_root_object = "index.html"
  aliases             = ["app.example.com"]

  # Static assets from S3
  origin {
    domain_name            = aws_s3_bucket.assets.bucket_regional_domain_name
    origin_id              = "s3-assets"
    origin_access_control_id = aws_cloudfront_origin_access_control.main.id
  }

  # API from ALB
  origin {
    domain_name = aws_lb.main.dns_name
    origin_id   = "alb-api"
    custom_origin_config {
      http_port              = 80
      https_port             = 443
      origin_protocol_policy = "https-only"
      origin_ssl_protocols   = ["TLSv1.2"]
    }
  }

  # Cache static assets for 1 year
  default_cache_behavior {
    target_origin_id       = "s3-assets"
    viewer_protocol_policy = "redirect-to-https"
    compress               = true
    cached_methods         = ["GET", "HEAD"]
    allowed_methods        = ["GET", "HEAD"]
    cache_policy_id        = "658327ea-f89d-4fab-a63d-7e88639e58f6" # Caching Optimized
  }

  # Forward /api/* to ALB with no caching
  ordered_cache_behavior {
    path_pattern           = "/api/*"
    target_origin_id       = "alb-api"
    viewer_protocol_policy = "https-only"
    allowed_methods        = ["GET", "HEAD", "OPTIONS", "PUT", "POST", "PATCH", "DELETE"]
    cached_methods         = ["GET", "HEAD"]
    cache_policy_id        = "4135ea2d-6df8-44a3-9df3-4b5a84be39ad" # CachingDisabled
    origin_request_policy_id = "b689b0a8-53d0-40ab-baf2-68738e2966ac" # AllViewer
  }

  viewer_certificate {
    acm_certificate_arn      = aws_acm_certificate.cert.arn
    ssl_support_method       = "sni-only"
    minimum_protocol_version = "TLSv1.2_2021"
  }
}

🔗 5.6 VPC Peering and Transit Gateway

As organisations grow, they create multiple VPCs for isolation — by environment, team, or compliance. Connecting them requires a connectivity strategy.

VPC Peering (point-to-point): VPC-A ←──────▶ VPC-B VPC-A ←──────▶ VPC-C VPC-B ←──────▶ VPC-C (3 peerings for 3 VPCs — grows as N²) Transit Gateway (hub-and-spoke): ┌──────────────┐ VPC-A ──────▶│ │◀────── VPC-D VPC-B ──────▶│ Transit │◀────── On-premises VPC-C ──────▶│ Gateway │◀────── VPN └──────────────┘ (1 attachment per VPC — scales linearly)
  • VPC Peering: Direct, private connection between two VPCs (same or different accounts/regions). Non-transitive — traffic cannot pass through a peer to reach a third VPC. Best for simple, bilateral connectivity.
  • Transit Gateway (TGW): A regional network hub that connects VPCs, VPNs, and Direct Connect in a hub-and-spoke model. Transitive routing — attach once, reach all. Supports route tables per attachment for segmentation.
  • PrivateLink: Expose a service from your VPC as an endpoint that other VPCs consume privately, without peering or internet. Used by AWS services and for ISV integrations.
  • Resource Access Manager (RAM): Share Transit Gateway, subnets, and other resources across AWS accounts in an Organization.
# Transit Gateway with shared route table (Terraform)
resource "aws_ec2_transit_gateway" "main" {
  description                     = "Central network hub"
  default_route_table_association = "disable"
  default_route_table_propagation = "disable"
  auto_accept_shared_attachments  = "enable"
  tags = { Name = "prod-tgw" }
}

resource "aws_ec2_transit_gateway_vpc_attachment" "app" {
  subnet_ids         = module.vpc_app.private_subnets
  transit_gateway_id = aws_ec2_transit_gateway.main.id
  vpc_id             = module.vpc_app.vpc_id
  tags = { Name = "app-vpc-attachment" }
}

resource "aws_ec2_transit_gateway_route" "default" {
  destination_cidr_block         = "0.0.0.0/0"
  transit_gateway_route_table_id = aws_ec2_transit_gateway_route_table.egress.id
  transit_gateway_attachment_id  = aws_ec2_transit_gateway_vpc_attachment.egress.id
}

🏢 5.7 VPN and Hybrid Networks

Hybrid networking connects on-premises data centres to cloud VPCs. There are two main approaches, chosen based on bandwidth requirements and latency sensitivity.

🔐 Site-to-Site VPN

Encrypted IPSec tunnel over the public internet. Quick to set up (~minutes). Up to 1.25 Gbps per tunnel. Two redundant tunnels per connection for HA. Use when bandwidth is moderate and setup speed matters.

⚡ AWS Direct Connect

Dedicated private fibre connection (1, 10, or 100 Gbps). Bypasses the public internet entirely. Consistent latency — no network jitter. Required for latency-sensitive workloads, large data transfers, and regulated industries.

# Site-to-Site VPN — two redundant tunnels (Terraform)
resource "aws_customer_gateway" "onprem" {
  bgp_asn    = 65000
  ip_address = "203.0.113.1"  # on-premises router public IP
  type       = "ipsec.1"
  tags       = { Name = "onprem-cgw" }
}

resource "aws_vpn_connection" "main" {
  customer_gateway_id   = aws_customer_gateway.onprem.id
  transit_gateway_id    = aws_ec2_transit_gateway.main.id
  type                  = "ipsec.1"
  static_routes_only    = false  # use BGP dynamic routing

  tunnel1_preshared_key = var.vpn_psk_1
  tunnel2_preshared_key = var.vpn_psk_2

  tags = { Name = "prod-vpn" }
}
BGP (Border Gateway Protocol): Dynamic routing protocol used with both VPN and Direct Connect. Automatically propagates route changes between on-premises and cloud networks without manual static route management.

🔍 5.8 VPC Flow Logs and Network Observability

VPC Flow Logs capture metadata about all IP traffic flowing through network interfaces in your VPC. Essential for security analysis, troubleshooting connectivity, and compliance.

# Enable VPC Flow Logs to S3 with Athena-friendly format (Terraform)
resource "aws_flow_log" "main" {
  vpc_id          = module.vpc.vpc_id
  traffic_type    = "ALL"       # ACCEPT | REJECT | ALL
  iam_role_arn    = aws_iam_role.flow_log.arn
  log_destination = aws_s3_bucket.flow_logs.arn
  log_destination_type = "s3"

  destination_options {
    file_format                = "parquet"   # efficient for Athena queries
    hive_compatible_partitions = true
    per_hour_partition         = true
  }
}
  • Query with Athena: Use AWS Athena to query flow logs in S3 — find top talkers, rejected traffic, unusual ports.
  • GuardDuty: Consumes VPC Flow Logs automatically to detect port scanning, exfiltration, and lateral movement.
  • Network Reachability Analyser: Verifies that a path between two endpoints is actually reachable given your security group and route table configs.

🕸️ 5.9 Service Mesh Deep Dive — Architecture

A service mesh adds a dedicated network infrastructure layer that handles all service-to-service communication. It is typically implemented as a sidecar proxy injected alongside every service instance.

Without service mesh: [Service A] ────HTTP────▶ [Service B] (app code handles retries, timeouts, auth) With service mesh (sidecar pattern): [Service A]──[Proxy A]──mTLS──[Proxy B]──[Service B] │ │ └──── Control Plane (config, certs, telemetry) App code is unmodified. Proxy handles: → Encryption (mTLS) → Retries / timeouts → Load balancing → Circuit breaking → Traffic splitting → Distributed tracing → Access policies → Golden metrics
The data plane (the proxies) handles actual traffic. The control plane configures the proxies centrally — pushing certificates, routing rules, and telemetry configuration without restarting services.

🔷 5.10 Envoy Proxy — The Universal Data Plane

Envoy is an open-source, high-performance proxy written in C++ originally built at Lyft. It is now the de facto standard data plane for service meshes, API gateways, and edge proxies.

  • L3/L4 and L7: Envoy understands HTTP/1.1, HTTP/2, gRPC, WebSocket, MongoDB, Redis, and more at the application layer.
  • xDS API: A standard control plane API (Listener Discovery Service, Route Discovery Service, Cluster Discovery Service) — any control plane that implements xDS can configure Envoy.
  • Observability: Built-in metrics (Prometheus format), access logs, and trace propagation (Zipkin, Jaeger, OpenTelemetry).
  • Extensibility: Lua scripts and WebAssembly (WASM) plugins — extend Envoy without recompilation.
  • Used by: Istio (sidecar), AWS App Mesh, Contour (Ingress), Envoy Gateway, Kong Gateway.
# Envoy static configuration example
static_resources:
  listeners:
  - name: app_listener
    address:
      socket_address: { address: 0.0.0.0, port_value: 8080 }
    filter_chains:
    - filters:
      - name: envoy.filters.network.http_connection_manager
        typed_config:
          "@type": type.googleapis.com/envoy.extensions.filters.network.http_connection_manager.v3.HttpConnectionManager
          stat_prefix: ingress_http
          route_config:
            virtual_hosts:
            - name: backend
              domains: ["*"]
              routes:
              - match: { prefix: "/api/" }
                route:
                  cluster: api_service
                  timeout: 5s
                  retry_policy:
                    retry_on: "5xx,connect-failure"
                    num_retries: 3
          http_filters:
          - name: envoy.filters.http.router

  clusters:
  - name: api_service
    connect_timeout: 1s
    lb_policy: ROUND_ROBIN
    load_assignment:
      cluster_name: api_service
      endpoints:
      - lb_endpoints:
        - endpoint:
            address:
              socket_address: { address: api-svc, port_value: 8080 }

🔗 5.11 Linkerd — Ultralight Service Mesh

Linkerd is a CNCF-graduated service mesh that prioritises simplicity, reliability, and minimal resource overhead. Unlike Istio, which uses Envoy, Linkerd uses its own purpose-built micro-proxy written in Rust.

  • Zero-config mTLS: Automatic mutual TLS for all meshed traffic with no configuration required. Certificate rotation is handled automatically by Linkerd.
  • Golden metrics — automatic: Success rate, request rate, and p50/p99/p999 latency for every service, deployment, and route — without any instrumentation.
  • HTTP/2 and gRPC native: Linkerd understands gRPC streams and reports per-route metrics natively.
  • Lightweight: The linkerd2-proxy adds ~10ms p99 latency and uses ~10MB RSS per pod — significantly less than Envoy-based meshes.
  • Traffic splitting (SMI): Weighted traffic splitting using the Service Mesh Interface (SMI) spec — no Istio-style CRD complexity.
  • Multi-cluster: Linkerd supports service mirroring across clusters — services in Cluster A are accessible from Cluster B using standard Kubernetes DNS.
# Install Linkerd
linkerd install --crds | kubectl apply -f -
linkerd install | kubectl apply -f -
linkerd check

# Install Viz extension (dashboard + metrics)
linkerd viz install | kubectl apply -f -
linkerd viz dashboard &

# Inject Linkerd into a namespace (all new pods get the proxy)
kubectl annotate namespace production linkerd.io/inject=enabled

# Or inject into an existing deployment
kubectl get deploy -n production -o yaml \
  | linkerd inject - \
  | kubectl apply -f -

# View per-service golden metrics
linkerd viz stat deployments -n production
linkerd viz stat deploy/checkout -n production

# Live traffic dashboard for a deployment
linkerd viz top deployment/checkout -n production

# Routes-level metrics (per HTTP path)
linkerd viz routes deployment/checkout -n production
# Linkerd traffic split — canary (SMI spec)
apiVersion: split.smi-spec.io/v1alpha2
kind: TrafficSplit
metadata:
  name: checkout-split
  namespace: production
spec:
  service: checkout
  backends:
  - service: checkout-v1
    weight: 900m   # 90%
  - service: checkout-v2
    weight: 100m   # 10%

⚡ 5.12 eBPF — The Next Generation Data Plane

eBPF (extended Berkeley Packet Filter) allows running sandboxed programs in the Linux kernel without changing kernel source code. It is revolutionising cloud-native networking.

  • Cilium: Kubernetes CNI (network plugin) built entirely on eBPF. Replaces kube-proxy, implements NetworkPolicies, and — with Cilium Service Mesh — can provide mesh functionality without any sidecar proxies.
  • No sidecar: eBPF-based meshes run at the kernel level — no proxy injection, no pod restart, lower overhead than sidecar-based approaches.
  • Hubble: Cilium's observability layer — real-time network flow visualisation, DNS monitoring, and connection tracking at the kernel level.
  • Tetragon: Cilium's security enforcement component — eBPF-based runtime threat detection and policy enforcement.
  • Sidecar vs eBPF: Sidecar proxies (Envoy, linkerd2-proxy) are mature and portable. eBPF requires Linux kernel 5.8+ but offers better performance and no pod restarts on mesh adoption.

🌐 5.13 Global Load Balancing and Anycast

  • AWS Global Accelerator: Uses the AWS global backbone (not the public internet) to route users to the optimal regional endpoint. Provides static Anycast IPs — users always connect to the nearest AWS edge, then traffic travels the fast private AWS network to your origin.
  • Anycast routing: The same IP address is announced from multiple locations. Users are automatically routed to the nearest one via BGP. Used by all major CDNs and DNS providers.
  • Azure Front Door / Google Cloud Load Balancing: Equivalent global load balancing services with integrated WAF and CDN.
  • Latency vs consistency: Global routing improves latency but introduces complexity — watch for split-brain in stateful systems.

🔒 5.14 Network Security Deep Dive

  • AWS Network Firewall: Stateful, managed firewall with Suricata-compatible IPS rules. Deploy in each VPC or centralise behind a TGW inspection VPC.
  • AWS WAF: Layer 7 firewall — block SQL injection, XSS, bad bots, geographic restrictions. Attach to CloudFront, ALB, or API Gateway.
  • AWS Shield Advanced: Enhanced DDoS protection with 24/7 DDoS Response Team, cost protection, and AWS WAF included.
  • Egress filtering: Use NAT Gateway + Network Firewall or a proxy (Squid, AWS Verified Access) to control and log outbound internet traffic from private subnets.
  • DNS Firewall (Route 53 Resolver): Block DNS queries to known malicious domains — first line of defence against C2 communication and data exfiltration.

resource "aws_route53_resolver_firewall_domain_list" "blocklist" {
