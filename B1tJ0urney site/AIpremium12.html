The Data Intelligence Stack

Big Data, Machine Learning, and Deep Learning are three distinct but deeply connected disciplines. Together they form the full pipeline from raw data at scale to intelligent predictions and decisions.

BIG DATA
Store, move & process data too large for a single machine
MACHINE LEARNING
Learn patterns from structured data without explicit rules
DEEP LEARNING
Learn hierarchical representations from raw data via neural networks

This module covers each layer in depth — when to use each, how they work, and how they connect.

How the three layers connect in a real production AI system:

1
Raw data arrives at scale — petabytes of clickstream logs, sensor readings, transactions land in S3 or HDFS. This is the Big Data layer: distributed storage and processing tools (Spark, Kafka, Flink) handle ingestion, cleaning, and feature extraction at scale.
2
Features are extracted and structured — Spark ETL pipelines transform raw events into clean feature tables. For tabular data (customer churn, fraud detection, pricing), this feeds directly into classical ML models (XGBoost, Random Forest) via Scikit-learn or Spark MLlib.
3
Unstructured data goes to Deep Learning — images, audio, free text, and raw sequences bypass manual feature engineering. A CNN, Transformer, or fine-tuned LLM learns representations directly from the raw input — but needs the Big Data infrastructure to handle training data at scale.
4
Models are served in production — trained models (PyTorch, XGBoost) are deployed as REST APIs, embedded in applications, or run as batch jobs. MLOps tools (MLflow, SageMaker, Vertex AI) track versions, monitor drift, and automate retraining pipelines.
🗄 Big Data

What Is Big Data?

Big Data refers to datasets too large, fast-moving, or complex to be processed by a single machine with traditional tools. It is characterized by the 4 Vs:

  • Volume — petabytes of data from logs, sensors, transactions
  • Velocity — real-time streams (clickstreams, IoT, social feeds)
  • Variety — structured tables, unstructured text, images, video
  • Veracity — data quality and trustworthiness challenges

A single company like Netflix processes petabytes of data per day. No single server can store or analyze this — distributed computing systems are required.

What "too big for one machine" actually means in practice:

Small Data
Fits in RAM (< 16GB)
pandas, NumPy, SQLite. A laptop can process in seconds. This covers most analytics tasks at small companies, personal projects, and ML prototyping. No distributed tools needed.
Medium Data
Fits on one disk (16GB–1TB)
PostgreSQL, DuckDB, Polars. A single powerful server handles it. Polars can process 100GB CSV files faster than Spark on a laptop. Many "Big Data" tools are overkill here.
Big Data
Multi-machine required (> 1TB)
Apache Spark, Flink, Kafka. Data is distributed across dozens or hundreds of nodes. Processing requires coordination. A full Spark cluster query on 10TB takes minutes — on one machine it would take days.
Massive Data
Petabyte scale (1PB+)
Cloud-native: BigQuery, Snowflake, Databricks. Managed services handle thousands of nodes automatically. Google processes ~1 exabyte/day. This scale requires dedicated data engineering teams and specialized tooling.
🗄 Big Data

The Hadoop Ecosystem

Hadoop was the first major open-source framework for distributed storage and processing. Its key components:

  • HDFS — splits files into 128MB blocks across many machines with replication for fault tolerance
  • MapReduce — Map (transform each record in parallel), Reduce (aggregate results). Slow for iterative tasks.
  • YARN — the resource manager that schedules jobs across the cluster
  • Hive — SQL-like queries over HDFS data
  • HBase — a NoSQL key-value store on top of HDFS for real-time reads/writes

Hadoop is largely legacy today — Spark has replaced MapReduce for most workloads. But HDFS concepts remain foundational.

Why MapReduce was slow — and what Spark did differently:

MapReduce Problem
Disk I/O Between Every Stage
After each Map and each Reduce operation, results were written to HDFS (disk). An ML training loop with 50 iterations would read/write disk 100 times. Disk is 100× slower than RAM — making iterative algorithms painfully slow.
Spark Solution
In-Memory RDDs
Spark's Resilient Distributed Datasets (RDDs) keep data in RAM across operations. After the initial read, subsequent transformations operate on in-memory data. 50 training iterations = 1 disk read, 49 in-memory passes. Up to 100× faster.
Spark Optimization
Lazy Evaluation & Query Plans
Spark doesn't execute transformations immediately — it builds a Directed Acyclic Graph (DAG) of operations. When an action (like .show()) is called, Spark optimizes the whole plan before running, combining stages and pushing filters down to reduce data movement.
When Hadoop Still Wins
Sequential Batch on Cold Storage
For one-time batch scans of cold data that doesn't need to be cached, HDFS + Hive is often cheaper than Spark (no cluster RAM required). Many companies keep HDFS for archival storage while using Spark for active processing.

Apache Spark — The Modern Standard

Apache Spark is the dominant Big Data processing engine. It keeps data in memory between operations — up to 100× faster than Hadoop MapReduce for iterative workloads like ML training.

from pyspark.sql import SparkSession
from pyspark.sql.functions import col, avg, count

spark = SparkSession.builder \
    .appName("SalesAnalysis") \
    .getOrCreate()

df = spark.read.csv("s3://data-lake/sales/*.csv", header=True, inferSchema=True)

result = df \
    .filter(col("year") == 2024) \
    .groupBy("region") \
    .agg(
        avg("revenue").alias("avg_revenue"),
        count("*").alias("num_transactions")
    ) \
    .orderBy(col("avg_revenue").desc())

result.show(20)

Production Spark patterns — optimizations used in real data engineering:

from pyspark.sql import SparkSession
from pyspark.sql.functions import col, to_date, year, month, sum as spark_sum
from pyspark.sql.window import Window
import pyspark.sql.functions as F

spark = SparkSession.builder \
    .appName("ProductionPipeline") \
    .config("spark.sql.adaptive.enabled", "true")       # auto-optimize query plan
    .config("spark.sql.shuffle.partitions", "200")       # tune for cluster size
    .getOrCreate()

# ── Pattern 1: Partition pruning — only read relevant data ────────────
transactions = spark.read.parquet("s3://datalake/transactions/") \
    .filter(col("year") == 2024)   # Parquet pushdown: reads only 2024 partitions

# ── Pattern 2: Broadcast join — avoid expensive shuffle for small tables
products = spark.read.parquet("s3://datalake/products/")   # small table (< 200MB)
joined = transactions.join(F.broadcast(products), "product_id")

# ── Pattern 3: Cache frequently reused DataFrames ─────────────────────
user_features = joined \
    .groupBy("user_id") \
    .agg(spark_sum("revenue").alias("total_spend"))
user_features.cache()    # keep in memory for multiple downstream uses

# ── Pattern 4: Window functions for time-series analytics ─────────────
window = Window.partitionBy("user_id").orderBy("event_date").rowsBetween(-6, 0)
transactions_with_rolling = transactions.withColumn(
    "rolling_7d_revenue",
    spark_sum("revenue").over(window)
)

# ── Pattern 5: Write optimized Parquet partitioned by date ────────────
user_features.write \
    .mode("overwrite") \
    .partitionBy("year", "month") \
    .parquet("s3://datalake/user_features_monthly/")
⚡ Spark Performance Quick Reference

Use Parquet, not CSV — Parquet is columnar: reading 3 of 100 columns reads 3% of the data. CSV reads everything.
Broadcast small tables — tables under ~200MB should be broadcast to avoid shuffle in joins.
Avoid .collect() — pulls all data to the driver; crashes on big datasets. Use .show(), .take(n), or write to storage instead.
Tune partitions — default 200 shuffle partitions is too many for small data. Aim for ~128MB per partition.

🗄 Big Data

Modern Data Architecture: Lakes & Warehouses

Real production systems separate raw storage from analytics-ready data:

  • Data Lake — stores raw, unprocessed data in any format (S3, Azure Data Lake, GCS). Cheap, flexible, schema-on-read.
  • Data Warehouse — structured, cleaned, analytics-ready (Snowflake, BigQuery, Redshift). Fast, schema-on-write.
  • Data Lakehouse — combines both: lake-scale storage with warehouse query performance (Delta Lake, Apache Iceberg).
  • ETL / ELT pipelines — Extract, Transform, Load between layers (Apache Airflow, dbt, Fivetran).

Most modern AI companies use a Lakehouse architecture: raw data lands in S3, processed by Spark, served as Delta Lake tables to data scientists and ML pipelines.

Full Lakehouse architecture — layers from raw ingestion to ML training:

LayerNameWhat It StoresFormatTools
BronzeRaw / LandingUnmodified source data — exactly as received from APIs, DBs, streamsJSON, CSV, Avro, raw logsKafka, Fivetran, S3
SilverCleaned / EnrichedDeduplicated, typed, joined. Schema enforced. Some business logic applied.Parquet, Delta LakeSpark, dbt
GoldBusiness / ML-ReadyAggregated metrics, feature tables, curated datasets for BI and model trainingDelta Lake, Parquetdbt, Spark, Feature Store
ServingApplication LayerPre-computed results for dashboards; real-time features for online inferenceSnowflake, BigQuery, RedisFeast, Tecton, Redshift
🏗️ Medallion Architecture (Bronze → Silver → Gold)

Popularized by Databricks, this pattern ensures every transformation is tracked and reversible. Raw data in Bronze is never modified — you can always re-derive Silver from Bronze if a bug is discovered. Delta Lake adds ACID transactions to S3, enabling time-travel (query data as it was 30 days ago), schema evolution, and audit logs — features previously only available in traditional databases.

🤖 Machine Learning

What Is Machine Learning?

Machine Learning is a method of building systems that learn patterns from data — without being explicitly programmed with rules.

TypeHow it learnsExamples
SupervisedFrom labeled input-output pairsSpam detection, price prediction, image classification
UnsupervisedFrom unlabeled data — finds structureCustomer segmentation, anomaly detection, topic modeling
ReinforcementFrom rewards and penalties via trial and errorGame playing, robotics, RLHF for LLMs

Choosing the right ML type — a practical decision framework:

Supervised
You have labeled data
The most common case. You have historical examples where you know the right answer: past emails labeled spam/not-spam, past transactions labeled fraud/legitimate, past house sales with prices. Key question: how much labeled data? Under 1K: try classical ML. Over 100K with raw input: consider DL.
Unsupervised
You have unlabeled data
Labeling is expensive or impossible. K-Means finds natural customer groups without you defining them. PCA reduces 100 features to 10 without telling it which matter. DBSCAN detects anomalies without labeled examples of what an anomaly looks like. Common in exploratory analysis.
Self-Supervised
Labels come from the data itself
The dominant paradigm for LLMs and vision models. Next-token prediction, masked word prediction (BERT), and contrastive image-text pairs (CLIP) — the model creates its own supervision signal from unlabeled data. Allows training on internet-scale data with no human labeling.
Reinforcement
You can define a reward signal
The agent learns by trial and error. RLHF (Reinforcement Learning from Human Feedback) is how Claude and GPT-4 are aligned — humans rate responses and a reward model is trained to predict these ratings, then the LLM is optimized against it.

The ML Workflow

Every ML project follows the same core pipeline:

  • 1. Data Collection — gather raw data from databases, APIs, files
  • 2. EDA — understand distributions, correlations, outliers
  • 3. Feature Engineering — create, transform, and select the variables that matter
  • 4. Train/Test Split — separate data to evaluate on unseen examples
  • 5. Model Training — fit the algorithm on training data
  • 6. Evaluation — measure performance (accuracy, F1, RMSE, AUC-ROC)
  • 7. Hyperparameter Tuning — optimize model configuration (GridSearchCV, Optuna)
  • 8. Deployment — serve predictions via API, batch job, or embedded model

Production ML workflow — beyond the basics, what actually matters in practice:

import pandas as pd
import numpy as np
from sklearn.pipeline import Pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.impute import SimpleImputer
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.model_selection import StratifiedKFold, cross_validate
from sklearn.metrics import roc_auc_score
import optuna
import mlflow

# ── Step 1: Robust preprocessing pipeline (prevents data leakage) ─────
numeric_features  = ['age', 'revenue', 'days_since_last_order']
categorical_features = ['country', 'product_category', 'channel']

numeric_transformer = Pipeline([
    ('imputer', SimpleImputer(strategy='median')),  # fill missing values
    ('scaler', StandardScaler())
])

categorical_transformer = Pipeline([
    ('imputer', SimpleImputer(strategy='most_frequent')),
    ('onehot', OneHotEncoder(handle_unknown='ignore', sparse_output=False))
])

preprocessor = ColumnTransformer([
    ('num', numeric_transformer, numeric_features),
    ('cat', categorical_transformer, categorical_features)
])

# ── Step 2: Full pipeline — preprocessing + model in one object ────────
# This prevents data leakage: scaler is fit only on training folds
full_pipeline = Pipeline([
    ('prep', preprocessor),
    ('model', GradientBoostingClassifier(n_estimators=200))
])

# ── Step 3: Stratified CV for reliable evaluation ─────────────────────
cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
cv_results = cross_validate(
    full_pipeline, X, y, cv=cv,
    scoring={'auc': 'roc_auc', 'f1': 'f1', 'precision': 'precision', 'recall': 'recall'},
    return_train_score=True    # detect overfitting: train >> test means overfitting
)
print(f"AUC: {cv_results['test_auc'].mean():.4f} ± {cv_results['test_auc'].std():.4f}")

# ── Step 4: Hyperparameter tuning with Optuna ─────────────────────────
def objective(trial):
    params = {
        'model__n_estimators':  trial.suggest_int('n_estimators', 100, 500),
        'model__max_depth':     trial.suggest_int('max_depth', 2, 6),
        'model__learning_rate': trial.suggest_float('learning_rate', 0.01, 0.3, log=True),
        'model__subsample':     trial.suggest_float('subsample', 0.6, 1.0),
    }
    full_pipeline.set_params(**params)
    return cross_val_score(full_pipeline, X_train, y_train, cv=3, scoring='roc_auc').mean()

study = optuna.create_study(direction='maximize')
study.optimize(objective, n_trials=50)
print(f"Best AUC: {study.best_value:.4f}")

# ── Step 5: MLflow tracking — log everything ──────────────────────────
with mlflow.start_run():
    mlflow.log_params(study.best_params)
    mlflow.log_metric("cv_auc", cv_results['test_auc'].mean())
    mlflow.sklearn.log_model(full_pipeline, "model")

Algorithm selection guide — a practical framework for choosing the right model:

Start Here
Always Baseline First
Before anything else, train a Logistic Regression or Linear Regression. It sets a performance floor, reveals whether your features have signal, and is fast to debug. If your fancy model can't beat the linear baseline, something is wrong with your data, not the model.
Tabular Data
Go Gradient Boosting
LightGBM is typically the best default for tabular classification and regression. It's faster than XGBoost, handles missing values natively, and requires minimal preprocessing. Use CatBoost when you have many high-cardinality categorical features (e.g. country, user_id).
Interpretability
Use SHAP for Any Model
Don't sacrifice accuracy for a simpler model just for explainability. Use SHAP (SHapley Additive exPlanations) to explain XGBoost or Random Forest predictions: shap.TreeExplainer(model) shows which features drove each individual prediction — even for complex ensembles.
Small Dataset
SVM or Regularized Linear
With under 1,000 labeled examples, tree ensembles can overfit. SVMs with RBF kernels generalize well in high-dimensional spaces with small data. Ridge/Lasso regression enforces strong regularization. Never use deep learning with <1K examples without a pre-trained backbone.
🤖 Machine Learning

Overfitting, Underfitting & Regularization

The most important challenge in ML is building models that generalize — performing well on new, unseen data, not just training data.

  • Underfitting — model too simple; high error on both train and test. Fix: more features, more complex model.
  • Overfitting — model memorizes training data; low train error, high test error. Fix: regularization, more data, simpler model.
  • L1 (Lasso) — penalty proportional to |weights|; drives irrelevant features to zero
  • L2 (Ridge) — penalty proportional to weights²; shrinks all weights, keeps all features
  • Dropout — randomly zeros neurons during training
  • Early Stopping — stop training when validation loss stops improving

Diagnosing model problems from learning curves — reading the bias-variance signals:

import numpy as np
import matplotlib.pyplot as plt
from sklearn.model_selection import learning_curve
from sklearn.ensemble import GradientBoostingClassifier

def plot_learning_curve(model, X, y, cv=5):
    """
    Learning curve plots training and validation score vs dataset size.
    Reveals underfitting, overfitting, and whether more data would help.
    """
    train_sizes, train_scores, val_scores = learning_curve(
        model, X, y, cv=cv, n_jobs=-1,
        train_sizes=np.linspace(0.1, 1.0, 10),
        scoring='roc_auc'
    )

    train_mean = train_scores.mean(axis=1)
    val_mean   = val_scores.mean(axis=1)
    gap = train_mean[-1] - val_mean[-1]

    # Diagnose the problem:
    if train_mean[-1] < 0.75:
        print("⚠️  HIGH BIAS (underfitting): both curves low")
        print("   Fix: add features, increase model complexity, remove regularization")
    elif gap > 0.1:
        print("⚠️  HIGH VARIANCE (overfitting): large train-val gap")
        print(f"   Gap: {gap:.3f}")
        print("   Fix: add data, increase regularization, reduce model complexity")
    else:
        print(f"✓ Good fit: train={train_mean[-1]:.3f}, val={val_mean[-1]:.3f}, gap={gap:.3f}")

    return train_mean[-1], val_mean[-1], gap

# ── Cross-validation: the right way to detect data leakage ───────────
# WRONG: fit scaler on all data, then do CV — scaler has seen test data!
# scaler.fit(X_all)  ← data leakage
# cross_val_score(model, scaler.transform(X_all), y)  ← inflated scores

# RIGHT: put scaler inside the pipeline — it only sees training folds
from sklearn.pipeline import Pipeline
pipeline = Pipeline([('scale', StandardScaler()), ('model', GradientBoostingClassifier())])
scores = cross_val_score(pipeline, X, y, cv=StratifiedKFold(5), scoring='roc_auc')
# Now scaler is fit fresh on each training fold ← no leakage
🔍 Data Leakage — The Silent Model Killer

Data leakage occurs when information from the test set contaminates training — producing models that look great in evaluation but fail in production. Common sources: fitting preprocessors on all data before splitting (StandardScaler, imputers); target-correlated features that wouldn't be available at prediction time (e.g. including "claim_paid" to predict "will_claim"); future data in time-series (using Tuesday's data to predict Monday). Always use sklearn Pipeline objects and time-based splits for temporal data.

🧠 Deep Learning

What Is Deep Learning?

Deep Learning uses artificial neural networks with many layers to learn hierarchical representations directly from raw data — without manual feature engineering.

Why "deep"? Layer 1 detects edges → Layer 2 detects shapes → Layer 3 detects objects. The depth enables this hierarchy.

When to use DL over classical ML:

  • Unstructured data: images, audio, raw text
  • Very large datasets (millions+ examples)
  • Tasks where feature engineering is impractical
  • State-of-the-art performance is required

When classical ML is better:

  • Structured/tabular data with <100K rows
  • When interpretability is required
  • Limited compute or training time

Neural Networks & Backpropagation

A neural network is a stack of layers. Each neuron computes a weighted sum and applies an activation function. Training adjusts weights using backpropagation.

import torch
import torch.nn as nn

class FeedforwardNet(nn.Module):
    def __init__(self, input_dim, hidden_dim, output_dim):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(input_dim, hidden_dim),
            nn.ReLU(),
            nn.Dropout(0.3),
            nn.Linear(hidden_dim, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, output_dim)
        )

    def forward(self, x):
        return self.net(x)

model     = FeedforwardNet(128, 256, 10)
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
criterion = nn.CrossEntropyLoss()

for epoch in range(50):
    for X_batch, y_batch in train_loader:
        optimizer.zero_grad()
        output = model(X_batch)
        loss   = criterion(output, y_batch)
        loss.backward()
        optimizer.step()

Production PyTorch training loop — what a real training script looks like:

import torch
import torch.nn as nn
from torch.optim.lr_scheduler import OneCycleLR
from torch.cuda.amp import GradScaler, autocast

def train_model(model, train_loader, val_loader, num_epochs=50):
    device    = torch.device('cuda' if torch.cuda.is_available() else 'cpu')
    model     = model.to(device)
    optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3, weight_decay=1e-4)
    criterion = nn.CrossEntropyLoss(label_smoothing=0.1)  # prevents overconfidence
    scaler    = GradScaler()  # mixed precision: 2x faster on modern GPUs

    # OneCycleLR: warm up then decay — often better than constant lr
    scheduler = OneCycleLR(optimizer, max_lr=1e-3,
                           steps_per_epoch=len(train_loader), epochs=num_epochs)

    best_val_acc, patience_counter, patience = 0, 0, 10

    for epoch in range(num_epochs):
        # ── Training phase ──────────────────────────────────────────
        model.train()
        train_loss = 0
        for X, y in train_loader:
            X, y = X.to(device), y.to(device)
            optimizer.zero_grad()

            with autocast():                   # float16 forward pass
                output = model(X)
                loss   = criterion(output, y)

            scaler.scale(loss).backward()      # scale gradients for float16
            scaler.unscale_(optimizer)
            torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)  # prevent exploding gradients
            scaler.step(optimizer)
            scaler.update()
            scheduler.step()
            train_loss += loss.item()

        # ── Validation phase ────────────────────────────────────────
        model.eval()
        correct = 0
        with torch.no_grad():
            for X, y in val_loader:
                X, y = X.to(device), y.to(device)
                preds = model(X).argmax(dim=1)
                correct += (preds == y).sum().item()

        val_acc = correct / len(val_loader.dataset)
        print(f"Epoch {epoch+1:3d} | loss: {train_loss/len(train_loader):.4f} | val_acc: {val_acc:.4f}")

        # ── Early stopping ──────────────────────────────────────────
        if val_acc > best_val_acc:
            best_val_acc = val_acc
            torch.save(model.state_dict(), 'best_model.pt')
            patience_counter = 0
        else:
            patience_counter += 1
            if patience_counter >= patience:
                print(f"Early stopping at epoch {epoch+1}")
                break

    model.load_state_dict(torch.load('best_model.pt'))
    return model
🧠 Deep Learning

Convolutional Neural Networks (CNNs)

CNNs are specialized for spatial data. Convolutional filters slide across the input, detecting local patterns regardless of position.

  • Convolution layer — applies learned filters to detect edges, textures, shapes
  • Pooling layer — reduces spatial dimensions (MaxPool, AvgPool)
  • Fully connected layer — combines features for final classification
class SimpleCNN(nn.Module):
    def __init__(self, num_classes=10):
        super().__init__()
        self.features = nn.Sequential(
            nn.Conv2d(3, 32, kernel_size=3, padding=1),
            nn.ReLU(), nn.MaxPool2d(2),
            nn.Conv2d(32, 64, kernel_size=3, padding=1),
            nn.ReLU(), nn.MaxPool2d(2)
        )
        self.classifier = nn.Linear(64 * 8 * 8, num_classes)

    def forward(self, x):
        return self.classifier(self.features(x).flatten(1))

Most practitioners use transfer learning — fine-tuning a pre-trained ImageNet model rather than training from scratch.

Transfer learning with a pre-trained CNN — the production approach for image tasks:

import torch
import torch.nn as nn
from torchvision import models, transforms
from torch.utils.data import DataLoader

# ── Load pre-trained ResNet50 (trained on 1.2M ImageNet images) ────────
backbone = models.resnet50(weights=models.ResNet50_Weights.DEFAULT)

# ── Strategy A: Feature extraction — freeze backbone, train only head ──
# Best when: your dataset is small (<1K images) or very different from ImageNet
for param in backbone.parameters():
    param.requires_grad = False

backbone.fc = nn.Sequential(
    nn.Linear(2048, 512),
    nn.ReLU(),
    nn.Dropout(0.3),
    nn.Linear(512, num_classes)    # only this head is trainable
)

# ── Strategy B: Fine-tuning — unfreeze later layers, use small lr ──────
# Best when: your dataset is medium-large or similar to ImageNet
for param in backbone.layer4.parameters():   # unfreeze last ResNet block

🧠 Deep Learning

Recurrent Networks & Transformers

RNNs / LSTMs process sequential data by maintaining a hidden state across time steps. They struggle with long-range dependencies.

Transformers replaced RNNs for most sequential tasks. Self-attention — every token attends to every other token in parallel — enables long-range dependencies and massive parallelism.

All modern LLMs (GPT-4, Claude, Gemini) are Transformer-based. BERT, T5, Whisper, DALL-E, Stable Diffusion — all Transformers. The most important architecture in AI today.

  • Encoder-only (BERT) — understanding tasks: classification, NER, embeddings
  • Decoder-only (GPT) — generation: text completion, chat, code
  • Encoder-decoder (T5, BART) — seq2seq: translation, summarization

Self-attention explained — the core mechanism that powers all Transformers:

import torch
import torch.nn.functional as F
import math

def scaled_dot_product_attention(Q, K, V, mask=None):
    """
    The core operation in every Transformer.
    
    Q (Query):  what this token is looking for
    K (Key):    what each token offers as a match
    V (Value):  what content is passed if the token is attended to
    
    Each token computes attention scores against ALL other tokens,
    then takes a weighted sum of their values.
    """
    d_k = Q.shape[-1]   # dimension of key vectors (for scaling)

    # Step 1: Compute raw attention scores — how much does Q "match" each K?
    scores = torch.matmul(Q, K.transpose(-2, -1))  # [batch, heads, seq, seq]

    # Step 2: Scale by sqrt(d_k) to prevent vanishing gradients in softmax
    scores = scores / math.sqrt(d_k)

    # Step 3: Mask future tokens (decoder only) — can't attend to the future
    if mask is not None:
        scores = scores.masked_fill(mask == 0, float('-inf'))

    # Step 4: Softmax → attention weights sum to 1 for each query position
    weights = F.softmax(scores, dim=-1)   # [batch, heads, seq_len, seq_len]

    # Step 5: Weighted sum of value vectors
    output = torch.matmul(weights, V)     # [batch, heads, seq_len, d_v]

    return output, weights

# Interpretation of weights[i][j]:
#   How much does token i attend to token j?
#   High weight = token i finds token j highly relevant
#   In "The cat sat on the mat", "sat" attends strongly to "cat" (subject) and "mat" (location)

# Multi-head attention: run this attention H times in parallel with different Q,K,V projections
# Each head learns to attend to different types of relationships
# Head 1: syntactic (subject-verb), Head 2: semantic (entity-property), etc.
🔢 Why Transformers Scale But RNNs Don't

RNNs are sequential: to process token N, you must first process tokens 1 through N-1. This serial dependency means you can't parallelize — training on a GPU that could run 10,000 operations simultaneously instead runs one at a time. Transformers are fully parallel: all tokens are processed simultaneously. Self-attention computes all pairwise relationships at once as a matrix multiplication — a single GPU operation. This is why scaling Transformer training to thousands of GPUs is straightforward, but scaling RNN training is nearly impossible.

🧠 Deep Learning

Transfer Learning & Fine-Tuning

Transfer learning takes a model pre-trained on a large dataset and adapts it to a new task — the dominant paradigm in modern DL.

from transformers import AutoModelForSequenceClassification, AutoTokenizer, TrainingArguments, Trainer

model = AutoModelForSequenceClassification.from_pretrained("bert-base-uncased", num_labels=3)

training_args = TrainingArguments(
    output_dir="./results", num_train_epochs=3,
    per_device_train_batch_size=16, learning_rate=2e-5, evaluation_strategy="epoch"
)

trainer = Trainer(model=model, args=training_args, train_dataset=train_ds, eval_dataset=eval_ds)
trainer.train()
  • Feature extraction — freeze pre-trained weights, only train a new head
  • Full fine-tuning — train all weights on the new task
  • PEFT / LoRA — train only small adapter weights (used for LLMs)

LoRA fine-tuning — how to adapt a 7B parameter LLM with a consumer GPU:

from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments
from peft import LoraConfig, get_peft_model, TaskType
from trl import SFTTrainer

# ── Load the base model ───────────────────────────────────────────────
model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3.2-3B",
    torch_dtype=torch.float16,     # half precision to fit in GPU memory
    device_map="auto"              # auto-distribute across available GPUs
)

# ── LoRA Configuration ────────────────────────────────────────────────
# Instead of updating 3B parameters, LoRA adds small low-rank matrices
# to specific layers. Only these adapters (~8M params) are trained.
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,                          # rank of the low-rank matrices (higher = more capacity)
    lora_alpha=32,                 # scaling factor
    target_modules=["q_proj", "v_proj"],  # which attention matrices to adapt
    lora_dropout=0.1,
    bias="none"
)

# ── Apply LoRA to the model ───────────────────────────────────────────
peft_model = get_peft_model(model, lora_config)
peft_model.print_trainable_parameters()
# Output: "trainable params: 8,388,608 || all params: 3,212,791,808 || trainable%: 0.26%"
# Only 0.26% of parameters are updated — but fine-tuning quality is close to full fine-tuning

# ── Train with SFTTrainer (supervised fine-tuning) ────────────────────
trainer = SFTTrainer(
    model=peft_model,
    train_dataset=formatted_dataset,
    args=TrainingArguments(
        output_dir="./lora_finetuned",
        num_train_epochs=3,
        per_device_train_batch_size=4,
        gradient_accumulation_steps=4,   # effective batch size = 16
        learning_rate=2e-4,
        fp16=True,
        logging_steps=10
    ),
    dataset_text_field="text"
)
trainer.train()

# ── Merge LoRA weights back into base model for deployment ────────────
merged_model = peft_model.merge_and_unload()
merged_model.save_pretrained("./final_model")
💡 When to Fine-Tune vs Prompt Engineering vs RAG

Start with prompt engineering — a well-crafted system prompt with few-shot examples often eliminates the need for fine-tuning entirely. Zero cost, immediate results.
Use RAG when the model needs access to private/recent information that isn't in its training data (company docs, current prices, internal knowledge bases).
Fine-tune (LoRA) when you need to: change the model's writing style or persona permanently, teach it a specialized format or domain language, or improve reliability on a narrow task where prompt engineering is inconsistent. Fine-tuning is not for adding new knowledge — use RAG for that.
