🐧 Why Linux in Cybersecurity?

Linux is the dominant operating system on servers, cloud, IoT devices and critical infrastructure. Most cybersecurity tools — Nmap, Wireshark, Metasploit, Burp Suite — run natively on Linux.

Distributions like Kali Linux and Parrot OS were created specifically for pentesting and come with hundreds of pre-installed tools. Without mastering the command line, you cannot use most of those tools effectively.

File System Structure

The Linux File System

Unlike Windows (C:\, D:\), Linux has a single directory tree starting at / (root). Knowing this structure is essential for locating logs, configuration files and binaries.

/ ← root of the entire file system ├── /bin ← essential binaries (ls, cp, mv, grep...) ├── /etc ← system configuration files │ ├── passwd ← list of users │ ├── shadow ← hashed passwords (root-only read) │ └── ssh/ ← SSH server configuration ├── /var/log ← system logs (CRITICAL for security) │ ├── syslog ← general system log │ ├── auth.log ← login attempts and authentication │ └── nginx/ ← web server logs ├── /home ← users' home folders (/home/john) ├── /root ← root user's home folder ├── /tmp ← temporary files (deleted on reboot) ├── /usr ← user-installed programmes ├── /proc ← information about running processes └── /dev ← hardware devices (disks, terminals)
🚨
/etc/shadow — The most sensitive file Contains the hashed passwords of all users. If an attacker gains access to this file, they can attempt to crack the hashes offline with tools like John the Ripper or Hashcat. That is why only root can read it — always verify with ls -la /etc/shadow.

tail -n 50 — viewing the last lines is more useful than the whole file — the most recent events are at the bottom.

bash — chmod & chown 
# Change permissions with chmod
$ chmod 755 script.sh    # owner: rwx, group: r-x, others: r-x
$ chmod 600 key.pem      # owner: rw-, group: ---, others: --- (SSH keys!)
$ chmod 644 config.txt   # owner: rw-, group: r--, others: r--
$ chmod +x script.sh      # add execute permission
$ chmod -w file.txt       # remove write permission

# Change owner with chown
$ chown john:john file.txt          # owner and group → john
$ chown -R www-data /var/www/html   # recursive: entire web folder

# View detailed permissions
$ ls -la /etc/shadow
-rw-r----- 1 root shadow 1024 Jan 15 auth.log
#          ^ only root and shadow group can read!
⚠️
chmod 777 is a security anti-pattern Granting full permissions to everyone (rwxrwxrwx) is a common beginner mistake. Any compromised user or process can modify or execute those files. Always use the minimum permissions necessary — principle of least privilege.
Users, Groups & Sudo

Root, Sudo and User Management

Linux clearly separates normal users from the superuser root. Root has full access to the system — which is why you should never work directly as root in day-to-day use. sudo allows executing specific commands with root privileges in a controlled way.

bash — users & sudo 
# Check the current user
$ whoami
john
$ id
uid=1000(john) gid=1000(john) groups=1000(john),27(sudo)

# Run command as root with sudo
$ sudo apt update
$ sudo cat /etc/shadow     # view root-only file
$ sudo -l                  # list what you can do with sudo

# User management
$ sudo useradd -m -s /bin/bash new_user    # create user
$ sudo passwd new_user                     # set password
$ sudo usermod -aG sudo new_user            # add to sudo group
$ sudo userdel -r username                  # delete user and home folder

# View system users
$ cat /etc/passwd | grep -v nologin        # users with a shell
$ last                                      # login history
$ who                                       # who is logged in right now
🚨
Privilege Escalation One of the main goals of an attacker after compromising a system is to escalate from a normal user to root. Tools like LinPEAS and linux-exploit-suggester automate the search for escalation vectors: misconfigured SUID files, misconfigured sudo, services running as root, etc.
Search, Grep & Log Analysis

grep, find, pipes — The investigation tools

In cybersecurity, you spend a lot of time analysing logs looking for suspicious patterns. grep, find and pipes (|) are your main tools.

bash — grep 
# grep — search for patterns in text
$ grep "Failed password" /var/log/auth.log          # failed login attempts
$ grep -i "error" /var/log/syslog                   # -i ignores case
$ grep -n "root" /etc/passwd                        # -n shows line number
$ grep -r "password" /var/www/html/                  # -r recursive search
$ grep -v "#" /etc/ssh/sshd_config                  # -v excludes lines with #
$ grep -E "[0-9]{1,3}\.[0-9]{1,3}" access.log        # regex: find IPs

# Pipes — chain commands together
$ cat auth.log | grep "Failed" | wc -l              # count failed attempts
$ cat auth.log | grep "Failed" | awk '{print $11}' | sort | uniq -c | sort -rn
#                                       attacker IP     count unique IPs  sort by frequency
bash — find 
# find — search for files in the system
$ find /home -name "*.txt"                           # find .txt files
$ find / -name "passwd" 2>/dev/null                   # search entire system
$ find / -perm -4000 2>/dev/null                      # SUID files (escalation risk!)
$ find /var/log -mtime -1                             # logs modified in last 24h
$ find /tmp -type f -executable                       # executables in /tmp (suspicious!)
$ find / -user root -writable 2>/dev/null             # root files you can write to
Real auth.log analysis — detecting brute force:

grep "Failed password" auth.log — finds all SSH authentication failure lines.
awk '{print $11}' — extracts only the IP address field (column 11).
sort | uniq -c — counts how many times each IP appears.
sort -rn — sorts from most frequent to least frequent.
Result: you instantly see which IPs are brute-forcing your server.

💡
Most important logs for security /var/log/auth.log — SSH logins, sudo, authentication failures.
/var/log/syslog — general system events.
/var/log/nginx/access.log — HTTP requests to the web server.
/var/log/fail2ban.log — automatically blocked IPs.

SSH — Secure Remote Access

SSH — Secure Shell

SSH is the standard protocol for secure remote access to Linux servers. It completely replaces Telnet (insecure) and is encrypted with TLS.

bash — SSH 
# Connect to a remote server
$ ssh john@192.168.1.10              # with password
$ ssh -i key.pem john@server.com     # with private key (more secure)
$ ssh -p 2222 john@server.com        # custom port

# Generate SSH key pair
$ ssh-keygen -t ed25519 -C "john@company.com"
# Creates: ~/.ssh/id_ed25519 (private) and ~/.ssh/id_ed25519.pub (public)

# Copy public key to server
$ ssh-copy-id john@server.com
# Then you can connect without a password!

# Protect the private key (MANDATORY)
$ chmod 600 ~/.ssh/id_ed25519
$ chmod 700 ~/.ssh/

Disable root loginPermitRootLogin no in /etc/ssh/sshd_configAttackers try root first

Basic Bash Scripting

Automating Tasks with Bash

A Bash script is a text file with commands that are executed in sequence. In cybersecurity, you use scripts to automate log analysis, scans, backups and hardening.

bash — log analysis script 
#!/bin/bash
# analyse_logs.sh — Detects SSH brute force attempts

LOG="/var/log/auth.log"
LIMIT=10          # more than 10 attempts = suspicious

echo "=== SSH Login Attempt Analysis ==="
echo "Date: $(date)"
echo ""

# Count total failures
TOTAL=$(grep "Failed password" $LOG | wc -l)
echo "Total failed attempts: $TOTAL"

# List suspicious IPs
echo ""
echo "IPs with more than $LIMIT attempts:"
grep "Failed password" $LOG \
  | awk '{print $11}' \
  | sort | uniq -c | sort -rn \
  | awk -v lim=$LIMIT '$1 > lim {print "  "$1" attempts - IP: "$2}'

# Check if fail2ban is active
if systemctl is-active --quiet fail2ban; then
    echo ""
    echo "✅ Fail2Ban is active — IPs blocked automatically"
else
    echo "⚠️  Fail2Ban is NOT active — install with: sudo apt install fail2ban"
fi
Script structure:

#!/bin/bash — shebang: tells the system this file is a Bash script.
VARIABLE=value — define variables (no spaces around =).
$VARIABLE — use the value of a variable.
$(command) — command substitution: run a command and use its output.
if ... then ... else ... fi — conditional structure.
systemctl is-active — check whether a service is active.

💡
How to run a Bash script: chmod +x analyse_logs.sh — grant execute permission.
./analyse_logs.sh — run it.
sudo ./analyse_logs.sh — run as root (required to read auth.log).
Basic Linux Hardening

Hardening — Reducing the Attack Surface

Hardening is the process of making a system more resistant to attacks by removing what is unnecessary and correctly configuring what remains.

ActionCommandImpact
Update the systemsudo apt update && sudo apt upgradeHigh — fixes known vulnerabilities
Disable unnecessary servicessudo systemctl disable serviceHigh — reduces attack surface
Configure UFW firewallsudo ufw enable && sudo ufw default denyHigh — blocks unnecessary ports
Install Fail2Bansudo apt install fail2banHigh — auto-blocks brute force
Disable SSH root loginPermitRootLogin no in sshd_configHigh — eliminates main attack vector
Remove unused packagessudo apt autoremoveMedium — less software = fewer vulnerabilities
Check SUID filesfind / -perm -4000 2>/dev/nullMedium — misconfigured SUID → privilege escalation

sudo apt install lynis && sudo lynis audit system
