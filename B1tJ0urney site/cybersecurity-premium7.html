🔐 DevSecOps Fundamentals

DevSecOps integrates Development, Security and Operations into a single continuous flow. Rather than treating security as a final phase, it is embedded from the very first commit all the way through to production.

In this module you will learn the DevSecOps model, the key analysis tools, how to protect CI/CD pipelines, manage secrets securely and apply supply chain security principles.

🔄

1. What is DevSecOps?

DevSecOps is an evolution of the DevOps model that makes security a shared responsibility across the entire team — not just security specialists. The goal is to eliminate the "wall of security" that occurred when projects reached the audit stage already in production.

⚠️
The problem with the traditional model In a Dev → QA → Sec → Ops flow, security issues are discovered too late. Fixing a vulnerability in production costs on average 30× more than fixing it during development.

2. Shift Left Security

"Shift Left" means moving security testing to earlier phases in the software development lifecycle — to the left of the pipeline. In practical terms, this involves:

Planning Phase — Threat Modeling
Identify threats before writing a single line of code. Frameworks like STRIDE and PASTA help map attack vectors directly at the architecture stage.
Code Phase — Pre-commit Hooks
Run security linters and secret detection before every commit. Tools like pre-commit, gitleaks and detect-secrets block issues locally before they ever reach the remote repository.
Pull Request — Automated SAST
Static analysis runs automatically on every PR. Results appear as comments in the code review, integrating security into the normal development workflow.
CI/CD — Quality Gates
The pipeline blocks deployments when it finds high or critical severity vulnerabilities, preventing insecure code from ever reaching production.
🔍

3. SAST, DAST, SCA and IAST

There are four main approaches for testing application security. Each analyses a different aspect and is used at a distinct stage of the pipeline.

SAST — Static Analysis
  • Analyses source code without executing the application
  • Detects: SQL Injection, XSS, buffer overflows in code
  • Phase: during development and CI
  • Tools: Semgrep, SonarQube, CodeQL
  • ⚡ Fast, but can produce false positives
DAST — Dynamic Analysis
  • Tests the application while it is running (black box)
  • Detects: runtime flaws, misconfigurations, exposures
  • Phase: staging / pre-production
  • Tools: OWASP ZAP, Burp Suite, Nikto
  • 🎯 Finds real vulnerabilities as seen in production
SCA — Dependency Analysis
  • Analyses third-party libraries and packages
  • Detects: CVEs in dependencies, problematic licences
  • Phase: build and continuous CI
  • Tools: Snyk, Dependabot, npm audit, Trivy
  • 📦 Essential — 85% of code comes from third parties
IAST — Interactive Analysis
  • Agent inside the application during testing
  • Detects: flaws with full execution context
  • Phase: during integration testing
  • Tools: Contrast Security, Hdiv Security
  • 🧠 Lowest false positive rate

4. Essential Ecosystem Tools

Basic pipeline security checks

# Detect secrets in the repository
git secrets --scan
gitleaks detect --source . -v

# Vulnerabilities in Node.js dependencies
npm audit --audit-level=high

# Static analysis with Semgrep
semgrep --config=auto .

# Docker image scan
docker scout cves myapp:latest
trivy image myapp:latest

# Filesystem scan (IaC, configs, code)
trivy fs . --severity HIGH,CRITICAL
Line-by-line explanation:

gitleaks detect — Searches for API keys, tokens and hardcoded passwords in the full Git history
npm audit --audit-level=high — Analyses dependencies and only fails if there are high or critical severity CVEs
semgrep --config=auto — Multi-language SAST with community rules, detects insecure code patterns
trivy image — Checks CVEs in container layers, including OS and application dependencies
trivy fs --severity HIGH,CRITICAL — Analyses local files, ignoring low-priority alerts

Semgrep
SAST
Multi-language static analysis with customisable rules. Integrates natively with GitHub Actions.
Trivy
SCA + Container
All-in-one scanner for containers, IaC, code and filesystems. Open source and very fast.
OWASP ZAP
DAST
Open source attack proxy for testing running web applications. Has headless mode for CI/CD integration.
Gitleaks
Secrets Detection
Detects hardcoded secrets in Git repositories, including the full history of old commits.
HashiCorp Vault
Secret Management
Centralised management of secrets, certificates and encryption keys with granular access control.
Checkov
IaC Security
Analyses Terraform, CloudFormation and Kubernetes files for security misconfigurations.

5. Secrets Management — What NEVER to do

🚫
Secrets in code are one of the #1 causes of breaches In 2023, more than 10 million secrets were exposed in public GitHub repositories. A leaked AWS key can result in $50,000+ in charges within minutes.

❌ Wrong vs ✅ Correct

# ❌ NEVER do this
DATABASE_URL = "postgresql://admin:SuperSecret123@prod-db:5432/app"
AWS_SECRET_KEY = "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY"
API_TOKEN = "sk-proj-abc123def456..."

# ✅ Use environment variables
DATABASE_URL = os.getenv("DATABASE_URL")

# ✅ Use a vault (e.g. HashiCorp Vault)
vault kv get secret/myapp/database

# ✅ GitHub Actions: use encrypted secrets
env:
  AWS_SECRET_KEY: ${{ secrets.AWS_SECRET_KEY }}
  • ✓
    Use environment variables for sensitive configuration in local development
  • ✓
    Use a secret manager (Vault, AWS Secrets Manager, Azure Key Vault) in production
  • ✓
    Add .env to .gitignore and never commit configuration files
  • !
    Rotate secrets periodically — even if they have never been exposed
  • ✗
    Store passwords, tokens or keys in source code, even "temporarily"
  • ✗
    Use the same secret in development and production

6. Secure CI/CD Pipeline — GitHub Actions

A complete DevSecOps pipeline integrates multiple automated security checks. Below is an example of a GitHub Actions workflow with security gates.

Example: GitHub Actions with Security Gates

name: DevSecOps Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:

jobs:
  security-scan:
    runs-on: ubuntu-latest
    steps:

      # 1. Checkout the code
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # full history for Gitleaks

      # 2. Secret detection
      - name: Gitleaks - Secrets Detection
        uses: gitleaks/gitleaks-action@v2

      # 3. SAST with Semgrep
      - name: Semgrep - Static Analysis
        uses: semgrep/semgrep-action@v1
        with:
          config: p/owasp-top-ten

      # 4. Vulnerable dependencies
      - name: npm audit
        run: npm audit --audit-level=high

      # 5. Build and container scan
      - name: Build Docker Image
        run: docker build -t myapp:${{ github.sha }} .

      - name: Trivy - Container Scan
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: myapp:${{ github.sha }}
          severity: HIGH,CRITICAL
          exit-code: '1'  # fails the pipeline if CVEs are found
Key points of this pipeline:

fetch-depth: 0 — Required for Gitleaks to analyse the full commit history, not just the latest commit
p/owasp-top-ten — Semgrep ruleset aligned with the OWASP Top 10 (the 10 most critical web vulnerabilities)
exit-code: '1' — Makes the step fail if Trivy finds High/Critical CVEs, automatically blocking the deployment
github.sha — Uses the commit hash as the image tag, ensuring full traceability

7. Supply Chain Security & SBOM

The 2020 SolarWinds attack showed the world that the software supply chain is a critical attack vector. A SBOM (Software Bill of Materials) is the "inventory" of all components in your software — libraries, transitive dependencies, versions and licences.

📰 Real Case
SolarWinds Attack (2020)
Attackers compromised SolarWinds' build pipeline and injected malicious code into the Orion software. The update was distributed to 18,000 customers, including US government agencies. This attack led to new regulations requiring SBOMs for software sold to the US government.

Generate an SBOM with Syft

# Install Syft (SBOM generator)
curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh

# Generate SBOM from a Docker image (SPDX format)
syft myapp:latest -o spdx-json > sbom.spdx.json

# Generate SBOM from a local project
syft dir:. -o cyclonedx-json > sbom.cyclonedx.json

# Check for vulnerabilities in the SBOM with Grype
grype sbom:./sbom.spdx.json

# Sign the SBOM with Cosign (supply chain integrity)
cosign sign-blob --key cosign.key sbom.spdx.json
✅
Supply Chain Security best practices Generate SBOMs on every build • Sign artefacts with Sigstore/Cosign • Use lock files (package-lock.json, Pipfile.lock) • Verify dependency checksums • Use private repositories for critical dependencies

8. Infrastructure as Code (IaC) Security

With IaC, infrastructure is defined as code — meaning configuration errors can be detected before any resource is created in the cloud.

Checkov — Terraform Scan

# Example of insecure Terraform code
resource "aws_s3_bucket" "data" {
  bucket = "my-company-data"
  acl    = "public-read"  # ⚠️ Public bucket!
}

# Run Checkov to detect the issue
checkov -d . --framework terraform

# Output:
# Check: CKV_AWS_20: "Ensure the S3 bucket has access control list (ACL) applied"
# FAILED for resource: aws_s3_bucket.data
# File: main.tf:1-5

# Fixed version
resource "aws_s3_bucket" "data" {
  bucket = "my-company-data"
}

resource "aws_s3_bucket_acl" "data_acl" {
  bucket = aws_s3_bucket.data.id
  acl    = "private"  # ✅ Private by default
}
