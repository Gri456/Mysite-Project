⚡ 1.1 Why Performance Matters

Performance is a feature. A slow website isn't just an inconvenience — it has measurable business impact. Google found that a 100ms delay in load time can reduce conversions by 7%. Amazon calculated that every additional second of latency costs them 1.6 billion dollars per year.


The web is accessed by billions of people on devices that range from the latest iPhone to low-end Android phones on 3G connections. Writing performant code is an act of inclusivity — you're ensuring your application works for everyone, not just users on fast networks with powerful hardware.

🔑 The Performance Mindset:

Performance is not something you bolt on after shipping a feature — it's a discipline you apply throughout the development process. The best time to fix a performance issue is before it exists: choose the right architecture, load only what you need, and measure everything.

🖥️ 1.2 How Browsers Render Pages — The Critical Rendering Path

Before you can optimise, you need to understand how the browser goes from a URL to painted pixels on screen. This pipeline is called the Critical Rendering Path (CRP) and every step is a potential bottleneck.

The 6 Steps of the CRP:
1. DNS Lookup → 2. TCP + TLS → 3. HTTP Request → 4. Parse HTML → DOM → 5. Build CSSOM + Render Tree → 6. Layout → Paint → Composite
🚧 Render-Blocking Resources

By default, CSS is render-blocking — the browser won't paint anything until all CSS is downloaded and parsed. JavaScript is both render-blocking and parser-blocking — it pauses HTML parsing completely.

<!-- BAD: blocks rendering -->
<script src="heavy-lib.js"></script>
<link rel="stylesheet" href="all-styles.css">
✅ Non-Blocking Techniques

Use defer and async to let HTML parse while scripts download. Use media queries on CSS to avoid blocking non-critical styles.

<!-- GOOD: non-blocking -->
<script src="app.js" defer></script>
<script src="analytics.js" async></script>
<link rel="stylesheet" href="print.css"
      media="print">
🔍 defer vs async — What's the Difference?
Attribute Download Execution Order Use for
defer Parallel After HTML parsed Preserved App scripts that need DOM
async Parallel As soon as downloaded Not guaranteed Independent scripts (analytics)
none Blocks Immediately Preserved Almost never

📊 1.3 Core Web Vitals — Google's Performance Metrics

Core Web Vitals are the metrics Google uses to measure user experience and factor into search rankings. They measure three fundamental aspects: loading, interactivity, and visual stability.

🖼️

LCP

Largest Contentful Paint

Time until the largest visible element (hero image, heading) is rendered.

✅ Good: < 2.5s
⚠ Needs work: 2.5–4s
❌ Poor: > 4s

Fix: Preload hero images, use CDN, eliminate render-blocking resources.

👆

INP

Interaction to Next Paint

Latency from user interaction (click, tap, key) to the next visual update.

✅ Good: < 200ms
⚠ Needs work: 200–500ms
❌ Poor: > 500ms

Fix: Break up long tasks, use web workers, avoid heavy event handlers.

📐

CLS

Cumulative Layout Shift

Amount of unexpected layout shifts — elements jumping around after load.

✅ Good: < 0.1
⚠ Needs work: 0.1–0.25
❌ Poor: > 0.25

Fix: Always set width/height on images and iframes, avoid inserting content above existing content.

HOW TO MEASURE CORE WEB VITALS IN CODE

// Using the web-vitals library (npm install web-vitals)
import { onLCP, onINP, onCLS } from 'web-vitals';

function sendToAnalytics({ name, value, rating }) {
  console.log(`${name}: ${value} (${rating})`);
  // Send to your analytics endpoint
  fetch('/analytics', {
    method: 'POST',
    body: JSON.stringify({ name, value, rating })
  });
}

onLCP(sendToAnalytics);
onINP(sendToAnalytics);
onCLS(sendToAnalytics);

🌐 1.4 Network Optimisation — Fewer and Faster Requests

The single most impactful thing you can do for performance is reduce the amount of data transferred over the network. Every byte has a cost — download time, parsing time, memory usage.

📦 HTTP/2 and HTTP/3

Modern HTTP protocols eliminate the "head-of-line blocking" of HTTP/1.1. HTTP/2 multiplexes multiple requests over one connection — making dozens of small files efficient. HTTP/3 uses QUIC (UDP-based) for even lower latency.

# Check if a site uses HTTP/2
curl -I --http2 https://example.com

# Nginx — enable HTTP/2
server {
  listen 443 ssl http2;
  # ...
}
🗜️ Compression

Gzip and Brotli compress text assets before sending them. Brotli typically achieves 15–25% better compression than Gzip. Most modern CDNs apply this automatically.

# Nginx — enable Brotli + Gzip
brotli on;
brotli_comp_level 6;
gzip on;
gzip_types text/css application/javascript
           application/json image/svg+xml;

RESOURCE HINTS — TELL THE BROWSER WHAT'S COMING

<!-- dns-prefetch: resolve DNS before the browser needs the resource -->
<link rel="dns-prefetch" href="//fonts.googleapis.com">

<!-- preconnect: DNS + TCP + TLS handshake in advance -->
<link rel="preconnect" href="https://api.example.com">

<!-- preload: high-priority fetch for resources you'll need soon -->
<link rel="preload" href="/hero.webp" as="image">
<link rel="preload" href="/fonts/inter.woff2" as="font" crossorigin>

<!-- prefetch: low-priority fetch for next-page resources -->
<link rel="prefetch" href="/dashboard.js">

<!-- modulepreload: preload + parse ES modules -->
<link rel="modulepreload" href="/app.js">
Code Splitting

Split JS bundles per route — only load what the current page needs

Tree Shaking

Remove unused code from your bundle during build (Vite, Webpack, Rollup)

Minification

Remove whitespace, comments, shorten variable names. esbuild is 10–100× faster than alternatives

Bundle Analysis

Use vite-bundle-visualizer or webpack-bundle-analyzer to find bloat

🖼️ 1.5 Image Optimisation — The Biggest Win

Images are typically 50–80% of a page's total byte weight. Optimising images is almost always the single highest-ROI performance improvement you can make.

📋 Modern Image Formats:
Format Best For vs JPEG

SVG Icons, logos Scales infinitely

✔ Use WebP or AVIF instead of JPEG/PNG
✔ Always specify width and height attributes

✔ Serve images via CDN with edge caching

✔ Use CSS for decorative images, not <img>

💾 1.6 Caching — The Fastest Request is the One Never Made

Caching is the most powerful performance technique because it completely eliminates the network round-trip. A properly cached asset loads in microseconds from memory, not hundreds of milliseconds from a server.

HTTP CACHE HEADERS — THE FUNDAMENTALS

# Cache immutable assets (hashed filenames) forever
Cache-Control: public, max-age=31536000, immutable
# app.a3f9c2.js → cached for 1 year, never revalidated

# Cache HTML documents — always revalidate
Cache-Control: no-cache
# Browser caches it but must check with server before using

# Never cache sensitive data
Cache-Control: no-store, private

# ETags — conditional revalidation
ETag: "33a64df551425fcc55e"
# Browser sends: If-None-Match: "33a64df551425fcc55e"
# Server responds: 304 Not Modified (no body!) or 200 with new content

# Last-Modified alternative
Last-Modified: Wed, 21 Oct 2023 07:28:00 GMT
# Browser sends: If-Modified-Since: Wed, 21 Oct 2023 07:28:00 GMT
🔑 Cache-Busting Strategy

Cache forever with immutable assets, but include a content hash in the filename. When the file changes, the hash changes, the browser downloads the new file.

# Vite does this automatically:
app.a3f9c2d1.js  ← hash from content
style.8b4e1f2a.css

# Old file = cached forever
# New file = different URL = fresh download
⚙️ Service Worker Cache

Service Workers intercept network requests and can serve from a programmatic cache — enabling offline support and instant repeat visits.

// service-worker.js
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request)
      .then(cached => cached || fetch(event.request))
  );
});

⚙️ 1.7 JavaScript Runtime Performance

The JavaScript engine (V8 in Chrome/Node) has a single-threaded event loop. Long-running JavaScript tasks block the main thread, preventing the browser from responding to user input, running animations, or rendering updates. This is what causes the "jank" users feel.

🚫 The Main Thread is a Shared Resource

When your JavaScript runs a 500ms task, the browser cannot respond to clicks, scroll events, or animations for that entire 500ms. A task longer than 50ms is considered a "Long Task" and directly impacts INP.

BREAKING UP LONG TASKS — SCHEDULER API

// BAD: Blocks the main thread for all 10,000 items
function processAllItems(items) {
  items.forEach(item => heavyProcess(item)); // 500ms+ blocking task
}

// GOOD: Yield to the browser between chunks
async function processInChunks(items, chunkSize = 100) {
  for (let i = 0; i < items.length; i += chunkSize) {
    const chunk = items.slice(i, i + chunkSize);
    chunk.forEach(item => heavyProcess(item));
    
    // Yield — let the browser handle pending interactions
    await scheduler.yield();          // Modern (Chrome 115+)
    // Or fallback: await new Promise(resolve => setTimeout(resolve, 0));
  }
}

// Web Workers — true parallelism for CPU-heavy tasks
const worker = new Worker('/heavy-worker.js');
worker.postMessage({ data: largeDataset });
worker.onmessage = ({ data }) => {
  // Result arrives without blocking main thread
  updateUI(data.result);
};
🔁 Debounce and Throttle

Prevent expensive functions from running too frequently in response to rapid events.

// Debounce: wait until user stops typing
const search = debounce((query) => {
  fetchResults(query);
}, 300);
input.addEventListener('input', (e) => search(e.target.value));

// Throttle: at most once per 16ms (~60fps)
const onScroll = throttle(() => {
  updateParallax();
}, 16);
window.addEventListener('scroll', onScroll);
📐 Memoisation

Cache the results of expensive pure functions so they only compute once per unique input.

function memoize(fn) {
  const cache = new Map();
  return function (...args) {
    const key = JSON.stringify(args);
    if (cache.has(key)) return cache.get(key);
    const result = fn.apply(this, args);
    cache.set(key, result);
    return result;
  };
}

const expensiveCalc = memoize((n) => {
  // ... heavy computation
  return result;
});

🎨 1.8 Rendering Pipeline — Layout, Paint, Composite

After the initial page load, every visual update triggers some portion of the browser's rendering pipeline. Understanding which CSS properties trigger which pipeline steps is essential to achieving smooth 60fps animations.

📐
Layout (Reflow)

Most expensive. Recalculates position and size of all affected elements. Triggered by: width, height, margin, padding, top, left, font-size.

🖌️
Paint

Fills in pixels. Triggered by: color, background, border-radius, box-shadow, outline. Skips layout step.

✨
Composite Only

Cheapest. Moves already-painted layers. Only triggered by: transform, opacity. Always use these for animations.

PERFORMANT ANIMATIONS — ALWAYS USE TRANSFORM + OPACITY

/* BAD — triggers layout on every frame (jank!) */
.box { transition: left 0.3s, top 0.3s, width 0.3s; }

/* GOOD — composite only, GPU-accelerated, silky smooth */
.box { transition: transform 0.3s, opacity 0.3s; }
.box.moved { transform: translateX(100px) scale(1.1); }
.box.hidden { opacity: 0; }

/* Force GPU layer promotion (use sparingly) */
.animated-element {
  will-change: transform;
  /* Or: transform: translateZ(0); — the old hack */
}

/* Use requestAnimationFrame for JS animations */
function animate() {
  element.style.transform = `translateX(${x}px)`;
  x += velocity;
  requestAnimationFrame(animate); // Syncs with display refresh
}
🚫 Forced Layout / Layout Thrashing

Reading layout properties (like offsetWidth) after writing style forces the browser to immediately flush and recalculate layout. In a loop, this creates "layout thrashing" — one of the most common performance bugs.

❌ BAD — layout thrashing

items.forEach(item => {
  item.style.width = container.offsetWidth + 'px';
  // Read after write = forced layout!
});

✅ GOOD — batch reads then writes

const width = container.offsetWidth; // Read once
items.forEach(item => {
  item.style.width = width + 'px'; // Write only
});

🏗️ 1.9 Modern Rendering Patterns — SSR, SSG, ISR, CSR

One of the most important architectural decisions in modern web development is where and when your HTML is generated. Each strategy has different performance characteristics.

📄 SSG — Static Site Generation

HTML generated at build time. Served directly from CDN. Fastest possible TTFB.

Best for: Blogs, docs, marketing sites, content that rarely changes.

Trade-off: Rebuild needed to update content. Doesn't scale to millions of pages.

⚙️ SSR — Server-Side Rendering

HTML generated on the server per request. Always fresh data. Good SEO.

Best for: Dashboards, e-commerce, user-specific content, real-time data.

Trade-off: Higher TTFB than SSG, server compute cost on every request.

🔄 ISR — Incremental Static Regen.

SSG with a revalidation timer. Pages rebuilt in background after a set interval.

Best for: E-commerce product pages, news sites — fast + fresh.

Trade-off: First request after revalidation gets stale page. Next.js feature.

🌐 CSR — Client-Side Rendering

HTML generated in the browser via JavaScript. Initial load = empty HTML shell.

Best for: Highly interactive apps, logged-in experiences, SPAs.

Trade-off: Slow initial load, poor SEO without hydration, blank screen flash.

NEXT.JS — MIXING RENDERING STRATEGIES PER PAGE

// SSG — runs at build time
export async function getStaticProps() {
  const posts = await fetchAllPosts();
  return { props: { posts } };
}

// SSR — runs on every request
export async function getServerSideProps({ req }) {
  const user = await getUser(req.cookies.token);
  return { props: { user } };
}

// ISR — SSG + revalidate every 60 seconds
export async function getStaticProps() {
  const products = await fetchProducts();
  return { props: { products }, revalidate: 60 };
}

// React Server Components (Next.js App Router — default)
async function ProductPage({ id }) {
  const product = await db.products.findById(id); // Runs on server!
  return <ProductDetail product={product} />;
}

✍️ 1.10 Font Optimisation — Eliminating Flash of Invisible Text

Web fonts are a surprisingly common source of performance problems. A poorly loaded font causes the dreaded FOIT (Flash of Invisible Text) or FOUT (Flash of Unstyled Text), degrading both perceived performance and CLS.

FONT LOADING BEST PRACTICES

/* 1. Self-host fonts instead of relying on Google Fonts CDN */
@font-face {
  font-family: 'Inter';
  src: url('/fonts/inter.woff2') format('woff2'); /* woff2 only — universal support */
  font-weight: 100 900;    /* Variable font — covers all weights */
  font-display: swap;      /* Show fallback font, swap when loaded */
  font-style: normal;
}

/* 2. Preload critical fonts in <head> */
<link rel="preload" href="/fonts/inter.woff2"
      as="font" type="font/woff2" crossorigin>

/* 3. Size-adjust fallback font to reduce layout shift */
@font-face {
  font-family: 'Inter-fallback';
  src: local('Arial');
  size-adjust: 107%;           /* Match Inter's metrics */
  ascent-override: 95%;
  descent-override: 25%;
}

body {
  font-family: 'Inter', 'Inter-fallback', sans-serif;
}
font-display: swap
Show system font immediately, swap when web font loads
Variable Fonts
One file for all weights and styles — huge file size reduction
Subsetting
Only include characters your site actually uses (latin subset)
Local Fonts
local('Arial') — use OS font if available, zero download

🎨 1.11 CSS Performance — Selectors, Containment and content-visibility

CSS has its own set of performance considerations beyond just file size. Complex selectors, unnecessary reflows, and painting large areas all impact rendering speed.

🔍 Selector Efficiency

Browsers match CSS selectors right-to-left. Complex selectors are slower — but in practice, the performance difference is negligible for normal pages. Focus instead on what CSS triggers.

/* Avoid: expensive selector */
.nav ul li a span { color: blue; }

/* Prefer: specific class */
.nav-link-text { color: blue; }

/* Avoid: universal selector in complex rules */
.container * { box-sizing: border-box; }

/* Prefer: set once at root */
*, *::before, *::after { box-sizing: border-box; }
📦 CSS Containment and content-visibility

New CSS properties that tell the browser to skip rendering work for off-screen content — dramatically speeding up initial load for long pages.

/* Skip layout, paint, style for off-screen elements */
.article-card {
  content-visibility: auto;
  contain-intrinsic-size: 0 300px; /* Estimated height */
}

/* Isolate element from rest of page */
.widget {
  contain: layout style paint;
  /* Changes inside don't affect outside */
}

📋 1.12 Performance Budgets and Monitoring

A performance budget is a set of limits on metrics that directly affect performance. It acts as a constraint that teams commit to — preventing performance regressions from slipping into production unnoticed.

Example Performance Budget:
Metric Budget Why
Total JS (compressed) < 200KB Parse + execute on mid-range phone
Total CSS (compressed) < 50KB Render-blocking cost
LCP < 2.5s Google "Good" threshold
INP < 200ms Responsive to all interactions
CLS < 0.1 No unexpected layout shifts
