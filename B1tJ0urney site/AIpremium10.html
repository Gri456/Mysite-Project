Introduction: What Is a Token?

A token is the basic unit of text that a language model processes. Models don't read words or characters — they read tokens. A token is roughly a word, part of a word, a punctuation mark, or a space, depending on the tokenizer.

As a rough rule of thumb:

  • 1 token ≈ 4 characters in English
  • 1 token ≈ ¾ of a word
  • 100 tokens ≈ 75 words
  • 1,000 tokens ≈ 750 words ≈ 1.5 pages of text

This module covers:

  • How tokenization works under the hood
  • Why tokens matter for cost and context limits
  • The BPE algorithm that powers most LLM tokenizers
  • How to count tokens with tiktoken
  • Practical implications for prompt engineering

Why the word→token gap matters in practice:

Intuition
Tokens ≠ Words
The word "tokenization" is 2 tokens: Token + ization. The word "AI" is 1 token. "Hello!" is 2 tokens: Hello + !. You can't reason about model limits or costs using word count alone.
Impact
Cost Is Always Tokens
Every LLM API invoice is in tokens, not words. A 10-word prompt might be 8 or 18 tokens depending on what those words are. Misjudging this can lead to unexpected bills and failed context limits in production.
Language
English Is Privileged
English text averages ~1.3 tokens/word. Chinese averages ~2 tokens/character. Arabic ~1.7 tokens/word. A 500-word English document and its Chinese translation are not the same token count — the Chinese version costs significantly more.
Context
Tokens Run Out
Every model has a finite context window. A GPT-4o call with 128K context sounds enormous — but a long codebase, system prompt, and conversation history can easily consume 100K+ tokens, leaving little room for the model to respond.

How Tokenization Works — BPE

Most modern LLMs use Byte Pair Encoding (BPE) — a compression algorithm adapted for text. It builds a vocabulary of common subword units by repeatedly merging the most frequent pairs of characters.

How BPE is trained:

1. Start with every character as its own token: ['h','e','l','l','o']

2. Find the most frequent pair: ('l','l') → merge into 'll'

3. Repeat until vocabulary size is reached (e.g. 50,000 tokens)

Result: common words become single tokens. Rare words are split into known subwords.

Why subword tokenization?

  • Handles unknown words by splitting them into known pieces
  • Common words ("the", "is") get their own token — efficient
  • Rare or made-up words are split into subparts — flexible
  • Works across languages with the same vocabulary

BPE step-by-step — watching the algorithm merge pairs on a real example:

1
Starting corpus (simplified): "low low low lower lower newest newest widest" — split every word into characters with a word-end marker: l o w </w>, l o w e r </w>, n e w e s t </w>, w i d e s t </w>
2
Count all adjacent pairs: (l,o)=5, (o,w)=5, (w,</w>)=3, (e,r)=2, (e,s)=2, (s,t)=2 … The most frequent is (l,o) and (o,w) tied at 5. Merge (o,w) → ow
3
After merge: l ow </w>, l ow e r </w>, n e w e s t </w>. Now count again. (l,ow)=5 is the most frequent. Merge → low
4
Continue: eventually (e,s)→es, then (es,t)→est, then (n,e)→ne, then (ne,w)→new, then (new,est)→newest. After enough merges, "newest" becomes a single token — but "newish" (not in training data) gets split into new + ish
5
Final vocabulary: a mix of characters, common subwords, and frequent whole words. The vocabulary size (e.g. 50,257 for GPT-2, 100,277 for GPT-4) is a hyperparameter — larger = more efficient encoding but bigger embedding tables
🔢 BPE vs SentencePiece vs WordPiece

BPE (GPT family, LLaMA 3, Mistral) — merges character pairs. Character-level fallback means it can tokenize any string.
WordPiece (BERT, DistilBERT) — similar to BPE but merges maximize likelihood instead of frequency. Uses ## prefix for subword continuations: token ##ization.
SentencePiece (LLaMA 1/2, Gemini, T5) — language-agnostic, treats input as raw bytes. Works well for multilingual and low-resource languages. Adds ▁ for word boundaries.

Token Vocabularies by Model

Each model family uses a different tokenizer with a different vocabulary size. Larger vocabularies mean fewer tokens per sentence (more efficient), but require more memory.

  • GPT-2 — 50,257 tokens (cl100k predecessor)
  • GPT-3.5 / GPT-4 — 100,277 tokens (cl100k_base via tiktoken)
  • GPT-4o — 200,019 tokens (o200k_base — more efficient)
  • Claude — uses its own internal tokenizer, similar density to cl100k
  • LLaMA / Mistral — SentencePiece tokenizer, ~32,000 tokens
  • Gemini — SentencePiece, ~256,000 vocabulary

The same sentence will produce different token counts depending on which model's tokenizer you use. Always count tokens using the correct tokenizer for your model.

⚠️ Never Mix Tokenizers

Using GPT-4's cl100k_base to count tokens for a Claude API call will give you wrong numbers. Claude's tokenizer produces slightly different counts for the same text. For exact counts: use the model's own token counter. For estimates across providers: add a 15–20% safety margin to your calculations.

Counting Tokens with tiktoken

OpenAI's tiktoken library lets you count tokens locally — before making an API call — so you can control costs and stay within context limits.

import tiktoken

# Load the tokenizer for a specific model
enc = tiktoken.encoding_for_model("gpt-4o")

text = "Tokenization is the foundation of all language models."

# Encode to tokens
tokens = enc.encode(text)
print(f"Token count : {len(tokens)}")   # → 10
print(f"Token IDs   : {tokens}")

# Decode back to text
decoded = enc.decode(tokens)
print(f"Decoded     : {decoded}")

# Count tokens in a full chat message
def count_tokens(messages, model="gpt-4o"):
    enc = tiktoken.encoding_for_model(model)
    total = 0
    for msg in messages:
        total += len(enc.encode(msg["content"])) + 4  # overhead per message
    return total

Production-grade token management — patterns used in real applications:

import tiktoken

class TokenBudgetManager:
    """Manages token budget for a multi-turn conversation."""

    def __init__(self, model: str = "gpt-4o", context_limit: int = 128_000,
                 max_output_reserve: int = 4_000):
        self.enc = tiktoken.encoding_for_model(model)
        self.context_limit = context_limit
        self.max_input_tokens = context_limit - max_output_reserve
        self.messages = []
        self.system_tokens = 0

    def set_system_prompt(self, system: str):
        self.system_tokens = len(self.enc.encode(system)) + 4
        self.system_prompt = system

    def count_msg(self, role: str, content: str) -> int:
        return len(self.enc.encode(content)) + 4  # 4 = per-message overhead

    def add_message(self, role: str, content: str):
        self.messages.append({"role": role, "content": content})

    def get_total_tokens(self) -> int:
        total = self.system_tokens + 3  # 3 = priming tokens
        for msg in self.messages:
            total += self.count_msg(msg["role"], msg["content"])
        return total

    def trim_to_budget(self):
        """Remove oldest non-system messages until we're within budget."""
        while self.get_total_tokens() > self.max_input_tokens and len(self.messages) > 1:
            removed = self.messages.pop(0)
            print(f"Trimmed: [{removed['role']}] {removed['content'][:40]}...")

    def remaining_budget(self) -> int:
        return self.max_input_tokens - self.get_total_tokens()

# Usage
manager = TokenBudgetManager("gpt-4o", context_limit=128_000, max_output_reserve=4_000)
manager.set_system_prompt("You are a helpful assistant.")

# Add messages and auto-trim if needed
manager.add_message("user", "Explain quantum computing in detail...")
manager.add_message("assistant", "Quantum computing uses qubits...")
manager.trim_to_budget()

print(f"Tokens used : {manager.get_total_tokens():,}")
print(f"Budget left : {manager.remaining_budget():,}")

The Context Window

The context window is the maximum number of tokens a model can process in a single call — including both the input (prompt) and the output (response).

  • GPT-4o — 128,000 tokens (~96,000 words)
  • Claude 3.5 Sonnet — 200,000 tokens (~150,000 words)
  • Gemini 1.5 Pro — 1,000,000 tokens (~750,000 words)
  • GPT-3.5 — 16,385 tokens
  • LLaMA 3 (8B) — 8,192 tokens

Important: the context window is shared. Input tokens + output tokens ≤ context limit. Sending 100K input tokens leaves only 28K for the response on GPT-4o.

Models have no memory between API calls — the entire conversation history must be resent every time.

How context window size affects what you can do with a model:

8K tokens
Small Context
Can handle: a short article, a few pages of code, a brief conversation. Cannot handle: long codebases, full research papers, extended chat history. Typical of: GPT-3.5 (original), local models like LLaMA 2 (7B), older API tiers.
128K tokens
Large Context
Can handle: an entire book chapter, a full codebase, a long email thread. One GPT-4o call can process ~96,000 words — roughly the length of a novel. Requires careful prompt design to keep the model focused on the relevant section.
200K tokens
Very Large Context (Claude)
Claude 3.5 Sonnet's 200K context can hold ~150,000 words — a full legal contract, an entire codebase, or hours of transcripts. Anthropic reports strong performance throughout the window, not just at the start.
1M tokens
Extreme Context (Gemini)
Gemini 1.5 Pro's 1M context can hold ~750,000 words — an hour of video, a full software repository, or years of meeting notes. Cost and latency scale with context size; using maximum context on every call is expensive.
📍 Lost in the Middle Problem

Research shows that LLMs retrieve information better from the beginning and end of their context window — and worse from the middle. On a 128K context call, important instructions buried in the middle of 100K tokens of documents may be partially ignored. Best practice: put key instructions at the start and end of the prompt, and use retrieval (RAG) rather than dumping entire documents into context.

Tokens & Pricing

AI APIs charge by the token — separately for input (prompt) and output (completion). Understanding this is critical for building cost-effective applications.

Example pricing (approximate, mid-2025):

  • GPT-4o — $2.50 / 1M input tokens · $10.00 / 1M output tokens
  • GPT-4o mini — $0.15 / 1M input · $0.60 / 1M output
  • Claude Sonnet 4 — $3.00 / 1M input · $15.00 / 1M output
  • Claude Haiku — $0.25 / 1M input · $1.25 / 1M output

Output tokens are always more expensive than input tokens — generating text is more compute-intensive than reading it. Reducing output length saves more money than shortening prompts.

💡 Prompt Caching — Cut Input Costs by 90%

Both Anthropic and OpenAI offer prompt caching: if you send the same prefix (system prompt, document, few-shot examples) repeatedly, the API caches the KV states from those tokens and charges a fraction of the normal rate for subsequent calls. Anthropic charges 10% of normal for cache hits. For apps with a fixed system prompt, this alone can reduce costs by 70–90%. Mark cacheable content with the cache_control parameter in the API.

Token Quirks & Edge Cases

Tokenization behaves in surprising ways that every AI developer must know:

  • Capitalization matters — "Token" and "token" may tokenize differently
  • Leading spaces matter — " hello" and "hello" are different tokens
  • Numbers split strangely — "1234567890" may become many tokens
  • Non-English is less efficient — Chinese, Arabic, and code often use more tokens per character
  • Emoji — a single emoji can be 1–3 tokens
  • Whitespace — tabs, newlines, and extra spaces all consume tokens
# Surprising token counts
"hello"           → 1 token
" hello"          → 1 token  (different token ID!)
"HELLO"           → 1 token  (different again)
"1234567890"      → 4 tokens
"2024-01-15"      → 5 tokens
"https://example.com/path?q=ai"  → 12+ tokens
"中文"            → 4 tokens  (2 Chinese chars = 4 tokens)
"🚀"              → 2 tokens

Why tokenization quirks cause real bugs — and how to avoid them:

Bug: Arithmetic
Numbers Are Unreliable
Large numbers are split across multiple tokens: 1234567 → [123][45][67]. This makes arithmetic hard — the model sees fragments, not a number. Reason GPT-3 famously failed at multi-digit multiplication. Use code execution (tools) for math, not raw generation.
Bug: Spelling
Character Counting Breaks
Ask a model "how many letters in 'strawberry'?" — it often answers wrong because it reasons at token level, not character level. The word "strawberry" is 3 tokens: straw + ber + ry. Character-level tasks require explicit instruction or tool use.
Bug: Encoding
Special Characters & Formatting
HTML tags, markdown, JSON brackets, and code indentation all consume tokens. A prompt with 30 lines of Python indented with tabs uses 30+ extra tokens just for whitespace. Prefer spaces over tabs in code prompts. Strip unnecessary formatting from documents before sending.
Bug: URLs
URLs Are Token Expensive
A single URL like https://api.example.com/v2/users?filter=active&sort=desc can be 20+ tokens. In RAG systems with dozens of source URLs, link metadata alone can consume hundreds of tokens. Shorten or omit URLs when not needed for the model's reasoning.

Tokens & Prompt Engineering

Understanding tokens changes how you write prompts. Every token costs money and consumes context. Here are practical implications:

  • Be concise — unnecessary filler words waste tokens with no benefit
  • Reuse system prompts — many APIs cache system prompt tokens
  • Truncate context — summarize old history instead of sending it all
  • Limit output length — use max_tokens to cap response length
  • Choose the right model — short tasks don't need a 200K context model
  • Structured outputs — JSON can be more token-efficient than prose
response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": prompt}],
    max_tokens=512,        # cap the output
    temperature=0.2        # lower temp = more focused = sometimes shorter
)

Advanced token optimization techniques used in production LLM applications:

import tiktoken

# ── Technique 1: Token-aware truncation ──────────────────────────────
def truncate_to_tokens(text: str, max_tokens: int, model: str = "gpt-4o") -> str:
    """Truncate text to at most max_tokens, cutting at word boundaries."""
    enc = tiktoken.encoding_for_model(model)
    tokens = enc.encode(text)
    if len(tokens) <= max_tokens:
        return text
    truncated_tokens = tokens[:max_tokens]
    return enc.decode(truncated_tokens)

# ── Technique 2: Conversation summarization ──────────────────────────
def summarize_history(messages: list, keep_last_n: int = 4) -> list:
    """Keep recent messages, summarize older ones into a single message."""
    if len(messages) <= keep_last_n:
        return messages

    old_messages = messages[:-keep_last_n]
    recent = messages[-keep_last_n:]

    # Summarize old context (call a cheaper/faster model)
    summary_text = call_llm(
        f"Summarize this conversation in 3 sentences:\n{format_messages(old_messages)}",
        model="gpt-4o-mini"
    )
    summary_msg = {"role": "system", "content": f"[Prior context summary]: {summary_text}"}
    return [summary_msg] + recent

# ── Technique 3: Structured output for efficiency ────────────────────
# Instead of: "Please extract the name, company, and email from this text and
#              explain your reasoning in a few sentences before giving the answer."
# Use: extract directly with structured output — no prose, fewer tokens

response = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": f"Extract entities from: {text}"}],
    response_format={"type": "json_object"},   # forces JSON, no prose
    max_tokens=200
)

# ── Technique 4: Measure compression ratio ───────────────────────────
def token_efficiency(text: str, model: str = "gpt-4o") -> dict:
    enc = tiktoken.encoding_for_model(model)
    tokens = len(enc.encode(text))
    chars = len(text)
    words = len(text.split())
    return {
        "tokens": tokens,
        "chars_per_token": round(chars / tokens, 2),
        "words_per_token": round(words / tokens, 2),
        "efficiency": "high" if chars / tokens > 4.5 else "low"
    }
🎯 The Token Optimization Hierarchy

In order of impact (biggest savings first):
1. Use a smaller model — switching GPT-4o → GPT-4o mini cuts cost by ~94% if quality holds.
2. Enable prompt caching — repeated system prompts cost 10% of normal after the first call.
3. Reduce output tokens — set max_tokens, ask for concise responses, use JSON instead of prose.
4. Use RAG instead of full context — retrieve 5 relevant chunks (500 tokens) instead of sending a 50K document.
5. Shorten input prompts — removing filler words saves tokens but has less impact than the above.
