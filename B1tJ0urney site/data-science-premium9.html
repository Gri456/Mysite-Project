🤖 9.1 — What is Generative AI? The New Paradigm

Generative AI refers to models capable of creating new content — text, code, images, audio — that never existed before. Unlike traditional ML models that predict a fixed label, generative models learn the underlying distribution of data and sample from it.

The revolution began with the Transformer architecture (Vaswani et al., 2017). Today, Large Language Models (LLMs) are pre-trained on trillions of tokens and can perform a vast range of tasks through natural language alone — a paradigm called in-context learning.

Raw Text (Internet) → Pre-training → Foundation Model → Fine-tuning → Specialised LLM → RLHF → Production LLM

Key Concepts

  • Tokens: The atomic units LLMs process. A token ≈ 0.75 words. GPT-4 has a context window of 128k tokens.
  • Temperature: Controls output randomness. 0 = deterministic, 2 = very creative/chaotic.
  • Context Window: The maximum number of tokens the model can "see" at once (input + output).
  • Hallucination: When the model generates confident but factually incorrect information — the #1 production risk.
  • Embeddings: Dense vector representations of text that capture semantic meaning, enabling similarity search.
💡 The LLMOps Challenge:

LLMs behave non-deterministically, have opaque reasoning, incur per-token costs, and can fail silently with confident-sounding wrong answers. Standard MLOps practices are insufficient — you need a new discipline built around evals, guardrails, cost control, and observability.

✍️ 9.2 — Prompt Engineering: The Art of Talking to LLMs

Prompt engineering is the discipline of crafting inputs to LLMs that reliably elicit the desired output. It is not a workaround — it is a core engineering skill in the GenAI stack.

Malicious users can embed instructions inside user-provided content to override your system prompt (e.g., "Ignore all previous instructions and..."). Always sanitise user input and use structural separators between system context and user data.

🔍 9.3 — RAG: Retrieval-Augmented Generation

RAG solves the two biggest limitations of LLMs: knowledge cutoffs (training data has a fixed date) and hallucination on private data (the model never saw your internal documents). By retrieving relevant context at inference time and injecting it into the prompt, the model answers grounded in real, up-to-date information.

User Query → Embed Query → Vector Search → Retrieve Chunks → Augment Prompt → LLM Answer

Building a RAG Pipeline from Scratch

from openai import OpenAI
import chromadb, textwrap

client = OpenAI()
chroma = chromadb.Client()
collection = chroma.create_collection("company_docs")

# --- INDEXING PHASE (done once, offline) ---
def index_documents(docs: list[dict]):
    """Chunk, embed, and store documents in the vector database."""
    for doc in docs:
        # Chunk text into overlapping windows
        chunks = chunk_text(doc["content"], size=500, overlap=50)
        
        for i, chunk in enumerate(chunks):
            # Create embedding via OpenAI
            embedding = client.embeddings.create(
                input=chunk,
                model="text-embedding-3-small"
            ).data[0].embedding
            
            collection.add(
                ids=[f"{doc['id']}_chunk_{i}"],
                embeddings=[embedding],
                documents=[chunk],
                metadatas=[{"source": doc["title"]}]
            )

def chunk_text(text: str, size: int, overlap: int) -> list[str]:
    words = text.split()
    chunks, start = [], 0
    while start < len(words):
        end = start + size
        chunks.append(" ".join(words[start:end]))
        start += size - overlap
    return chunks


# --- RETRIEVAL & GENERATION PHASE (at inference time) ---
def rag_query(user_question: str, k: int = 3) -> str:
    # 1. Embed the user's question
    q_embedding = client.embeddings.create(
        input=user_question,
        model="text-embedding-3-small"
    ).data[0].embedding

    # 2. Retrieve the top-k most relevant chunks
    results = collection.query(
        query_embeddings=[q_embedding],
        n_results=k
    )

    context = "\n\n---\n\n".join(results["documents"][0])
    sources = [m["source"] for m in results["metadatas"][0]]

    # 3. Augment prompt with retrieved context
    prompt = f"""Answer the question using ONLY the context below.
If the answer is not in the context, say "I don't have that information."

CONTEXT:
{context}

QUESTION: {user_question}

ANSWER:"""

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        temperature=0
    )

    return {
        "answer": response.choices[0].message.content,
        "sources": list(set(sources)),
        "tokens_used": response.usage.total_tokens
    }

🏗️ 9.4 — LLMOps: Operating LLMs in Production

LLMOps is the discipline of deploying, monitoring, evaluating, and improving LLM-powered applications in production. It extends MLOps with new concerns unique to foundation models: non-determinism, cost per-token, prompt versioning, and output quality assessment.

The LLMOps Stack

  • Orchestration: LangChain, LlamaIndex — coordinate multi-step LLM workflows and tool use.
  • Prompt Management: LangSmith, PromptLayer — version, A/B test, and track prompts as first-class assets.
  • Vector Stores: Pinecone, Weaviate, ChromaDB — efficient semantic search over millions of embeddings.
  • Observability: LangSmith, Helicone, Arize Phoenix — trace every LLM call with cost, latency, and token counts.
  • Evals: RAGAS, DeepEval — automated evaluation of answer quality, faithfulness, and retrieval relevance.
  • Guardrails: NeMo Guardrails, LlamaGuard — detect and filter harmful, off-topic, or sensitive outputs.

In LLMOps, you can't improve what you can't measure. Before shipping any LLM feature, define your evaluation dataset (golden Q&A pairs) and your metrics. Treat every prompt change like a code change — run evals before merging to production.

Cost Control & Caching Strategy

import hashlib, redis
from functools import wraps

redis_client = redis.Redis(host='localhost', port=6379, db=0)

def semantic_cache(ttl_seconds: int = 3600):
    """
    Cache LLM responses by prompt hash.
    Identical or near-identical prompts return cached answers instantly.
    Typical cost reduction: 30-60% in production.
    """
    def decorator(func):
        @wraps(func)
        def wrapper(prompt: str, **kwargs):
            # Create a deterministic cache key from the prompt
            cache_key = f"llm:{hashlib.sha256(prompt.encode()).hexdigest()}"
            
            # Check cache first
            cached = redis_client.get(cache_key)
            if cached:
                return {"response": cached.decode(), "cached": True, "cost": 0}
            
            # Call the LLM
            result = func(prompt, **kwargs)
            
            # Store in cache with expiry
            redis_client.setex(cache_key, ttl_seconds, result["response"])
            
            return {**result, "cached": False}
        return wrapper
    return decorator

@semantic_cache(ttl_seconds=86400)  # Cache for 24 hours

🔧 9.5 — Fine-Tuning vs. RAG vs. Prompting: When to Use What

This is the most strategic decision in any GenAI project. Getting it wrong wastes weeks of engineering and thousands in GPU costs.

PROMPT ENGINEERING ✓
  • New behaviour/format needed
  • Limited data available
  • Need to ship fast
  • Task is general purpose
RAG ✓
  • Private / dynamic knowledge
  • Knowledge cutoff is an issue
  • Answers need citations
  • Data changes frequently
FINE-TUNING ✓
  • Domain-specific style/tone
  • 1000+ labelled examples exist
  • Cost/latency critical at scale
  • Privacy: can't send data to API
⚠️ The Fine-Tuning Trap:

Most teams jump to fine-tuning too early. It's expensive, slow, and adds an ongoing maintenance burden. The 80/20 rule: 80% of GenAI use cases can be solved excellently with good prompts + RAG. Fine-tune only when those options are exhausted.

🤖 9.6 — AI Agents & Tool Use: LLMs That Act

An AI Agent is an LLM that doesn't just generate text — it takes actions. By giving models access to tools (web search, code execution, APIs, databases), they can reason over multiple steps, call external services, and achieve complex goals autonomously.

The core loop of an agent is called ReAct (Reason + Act): the model alternates between thinking about what to do and executing a tool call, iterating until the task is complete.

User Goal → Thought → Choose Tool → Action → Observe Result → Reflect → Final Answer

Building an Agent with Tool Calling (OpenAI)

import openai, json, requests

client = openai.OpenAI()

# --- Define tools the agent can call ---
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_stock_price",
            "description": "Returns the current stock price for a given ticker symbol.",
            "parameters": {
                "type": "object",
                "properties": {
                    "ticker": {
                        "type": "string",
                        "description": "Stock ticker, e.g. AAPL, MSFT, TSLA"
                    }
                },
                "required": ["ticker"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "calculate_portfolio_value",
            "description": "Calculates total portfolio value given holdings.",
            "parameters": {
                "type": "object",
                "properties": {
                    "holdings": {
                        "type": "object",
                        "description": "Dict of {ticker: shares}, e.g. {'AAPL': 10, 'MSFT': 5}"
                    }
                },
                "required": ["holdings"]
            }
        }
    }
]

# --- Tool implementations ---
def get_stock_price(ticker: str) -> dict:
    # In production: call a real financial API
    mock_prices = {"AAPL": 189.45, "MSFT": 415.20, "TSLA": 248.90}
    price = mock_prices.get(ticker.upper(), 0)
    return {"ticker": ticker, "price": price, "currency": "USD"}

def calculate_portfolio_value(holdings: dict) -> dict:
    total = 0
    breakdown = {}
    for ticker, shares in holdings.items():
        price_data = get_stock_price(ticker)
        value = price_data["price"] * shares
        breakdown[ticker] = {"shares": shares, "price": price_data["price"], "value": value}
        total += value
    return {"total_usd": round(total, 2), "breakdown": breakdown}

# --- Agentic loop ---
def run_agent(user_message: str) -> str:
    messages = [{"role": "user", "content": user_message}]
    
    while True:
        response = client.chat.completions.create(
            model="gpt-4o",
            messages=messages,
            tools=tools,
            tool_choice="auto"
        )
        
        msg = response.choices[0].message
        
        # If no tool call → agent has finished
        if not msg.tool_calls:
            return msg.content
        
        # Execute each tool the agent requested
        messages.append(msg)
        for tool_call in msg.tool_calls:
            fn_name = tool_call.function.name
            fn_args = json.loads(tool_call.function.arguments)
            
            # Dispatch to the correct function
            if fn_name == "get_stock_price":
                result = get_stock_price(**fn_args)
            elif fn_name == "calculate_portfolio_value":
                result = calculate_portfolio_value(**fn_args)
            
            # Feed the result back to the agent
            messages.append({
                "role": "tool",
                "tool_call_id": tool_call.id,
                "content": json.dumps(result)
            })

# Usage
answer = run_agent("I own 10 shares of AAPL and 5 of MSFT. What is my total portfolio value?")
print(answer)
# "Your portfolio is worth $2,970.50: AAPL (10 × $189.45 = $1,894.50) + MSFT (5 × $415.20 = $2,076.00)"

Multi-Agent Systems

For complex workflows, you can chain multiple specialised agents together. Each agent has a defined role and they pass results between each other — like a software team where each developer owns a module.

🧠 Orchestrator Agent

Receives the high-level goal, decomposes it into sub-tasks, delegates to specialist agents, and synthesises their outputs into a final result.

⚙️ Specialist Agent

Focused on one domain (e.g., data retrieval, code execution, email drafting). Receives a specific sub-task and returns a structured result to the orchestrator.

💡 When to use Agents: Use agents when the task requires dynamic decision-making (you don't know the exact steps in advance), multiple tool calls, or iterative refinement. For simple, single-step tasks, a direct LLM call is faster and cheaper.

🧠 9.7 — Memory & State in LLM Applications

LLMs are stateless by default — each API call is independent, with no memory of previous conversations. Building memory is essential for chatbots, agents, and any multi-turn application. There are four fundamentally different types of memory, each with different trade-offs.

Memory Type How It Works Best For Limitation
In-Context Append full conversation history to every prompt Short sessions (<10 turns) Hits context window limit quickly; expensive
Summary Periodically summarise old turns; keep summary in prompt Medium-length sessions Loses specific details during compression
Vector (Episodic) Embed past interactions; retrieve relevant ones at query time Long-term personal assistants Retrieval can miss edge cases; adds latency
External (DB) Write structured facts to a DB; query it when needed CRM-style user profiles, task tracking Requires explicit write/read logic

Implementing Sliding Window + Summary Memory

from openai import OpenAI
from collections import deque

client = OpenAI()

class ConversationMemory:
    def __init__(self, window_size: int = 6, max_summary_tokens: int = 300):
        self.window = deque(maxlen=window_size)  # Keep last N turns in full
        self.summary = ""                          # Compressed older history
        self.window_size = window_size
    
    def add_turn(self, role: str, content: str):
        self.window.append({"role": role, "content": content})
    
    def compress(self):
        """Summarise the oldest turns when window is full."""
        if len(self.window) < self.window_size:
            return
        
        # Take the oldest half of the window and compress it
        to_compress = list(self.window)[:self.window_size // 2]
        history_text = "\n".join(f"{m['role']}: {m['content']}" for m in to_compress)
        
        response = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{
                "role": "user",
                "content": f"Summarise this conversation in 2-3 sentences, preserving key facts:\n\n{history_text}"
            }],
            max_tokens=150
        )
        self.summary = response.choices[0].message.content
        
        # Remove compressed turns from window
        for _ in range(self.window_size // 2):
            self.window.popleft()
    
    def build_messages(self, system_prompt: str, user_input: str) -> list:
        messages = [{"role": "system", "content": system_prompt}]
        
        # Inject compressed history if it exists
        if self.summary:
            messages.append({
                "role": "system",
                "content": f"[Previous conversation summary]: {self.summary}"
            })
        
        # Add recent turns from the sliding window
        messages.extend(list(self.window))
        messages.append({"role": "user", "content": user_input})
        return messages

# --- Usage ---
memory = ConversationMemory(window_size=6)
system = "You are a helpful financial advisor. Remember user preferences."

def chat(user_input: str) -> str:
    messages = memory.build_messages(system, user_input)
    
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=messages,
        temperature=0.7
    )
    
    assistant_reply = response.choices[0].message.content
    
    # Store both turns in memory
    memory.add_turn("user", user_input)
    memory.add_turn("assistant", assistant_reply)
    memory.compress()  # Compress if needed
    
    return assistant_reply
⚠️ The Context Window Cost Trap:

Naively appending the full chat history to every API call means your costs grow quadratically with conversation length. A 100-turn conversation can cost 50× more than a 10-turn one. Always implement a memory strategy — even a simple sliding window dramatically reduces costs without sacrificing quality.

🛡️ 9.8 — Guardrails, Safety & Responsible GenAI

Deploying an LLM without guardrails is like opening a server with no firewall. LLMs can generate harmful content, leak sensitive data, be manipulated into ignoring your policies, or confidently give dangerous advice. Guardrails are the safety layer that sits around your LLM calls in production.

The Three Layers of GenAI Safety

🔒 Input Guardrails

Validate, sanitise, and filter user input before it reaches the LLM. Detect prompt injection, PII, toxic language, or off-topic requests.

⚙️ Model-Level Safety

System prompt hardening, instruction hierarchy, temperature capping. The model itself is configured to resist manipulation and stay on-task.

🔍 Output Guardrails

Validate model output before showing it to users. Check for hallucination signals, blocked topics, PII leakage, or format violations.

Custom Output Validator

import re
from dataclasses import dataclass

@dataclass
class ValidationResult:
    passed: bool
    issues: list[str]
    sanitised_output: str

def validate_llm_output(raw_output: str, config: dict) -> ValidationResult:
    issues = []
    output = raw_output
    
    # 1. Check for PII patterns
    email_pattern = r'\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}\b'
    phone_pattern = r'\b(\+?\d[\d\s\-()]{7,}\d)\b'
    
    if re.search(email_pattern, output):
        issues.append("PII: email address detected in output")
        output = re.sub(email_pattern, "[EMAIL REDACTED]", output)
    
    if re.search(phone_pattern, output):
        issues.append("PII: phone number detected in output")
        output = re.sub(phone_pattern, "[PHONE REDACTED]", output)
    
    # 2. Check for blocked topics
    for blocked_topic in config.get("blocked_topics", []):
        if blocked_topic.lower() in output.lower():
            issues.append(f"Blocked topic detected: '{blocked_topic}'")
            return ValidationResult(passed=False, issues=issues, sanitised_output="")
    
    # 3. Check minimum length (guard against empty responses)
    if len(output.strip()) < config.get("min_length", 10):
        issues.append("Output too short — possible model refusal or error")
    
    # 4. Detect hallucination signals
    uncertainty_phrases = ["I think", "I believe", "I'm not sure", "might be", "possibly"]
    if config.get("require_factual") and any(p in output for p in uncertainty_phrases):
        issues.append("Uncertainty language detected in factual-mode response")
    
    return ValidationResult(
        passed=len([i for i in issues if "Blocked" in i]) == 0,
        issues=issues,
        sanitised_output=output
    )

# Usage in production
def production_llm_call(prompt: str) -> str:
    raw = call_llm(prompt)  # Your LLM wrapper
    
    result = validate_llm_output(raw, config={
        "blocked_topics": ["competitor pricing", "internal salaries"],
        "min_length": 20,
        "require_factual": True
    })
    
    if not result.passed:
        return "I'm sorry, I can't provide that information."
    
    if result.issues:
        log_warning(f"Output had issues: {result.issues}")  # Log for review
    
    return result.sanitised_output
💡 The Responsible GenAI Checklist: Before shipping any LLM feature, verify: (1) Input sanitisation against prompt injection; (2) Output validation for PII and blocked topics; (3) Rate limiting per user to prevent abuse; (4) Audit logging of every LLM call for compliance; (5) Human-in-the-loop escalation for high-stakes decisions; (6) A clear disclosure that users are interacting with an AI.

⚡ 9.9 — Performance Optimisation: Speed, Cost & Scalability

In production, LLM cost and latency are engineering problems, not afterthoughts. A single GPT-4o call costs ~$0.005–$0.015. At 100,000 daily users making 5 calls each, that's $2,500–$7,500 per day — before any optimisation. The following techniques can reduce this by 60–90%.

Model Routing: Right-Size Every Request

from openai import OpenAI
import re

client = OpenAI()

def classify_query_complexity(query: str) -> str:
    """
    Route queries to the cheapest model that can handle them.
    Simple queries → gpt-4o-mini ($0.15/1M tokens)
    Complex queries → gpt-4o ($5.00/1M tokens)  — 33× more expensive!
    """
    query_lower = query.lower()
    
    # Signals of HIGH complexity
    complex_signals = [
        len(query.split()) > 50,                           # Long query
        any(w in query_lower for w in ["analyse", "compare", "reason", "evaluate"]),
        "?" in query and query.count("?") > 1,             # Multiple questions
        re.search(r'\d+[\+\-\*/]\d+', query) is not None, # Math expressions
    ]
    
    # Signals of LOW complexity
    simple_signals = [
        len(query.split()) < 15,
        any(w in query_lower for w in ["hello", "thanks", "what is", "define"]),
        query.endswith("?") and len(query) < 60,
    ]
    
    complex_score = sum(complex_signals)
    simple_score = sum(simple_signals)
    
    if complex_score >= 2:
        return "gpt-4o"           # Heavy-duty reasoning
    elif simple_score >= 2:
        return "gpt-4o-mini"      # Fast & cheap
    else:
        return "gpt-4o-mini"      # Default to cheap; escalate if needed

def routed_llm_call(query: str, system: str = "") -> dict:
    model = classify_query_complexity(query)
    
    response = client.chat.completions.create(
        model=model,
        messages=[
            {"role": "system", "content": system},
            {"role": "user", "content": query}
        ]
    )
    
    cost_per_1m = {"gpt-4o": 5.00, "gpt-4o-mini": 0.15}
    tokens = response.usage.total_tokens
    cost = (tokens / 1_000_000) * cost_per_1m[model]
    
    return {
        "answer": response.choices[0].message.content,
        "model_used": model,
        "tokens": tokens,
        "cost_usd": round(cost, 6)
    }

# Without streaming: user waits 5-15 seconds for the full response
# With streaming: first token appears in <1s, reducing perceived latency by 80%

import asyncio

async def stream_response(prompt: str):
    """Stream tokens as they are generated — dramatically improves UX."""
    stream = await client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[{"role": "user", "content": prompt}],
        stream=True  # ← This is all it takes
    )
    
    full_response = ""
    async for chunk in stream:
        delta = chunk.choices[0].delta.content
        if delta:
            print(delta, end="", flush=True)  # Display as it arrives
            full_response += delta
    
    return full_response

from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()

@app.post("/chat/stream")
async def chat_stream(query: str):
    async def token_generator():
        stream = await client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[{"role": "user", "content": query}],
            stream=True
        )
        async for chunk in stream:
            delta = chunk.choices[0].delta.content
            if delta:
                yield f"data: {delta}\n\n"  # Server-Sent Events format
    
    return StreamingResponse(token_generator(), media_type="text/event-stream")

Batch Processing for Offline Workloads

# OpenAI Batch API: 50% cost discount for non-real-time workloads
# Ideal for: nightly report generation, bulk classification, offline enrichment

import json
from openai import OpenAI

client = OpenAI()

def create_batch_job(requests: list[dict]) -> str:
    """
    Submit 1000s of requests as a batch — processed within 24h at half price.
    """
    # Format as JSONL
    batch_lines = []
    for i, req in enumerate(requests):
        batch_lines.append(json.dumps({
            "custom_id": f"request-{i}",
            "method": "POST",
            "url": "/v1/chat/completions",
            "body": {
                "model": "gpt-4o-mini",
                "messages": [{"role": "user", "content": req["prompt"]}],
                "max_tokens": req.get("max_tokens", 500)
            }
        }))
    
    # Upload the JSONL file
    batch_file = client.files.create(
        file=("\n".join(batch_lines)).encode(),
        purpose="batch"
    )
    
    # Submit the batch job
    batch = client.batches.create(
        input_file_id=batch_file.id,
        endpoint="/v1/chat/completions",
        completion_window="24h"
    )
    
    print(f"✅ Batch submitted: {batch.id}")
    print(f"💰 Expected cost saving: ~50% vs. real-time API")
    return batch.id

# Check status later
def get_batch_results(batch_id: str) -> list[dict]:
    batch = client.batches.retrieve(batch_id)
    
    if batch.status != "completed":
        print(f"Status: {batch.status} — not ready yet")
        return []
    
    # Download and parse results
    result_file = client.files.content(batch.output_file_id)
    results = []
    for line in result_file.text.strip().split("\n"):
        item = json.loads(line)
        results.append({
            "id": item["custom_id"],
            "answer": item["response"]["body"]["choices"][0]["message"]["content"]
        })
    return results
