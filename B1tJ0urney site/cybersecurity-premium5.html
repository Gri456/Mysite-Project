Defensive Security Fundamentals

Defensive security (Blue Team) focuses on protecting systems, detecting threats and responding to incidents before they cause real damage.

In this module you will learn about SOC structure, log analysis, SIEM, IDS/IPS, incident response and system hardening — the core skills of any security analyst.

🏢 What is a SOC?

A Security Operations Center (SOC) is a team (and infrastructure) dedicated to monitoring, detecting, analysing and responding to security threats 24/7.

Tier 1 — Triage Analyst

Monitors alerts in real time, performs initial triage and decides whether it is a false positive or a real incident. This is the first response level — speed is essential.

Tier 2 — Incident Analyst

Investigates incidents escalated by Tier 1 in greater depth. Analyses correlated logs, timelines and indicators of compromise (IoCs).

Tier 3 — Threat Hunter / Specialist

Proactively hunts for threats that have not yet generated alerts. Develops new detection rules and responds to critical incidents.

Alert vs Incident

An alert is an automatically generated notification (may be a false positive). An incident is an alert confirmed as a real threat that requires a response. Knowing how to distinguish the two saves hours of unnecessary work.

📋 Log Analysis — Linux

Logs are the primary source of evidence in a SOC. Knowing where they are and how to read them is a fundamental skill.

# Essential log files on Linux
/var/log/auth.log        # SSH auth, sudo, su (Debian/Ubuntu)
/var/log/secure          # equivalent on CentOS/RHEL
/var/log/syslog          # general system log
/var/log/messages        # kernel and system messages
/var/log/kern.log        # kernel events
/var/log/apache2/        # Apache web server logs
/var/log/nginx/          # Nginx web server logs
/var/log/dpkg.log        # installed/removed packages

# Essential analysis commands
cat /var/log/auth.log                          # view everything (careful with large files)
tail -f /var/log/auth.log                      # monitor in real time
tail -n 100 /var/log/auth.log                  # last 100 lines
grep "Failed password" /var/log/auth.log       # filter failed logins
grep "Accepted password" /var/log/auth.log     # successful logins
grep "sudo" /var/log/auth.log                  # sudo usage

# Count failures per IP (brute force detection)
grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -rn | head -20

# View recent logins
last                    # login history
lastb                   # failed login attempts
lastlog                 # last login for each user
who                     # who is currently logged in
w                       # who is logged in and what they are doing
journalctl — modern logs

On systems with systemd (most modern distros), use journalctl to access centralised logs:

journalctl -xe                     # latest events with details
journalctl -u ssh                  # SSH service logs
journalctl --since "1 hour ago"    # last hour
journalctl --since "2024-01-01" --until "2024-01-02"
journalctl -p err                  # errors only
journalctl -f                      # follow mode (real time)

🚨 Threat Detection in Logs

Recognising suspicious patterns in logs is the most important skill of a Tier 1/2 analyst.

## 1. SSH BRUTE FORCE — many failures from the same IP
grep "Failed password" /var/log/auth.log | grep "192.168.1.50" | wc -l
# If > 20 attempts in a short period → likely brute force

## 2. LOGIN OUTSIDE HOURS — access at 3am
grep "Accepted" /var/log/auth.log | grep "03:"

## 3. LOGIN FROM GEOGRAPHICALLY UNLIKELY IP
grep "Accepted" /var/log/auth.log | awk '{print $11}'
# Check IPs at https://ipinfo.io or abuseipdb.com

## 4. SUDO BY UNEXPECTED USER
grep "sudo" /var/log/auth.log | grep -v "root\|admin"

## 5. NEW USER CREATION (possible persistence)
grep "new user\|useradd" /var/log/auth.log

## 6. SUSPICIOUS RUNNING PROCESSES
ps aux                             # all processes
ps aux | grep -i "nc\|netcat\|bash -i"   # suspicious shells
top                                # real-time processes
htop                               # improved version of top

## 7. SUSPICIOUS NETWORK CONNECTIONS
ss -tlnp                           # open ports and processes
netstat -tulnp                     # alternative to ss
ss -anp | grep ESTABLISHED         # active connections
Indicators of Compromise (IoCs)

IoCs are evidence that a system has been compromised: malicious IPs, hashes of suspicious files, C2 (command and control) domains, or processes with unusual names. Public databases like VirusTotal, AbuseIPDB and Shodan help verify IoCs.

🖥️ SIEM — Security Information and Event Management

SIEM is the heart of a SOC. It centralises logs from multiple sources, correlates events and generates automatic alerts when it detects suspicious patterns.

What does a SIEM do?

Collects logs from firewalls, servers, endpoints, applications and network devices. Normalises the data into a common format. Correlates events from multiple sources to detect complex attacks. Alerts the team when rules are triggered.

Splunk — Query Example

Splunk uses its own search language (SPL). Example of brute force detection:

# Splunk SPL — detect SSH brute force
index=linux sourcetype=syslog "Failed password"
| stats count by src_ip
| where count > 20
| sort -count

# ELK Stack — Kibana/Elasticsearch query (KQL)
message: "Failed password" and event.dataset: "system.auth"

# ELK — group by attacking IP
{
  "aggs": {
    "top_ips": {
      "terms": { "field": "source.ip", "size": 10 }
    }
  }
}
Most used SIEMs

Splunk — market leader, very powerful but expensive. ELK Stack (Elasticsearch + Logstash + Kibana) — open source, widely used in labs and mid-sized companies. Microsoft Sentinel — native Azure cloud. QRadar (IBM) — common in large enterprises. For learning, use ELK Stack or Wazuh (open source with SIEM + EDR).

🛡️ IDS vs IPS — Detection and Prevention

IDS and IPS are systems that analyse network traffic looking for malicious patterns, with a critical difference in how they react.

IDS — Intrusion Detection System

Monitors traffic passively and alerts when it detects something suspicious. It does not block anything — it only observes and reports. Like an alarm without security.

IPS — Intrusion Prevention System

Operates inline (in the middle of traffic) and can actively block malicious packets in real time. Like a guard who stops the attacker at the door.

NIDS vs HIDS

NIDS (Network-based) monitors network traffic — sees attacks at network level. HIDS (Host-based) runs on a specific server — monitors local files, processes and logs. Wazuh is an excellent open source HIDS.

# Snort — most popular open source IDS/IPS
snort -v                            # verbose mode (shows packets)
snort -A console -c /etc/snort/snort.conf  # with rules

# Example Snort rule (detect port scan)
alert tcp any any -> $HOME_NET any (msg:"Port Scan Detected"; flags:S; threshold:type both, track by_src, count 20, seconds 5; sid:1000001; rev:1;)

# Suricata — modern alternative to Snort
suricata -c /etc/suricata/suricata.yaml -i eth0

# View generated alerts
tail -f /var/log/suricata/fast.log

🔥 Incident Response — PICERL Process

Incident response follows a structured process to minimise impact and ensure the attacker is expelled and does not return.

1. Preparation

Have playbooks ready, tools installed, team trained and ticketing system configured before any incident. Preparation defines response speed.

2. Identification

Confirm the alert is a real incident (not a false positive). Collect initial evidence: logs, timestamps, IPs involved, affected systems.

3. Containment

Isolate the compromised system from the network to prevent spread. This may mean blocking an IP at the firewall, disconnecting a VM from the network, or revoking compromised credentials.

4. Eradication

Remove the threat from the system: delete malware, close backdoors, remove users created by the attacker, fix the exploited vulnerability.

5. Recovery

Restore the system to normal operation. Monitor actively in the following hours/days to ensure the attacker has not returned.

6. Lessons Learned

Post-incident meeting to document what happened, what went well, what failed and how to improve. The final report is essential to prevent recurrence.

⚡ Immediate Response Commands

When an incident is confirmed, these are the commands you use to investigate and contain quickly.

# QUICK INVESTIGATION OF COMPROMISED SYSTEM
whoami && id                       # confirm current user
ps aux --sort=-%cpu | head -20     # highest CPU processes
ss -anp | grep ESTABLISHED         # active connections and processes
netstat -tlnp                      # open ports
lsof -i                            # open files with connections
find /tmp /var/tmp -type f -newer /etc/passwd   # recent suspicious files
crontab -l                         # user scheduled tasks
cat /etc/crontab                   # system scheduled tasks
ls /etc/cron.d/                    # additional crons

# CONTAINMENT — block attacking IP
iptables -A INPUT -s 192.168.1.50 -j DROP
iptables -A OUTPUT -d 192.168.1.50 -j DROP
ufw deny from 192.168.1.50         # using UFW (simpler)

# KILL SUSPICIOUS PROCESS
kill -9 PID                        # terminate process by PID
pkill -f "process_name"            # terminate by name

# REVOKE ACTIVE SSH SESSION
who                                # view active sessions and TTY
pkill -9 -t pts/1                  # kill session on terminal pts/1

# LOCK COMPROMISED USER
passwd -l username                 # lock account
usermod -L username                # alternative

🎯 False Positives vs True Positives

Managing the false positive rate is one of the biggest challenges in a SOC. Too many false alerts cause alert fatigue and make the team ignore real alerts.

False Positive (FP)

The system generated an alert, but there is no real threat. Example: the antivirus scanner detects a legitimate pentest tool as malware. Reducing FPs improves SOC efficiency.

False Negative (FN)

There is a real threat, but no alert was generated. This is the most dangerous scenario — the attacker goes unnoticed. Outdated detection rules cause FNs.

True Positive (TP)

The alert was generated and a real threat exists. The SOC's goal. Well-tuned rules and current threat intelligence increase the TP rate.

Triage — Triage Process

When receiving an alert, the Tier 1 analyst checks: (1) is this known legitimate activity? (2) Does the IP/user have history? (3) Is there correlation with other events? Based on this, classifies as FP or escalates as an incident.

🔒 Hardening — Reducing the Attack Surface

Hardening is the process of securing a system by reducing its vulnerabilities before it is attacked. It is the proactive side of defence.

# SSH Hardening (/etc/ssh/sshd_config)
PermitRootLogin no              # do not allow direct login as root
PasswordAuthentication no       # use SSH keys only
Port 2222                       # change default port (makes scanning harder)
MaxAuthTries 3                  # maximum authentication attempts
AllowUsers alice bob            # only these users via SSH

# After editing, restart SSH
systemctl restart sshd

# Basic firewall with UFW
ufw enable
ufw default deny incoming
ufw default allow outgoing
ufw allow 22/tcp                # SSH (or the port you chose)
ufw allow 80/tcp                # HTTP
ufw allow 443/tcp               # HTTPS
ufw status verbose

# Check unnecessary services running
systemctl list-units --type=service --state=running
systemctl disable service_name  # disable unnecessary service

# Check permissions on sensitive files
ls -la /etc/shadow              # should be 640 or 000
ls -la /etc/passwd              # should be 644
chmod 640 /etc/shadow
chown root:shadow /etc/shadow
Principle of Least Privilege

Every user and service should have only the permissions strictly necessary for their function. A database user does not need SSH access. A web server does not need to run as root.

🔎 Threat Hunting — Proactive Threat Hunting

Threat hunting is the proactive search for threats already inside the network that have not yet generated alerts. It assumes the system is already compromised.

Hypothesis-driven hunting

The threat hunter creates a hypothesis (e.g. "an attacker may be using cron jobs for persistence") and then searches for evidence that confirms or refutes it in the available logs and telemetry.

# Hunting for persistence via cron
find /var/spool/cron /etc/cron* -type f | xargs ls -la
cat /etc/crontab
crontab -l -u www-data    # crons of the web server user

# Hunting for suspicious SUID binaries (recently installed)
find / -perm -4000 -newer /etc/passwd 2>/dev/null

# Hunting for beaconing connections (regular outbound traffic)
# Pattern: connections at fixed intervals to the same external IP
ss -anp | grep ESTABLISHED | awk '{print $5}' | cut -d: -f1 | sort | uniq -c | sort -rn

# Hunting via shell history (executed commands)
cat ~/.bash_history
find /home -name ".bash_history" -exec cat {} \;
cat /root/.bash_history

# Hunting for recently modified files in critical areas
find /etc /bin /usr/bin -newer /tmp -type f 2>/dev/null
find /var/www -name "*.php" -newer /etc/passwd    # webshells
Beaconing — C2 Signal

When a system is compromised, malware typically performs regular check-ins with the attacker's server (C2 — Command & Control). These regular, periodic connections are called beaconing and can be identified by network traffic analysis over time.
