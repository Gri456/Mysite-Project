🔐 4.1 The Shared Responsibility Model

Cloud security is a joint responsibility between the provider and the customer. Understanding where the boundary lies is the foundation of everything else.

┌─────────────────────────────────────────────────────────────┐ │ PROVIDER responsibility │ │ Physical security · Hardware · Hypervisor · Global network │ ├────────────────────┬────────────────┬───────────────────────┤ │ IaaS │ PaaS │ SaaS │ │ Customer manages: │ Provider mgmt: │ Provider manages: │ │ OS, middleware, │ OS + runtime │ Everything │ │ app, data, IAM, │ Customer: │ Customer: │ │ network config │ app, data, IAM │ data + access only │ └────────────────────┴────────────────┴───────────────────────┘
Key rule: The provider is responsible for security OF the cloud (infrastructure). The customer is responsible for security IN the cloud (data, identities, configurations).

🪪 4.2 IAM — Identity and Access Management

IAM is the foundation of cloud security. It answers: who can do what, on which resources, under what conditions.

  • Users: Human identities. Avoid long-lived access keys — use roles and SSO instead.
  • Groups: Collections of users sharing the same permissions. Manage at group level, not individual user level.
  • Roles: Identities assumed by services, applications, or users temporarily. Issue short-lived credentials.
  • Policies: JSON documents defining allowed/denied actions on resources.
  • Service Accounts: Machine identities for automated workloads (Lambda, EC2, K8s pods).
  • MFA: Mandatory for all privileged accounts and the root account.
  • SSO / Identity Federation: Centralise identity via IdP (Okta, Azure AD, Google Workspace).
# AWS IAM Policy — read-only S3 access (Principle of Least Privilege)
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadOnlyS3Access",
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket",
        "s3:GetObjectVersion"
      ],
      "Resource": [
        "arn:aws:s3:::my-app-data",
        "arn:aws:s3:::my-app-data/*"
      ],
      "Condition": {
        "StringEquals": {
          "aws:RequestedRegion": "eu-west-1"
        }
      }
    }
  ]
}
Principle of Least Privilege (PoLP): Grant only the minimum permissions required to perform a task. Nothing more. Review and prune permissions regularly.

🛡️ 4.3 Zero Trust Architecture

Traditional security assumed a trusted internal network. Zero Trust eliminates that assumption: never trust, always verify — regardless of where the request originates.

Old model: [Internet] → FIREWALL → [Internal network: TRUSTED ✓] "Trust but verify" — once inside, free movement Zero Trust: [Internet] → [Identity check] → [Device check] → [Context check] → [Resource] (every request verified) No implicit trust. Even internal traffic is untrusted.
  • Verify explicitly: Authenticate and authorise every request using identity, device health, location, and risk signals.
  • Least privilege access: Just-in-time (JIT) and just-enough-access (JEA) — grant permissions only when needed.
  • Assume breach: Design as if the attacker is already inside. Limit blast radius with micro-segmentation.
  • Micro-segmentation: Divide networks into small zones. A breach in one zone cannot easily spread to others.
  • Continuous validation: Re-verify trust at every step — session tokens expire, device health is re-checked.

🔒 4.4 Encryption — At Rest

Data at rest is stored data — in databases, object storage, disks, and backups. It must always be encrypted.

  • AES-256: Industry standard for symmetric encryption. Used by AWS, Azure, and GCP by default.
  • AWS KMS (Key Management Service): Managed service for creating, rotating, and auditing encryption keys. Integrates with S3, EBS, RDS, Secrets Manager.
  • Azure Key Vault: Equivalent managed key and secrets store for Azure resources.
  • GCP Cloud KMS: Google's equivalent — also supports Cloud HSM for hardware-backed keys.
  • Customer-Managed Keys (CMK): You control key rotation and revocation. Higher responsibility, higher control.
  • Envelope encryption: Data is encrypted with a data key; the data key itself is encrypted with a master key in KMS.
# Encrypt an S3 bucket with KMS (Terraform)
resource "aws_s3_bucket_server_side_encryption_configuration" "example" {
  bucket = aws_s3_bucket.data.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm     = "aws:kms"
      kms_master_key_id = aws_kms_key.app_key.arn
    }
    bucket_key_enabled = true  # reduces KMS API call costs
  }
}

🔐 4.5 Encryption — In Transit

Data in transit moves over networks. It must be encrypted to prevent interception and man-in-the-middle attacks.

  • TLS 1.2+: Minimum standard. TLS 1.3 is preferred for new deployments — faster handshake, stronger ciphers.
  • TLS certificates: Use AWS Certificate Manager (ACM), Let's Encrypt, or Azure-managed certs. Auto-renew.
  • mTLS (Mutual TLS): Both client and server present certificates. Used for service-to-service authentication in microservices.
  • HSTS: HTTP Strict Transport Security — forces browsers to only connect via HTTPS.
  • Cipher suites: Disable weak ciphers (RC4, 3DES, MD5). Use AES-GCM, ChaCha20-Poly1305.
  • End-to-end encryption: Ensure traffic is not decrypted at load balancers unless required for inspection.
# Enforce HTTPS-only on an AWS ALB (Terraform)
resource "aws_lb_listener" "http_redirect" {
  load_balancer_arn = aws_lb.main.arn
  port              = "80"
  protocol          = "HTTP"

  default_action {
    type = "redirect"
    redirect {
      port        = "443"
      protocol    = "HTTPS"
      status_code = "HTTP_301"
    }
  }
}

resource "aws_lb_listener" "https" {
  load_balancer_arn = aws_lb.main.arn
  port              = "443"
  protocol          = "HTTPS"
  ssl_policy        = "ELBSecurityPolicy-TLS13-1-2-2021-06"
  certificate_arn   = aws_acm_certificate.cert.arn
}

🗝️ 4.6 Secrets Management

Secrets — passwords, API keys, certificates, tokens — must never be stored in code, environment variables, or logs.

BAD: app/config.py → DB_PASSWORD = "super_secret_123" ← in Git! ❌ BAD: docker run -e DB_PASSWORD=super_secret_123 ← in shell history ❌ BAD: kubectl create secret generic db --from-literal=pass=secret ← base64 ≠ encrypted ❌ GOOD: App → AWS Secrets Manager → retrieve at runtime ✓ GOOD: App → HashiCorp Vault → dynamic secret (auto-expires) ✓ GOOD: K8s + External Secrets Operator → Vault/KMS → K8s Secret ✓
  • AWS Secrets Manager: Store, rotate, and retrieve secrets programmatically. Auto-rotation for RDS, Redshift, and custom Lambda functions.
  • HashiCorp Vault: Open-source secrets management platform. Dynamic secrets — generate short-lived DB credentials on demand.
  • Azure Key Vault: Secrets, keys, and certificates in one managed service.
  • External Secrets Operator: Sync secrets from Vault/AWS/Azure into Kubernetes Secrets automatically.
  • Detect secrets in code: git-secrets, gitleaks, TruffleHog — scan commits before push.
# Retrieve a secret from AWS Secrets Manager (Python)
import boto3, json

client = boto3.client('secretsmanager', region_name='eu-west-1')
response = client.get_secret_value(SecretId='prod/db/credentials')
secret = json.loads(response['SecretString'])
db_password = secret['password']  # never logged or hardcoded

# HashiCorp Vault — dynamic database credentials
vault write database/roles/app-role \
  db_name=my-postgres \
  creation_statements="CREATE ROLE..." \
  default_ttl="1h" \
  max_ttl="24h"

# Returns a unique username + password, expires in 1h automatically

📡 4.7 SIEM and Threat Detection

SIEM (Security Information and Event Management) aggregates logs and events from across your environment and correlates them to detect threats.

  • AWS CloudTrail: Logs every API call made in your AWS account — who did what, from where, and when. Essential for audit trails.
  • AWS GuardDuty: ML-powered threat detection. Analyses CloudTrail, VPC Flow Logs, and DNS logs to detect credential compromise, unusual API patterns, cryptomining, and port scanning.
  • AWS Security Hub: Aggregates findings from GuardDuty, Inspector, Macie, IAM Access Analyzer into a unified dashboard with compliance scores.
  • Azure Microsoft Sentinel: Cloud-native SIEM with AI-driven analytics and automation playbooks.
  • Splunk / Elastic SIEM: Popular enterprise SIEM solutions — ingest, index, and correlate logs at scale.
  • WAF (Web Application Firewall): Inspect HTTP/HTTPS traffic and block OWASP Top 10 threats (SQL injection, XSS, RFI, bot attacks).
# CloudTrail alert — detect root account usage (Terraform)
resource "aws_cloudwatch_metric_alarm" "root_account_usage" {
  alarm_name          = "root-account-usage"
  comparison_operator = "GreaterThanOrEqualToThreshold"
  evaluation_periods  = 1
  metric_name         = "RootAccountUsage"
  namespace           = "CloudTrailMetrics"
  period              = 300
  statistic           = "Sum"
  threshold           = 1
  alarm_description   = "ALERT: Root account used — investigate immediately"
  alarm_actions       = [aws_sns_topic.security_alerts.arn]
}

🕵️ 4.8 Vulnerability Management

Vulnerabilities must be found and remediated before attackers exploit them. This requires scanning at every layer of the stack.

  • Container image scanning (Trivy): Scan Docker images for known CVEs in OS packages and application dependencies.
  • AWS Inspector / ECR scanning: Automated vulnerability assessment for EC2 instances and container images.
  • DAST (Dynamic Application Security Testing): Test running applications — OWASP ZAP, Burp Suite.
  • SAST (Static Application Security Testing): Scan source code before deployment — Snyk Code, SonarQube, Semgrep.
  • CSPM (Cloud Security Posture Management): Continuously scan cloud configs for misconfigurations — Wiz, Prisma Cloud, Orca Security.
  • Penetration testing: Red team exercises to find what scanners miss. AWS allows pen testing with prior notification.
# Trivy — scan a container image in CI
trivy image --severity HIGH,CRITICAL \
  --exit-code 1 \
  myregistry/api:v2.1.0

📋 4.9 Compliance Frameworks

Compliance frameworks define standards for how organisations must handle security, privacy, and data protection. Cloud providers offer compliance-ready infrastructure, but the customer must configure and operate it correctly.

SOC 2 Type II

Audits organisational controls over security, availability, processing integrity, confidentiality, and privacy. Type II covers a period of time (usually 6–12 months). Essential for B2B SaaS.

ISO 27001

International standard for Information Security Management Systems (ISMS). Requires a formal risk assessment process, documented controls, and annual audits. Globally recognised.

GDPR

EU regulation governing personal data. Requires data minimisation, consent, right to erasure, breach notification within 72h, and data residency controls. Fines up to 4% of global revenue.

PCI DSS

Payment Card Industry Data Security Standard. Required for any organisation that stores, processes, or transmits cardholder data. 12 requirements covering network, access, monitoring, and testing.

HIPAA

US Health Insurance Portability and Accountability Act. Protects Protected Health Information (PHI). Requires BAAs with cloud providers and strict access controls on health data.

NIS2 / DORA

EU directives for critical infrastructure (NIS2) and financial sector resilience (DORA). Require incident reporting, supply chain security, and business continuity planning.

Cloud provider compliance evidence: AWS Artifact, Azure Compliance Manager, and GCP Compliance Hub provide audit reports, certifications, and shared responsibility documentation for regulators.

🔍 4.10 IAM Security Best Practices

# AWS: List all users with console access and no MFA
aws iam generate-credential-report
aws iam get-credential-report --query 'Content' --output text \
  | base64 -d | csvtool col 1,4,8 - \
  | awk -F',' '$2=="true" && $3=="false"'

# Detect over-permissive policies (wildcard actions)
aws iam list-policies --scope Local \
  | jq '.Policies[].PolicyName' \
  | xargs -I{} aws iam get-policy-version \
    --policy-arn arn:aws:iam::123456789:policy/{} \
    --version-id v1 \
  | jq 'select(.PolicyVersion.Document.Statement[].Action == "*")'
  • Enable MFA for all human users — especially privileged accounts and root.
  • Never use root credentials for day-to-day operations. Lock root with MFA.
  • Rotate access keys regularly. Use IAM Access Analyzer to detect unused permissions.
  • Use IAM roles with AssumeRole — no long-lived keys on EC2 instances or Lambda.
  • Enable IAM Access Analyzer — alerts when resources are shared outside your account.
  • Use Service Control Policies (SCPs) in AWS Organizations to enforce guardrails across all accounts.

🌐 4.11 Network Security in the Cloud

  • Security Groups: Stateful instance-level firewall — define inbound/outbound rules by port, protocol, and CIDR or security group source.
  • NACLs (Network ACLs): Stateless subnet-level firewall — evaluated in order, explicit deny is possible.
  • VPC Flow Logs: Capture metadata for all traffic in your VPC. Essential for detecting lateral movement and anomalous traffic.
  • AWS Network Firewall: Managed deep packet inspection with Suricata-compatible rules.
  • PrivateLink: Access services without exposing traffic to the public internet.
  • Shield / DDoS protection: AWS Shield Standard (free) + Shield Advanced for layer 3/4/7 DDoS mitigation.
# Restrictive Security Group — allow HTTPS only from CloudFront
resource "aws_security_group_rule" "https_from_cloudfront" {
  type              = "ingress"
  from_port         = 443
  to_port           = 443
  protocol          = "tcp"
  security_group_id = aws_security_group.alb.id
  # Only allow traffic from CloudFront managed prefix list
  prefix_list_ids   = ["pl-fab65393"]  # AWS CloudFront PL
  description       = "HTTPS from CloudFront only"
}

⚙️ 4.12 Security Automation and DevSecOps

Security must be integrated into every stage of the development pipeline — "shift left" so issues are caught early and cheaply.

DevSecOps Pipeline: Code → SAST (Semgrep, SonarQube) ← catch logic bugs Commit → Secret scanning (gitleaks) ← stop credential leaks Build → Container scan (Trivy, Snyk) ← patch CVEs before deploy Deploy → IaC scan (Checkov, tfsec) ← fix misconfigs before apply Runtime → CSPM + SIEM (Wiz, GuardDuty) ← detect anomalies live Audit → CloudTrail + compliance reports ← evidence for regulators
  • Checkov / tfsec: Scan Terraform and CloudFormation for security misconfigurations before applying.
  • gitleaks / TruffleHog: Scan Git repositories for accidentally committed secrets.
  • OPA (Open Policy Agent): Policy-as-code engine — enforce custom security rules at any layer.
  • AWS Config: Continuously monitors AWS resource configurations against desired-state rules. Auto-remediation actions available.
# Checkov — scan Terraform for security issues
checkov -d . --framework terraform \
  --check CKV_AWS_18,CKV_AWS_19,CKV_AWS_20

# gitleaks — detect secrets in git history
gitleaks detect --source . --verbose

# tfsec — inline security scanning
tfsec . --minimum-severity HIGH

🔑 4.13 Certificate and PKI Management

  • ACM (AWS Certificate Manager): Provision, manage, and auto-renew TLS certificates. Free for ALB, CloudFront, and API Gateway.
  • Private CA: ACM Private CA or Vault PKI Secrets Engine — issue internal certificates for service-to-service mTLS.
  • Certificate rotation: Automate rotation — manually managed certs expire and cause outages.
  • Certificate Transparency (CT): Monitor CT logs (crt.sh) for unauthorised certificates issued for your domain.
  • SPIFFE/SPIRE: Open standard for workload identity — issues short-lived X.509 certificates to services automatically.

🚨 4.14 Incident Response in the Cloud

  • Preparation: Define playbooks for common incidents (credential compromise, data breach, ransomware). Practice tabletop exercises.
  • Detection: GuardDuty + SIEM alerts. Tune thresholds — too many alerts = alert fatigue.
  • Containment: Isolate compromised resources — revoke credentials, quarantine EC2 instances, snapshot for forensics.
  • Eradication and recovery: Remove persistence mechanisms, rotate secrets, redeploy from clean IaC state.
  • Post-incident: Blameless postmortem, update runbooks, improve detection rules.
  • AWS: Use IAM policies to quarantine (attach deny-all policy). Use Systems Manager for forensic data collection without SSH.
# Quarantine a compromised IAM user immediately
aws iam put-user-policy \
  --user-name compromised-user \
  --policy-name QUARANTINE \
  --policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*"
    }]
  }'

# Revoke all active sessions
aws iam create-policy-version \
  --policy-arn arn:aws:iam::123456789:policy/DenyAll \
  --policy-document file://deny-all.json \
  --set-as-default
