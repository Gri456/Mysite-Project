☁️ 1.1 What is Cloud Computing?

Cloud computing is the delivery of computing services — servers, storage, databases, networking, software — over the internet ("the cloud") to offer faster innovation, flexible resources, and economies of scale.

Core characteristics (NIST definition):
• On-demand self-service — provision resources without human interaction.
• Broad network access — accessible over standard networks from any device.
• Resource pooling — multi-tenant model serving multiple consumers.
• Rapid elasticity — scale up/down quickly based on demand.
• Measured service — pay only for what you use.

🏗️ 1.2 Service Models: IaaS, PaaS, SaaS

  • IaaS (Infrastructure as a Service): Raw compute, storage, and network. You manage OS and above. Examples: AWS EC2, Azure VMs, GCP Compute Engine.
  • PaaS (Platform as a Service): Managed runtime for deploying applications. Provider manages infrastructure + OS. Examples: Heroku, AWS Elastic Beanstalk, Google App Engine.
  • SaaS (Software as a Service): Fully managed applications. You just use the software. Examples: Gmail, Salesforce, Slack.
A useful analogy: IaaS = renting land, PaaS = renting a kitchen, SaaS = ordering a meal.

🌍 1.3 Deployment Models

  • Public Cloud: Resources owned and operated by a third-party provider. Shared infrastructure, pay-as-you-go. Best for scalable workloads.
  • Private Cloud: Dedicated infrastructure for a single organisation. Higher control and security, higher cost.
  • Hybrid Cloud: Combination of public and private. Workloads move between environments. Enables data sovereignty + cloud flexibility.
  • Multi-Cloud: Using multiple cloud providers. Avoids vendor lock-in, improves resilience.

🗺️ 1.4 Regions, Availability Zones and Edge Locations

Major providers divide their global infrastructure into geographic areas for fault tolerance and low latency.

  • Region: A geographic area containing multiple data centers (e.g., eu-west-1, West Europe).
  • Availability Zone (AZ): Isolated data centers within a region. Deploying across AZs ensures high availability.
  • Edge Locations: CDN endpoints closer to end users to reduce latency (used by CloudFront, Azure CDN, etc.).
Best practice: Always deploy production workloads across at least 2 AZs for resilience.

☁️ 1.5 Major Cloud Providers

  • AWS (Amazon Web Services): Market leader, widest service catalogue (~200+ services). Strong in compute, storage, serverless (Lambda).
  • Microsoft Azure: Strong enterprise adoption, deep Active Directory and .NET integration. Leader in hybrid cloud.
  • GCP (Google Cloud Platform): Strong in data analytics, Kubernetes (GKE originated here), and AI/ML services.
  • Others: Oracle Cloud, IBM Cloud, Alibaba Cloud — popular in specific industries or regions.

📐 1.6 Cloud Architecture Patterns

Modern cloud architectures are built around proven patterns for scalability, resilience, and cost efficiency.

  • Monolithic vs Microservices: Monoliths are single deployable units. Microservices decompose an application into small, independently deployable services.
  • Event-Driven Architecture: Services communicate via events (Kafka, SQS, EventBridge). Decoupled and scalable.
  • Serverless: Execute code without managing servers. Pay per invocation (AWS Lambda, Azure Functions).
  • CQRS + Event Sourcing: Separate read and write models; store state changes as a sequence of events.

🔩 1.7 Well-Architected Framework

AWS, Azure, and GCP all publish well-architected frameworks. AWS's is the most widely referenced, built on 6 pillars:

  • Operational Excellence — run and monitor systems effectively.
  • Security — protect data and systems.
  • Reliability — recover from failures automatically.
  • Performance Efficiency — use resources efficiently.
  • Cost Optimisation — avoid unnecessary costs.
  • Sustainability — minimise environmental impact.

🔄 1.8 Shared Responsibility Model

Security and compliance in the cloud is a shared responsibility between the provider and the customer.

Provider is responsible for: Security OF the cloud (physical hardware, networking, hypervisor).
Customer is responsible for: Security IN the cloud (data, IAM, OS patching, encryption, network configuration).
  • In IaaS — customer manages most of the stack.
  • In PaaS — provider takes over more of the responsibility.
  • In SaaS — customer is mainly responsible for data and access.

⚡ 1.9 Elasticity, Scalability and High Availability

  • Scalability: The ability to handle growing workloads — vertically (bigger instances) or horizontally (more instances).
  • Elasticity: Automatically scale resources up and down in response to demand.
  • High Availability (HA): System remains operational despite component failures. Target: 99.9%+ uptime.
  • Fault Tolerance: System continues operating even when components fail. Uses redundancy and failover.
  • Disaster Recovery (DR): Plan for recovering systems after catastrophic failure. Key metrics: RPO and RTO.

🗄️ 1.10 Core Cloud Services Overview

  • Compute: EC2 / Azure VMs / GCE — virtual machines to run workloads.
  • Object Storage: S3 / Azure Blob / GCS — store unstructured data at scale.
  • Managed Databases: RDS, DynamoDB / Azure SQL / Cloud Spanner.
  • Networking: VPC / VNet — isolated virtual networks.
  • IAM: Identity and Access Management — control who can do what.
  • CDN: CloudFront / Azure CDN / Cloud CDN — edge caching for performance.

📦 1.11 Containers and Kubernetes

Containers package applications and their dependencies into isolated units, ensuring consistent behaviour across environments.

# Build and run a container
docker build -t myapp:1.0 .
docker run -p 8080:80 myapp:1.0
  • Docker: The standard container runtime.
  • Kubernetes (K8s): Orchestrates containers at scale. Manages deployment, scaling, and self-healing.
  • Managed K8s: EKS (AWS), AKS (Azure), GKE (GCP) — removes the burden of managing the control plane.

💡 1.12 Cloud-Native Principles

  • Design for failure — assume components will fail.
  • Immutable infrastructure — replace, don't patch.
  • Stateless services where possible.
  • Automate everything — infrastructure, testing, deployment.
  • Observability built in from the start.
Cloud-native is not just "running in the cloud" — it's about leveraging cloud capabilities to build more resilient, scalable, and manageable systems.

🌐 1.13 Networking in the Cloud — VPC, Subnets and Security Groups

Every cloud workload lives inside a virtual network. Understanding how cloud networking is structured is essential for designing secure, well-connected systems.

  • VPC / VNet: A Virtual Private Cloud (AWS) or Virtual Network (Azure) is a logically isolated network you control. You define the IP address range, subnets, route tables, and gateways.
  • Public subnet: Resources here can reach the internet via an Internet Gateway. Load balancers and bastion hosts live here.
  • Private subnet: No direct internet access. Application servers and databases live here for security.
  • Security Groups: Stateful firewall rules at the instance level — define what traffic is allowed in and out.
  • Network ACLs: Stateless firewall rules at the subnet level — an additional layer of control.
  • NAT Gateway: Allows instances in a private subnet to initiate outbound internet connections (e.g. software updates) without being reachable from the internet.
Classic 3-tier layout:
• Public subnet — Load Balancer (ALB / NLB)
• Private subnet (app) — EC2 / ECS / Lambda
• Private subnet (data) — RDS / ElastiCache / DynamoDB

🔐 1.14 Identity and Access Management (IAM)

IAM is the foundation of cloud security. It controls who — humans and services — can perform what actions on which resources.

  • Users: Individual identities for humans. Best practice: use SSO via an identity provider (Okta, Azure AD, Google Workspace) rather than managing IAM users directly.
  • Groups: Collections of users sharing the same permissions. Assign policies to groups, not individuals.
  • Roles: Identities assumed by services, Lambda functions, or EC2 instances. No long-lived credentials — roles issue temporary tokens via STS.
  • Policies: JSON documents defining what actions are allowed or denied on which resources. Follow the principle of least privilege — grant only what is needed.
  • MFA: Multi-Factor Authentication is mandatory for all human access to production environments.
// IAM policy — allow read-only S3 access to a specific bucket
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:GetObject", "s3:ListBucket"],
    "Resource": [
      "arn:aws:s3:::my-app-bucket",
      "arn:aws:s3:::my-app-bucket/*"
    ]
  }]
}

🗄️ 1.15 Storage Types in the Cloud

Cloud providers offer multiple storage types for different workload requirements. Choosing the right one affects performance, cost, and durability.

  • Object Storage (S3 / Azure Blob / GCS): Unlimited flat storage for files, backups, static assets, and data lakes. Accessed via HTTP. 11 nines durability. Not mountable as a filesystem.
  • Block Storage (EBS / Azure Managed Disk): Persistent disk volumes attached to VMs. Low latency, suitable for databases and OS volumes. Tied to a single AZ.
  • File Storage (EFS / Azure Files / Filestore): Shared network filesystem (NFS / SMB). Mountable by multiple instances simultaneously — useful for shared content and CMS systems.
  • Ephemeral / Instance Storage: Physically attached to the host — extremely fast, but data is lost when the instance stops. Use for caches and temp files only.
  • Archive Storage (S3 Glacier / Azure Archive): Long-term cold storage at very low cost. Retrieval takes minutes to hours. Ideal for compliance archives.
Rule of thumb: Object storage for data at rest and distribution. Block storage for databases and VM disks. File storage for shared workloads needing concurrent access.

📊 1.16 Managed Databases — Relational and Non-Relational

Managed database services eliminate patching, backups, and replication complexity — letting teams focus on application logic.

  • Relational (SQL): Amazon RDS (MySQL, PostgreSQL, MariaDB), Aurora (MySQL/PostgreSQL-compatible, up to 5× faster than standard MySQL), Azure SQL Database, Cloud SQL. Best for structured data, ACID transactions, and complex joins.
  • Key-Value / Wide-Column: DynamoDB (AWS), Azure Cosmos DB, Bigtable. Single-digit millisecond reads at any scale. Best for sessions, carts, and leaderboards.
  • Document: MongoDB Atlas, Firestore. Flexible JSON schemas. Best for content management and product catalogues.
  • Time-Series: Amazon Timestream, InfluxDB Cloud. Optimised for metrics, IoT telemetry, and monitoring data with automatic downsampling.
  • Serverless databases: Aurora Serverless v2, DynamoDB on-demand — scale to zero, pay per request. Ideal for unpredictable or bursty workloads.

🚀 1.17 Auto Scaling and Load Balancing

Auto scaling and load balancing work together to handle variable traffic, maximise availability, and minimise cost by running only the capacity you need.

  • Auto Scaling Group (ASG): Automatically adds or removes EC2 instances based on policies (CPU%, custom CloudWatch metrics, schedules). Defines min, desired, and max capacity.
  • Application Load Balancer (ALB): Layer 7 (HTTP/HTTPS). Routes traffic based on URL path, hostname, or headers. Supports WebSockets and gRPC. Best for microservices and web apps.
  • Network Load Balancer (NLB): Layer 4 (TCP/UDP). Ultra-low latency, static IPs, handles millions of requests per second. Best for gaming, VoIP, and financial systems.
  • Target Groups: Logical groups of backends (EC2 instances, ECS tasks, Lambda functions, or IPs) that ALB/NLB routes traffic to. Each target is health-checked independently.
  • Health checks: Load balancers probe targets on a defined interval. Unhealthy targets are automatically removed from rotation until they recover.
# Target tracking scaling — maintain 70% average CPU
aws autoscaling put-scaling-policy   --auto-scaling-group-name my-asg   --policy-name cpu-target-70   --policy-type TargetTrackingScaling   --target-tracking-configuration '{
    "TargetValue": 70.0,
    "PredefinedMetricSpecification": {
      "PredefinedMetricType": "ASGAverageCPUUtilization"
    }
  }'

🔄 1.18 CI/CD and DevOps in the Cloud

Continuous Integration and Continuous Delivery are the engines that allow cloud teams to ship reliably and frequently without sacrificing quality.

  • Continuous Integration (CI): Developers merge code frequently — every push triggers automated build, lint, and test pipelines. Catches bugs early, before they compound.
  • Continuous Delivery (CD): Every passing build is automatically deployable. Release to production is either automatic (deployment) or a single manual approval step (delivery).
  • Pipeline tools: GitHub Actions, GitLab CI/CD, AWS CodePipeline, Azure DevOps, CircleCI, Jenkins. Most teams today use GitHub Actions for its simplicity and ecosystem.
  • Deployment strategies:
    • Blue/Green: Run two identical environments — switch traffic instantly. Easy rollback.
    • Canary: Gradually route a percentage of traffic to the new version. Monitor, then expand or rollback.
    • Rolling: Replace instances one batch at a time. Zero downtime, slower rollback.
  • GitOps: Git is the single source of truth for infrastructure and application config. ArgoCD and Flux automatically reconcile Kubernetes clusters to the desired state declared in Git.
The DevOps loop: Plan → Code → Build → Test → Release → Deploy → Operate → Monitor → (repeat)

🌩️ 1.19 Cloud Cost Management Fundamentals

Cloud bills scale with usage — without visibility and discipline, costs grow quickly. Understanding the pricing model early prevents expensive surprises.

  • On-Demand: Pay per second or minute with no commitment. Most flexible, highest unit price. Best for unpredictable or short-lived workloads.
  • Reserved Instances / Savings Plans: Commit to 1 or 3 years of usage in exchange for up to 72% discount. Best for stable, always-on workloads like production databases.
  • Spot Instances: Use unused cloud capacity at up to 90% discount. Can be interrupted with a 2-minute warning. Best for batch jobs, CI runners, and ML training.
  • Free Tier: AWS, Azure, and GCP all offer free tiers — useful for learning, prototyping, and small hobby projects.
  • Cost Explorer / Budgets: Native tools to visualise spend, identify top cost drivers, set budget thresholds, and receive alerts before overspending.
  • Right-sizing: Match instance type and size to actual workload demand. The most common source of waste is over-provisioned instances running at 10–20% CPU utilisation.
FinOps principle: Every engineer is responsible for cost. Tag all resources with team and environment, review Cost Explorer weekly, and treat cloud spend as an engineering metric.

🔒 1.20 Cloud Security Fundamentals

Security in the cloud is a continuous practice. These are the non-negotiable basics every cloud practitioner must understand from day one.

  • Encryption at rest: All storage (S3, EBS, RDS) must be encrypted using managed keys. AWS KMS, Azure Key Vault, and GCP Cloud KMS provide centralised key management. Often a compliance requirement.
  • Encryption in transit: All network communication must use TLS 1.2+. Use ACM (AWS Certificate Manager) for free, auto-renewing certificates.
  • Secrets management: Never hardcode passwords or API keys in code or IaC. Use AWS Secrets Manager, Azure Key Vault, or HashiCorp Vault — rotate secrets automatically on a schedule.
  • Cloud security posture: AWS Security Hub, Microsoft Defender for Cloud, and GCP Security Command Center continuously scan for misconfigurations and compliance violations.
  • Zero Trust: Never trust, always verify. Authenticate every request regardless of network origin. Apply least-privilege IAM everywhere — no wildcard permissions.
  • Compliance frameworks: SOC 2, ISO 27001, GDPR, HIPAA, PCI DSS — cloud providers publish shared responsibility documentation and compliance programmes for each standard.

📈 1.21 Monitoring, Logging and Observability Basics

You cannot manage what you cannot measure. Observability is built on three pillars — and cloud providers offer native tooling for all three out of the box.

  • Metrics: Numeric measurements over time — CPU%, memory, request rate, error rate, p99 latency. AWS CloudWatch, Azure Monitor, GCP Cloud Monitoring. Set alarms on key thresholds to trigger alerts or auto-scaling.
  • Logs: Timestamped event records from applications and infrastructure. CloudWatch Logs, Azure Log Analytics, GCP Cloud Logging. Aggregate centrally, query with structured log formats (JSON).
  • Traces: Follow a single request as it traverses multiple services. AWS X-Ray, Azure Application Insights, GCP Cloud Trace. Essential for debugging latency and errors in distributed microservices.
  • Dashboards: Grafana (open-source, connects to any data source), CloudWatch Dashboards, Azure Workbooks — visualise all signals in one operational view.
  • Alerting: PagerDuty, OpsGenie, or native cloud alerting routes notifications to the right person or team when thresholds are breached. Design alerts to be actionable — every alert should require a human response.
The three pillars: Logs tell you what happened. Metrics tell you how the system is behaving over time. Traces tell you where a specific request went wrong across services.
