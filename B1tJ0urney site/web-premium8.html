🟦 1.1 What is TypeScript and why use it?

JavaScript is dynamic and flexible — but that flexibility has a cost: bugs that only surface at runtime, code that's hard to refactor, and chaotic collaboration in large teams. TypeScript solves this by adding a static type system on top of JavaScript.


TypeScript compiles to JavaScript and runs in any environment where JavaScript runs. The tsc compiler validates your code before it executes, catching errors that would otherwise only appear in production.

🐛
Fewer Bugs

The compiler catches type errors before runtime

📚
Auto-complete

IDEs like VS Code offer intelligent suggestions based on types

🔄
Safe Refactoring

Renaming a property automatically updates all usages

🤝
Living Documentation

Types are documentation that the compiler verifies

INITIAL SETUP

# Install TypeScript globally
npm install -g typescript

# Check version
tsc --version

# Initialise a TypeScript project
tsc --init         # Creates tsconfig.json

# Compile a file
tsc app.ts

# Watch mode (recompiles on save)
tsc --watch

# In modern Node.js projects
npm install -D typescript ts-node @types/node
⚙️ tsconfig.json — Recommended Configuration for Serious Projects: 
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": true,            // Enables ALL strict checks
    "noUncheckedIndexedAccess": true,
    "noImplicitReturns": true,
    "noFallthroughCasesInSwitch": true,
    "exactOptionalPropertyTypes": true,
    "outDir": "./dist",
    "rootDir": "./src",
    "declaration": true,       // Generates .d.ts files
    "sourceMap": true
  }
}

Golden rule: always enable "strict": true. It may feel painful at first, but it saves days of debugging later.

🔤 1.2 Basic Types and Type Annotations

TypeScript adds type annotations to JavaScript. Most of the time the compiler infers types automatically — you don't need to annotate everything. But knowing the base types is essential.

PRIMITIVES AND BASIC TYPES

// Primitives
let name: string = "Alice";
let age: number = 30;
let active: boolean = true;
let nothing: null = null;
let undef: undefined = undefined;
let bigInt: bigint = 100n;
let sym: symbol = Symbol("id");

// Arrays
let scores: number[] = [1, 2, 3];
let tags: Array<string> = ["ts", "js"];

// Tuples — fixed-size arrays with fixed types
let point: [number, number] = [10, 20];
let entry: [string, number] = ["age", 25];

// Object types
let user: { name: string; age: number } = { name: "Bob", age: 25 };

// Automatic inference — TypeScript already knows the type!
let count = 42;           // TypeScript infers: number
let greeting = "Hello";  // TypeScript infers: string
count = "oops";           // ERROR: Type 'string' is not assignable to type 'number'
✅ any vs unknown 
// any — disables checks (avoid!)
let dangerous: any = "hello";
dangerous.toFixed(); // No error, but crashes

// unknown — type-safe "any"
let safe: unknown = getData();
// Must check before using:
if (typeof safe === "string") {
  console.log(safe.toUpperCase()); // OK
}

// never — value that never exists
function crash(): never {
  throw new Error("Fatal!");
}
🔀 Union and Intersection Types 
// Union — can be A OR B
type StringOrNumber = string | number;
let id: StringOrNumber = "abc-123";
id = 42; // also valid

// Intersection — has properties of A AND B
type Admin = User & { adminLevel: number };

// Literal types — exact values
type Direction = "north" | "south" | "east" | "west";
type DiceRoll = 1 | 2 | 3 | 4 | 5 | 6;

let dir: Direction = "north"; // OK
dir = "up"; // ERROR!

📐 1.3 Interfaces and Type Aliases — Structuring your Data

To describe the shape of complex objects, use interface or type. Both work very similarly, but have important differences.

INTERFACES — PREFERRED FOR OBJECTS AND CLASSES

interface User {
  id: number;
  name: string;
  email: string;
  age?: number;          // Optional property
  readonly createdAt: Date; // Read-only — can't change after creation
}

// Interface extension — inheritance
interface AdminUser extends User {
  permissions: string[];
  adminLevel: 1 | 2 | 3;
}

// Implementation in classes
class UserService implements User {
  id: number;
  name: string;
  email: string;
  readonly createdAt: Date;
  
  constructor(data: Omit<User, 'createdAt'>) {
    this.id = data.id;
    this.name = data.name;
    this.email = data.email;
    this.createdAt = new Date();
  }
}

// Declaration merging — you can extend existing interfaces
interface Window {
  myCustomProperty: string; // Add to the global Window type
}

TYPE ALIASES — PREFERRED FOR UNIONS, INTERSECTIONS AND COMPLEX TYPES

// Type alias for objects
type Product = {
  id: string;
  name: string;
  price: number;
  category: "electronics" | "clothing" | "food";
};

// Types for functions
type Transformer<T, R> = (input: T) => R;
type EventHandler = (event: Event) => void;

// Conditional types — types that depend on other types
type IsString<T> = T extends string ? "yes" : "no";
type Result1 = IsString<string>; // "yes"
type Result2 = IsString<number>; // "no"

// Template literal types
type EventName = "click" | "focus" | "blur";
type Handler = `on${Capitalize<EventName>}`;
// Handler = "onClick" | "onFocus" | "onBlur"
⚖️ Interface vs Type — When to use each?
Situation Use Why
Describing objects / classes interface Supports declaration merging and extends
Unions, intersections, tuples type Interfaces can't express unions
Public APIs / libraries interface Users can extend interfaces

🧬 1.4 Generics — Reusable and Type-Safe Code

Generics are one of TypeScript's most powerful features. They allow you to write code that works with any type, while maintaining full type safety — without using any.

😩 Without Generics 
// Must duplicate code for each type
function getFirstString(arr: string[]): string {
  return arr[0];
}
function getFirstNumber(arr: number[]): number {
  return arr[0];
}
// Or lose safety with any:
function getFirst(arr: any[]): any {
  return arr[0]; // Type lost!
}
😊 With Generics 
// One function that works with any type!
function getFirst<T>(arr: T[]): T {
  return arr[0];
}

const s = getFirst(["a", "b"]); // string
const n = getFirst([1, 2, 3]);   // number
const u = getFirst([{ id: 1 }]); // { id: number }
// TypeScript infers T automatically!

ADVANCED GENERICS — CONSTRAINTS AND MULTIPLE PARAMETERS

// Constraint — T must have an 'id' property
function findById<T extends { id: number }>(items: T[], id: number): T | undefined {
  return items.find(item => item.id === id);
}

// Multiple type parameters
function merge<T extends object, U extends object>(obj1: T, obj2: U): T & U {
  return { ...obj1, ...obj2 };
}

const result = merge({ name: "Alice" }, { age: 30 });
// result: { name: string } & { age: number }

// Generics in interfaces
interface Repository<T> {
  findAll(): Promise<T[]>;
  findById(id: number): Promise<T | null>;
  create(data: Omit<T, 'id'>): Promise<T>;
  update(id: number, data: Partial<T>): Promise<T>;
  delete(id: number): Promise<void>;
}

// Strongly typed implementation
class UserRepository implements Repository<User> {
  async findAll(): Promise<User[]> { return []; }
  async findById(id: number): Promise<User | null> { return null; }
  async create(data: Omit<User, 'id'>): Promise<User> { return {} as User; }
  async update(id: number, data: Partial<User>): Promise<User> { return {} as User; }
  async delete(id: number): Promise<void> { }
}

🛠️ 1.5 Utility Types — Transforming Existing Types

TypeScript includes a set of built-in Utility Types that transform existing types. They are invaluable day-to-day and avoid code duplication.

THE MOST COMMONLY USED UTILITY TYPES

interface User {
  id: number;
  name: string;
  email: string;
  password: string;
  createdAt: Date;
}

// Partial<T> — all properties become optional
type UpdateUserDTO = Partial<User>;
// { id?: number; name?: string; email?: string; ... }

// Required<T> — all properties become required
type FullUser = Required<User>;

// Pick<T, K> — select only some properties
type PublicUser = Pick<User, 'id' | 'name' | 'email'>;
// Never exposes the password!

// Omit<T, K> — exclude specific properties
type CreateUserDTO = Omit<User, 'id' | 'createdAt'>;
// User doesn't have an id or creation date yet

// Readonly<T> — all properties are read-only
type ImmutableUser = Readonly<User>;

// Record<K, V> — create an object with keys K and values V
type UsersByRole = Record<'admin' | 'user' | 'guest', User[]>;

// NonNullable<T> — removes null and undefined
type SafeString = NonNullable<string | null | undefined>; // string

// ReturnType<T> — extracts the return type of a function
type FetchResult = ReturnType<typeof fetch>; // Promise<Response>

// Parameters<T> — extracts function parameters as a tuple
type SetTimeoutParams = Parameters<typeof setTimeout>;
Partial<T>

All optional — for partial updates

Omit<T,K>

Remove properties — for DTOs

Pick<T,K>

Select properties — for projections

Record<K,V>

Typed dictionaries

🎯 1.6 Type Narrowing and Type Guards

When working with union types, TypeScript needs to know which "branch" of the type you're working with. The process of refining types within a block of code is called narrowing.

NARROWING TECHNIQUES

// typeof narrowing
function formatValue(value: string | number): string {
  if (typeof value === "string") {
    return value.toUpperCase(); // TypeScript knows it's a string here
  }
  return value.toFixed(2); // TypeScript knows it's a number here
}

// instanceof narrowing
function processShape(shape: Circle | Rectangle) {
  if (shape instanceof Circle) {
    return Math.PI * shape.radius ** 2;
  }
  return shape.width * shape.height;
}

// Discriminated Unions — recommended pattern
type Shape =
  | { kind: "circle"; radius: number }
  | { kind: "rectangle"; width: number; height: number }
  | { kind: "triangle"; base: number; height: number };

function getArea(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;
    case "rectangle":
      return shape.width * shape.height;
    case "triangle":
      return (shape.base * shape.height) / 2;
    // TypeScript ensures you've covered all cases!
  }
}

// Custom Type Guard — type predicate
function isUser(value: unknown): value is User {
  return (
    typeof value === "object" &&
    value !== null &&
    "id" in value &&
    "name" in value
  );
}

const data: unknown = fetchFromAPI();
if (isUser(data)) {
  console.log(data.name); // TypeScript knows it's a User!
}

⚡ 1.7 Advanced Functions — Overloads, this and Parameters

TypeScript adds powerful expressiveness to functions — you can define multiple signatures, explicitly type this, and use advanced parameter modifiers.

FUNCTION OVERLOADS AND ADVANCED TYPING

// Function overloads — multiple signatures
function createElement(tag: "div"): HTMLDivElement;
function createElement(tag: "span"): HTMLSpanElement;
function createElement(tag: "input"): HTMLInputElement;
function createElement(tag: string): HTMLElement {
  return document.createElement(tag);
}

const div = createElement("div");   // HTMLDivElement
const span = createElement("span"); // HTMLSpanElement

// Strongly typed rest parameters
function buildUrl(base: string, ...segments: string[]): string {
  return [base, ...segments].join("/");
}

// Functions as types
type Predicate<T> = (item: T) => boolean;
type Mapper<T, R> = (item: T, index: number) => R;
type Reducer<T, R> = (acc: R, item: T) => R;

// Type-safe higher-order functions
function pipe<T>(...fns: Array<(x: T) => T>): (x: T) => T {
  return (x) => fns.reduce((v, f) => f(v), x);
}

const process = pipe(
  (s: string) => s.trim(),
  (s) => s.toLowerCase(),
  (s) => s.replace(/\s+/g, "-")
);

console.log(process("  Hello World  ")); // "hello-world"

🏛️ 1.8 Classes and Object-Oriented Programming

TypeScript transforms JavaScript classes with access modifiers, readonly properties, abstract classes, and full integration with the type system.

ADVANCED CLASSES IN TYPESCRIPT

abstract class BaseEntity {
  readonly id: string;
  readonly createdAt: Date;

  constructor() {
    this.id = crypto.randomUUID();
    this.createdAt = new Date();
  }

  // Abstract method — subclasses must implement
  abstract validate(): boolean;

  toJSON(): Record<string, unknown> {
    return { id: this.id, createdAt: this.createdAt };
  }
}

class User extends BaseEntity {
  #password: string; // Native private field (not accessible externally)

  constructor(
    public name: string,            // public — creates and exposes property
    public email: string,
    private role: "admin" | "user", // private — only within the class
    password: string
  ) {
    super();
    this.#password = this.hashPassword(password);
  }

  validate(): boolean {
    return this.email.includes("@") && this.name.length > 0;
  }

  private hashPassword(raw: string): string {
    return `hashed_${raw}`; // Simplified
  }

  // Getter — accessed like a property
  get displayName(): string {
    return `${this.name} (${this.role})`;
  }

  // Static factory method
  static create(name: string, email: string, password: string): User {
    return new User(name, email, "user", password);
  }
}

📦 1.9 Modules, Namespaces and Code Organisation

In scalable TypeScript projects, code organisation is critical. TypeScript supports native ES Modules, module declarations for third-party libraries, and export patterns that maximise tree-shaking.

RECOMMENDED MODULE PATTERNS

// types/index.ts — centralise all types
export interface User { id: number; name: string; email: string; }
export type UserId = User['id']; // Extract type from property
export type UserRole = 'admin' | 'user' | 'guest';

// utils/validators.ts
export const isValidEmail = (email: string): boolean =>
  /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email);

// services/UserService.ts
import type { User, UserId } from '../types';
// 'import type' — only imports types, zero bundle impact!

export class UserService {
  private users: Map<UserId, User> = new Map();

  getUser(id: UserId): User | undefined {
    return this.users.get(id);
  }
}

// Module declaration for libraries without types
declare module 'some-untyped-lib' {
  export function doSomething(x: string): number;
  export const VERSION: string;
}

src/
├── types/          # Types and interfaces
│   └── index.ts
├── models/         # Domain classes
├── services/       # Business logic
├── repositories/   # Data access
├── controllers/    # HTTP handlers
├── utils/          # Utility functions
├── config/         # Configuration
└── index.ts        # Entry point
❌ Anti-patterns to Avoid
  • → Using any as an escape (use unknown)
  • → Unnecessary type assertions (as X)
  • → Suppressing errors with // @ts-ignore
  • → Giant, undecomposed interfaces
  • → Mixing null and undefined without criteria

🎨 1.10 Decorators — Metaprogramming

Decorators are an experimental feature (but widely used in frameworks like Angular, NestJS, and TypeORM) that allows adding behaviour to classes, methods, and properties in a declarative way.

PRACTICAL DECORATORS

// Enable in tsconfig: "experimentalDecorators": true

// Class decorator — adds metadata
function Controller(path: string) {
  return function (target: Function) {
    Reflect.defineMetadata('path', path, target);
  };
}

// Method decorator — automatic logging
function Log(target: any, key: string, descriptor: PropertyDescriptor) {
  const original = descriptor.value;
  descriptor.value = function (...args: any[]) {
    console.log(`[LOG] ${key} called with:`, args);
    const result = original.apply(this, args);
    console.log(`[LOG] ${key} returned:`, result);
    return result;
  };
  return descriptor;
}

// Property decorator — validation
function MinLength(min: number) {
  return function (target: any, propertyKey: string) {
    let value: string;
    Object.defineProperty(target, propertyKey, {
      get: () => value,
      set: (newValue: string) => {
        if (newValue.length < min)
          throw new Error(`${propertyKey} must have at least ${min} characters`);
        value = newValue;
      }
    });
  };
}

// Combined usage — as in NestJS
@Controller('/users')
class UserController {
  @MinLength(3)
  name: string = '';

  @Log
  getUsers(): string[] {
    return ['Alice', 'Bob'];
  }
}

🗺️ 1.11 Mapped Types and Advanced Conditional Types

Mapped Types allow you to create new types by programmatically transforming existing ones — they are the foundation of how TypeScript's Utility Types are implemented internally.

MAPPED TYPES AND ADVANCED INFERENCE

// Mapped type — transform all properties
type Nullable<T> = {
  [K in keyof T]: T[K] | null;
};

type Getters<T> = {
  [K in keyof T as `get${Capitalize<string & K>}`]: () => T[K];
};

type UserGetters = Getters<User>;
// { getId: () => number; getName: () => string; ... }

// Advanced conditional types
type DeepReadonly<T> = {
  readonly [K in keyof T]: T[K] extends object ? DeepReadonly<T[K]> : T[K];
};

// infer — extract types within conditional types
type UnpackPromise<T> = T extends Promise<infer U> ? U : T;

type ApiResult = UnpackPromise<Promise<User[]>>; // User[]
type SyncResult = UnpackPromise<string>;           // string

// Extract all methods from a class
type MethodsOf<T> = {
  [K in keyof T as T[K] extends Function ? K : never]: T[K];
};

// Satisfies operator (TS 4.9+) — validates without widening
const config = {
  port: 3000,
  host: "localhost",
  db: { url: "postgres://..." }
} satisfies AppConfig; // Validates against AppConfig but keeps precise type

🚀 1.12 Compilation Performance and Best Practices at Scale

In large projects, TypeScript compiler performance can become a bottleneck. There are specific strategies to maintain fast feedback even in monorepos with millions of lines of code.

⚡ Compilation Optimisations: 
// tsconfig.json — large projects
{
  "compilerOptions": {
    "incremental": true,       // Incremental cache
    "tsBuildInfoFile": ".tsbuildinfo",
    "skipLibCheck": true,      // Skip node_modules types
    "isolatedModules": true    // Compatible with esbuild/swc
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
🏗️ Project References (Monorepos): 
// tsconfig.json at the monorepo root
{
  "references": [
    { "path": "./packages/core" },
    { "path": "./packages/api" },
    { "path": "./packages/web" }
  ]
}
