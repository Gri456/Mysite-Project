🌐 6.0 Understanding Backend Architecture

The backend is the invisible engine of every modern application. While the frontend handles visuals and user interaction, the backend is responsible for logic, authentication, data processing, storage, and security.

Frontend vs Backend:
• Frontend: What the user sees (HTML, CSS, JavaScript).
• Backend: What processes requests and manages data.
• Database: Persistent storage system.

A typical architecture follows this flow:

  • User sends request (browser → server)
  • Server processes logic
  • Server queries database
  • Server returns response (usually JSON)

🟢 6.1 Node.js – JavaScript on the Server

Node.js allows JavaScript to run outside the browser. It is event-driven and non-blocking, meaning it can handle many simultaneous connections efficiently. 

// Basic Node server example
const http = require('http');

const server = http.createServer((req, res) => {
    res.writeHead(200, {'Content-Type': 'text/plain'});
    res.end('Hello Backend World');
});

server.listen(3000);

Node is ideal for APIs, real-time apps, microservices, and scalable systems.

⚡ 6.2 Express – Simplifying Server Creation

Express is a minimal framework built on top of Node.js that simplifies routing and middleware handling. 

const express = require('express');
const app = express();

app.get('/users', (req, res) => {
    res.json({ message: "List of users" });
});

app.listen(3000);

Express makes it easy to build REST APIs using HTTP methods like GET, POST, PUT and DELETE.

🔗 6.3 REST API Principles

REST (Representational State Transfer) is an architectural style for designing APIs.

  • Stateless: Each request contains all necessary information.
  • Resource-Based: Everything is treated as a resource (users, products).
  • HTTP Methods: GET, POST, PUT, DELETE.
  • JSON Format: Standard response format.
Example endpoint structure:
GET /users
GET /users/1
POST /users
PUT /users/1
DELETE /users/1

🔐 6.5 Authentication & Authorization

Authentication verifies identity. Authorization verifies permissions.

  • Password hashing (bcrypt)
  • JWT (JSON Web Tokens)
  • Session-based authentication
  • Role-based access control (RBAC)
// Example JWT usage
const jwt = require('jsonwebtoken');

const token = jwt.sign({ userId: 1 }, "secretKey", { expiresIn: "1h" });

🌍 6.8 HTTP Protocol – Deep Technical Understanding

HTTP (Hypertext Transfer Protocol) is the foundation of all web communication. Every frontend-to-backend interaction is an HTTP request-response cycle.

Request Structure:
• Method (GET, POST, PUT, DELETE)
• URL (endpoint)
• Headers (metadata)
• Body (optional data payload)

HTTP is stateless — meaning the server does not remember previous requests. Authentication tokens or sessions are used to simulate state.

📌 Status Codes Breakdown

  • 200 OK: Successful request
  • 201 Created: Resource created
  • 204 No Content: Success without body
  • 400 Bad Request: Client input error
  • 401 Unauthorized: Authentication required
  • 403 Forbidden: Authenticated but no permission
  • 404 Not Found: Resource does not exist
  • 500 Internal Server Error: Server failure

Proper status codes are critical for clean API design and frontend integration.

🧭 6.9 Advanced Routing & Modularization

Professional backends do not define all routes in a single file. Routes are modularized by feature. 

// routes/user.routes.js
const router = require("express").Router();
router.get("/", getUsers);
router.post("/", createUser);
module.exports = router;

Then imported into the main server: 

app.use("/api/v1/users", userRoutes);

📌 Best Practices

  • Versioned routes (/api/v1/)
  • Feature-based folder structure
  • Controller separation
  • Input validation middleware before controller logic

⚡ 6.13 Query Optimization – Real Performance Engineering

Poorly written queries are one of the main causes of backend bottlenecks.

🔎 Indexing Strategy

Indexes dramatically improve read performance but slightly slow writes. They should be applied on frequently filtered or joined columns. 

CREATE INDEX idx_user_email ON users(email);

🚨 N+1 Problem

Happens when fetching related data in loops instead of using joins.

  • Bad: 1 query for users + 1 query per user for orders
  • Good: Single JOIN query

📊 Execution Plans

Use EXPLAIN to analyze how the database executes queries.

📬 6.15 Message Queues & Asynchronous Systems

In scalable systems, heavy tasks should not block HTTP responses. Instead, they are processed asynchronously via message queues.

Example Use Cases

  • Sending emails
  • Generating reports
  • Image processing
  • Payment confirmation

Flow:

Client → API → Queue → Worker → Database

This decouples services and improves reliability.

🧩 6.16 Microservices Architecture

Microservices divide a system into independently deployable services.

Core Principles

  • Each service has its own database
  • Services communicate via HTTP or message broker
  • Independent deployment pipelines
  • Failure isolation

Trade-Off

Microservices increase operational complexity and require DevOps maturity.
