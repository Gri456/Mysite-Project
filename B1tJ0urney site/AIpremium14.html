Introduction: The Gap Between Notebook and Production

Building a model that works in a Jupyter notebook is the easy part. Getting that model into production — serving real users, staying accurate over time, retraining automatically, and rolling back safely — is where most AI projects fail.

MLOps (Machine Learning Operations) is the discipline that bridges this gap. It applies DevOps principles — automation, version control, CI/CD, monitoring — to the unique challenges of ML systems.

This module covers:

  • The MLOps lifecycle and maturity model
  • Experiment tracking with MLflow
  • Model registry, versioning, and lineage
  • CI/CD pipelines for ML
  • Containerization with Docker
  • Model serving: REST APIs, batch inference, streaming
  • Production monitoring and data/model drift detection
  • Scaling with Kubernetes and cloud ML platforms

Why ML systems fail in production — the unique challenges that MLOps addresses:

Challenge 1
Reproducibility
A model trained in February may not reproduce in August. Dependencies change, random seeds aren't set, data transforms differ between training and serving. MLOps enforces reproducibility through version-controlled code, pinned dependencies, and experiment tracking.
Challenge 2
Silent Degradation
Unlike software bugs (which throw errors), ML models fail silently. Accuracy degrades as the world changes. Users get bad recommendations, wrong fraud flags, and incorrect predictions — with no error in the logs. Monitoring is the only defense.
Challenge 3
Data Dependency
ML systems depend on data pipelines. If upstream data changes (schema, volume, quality), the model breaks. Data validation, lineage tracking, and feature store consistency are essential — software CI/CD only tests code, not data.
Challenge 4
Multiple Artifacts
A software deployment ships code. An ML deployment ships code + model weights + preprocessing pipeline + feature schema + evaluation metrics + training data version. Versioning all these together is a coordination problem that ML-specific tools solve.
📊 The Hidden Cost of No MLOps

Gartner estimates that 85% of AI projects never reach production. Of those that do, the average time from model training to production deployment is 6–12 months without MLOps tooling. With mature MLOps pipelines, world-class teams deploy in hours. MLOps is not an accessory — it's what determines whether your AI investment actually generates value.

The key insight: the loop never ends. Models decay as the world changes. A production ML system must automatically detect when it's degrading and trigger retraining — without human intervention for every cycle.

What makes ML CI/CD different from software CI/CD — the extra dimensions:

1
Code changes trigger software CI/CD. In ML, three things can independently trigger a new pipeline run: (a) code change, (b) new training data arrival, (c) drift alert from monitoring. Your pipeline must handle all three entry points.
2
Tests include data validation, not just code tests. Before training, you must validate that data conforms to the expected schema, that there are no missing columns, and that feature distributions haven't shifted drastically from the last training run.
3
The quality gate is statistical, not binary. Software deploys if tests pass or fail. ML deploys only if the new model outperforms the current production model on held-out evaluation data by a defined margin (e.g. AUC must improve by at least +0.01).
4
Rollback requires re-serving an old model artifact. In software, a git revert undoes a code change. In ML, rollback means loading a specific model version from the registry and re-deploying it. The model registry makes this a one-command operation.
5
Deployment strategies are probabilistic. You often want to test a new model on a subset of traffic before full rollout. Canary deployments (5% → 20% → 100%), A/B tests, and shadow mode (new model runs but doesn't serve) are standard patterns in production ML.

MLOps Maturity Levels

Google and Microsoft define MLOps maturity as a progression from manual to fully automated:

LevelDescriptionCharacteristics
Level 0Manual processNotebooks, manual training, no tracking, no CI/CD
Level 1ML pipeline automationAutomated training, experiment tracking, model registry, manual deployment
Level 2CI/CD pipeline automationAutomated testing, automated deployment, monitoring, auto-retraining on drift

Most companies operate at Level 0–1. Level 2 is the goal for any team with models serving significant traffic. The jump from 0 to 1 delivers the most immediate value.

What each level looks like in practice — concrete differences in day-to-day ML work:

Level 0 Reality
Notebook Chaos
A data scientist trains a model in model_final_v3_REAL.ipynb. Parameters are hardcoded. Results exist only in their memory. Deploying means emailing a pickle file. Reproducing the experiment 3 months later is impossible. This describes ~60% of ML teams at companies without dedicated ML engineering.
Level 1 Reality
Tracked & Versioned
Every training run is logged to MLflow automatically. The best model is registered with a version number. Deploying means promoting from Staging to Production in the registry. A new team member can reproduce any historical experiment in 10 minutes. Retraining is a script, not a manual process. Most well-run data teams here.
Level 2 Reality
Fully Automated
New training data arrives → pipeline triggers automatically → model trains → quality gate compares vs production → if better, deploys to staging → integration tests run → promotes to production. Drift detected at 2am → alert fires → retraining triggered → new model deployed by morning. Humans approve only irreversible actions. Top-tier ML teams here.

Experiment Tracking with MLflow

MLflow is the most widely used open-source platform for ML experiment tracking, model packaging, and registry. It records every training run so you can reproduce, compare, and promote the best models.

import mlflow
import mlflow.sklearn
from sklearn.ensemble import RandomForestClassifier
from sklearn.metrics import accuracy_score, f1_score

mlflow.set_experiment("fraud-detection-v2")

with mlflow.start_run(run_name="rf-depth8-200trees"):
    params = {"n_estimators": 200, "max_depth": 8, "min_samples_split": 5}
    mlflow.log_params(params)

    model = RandomForestClassifier(**params)
    model.fit(X_train, y_train)
    preds = model.predict(X_test)

    mlflow.log_metric("accuracy", accuracy_score(y_test, preds))
    mlflow.log_metric("f1_score", f1_score(y_test, preds))
    mlflow.log_metric("train_size", len(X_train))

    mlflow.sklearn.log_model(
        model, artifact_path="model",
        input_example=X_train[:5],
        registered_model_name="FraudDetector"
    )
    mlflow.log_artifact("confusion_matrix.png")

MLflow in production — advanced patterns: auto-logging, hyperparameter search, and comparing runs programmatically:

import mlflow
import mlflow.sklearn
from mlflow.tracking import MlflowClient
from sklearn.model_selection import cross_val_score
import optuna

# ── Auto-logging: log everything with one line ─────────────────────────
mlflow.sklearn.autolog()   # logs params, metrics, model, feature names automatically
# also available: mlflow.pytorch.autolog(), mlflow.xgboost.autolog(), etc.

# ── Hyperparameter optimization with Optuna + MLflow ──────────────────
mlflow.set_experiment("fraud-hparam-search")

def objective(trial):
    params = {
        "n_estimators": trial.suggest_int("n_estimators", 50, 500),
        "max_depth":    trial.suggest_int("max_depth", 3, 15),
        "min_samples_split": trial.suggest_int("min_samples_split", 2, 20),
        "max_features": trial.suggest_categorical("max_features", ["sqrt", "log2"])
    }
    with mlflow.start_run(nested=True):   # nested runs for each trial
        mlflow.log_params(params)
        model = RandomForestClassifier(**params, random_state=42)
        cv_scores = cross_val_score(model, X_train, y_train, cv=5, scoring="f1")
        mean_f1 = cv_scores.mean()
        mlflow.log_metric("cv_f1_mean", mean_f1)
        mlflow.log_metric("cv_f1_std", cv_scores.std())
        return mean_f1

with mlflow.start_run(run_name="optuna-search-50trials"):
    study = optuna.create_study(direction="maximize")
    study.optimize(objective, n_trials=50)
    best_params = study.best_params
    mlflow.log_params(best_params)
    mlflow.log_metric("best_cv_f1", study.best_value)
    print(f"Best params: {best_params}, Best F1: {study.best_value:.4f}")

# ── Compare runs programmatically to find the best ────────────────────
client = MlflowClient()

runs = client.search_runs(
    experiment_ids=["1"],        # experiment ID
    filter_string="metrics.f1_score > 0.90",
    order_by=["metrics.auc_roc DESC"],
    max_results=5
)

for run in runs:
    print(f"Run: {run.info.run_id[:8]}  "
          f"F1: {run.data.metrics['f1_score']:.4f}  "
          f"AUC: {run.data.metrics.get('auc_roc', 'N/A')}")

best_run_id = runs[0].info.run_id
print(f"Best run: {best_run_id}")
🔧 MLflow vs Weights & Biases — When to Use Each

MLflow: Open-source, self-hosted option, built into Databricks. Best for teams that want full control, already use Databricks, or need air-gapped deployment. The model registry and pyfunc serving interface are best-in-class.

Weights & Biases (W&B): Best-in-class UI for visualizing training, sweep optimization, and artifact tracking. Better for deep learning / computer vision teams with complex loss curves. Managed cloud service — faster to set up, no infrastructure to maintain. Free tier is generous for individuals.

# ── Safe rollback in production ───────────────────────────────────────
def rollback_to_version(model_name: str, version: int):
    """Instantly roll back to a previous model version."""
    # Archive current production model
    current = client.get_latest_versions(model_name, stages=["Production"])
    if current:
        client.transition_model_version_stage(
            model_name, current[0].version, "Archived"
        )
    # Promote the target version back to production
    client.transition_model_version_stage(
        model_name, str(version), "Production"
    )
    print(f"Rolled back {model_name} to version {version}")

# E.g.: rollback_to_version("FraudDetector", 12)

CI/CD for Machine Learning

CI/CD automates the pipeline from code change to production deployment. For ML, this includes data validation and model quality gates — not just code tests.

# .github/workflows/ml-pipeline.yml
name: ML CI/CD Pipeline
on:
  push:
    branches: [main]
jobs:
  train-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Validate data
        run: python scripts/validate_data.py
      - name: Run unit tests
        run: pytest tests/ -v
      - name: Train model
        run: python scripts/train.py
      - name: Evaluate vs production
        run: python scripts/evaluate.py
      - name: Deploy to staging
        if: success()
        run: kubectl apply -f k8s/staging/
      - name: Promote to production
        if: success()
        run: kubectl apply -f k8s/production/

Production-grade ML CI/CD with data validation, quality gates, and canary deployment:

# scripts/validate_data.py — data quality gate
import pandas as pd
import sys
from scipy import stats

def validate_training_data(path: str) -> bool:
    df = pd.read_parquet(path)

    # ── Schema validation ─────────────────────────────────────────────
    required_cols = ["transaction_amount", "merchant_category",
                     "user_age_days", "hour_of_day", "is_fraud"]
    missing = [c for c in required_cols if c not in df.columns]
    if missing:
        print(f"FAIL: Missing columns: {missing}")
        return False

    # ── Volume check ──────────────────────────────────────────────────
    if len(df) < 10_000:
        print(f"FAIL: Dataset too small: {len(df)} rows (min 10,000)")
        return False

    # ── Null check ────────────────────────────────────────────────────
    null_pct = df.isnull().mean()
    high_null = null_pct[null_pct > 0.05]
    if not high_null.empty:
        print(f"FAIL: High null rate in columns: {high_null.to_dict()}")
        return False

    # ── Label distribution check ──────────────────────────────────────
    fraud_rate = df["is_fraud"].mean()
    if not (0.001 <= fraud_rate <= 0.20):
        print(f"FAIL: Unusual fraud rate: {fraud_rate:.3f} (expected 0.001–0.20)")
        return False

    print(f"PASS: Data validation OK — {len(df):,} rows, fraud rate: {fraud_rate:.3%}")
    return True

if not validate_training_data("data/training.parquet"):
    sys.exit(1)


# scripts/evaluate.py — model quality gate
import mlflow
from mlflow.tracking import MlflowClient
import sys

def should_deploy(new_run_id: str, min_improvement: float = 0.01) -> bool:
    client = MlflowClient()

    # Get new model metrics
    new_run = client.get_run(new_run_id)
    new_auc = new_run.data.metrics["auc_roc"]

    # Get current production model metrics
    prod_versions = client.get_latest_versions("FraudDetector", stages=["Production"])
    if not prod_versions:
        print("No production model found. Approving first deployment.")
        return True

    prod_run_id = prod_versions[0].run_id
    prod_run = client.get_run(prod_run_id)
    prod_auc = prod_run.data.metrics["auc_roc"]

    improvement = new_auc - prod_auc
    print(f"Production AUC: {prod_auc:.4f} | New AUC: {new_auc:.4f} | Δ: {improvement:+.4f}")

    if improvement >= min_improvement:
        print(f"PASS: New model is better by {improvement:.4f} (threshold: {min_improvement})")
        return True
    else:
        print(f"FAIL: Insufficient improvement. Model will NOT be promoted.")
        return False

import os
new_run_id = os.environ["MLFLOW_RUN_ID"]
if not should_deploy(new_run_id):
    sys.exit(1)

# ── Multi-stage build: separates build environment from runtime ────────
# This cuts image size from ~2GB to ~400MB for Python ML images

FROM python:3.11-slim AS builder
WORKDIR /build
COPY requirements.txt .
# Install into a virtual env so we can copy it cleanly
RUN python -m venv /opt/venv && \
    /opt/venv/bin/pip install --no-cache-dir -r requirements.txt

# ── Runtime stage: only what's needed to serve predictions ────────────
FROM python:3.11-slim AS runtime
WORKDIR /app

# Copy only the installed packages (not build tools)
COPY --from=builder /opt/venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"

# Non-root user for security — never run containers as root
RUN useradd -m -u 1000 mlserve
USER mlserve

# Copy application code
COPY --chown=mlserve:mlserve app/ ./app/

EXPOSE 8080

# Readiness check: waits for model to load before accepting traffic
HEALTHCHECK --interval=15s --timeout=5s --start-period=30s --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1

Production FastAPI serving — with input logging, prediction tracking, and async batch support:

from fastapi import FastAPI, HTTPException, BackgroundTasks
from pydantic import BaseModel, Field
import mlflow.pyfunc
import pandas as pd
import logging
import time
import uuid
from datetime import datetime

logger = logging.getLogger(__name__)
app    = FastAPI(title="Fraud Detection API", version="1.0.0")

# Load model once at startup (not per request)
model = None

@app.on_event("startup")
async def load_model():
    global model
    model = mlflow.pyfunc.load_model("models:/FraudDetector/Production")
    logger.info("Model loaded: FraudDetector/Production")

class PredictionRequest(BaseModel):
    transaction_amount: float = Field(gt=0, description="Transaction amount in USD")
    merchant_category:  str
    user_age_days:      int = Field(ge=0)
    hour_of_day:        int = Field(ge=0, le=23)

class PredictionResponse(BaseModel):
    prediction_id:     str
    fraud_probability: float
    flag:              bool
    model_version:     str
    latency_ms:        float

def log_prediction(req_dict: dict, response_dict: dict):
    """Background task: log every prediction for drift monitoring."""
    logger.info({
        "event": "prediction",
        "timestamp": datetime.utcnow().isoformat(),
        **req_dict,
        **response_dict
    })

@app.post("/predict", response_model=PredictionResponse)
async def predict(req: PredictionRequest, background_tasks: BackgroundTasks):
    if model is None:
        raise HTTPException(status_code=503, detail="Model not loaded")

    start = time.perf_counter()
    prediction_id = str(uuid.uuid4())[:8]

    features = pd.DataFrame([{
        "transaction_amount": req.transaction_amount,
        "merchant_category":  req.merchant_category,
        "user_age_days":      req.user_age_days,
        "hour_of_day":        req.hour_of_day
    }])

    prob = float(model.predict(features)[0])
    latency_ms = (time.perf_counter() - start) * 1000

    response = PredictionResponse(
        prediction_id=prediction_id,
        fraud_probability=round(prob, 4),
        flag=prob > 0.85,
        model_version="12",
        latency_ms=round(latency_ms, 2)
    )

    # Log asynchronously — doesn't block the response
    background_tasks.add_task(log_prediction, req.dict(), response.dict())

    return response

@app.post("/predict/batch")
async def predict_batch(requests: list[PredictionRequest]):
    """Score up to 1000 records in a single call."""
    if len(requests) > 1000:
        raise HTTPException(status_code=400, detail="Max batch size: 1000")

    features = pd.DataFrame([r.dict() for r in requests])
    probs = model.predict(features).tolist()
    return [{"fraud_probability": round(p, 4), "flag": p > 0.85}
            for p in probs]

@app.get("/health")
def health():
    return {"status": "ok", "model_loaded": model is not None}

Production Monitoring & Drift Detection

Models degrade silently in production. Monitoring is not optional.

What to monitor:

  • Prediction distribution — are the outputs shifting?
  • Data drift — has the distribution of input features changed from training?
  • Model drift (concept drift) — has the relationship between features and labels changed?
  • Latency & throughput — p50, p95, p99 response times.
  • Business metrics — the ultimate signal: revenue, click-through rate, conversion, churn.
from evidently.report import Report
from evidently.metric_preset import DataDriftPreset, ClassificationPreset

report = Report(metrics=[DataDriftPreset(), ClassificationPreset()])
report.run(reference_data=training_df, current_data=production_df)
report.save_html("drift_report.html")

Complete monitoring system — drift detection, alerting, and automated retraining trigger:

from evidently.report import Report
from evidently.test_suite import TestSuite
from evidently.metric_preset import DataDriftPreset
from evidently.tests import TestNumberOfDriftedColumns
import pandas as pd
import mlflow
import smtplib
import json
from datetime import datetime, timedelta

def run_drift_monitoring(
    reference_path: str,
    production_path: str,
    drift_threshold: float = 0.2,   # trigger if >20% of features drift
    alert_email: str = "ml-team@company.com"
) -> dict:
    """
    Full drift monitoring pipeline:
    1. Load reference (training) and current (production) data
    2. Run statistical drift tests
    3. Alert if drift exceeds threshold
    4. Trigger retraining if critical drift detected
    """

    reference_df  = pd.read_parquet(reference_path)
    production_df = pd.read_parquet(production_path)

    # ── Statistical drift tests ───────────────────────────────────────
    tests = TestSuite(tests=[
        TestNumberOfDriftedColumns(lt=int(len(reference_df.columns) * drift_threshold))
    ])
    tests.run(reference_data=reference_df, current_data=production_df)
    test_results = tests.as_dict()

    # ── Detailed per-feature report ───────────────────────────────────
    report = Report(metrics=[DataDriftPreset()])
    report.run(reference_data=reference_df, current_data=production_df)
    report_dict = report.as_dict()

    # Extract drifted features
    feature_results = report_dict["metrics"][0]["result"]["drift_by_columns"]
    drifted = {col: res for col, res in feature_results.items()
               if res.get("drift_detected")}

    result = {
        "timestamp": datetime.utcnow().isoformat(),
        "n_features": len(feature_results),
        "n_drifted": len(drifted),
        "drift_share": len(drifted) / len(feature_results),
        "drifted_features": list(drifted.keys()),
        "tests_passed": test_results["summary"]["all_passed"]
    }

    # ── Log drift metrics to MLflow for tracking over time ────────────
    with mlflow.start_run(run_name=f"drift-check-{datetime.now().strftime('%Y%m%d')}",
                          experiment_id=mlflow.get_experiment_by_name("monitoring").experiment_id):
        mlflow.log_metrics({
            "drift_share": result["drift_share"],
            "n_drifted_features": result["n_drifted"]
        })
        mlflow.log_dict(result, "drift_result.json")

    # ── Alert and auto-retrain if drift exceeds threshold ─────────────
    if result["drift_share"] > drift_threshold:
        print(f"⚠️ DRIFT ALERT: {result['drift_share']:.1%} of features drifted")
        print(f"   Drifted: {result['drifted_features']}")

        # Trigger retraining pipeline
        import subprocess
        subprocess.Popen(["python", "scripts/train.py",
                          "--trigger", "drift-alert",
                          "--drifted-features", json.dumps(result["drifted_features"])])
        print("✓ Retraining pipeline triggered")
    else:
        print(f"✓ No significant drift — {result['drift_share']:.1%} of features")

    return result

# Schedule this to run daily via cron or Airflow
# 0 6 * * * python scripts/monitor_drift.py
📐 Drift Detection Methods — Which Statistical Test to Use

Kolmogorov-Smirnov (KS test): compares CDFs of two distributions. Good for continuous features (transaction amount, age). Sensitive to shape differences.
Population Stability Index (PSI): industry standard for credit risk. Buckets the distribution and measures shift. PSI >0.2 = significant drift.
Jensen-Shannon Divergence: symmetric divergence between distributions. Good for both continuous and categorical. Bounded [0,1].
Chi-squared test: best for categorical features (merchant category, device type). Tests if observed frequencies match expected.

Feature Stores

A feature store is a centralized repository for ML features — computed, stored, and served consistently between training and serving. It solves the training-serving skew problem.

  • Offline store — historical features for training (S3, BigQuery, Delta Lake)
  • Online store — low-latency feature lookup for real-time serving (Redis, DynamoDB)
  • Feature pipelines — compute features once, use everywhere
  • Point-in-time correctness — retrieve features as they were at any historical timestamp (prevents data leakage)

Tools: Feast (open-source), Tecton, Hopsworks, AWS SageMaker Feature Store, Vertex AI Feature Store.

Training-serving skew is one of the most common silent bugs in production ML. The model trains on features computed one way and serves using features computed a different way — causing invisible accuracy degradation.

Feature store implementation with Feast — defining, materializing, and serving features:

from feast import FeatureStore, Entity, FeatureView, Field, FileSource
from feast.types import Float32, Int64, String
from datetime import timedelta

# ── Define the feature store ──────────────────────────────────────────
store = FeatureStore(repo_path="feature_repo/")

# ── Define entities (the "keys" for feature lookup) ───────────────────
user = Entity(name="user_id", join_keys=["user_id"])
transaction = Entity(name="transaction_id", join_keys=["transaction_id"])

# ── Define feature views (groups of related features) ─────────────────
user_features_view = FeatureView(
    name="user_features",
    entities=[user],
    ttl=timedelta(days=30),   # features expire after 30 days
    schema=[
        Field(name="user_age_days",        dtype=Int64),
        Field(name="lifetime_txn_count",   dtype=Int64),
        Field(name="avg_txn_amount_30d",   dtype=Float32),
        Field(name="preferred_category",   dtype=String),
    ],
    source=FileSource(
        path="data/user_features.parquet",
        timestamp_field="event_timestamp"  # enables point-in-time joins
    )
)

txn_features_view = FeatureView(
    name="transaction_features",
    entities=[transaction],
    ttl=timedelta(hours=1),   # transaction features expire fast
    schema=[
        Field(name="transaction_amount", dtype=Float32),
        Field(name="merchant_category",  dtype=String),
        Field(name="hour_of_day",        dtype=Int64),
    ],
    source=FileSource(path="data/transaction_features.parquet",
                      timestamp_field="event_timestamp")
)

# ── Materialize features to the online store (Redis) ──────────────────
# Run offline → online sync before serving goes live
from datetime import datetime
store.materialize_incremental(end_date=datetime.utcnow())

# ── Training: point-in-time correct feature retrieval ─────────────────
# Prevents data leakage: retrieves features as they existed at label time
entity_df = pd.DataFrame({
    "user_id":        [101, 102, 103],
    "event_timestamp": ["2024-10-01", "2024-10-02", "2024-10-03"]  # label timestamps
})

training_features = store.get_historical_features(
    entity_df=entity_df,
    features=["user_features:user_age_days",
              "user_features:avg_txn_amount_30d",
              "transaction_features:transaction_amount"]
).to_df()

# ── Serving: real-time feature lookup from online store ───────────────
# Latency: ~2-5ms from Redis
online_features = store.get_online_features(
    features=["user_features:user_age_days",
              "user_features:avg_txn_amount_30d"],
    entity_rows=[{"user_id": 101}]
).to_dict()

print(online_features)
# → {"user_age_days": [182], "avg_txn_amount_30d": [127.43]}
