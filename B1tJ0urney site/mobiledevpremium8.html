Why Architecture Matters

Architecture is the set of decisions that determine how code is organized, how components communicate, and how state flows through the application. Bad architecture doesn't fail immediately — it accumulates as technical debt until the codebase becomes unmaintainable.

  • Separating concerns — UI, business logic, and data should never be mixed
  • Testability — well-architected code can be unit tested without a simulator
  • Scalability — new features should not require rewriting existing ones
  • Predictable state — knowing exactly what your app will display at any moment
  • Team collaboration — clear boundaries allow parallel development without conflicts
  • Maintainability — future developers (and your future self) can understand the code
  • Debuggability — when things go wrong, you can trace state changes and find the cause

MVC vs MVVM — The Most Important Transition

MVC is where iOS development started. MVVM is where serious iOS development lives today.

MVC (Model-View-Controller): The Controller handles everything — UI events, business logic, networking, and state updates. In practice, this creates "Massive View Controllers" that are impossible to test.

MVVM (Model-View-ViewModel): The ViewModel contains all business logic and state. The View only observes and renders. The ViewModel has no UIKit/SwiftUI imports — it's plain Swift, 100% unit-testable.

// MVVM in SwiftUI — ViewModel holds all logic
class CounterViewModel: ObservableObject {
    @Published var count: Int = 0
    @Published var isLoading: Bool = false
    @Published var errorMessage: String? = nil

    func increment() { count += 1 }
    func decrement() { count = max(0, count - 1) }

    func fetchCount() async {
        isLoading = true
        do {
            count = try await CounterService.shared.getCount()
        } catch {
            errorMessage = error.localizedDescription
        }
        isLoading = false
    }
}

// View only observes — zero business logic here
struct CounterView: View {
    @StateObject var vm = CounterViewModel()

    var body: some View {
        VStack {
            if vm.isLoading { ProgressView() }
            else { Text("\(vm.count)").font(.largeTitle) }
            HStack {
                Button("-") { vm.decrement() }
                Button("+") { vm.increment() }
            }
        }
        .task { await vm.fetchCount() }
    }
}

Clean Architecture — Separation in Layers

Clean Architecture, popularized by Robert C. Martin ("Uncle Bob"), organizes code into concentric layers where dependencies only point inward — outer layers can depend on inner layers, but inner layers must never know about outer layers.

🎨 Presentation Layer
ViewModels, Views, UI components. Observes state. Calls use cases. No business logic.
⚙️ Domain Layer
Use Cases (Interactors), Entities, Repository interfaces. Pure business logic. Zero framework dependencies.
💾 Data Layer
Repository implementations, API clients, database DAOs. Implements interfaces defined in Domain.

The Dependency Rule: Source code dependencies must always point toward higher-level policies. A ViewModel can call a UseCase; a UseCase cannot import UIKit or Retrofit.

// Clean Architecture — Repository Pattern
// Domain layer: defines the interface (no implementation detail)
protocol UserRepository {
    func getUser(id: String) async throws -> User
    func saveUser(_ user: User) async throws
}

// Domain layer: use case (pure business logic)
class GetUserProfileUseCase {
    private let repository: UserRepository  // depends on protocol, not concrete type

    init(repository: UserRepository) { self.repository = repository }

    func execute(userId: String) async throws -> UserProfile {
        let user = try await repository.getUser(id: userId)
        return UserProfile(user: user, isVerified: user.email.contains("@"))
    }
}

// Data layer: concrete implementation (hidden from domain)
class RemoteUserRepository: UserRepository {
    func getUser(id: String) async throws -> User {
        return try await APIClient.shared.get("/users/\(id)")
    }
    func saveUser(_ user: User) async throws {
        try await APIClient.shared.post("/users", body: user)
    }
}

Unidirectional Data Flow — Redux & MVI

Unidirectional Data Flow (UDF) solves one of the hardest problems in app development: state that can be mutated from multiple places becomes unpredictable. UDF enforces a single, predictable cycle.

Action / Intent
→
Reducer
→
New State
→
View Renders
→
User Interacts
→
Action / Intent

Redux core principles: (1) Single source of truth — one Store holds all app state. (2) State is read-only — the only way to change state is to dispatch an Action. (3) Changes via pure functions — Reducers are pure functions: (State, Action) → NewState.

// Redux-style state management in Swift
// 1. Define state
struct AppState {
    var posts: [Post] = []
    var isLoading: Bool = false
    var error: String? = nil
}

// 2. Define actions (what CAN happen)
enum AppAction {
    case fetchPosts
    case postsLoaded([Post])
    case fetchFailed(String)
}

// 3. Pure reducer: (State, Action) → NewState
func appReducer(state: inout AppState, action: AppAction) {
    switch action {
    case .fetchPosts:
        state.isLoading = true
        state.error = nil
    case .postsLoaded(let posts):
        state.posts = posts
        state.isLoading = false
    case .fetchFailed(let msg):
        state.error = msg
        state.isLoading = false
    }
}

// 4. Store dispatches actions, reducer updates state
class Store: ObservableObject {
    @Published private(set) var state = AppState()
    func dispatch(_ action: AppAction) { appReducer(state: &state, action: action) }
}

BLoC Pattern — Flutter's State Solution

BLoC (Business Logic Component) is Flutter's dominant state management pattern. It separates UI from business logic using streams: the UI sends Events in, and receives States out. The bloc_library package makes this ergonomic.

📤 Events
User actions or external triggers. Sent from UI to BLoC. Examples: LoginButtonPressed, FetchDataRequested.
⚙️ BLoC
Receives events, runs business logic (async calls, validation), emits new states via streams.
📥 States
UI representation of data. Examples: Initial, Loading, Success, Failure. UI rebuilds on each new state.
// Flutter BLoC — Counter example
// Events
abstract class CounterEvent {}
class IncrementPressed extends CounterEvent {}
class DecrementPressed extends CounterEvent {}

// States
abstract class CounterState { final int count; const CounterState(this.count); }
class CounterInitial extends CounterState { const CounterInitial() : super(0); }
class CounterUpdated extends CounterState { const CounterUpdated(int count) : super(count); }

// BLoC — pure business logic, no Flutter imports needed
class CounterBloc extends Bloc<CounterEvent, CounterState> {
  CounterBloc() : super(const CounterInitial()) {
    on<IncrementPressed>((event, emit) {
      emit(CounterUpdated(state.count + 1));
    });
    on<DecrementPressed>((event, emit) {
      emit(CounterUpdated(state.count > 0 ? state.count - 1 : 0));
    });
  }
}

// UI — just sends events and reacts to states
BlocBuilder<CounterBloc, CounterState>(
  builder: (context, state) {
    return Column(
      children: [
        Text('${state.count}', style: TextStyle(fontSize: 48)),
        ElevatedButton(
          onPressed: () => context.read<CounterBloc>().add(IncrementPressed()),
          child: Text('+'),
        ),
      ],
    );
  },
)

The Composable Architecture (TCA) — SwiftUI's Power Pattern

TCA, created by Point-Free, is the most structured Swift architecture available. It brings Redux-style unidirectional flow to SwiftUI with composability, testability, and side effect management built in.

  • State — a struct describing the full state of a feature or screen
  • Action — an enum of all possible user interactions and effects for that feature
  • Reducer — a function taking State + Action → updated State + Effects to run
  • Store — holds State, receives Actions, runs the Reducer, and drives the View
  • Effect — async operations (API calls, timers) that return Actions back into the system
import ComposableArchitecture

// TCA — Feature definition
@Reducer
struct CounterFeature {
    struct State: Equatable {
        var count = 0
        var isLoading = false
    }

    enum Action {
        case incrementButtonTapped
        case decrementButtonTapped
        case fetchCount
        case countResponse(Int)
    }

    var body: some Reducer<State, Action> {
        Reduce { state, action in
            switch action {
            case .incrementButtonTapped:
                state.count += 1
                return .none
            case .decrementButtonTapped:
                state.count -= 1
                return .none
            case .fetchCount:
                state.isLoading = true
                return .run { send in          // side effects are explicit
                    let count = try await CounterClient.live.fetch()
                    await send(.countResponse(count))
                }
            case .countResponse(let count):
                state.count = count
                state.isLoading = false
                return .none
            }
        }
    }
}

// View — minimal, just sends actions and reads state
struct CounterView: View {
    let store: StoreOf<CounterFeature>
    var body: some View {
        WithViewStore(store, observe: { $0 }) { viewStore in
            VStack {
                Text("\(viewStore.count)")
                Button("+") { viewStore.send(.incrementButtonTapped) }
            }
        }
    }
}

Dependency Injection — Testable Architecture

Dependency Injection (DI) is the practice of providing dependencies to an object from the outside rather than creating them inside. It is the foundation of testable architecture — without it, you cannot swap real implementations for mocks in tests.

Without DI (tightly coupled): The ViewModel creates its own NetworkService, making it impossible to test without a real network. Any test will fire real API calls.

With DI (loosely coupled): The ViewModel receives a protocol-typed dependency. In tests, inject a MockNetworkService. In production, inject the RealNetworkService. Zero code change needed in the ViewModel.

// Protocol-based DI — makes mocking trivial
protocol PostService {
    func fetchPosts() async throws -> [Post]
}

class PostViewModel: ObservableObject {
    @Published var posts: [Post] = []
    private let service: PostService

    // Dependency injected via initializer
    init(service: PostService) { self.service = service }

    func load() async {
        posts = (try? await service.fetchPosts()) ?? []
    }
}

// Production: inject real implementation
let vm = PostViewModel(service: RealPostService())

// Tests: inject fast, controlled mock
class MockPostService: PostService {
    var stubbedPosts: [Post] = [Post(title: "Test")]
    func fetchPosts() async throws -> [Post] { return stubbedPosts }
}
let testVM = PostViewModel(service: MockPostService())

// Android Kotlin — Hilt DI (industry standard)
@HiltViewModel
class PostViewModel @Inject constructor(
    private val repository: PostRepository   // Hilt injects this
) : ViewModel() {
    fun loadPosts() = viewModelScope.launch {
        _posts.value = repository.getPosts()
    }
}

Popular DI frameworks:

  • Hilt (Android) — Google's recommended DI, built on Dagger 2. Compile-time verification.
  • Koin (Android/KMP) — lightweight, runtime DI with a DSL. Easier to learn than Hilt.
  • Swinject (iOS) — DI container for Swift projects.
  • TCA's DependencyValues — TCA's built-in DI system for SwiftUI, globally overridable in tests.

State Management Strategies — Choosing the Right Scope

Not all state is equal. A common mistake is putting all state in a global store when most state should be local. Choose the narrowest scope that works.

State TypeExamplesiOS SolutionAndroid/Flutter Solution
Ephemeral / LocalTextField focus, button press animation, dropdown open@State, @FocusStateremember {}, MutableState
Screen-levelLoading flag, form data, displayed list@StateObject ViewModelViewModel + StateFlow
Shared / FeatureLogged-in user, shopping cart, current theme@EnvironmentObject, TCA StoreShared ViewModel, Hilt scoped
Global / AppAuth state, navigation stack, global configApp-level TCA Store / singletonApplication-scoped ViewModel / Redux
PersistentUser preferences, cached data, offline stateUserDefaults, Core Data, SwiftDataDataStore, Room, SharedPreferences
