Introduction: LLM Engineering

LLM Engineering is the discipline of building reliable, cost-effective, production-grade applications on top of Large Language Models. It sits at the intersection of software engineering, prompt design, and AI systems thinking.

This module covers two major pillars:

  • LLM Engineering — working with the API, prompt engineering, structured outputs, function calling, fine-tuning, evals
  • RAG (Retrieval-Augmented Generation) — connecting LLMs to your own data via embeddings, vector databases, and retrieval pipelines

The most common failure mode in LLM applications is not model quality — it's poor prompt design, lack of retrieval, or not measuring what matters. This module fixes all three.

The LLM application stack — layers from API call to production system:

Layer 1
Model API
The foundation. A raw HTTP call to OpenAI, Anthropic, or an open-source model via Ollama/vLLM. Input: JSON with messages. Output: completion text or structured JSON. Everything else builds on top of this.
Layer 2
Prompt Layer
Templates, few-shot examples, chain-of-thought instructions, system personas. The difference between a bad and good prompt can mean 40%+ accuracy improvement without changing the model. This is where most production gains come from.
Layer 3
Retrieval Layer
RAG: embeddings, vector databases, chunking strategies, re-ranking. Connects the model to your own documents. Makes answers grounded and factual. The most widely deployed LLM engineering pattern in enterprise settings.
Layer 4
Orchestration Layer
LangChain, LlamaIndex, multi-step chains, agents. Coordinates multiple model calls, tools, memory. Adds complexity — only add this layer when simpler approaches don't solve the problem.
Layer 5
Evaluation Layer
RAGAS, LLM-as-judge, human evals. You cannot improve what you don't measure. Most teams skip this layer and wonder why their app regresses. Build evals before you optimize — it is the most important non-negotiable in production LLM systems.
Layer 6
Observability Layer
LangSmith, Weave, Helicone. Traces every LLM call: prompt, response, token count, latency, cost. In production you are blind without this. Enables debugging, cost monitoring, and catching prompt injection attacks.
⚙️ LLM Engineering

The OpenAI / Anthropic API Pattern

All major LLM APIs share the same core structure: a list of messages with roles, and a set of parameters that control generation.

from openai import OpenAI

client = OpenAI()

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system",  "content": "You are a senior software engineer. Be concise and precise."},
        {"role": "user",    "content": "Explain the difference between a mutex and a semaphore."}
    ],
    temperature=0.2,
    max_tokens=512,
    top_p=0.9,
    frequency_penalty=0.1
)

print(response.choices[0].message.content)
print(f"Tokens used: {response.usage.total_tokens}")

Key parameters:

  • temperature — 0 = deterministic, 1 = creative. Use low for factual tasks, higher for creative ones.
  • max_tokens — hard cap on output length. Always set it to avoid runaway costs.
  • top_p — nucleus sampling. Controls diversity by sampling from top-p probability mass.
  • frequency_penalty — reduces probability of tokens that have already appeared.

Production API patterns — what separates a demo from a production system:

from openai import OpenAI, APITimeoutError, RateLimitError, APIError
import time
import logging

client = OpenAI()
logger = logging.getLogger(__name__)

def call_llm_with_retry(
    messages: list[dict],
    model: str = "gpt-4o",
    temperature: float = 0.2,
    max_tokens: int = 1024,
    max_retries: int = 3,
    fallback_model: str = "gpt-4o-mini"
) -> str:
    """
    Production LLM call with: retry logic, exponential backoff,
    model fallback, timeout, and structured logging.
    """
    for attempt in range(max_retries):
        try:
            response = client.chat.completions.create(
                model=model,
                messages=messages,
                temperature=temperature,
                max_tokens=max_tokens,
                timeout=30.0,         # always set a timeout
                seed=42               # reproducibility (same seed → same output for same input)
            )
            content = response.choices[0].message.content
            logger.info({
                "model": model, "attempt": attempt + 1,
                "input_tokens": response.usage.prompt_tokens,
                "output_tokens": response.usage.completion_tokens,
                "cost_usd": (response.usage.prompt_tokens * 2.50 +
                             response.usage.completion_tokens * 10.0) / 1_000_000
            })
            return content

        except RateLimitError:
            wait = 2 ** attempt  # exponential backoff: 1s, 2s, 4s
            logger.warning(f"Rate limit hit, retrying in {wait}s...")
            time.sleep(wait)

        except APITimeoutError:
            logger.warning(f"Timeout on attempt {attempt + 1}")
            if attempt == max_retries - 1:
                # Fall back to a faster, smaller model
                logger.warning(f"Falling back to {fallback_model}")
                model = fallback_model

        except APIError as e:
            logger.error(f"API error: {e}")
            raise  # don't retry on 400/401 errors

    raise RuntimeError(f"All {max_retries} attempts failed")

# Anthropic (Claude) — same pattern, different SDK
from anthropic import Anthropic

def call_claude(prompt: str) -> str:
    message = anthropic_client.messages.create(

💡 Model Selection Guide (mid-2025)

GPT-4o: Best general-purpose. Strong at reasoning, code, structured outputs. $2.50/1M input, $10/1M output.
GPT-4o mini: 94% cheaper, 80–85% of the quality. Use for high-volume tasks where cost matters. First choice for routing, classification, summarization.
Claude Sonnet 4: Strongest at long document processing, nuanced writing, following complex instructions. 200K context window. $3/1M input.
Claude Haiku: Anthropic's fast, cheap tier. Comparable to GPT-4o mini in speed and cost.
LLaMA 3 / Mistral: Open-source, self-hosted. Zero API cost at scale. Lower quality ceiling but full control over data and privacy.

Prompt Engineering — Principles & Patterns

Prompt engineering is the art and science of writing inputs that reliably produce the outputs you need. A well-designed prompt is worth far more than a model upgrade.

Core principles:

  • Be explicit — state exactly what you want: format, length, tone, what to avoid
  • Use examples (few-shot) — showing is more reliable than telling
  • Chain-of-thought — add "Think step by step" for reasoning tasks
  • Give a role — "You are a medical expert" shapes vocabulary and reasoning style
  • Positive framing — "Be concise" works better than "Don't be verbose"
  • Separate instructions from data — use XML tags or delimiters to mark boundaries
# ❌ Weak prompt
"Summarize this."

# ✅ Strong prompt
"""
You are a financial analyst writing for non-technical executives.

Summarize the following earnings call transcript in exactly 3 bullet points.
Each bullet must:
- Start with a key metric or decision
- Be under 25 words
- Use plain English, no jargon


{transcript_text}

"""

The prompt engineering toolkit — all major techniques with when to use each:

TechniqueWhat It DoesWhen to UseExample Signal
Zero-shot Instruction only, no examples. Model uses pre-trained knowledge. Simple, well-defined tasks. Classification with obvious categories. Summarization. "Classify this review as positive or negative."
Few-shot 2–10 input/output examples in the prompt. Teaches format + behavior by demonstration. Custom output format. Domain-specific style. When zero-shot produces wrong format. Model keeps using markdown when you want plain JSON.
Chain-of-Thought "Think step by step" before final answer. Forces explicit reasoning trace. Math, logic, multi-step reasoning. Anywhere accuracy on complex tasks matters. Model gives wrong answer to multi-step calculation.
Self-Consistency Run the same prompt 3–5× at temperature 0.7, majority-vote the answers. High-stakes single questions where one wrong inference could fail. Classification with nuance. Medical or legal reasoning where you need confidence, not speed.
Role Prompting Assign a persona: "You are a senior security auditor..." Domain vocabulary, depth of analysis, writing style control. Output uses wrong jargon or reads like a general summary instead of expert analysis.
Decomposition Break a complex task into subtasks across multiple prompts / model calls. Tasks too complex for one call. Workflows where intermediate results need validation. "Write a complete business plan" — too broad for one reliable call.

Structured Outputs & Function Calling

Production applications rarely need free-form text — they need structured, parseable data. Modern LLM APIs support structured outputs that guarantee valid JSON matching a schema.

from openai import OpenAI
from pydantic import BaseModel
from typing import List

client = OpenAI()

class MovieReview(BaseModel):
    title: str
    sentiment: str
    score: float
    key_themes: List[str]
    one_line_summary: str

response = client.beta.chat.completions.parse(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": "Extract structured movie review data."},
        {"role": "user",   "content": f"Review: {review_text}"}
    ],
    response_format=MovieReview
)

review = response.choices[0].message.parsed
print(f"Sentiment: {review.sentiment}, Score: {review.score}/10")

Real-world structured output patterns for common production use cases:

from openai import OpenAI
from pydantic import BaseModel, Field, field_validator
from typing import List, Literal, Optional
from enum import Enum

client = OpenAI()

# ── Pattern 1: Classification with confidence ────────────────────────
class SupportTicketClassification(BaseModel):
    category: Literal["billing", "technical", "account", "feature_request", "complaint"]
    priority: Literal["low", "medium", "high", "critical"]
    confidence: float = Field(ge=0.0, le=1.0, description="Model confidence 0-1")
    requires_human: bool
    suggested_response_time_hours: int
    brief_summary: str = Field(max_length=100)

    @field_validator("confidence")
    def confidence_must_be_valid(cls, v):
        if v < 0 or v > 1:
            raise ValueError("Confidence must be between 0 and 1")
        return v

def classify_ticket(ticket_text: str) -> SupportTicketClassification:
    response = client.beta.chat.completions.parse(
        model="gpt-4o-mini",   # cheap model for classification
        messages=[
            {"role": "system", "content": "Classify this customer support ticket precisely."},
            {"role": "user",   "content": ticket_text}
        ],
        response_format=SupportTicketClassification
    )
    return response.choices[0].message.parsed

# ── Pattern 2: Information extraction with validation ────────────────
class ContractParty(BaseModel):
    name: str
    role: Literal["buyer", "seller", "vendor", "client", "contractor"]
    jurisdiction: Optional[str] = None

class ContractExtraction(BaseModel):
    parties: List[ContractParty]
    effective_date: Optional[str] = None  # ISO 8601 format
    expiry_date: Optional[str] = None
    total_value_usd: Optional[float] = None
    governing_law: Optional[str] = None
    auto_renews: bool
    key_obligations: List[str] = Field(max_length=5)  # max 5 items

def extract_contract(contract_text: str) -> ContractExtraction:
    response = client.beta.chat.completions.parse(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": """Extract key contract metadata.
            For dates use ISO 8601 (YYYY-MM-DD).
            If information is not present, use null.
            Key obligations: max 5, each under 20 words."""},
            {"role": "user", "content": contract_text}
        ],
        response_format=ContractExtraction
    )
    return response.choices[0].message.parsed

# ── Pattern 3: Streaming structured output ───────────────────────────
with client.beta.chat.completions.stream(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Analyse this dataset: ..."}],
    response_format=SupportTicketClassification
) as stream:
    for event in stream:
        if event.type == "content.delta":
            print(event.delta, end="", flush=True)  # stream to UI as JSON builds
🔧 Function Calling vs Structured Outputs

Structured Outputs (response_format=MyModel): use when you want guaranteed valid JSON back from the model. The model always responds with data — never calls external code. Best for extraction, classification, transformation.

Function Calling / Tools: use when you want the model to optionally decide to invoke an external function (search, database, API). The model outputs a call description; your code executes it. Best for agents and dynamic data fetching. Under the hood, these are the same mechanism — function calling is structured output applied to tool invocations.

LLM Evaluation — Measuring What Matters

You cannot improve what you don't measure. Building an evaluation suite before optimizing is the most important habit in LLM engineering.

Types of evals:

  • Unit tests — exact match, regex, or schema validation on known inputs
  • LLM-as-judge — use a second LLM to score outputs on accuracy, helpfulness, safety
  • Human evaluation — gold standard; expensive but necessary for production sign-off
  • RAG-specific — faithfulness, relevance, context recall
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_recall

results = evaluate(
    dataset=test_dataset,
    metrics=[faithfulness, answer_relevancy, context_recall]
)
print(results.to_pandas())

Building an LLM-as-judge evaluation system from scratch:

from openai import OpenAI
from pydantic import BaseModel
from typing import Literal
import json

client = OpenAI()

class EvalScore(BaseModel):
    accuracy: int        # 1-5: is the answer factually correct?
    completeness: int    # 1-5: does it fully answer the question?
    conciseness: int     # 1-5: is it appropriately brief?
    reasoning: str       # 2-3 sentence explanation of scores

def llm_judge(
    question: str,
    reference_answer: str,
    model_answer: str,
    judge_model: str = "gpt-4o"   # use a strong model as judge
) -> EvalScore:
    """Score a model's answer against a reference using an LLM judge."""

    prompt = f"""You are an expert evaluator. Score the model's answer on three criteria.
Be strict and objective. Use the full 1-5 scale.

Question: {question}
Reference Answer (ground truth): {reference_answer}
Model Answer (to evaluate): {model_answer}

Evaluate the model answer. Return scores 1-5 for each criterion."""

    response = client.beta.chat.completions.parse(
        model=judge_model,
        messages=[{"role": "user", "content": prompt}],
        response_format=EvalScore,
        temperature=0   # deterministic judging
    )
    return response.choices[0].message.parsed

# Run eval on a test set
def run_eval_suite(test_cases: list[dict], model_fn) -> dict:
    scores = []
    for case in test_cases:
        answer = model_fn(case["question"])
        score = llm_judge(
            question=case["question"],
            reference_answer=case["expected_answer"],
            model_answer=answer
        )
        scores.append({
            "question": case["question"][:50],
            "accuracy": score.accuracy,
            "completeness": score.completeness,
            "conciseness": score.conciseness
        })

    avg_accuracy     = sum(s["accuracy"] for s in scores) / len(scores)
    avg_completeness = sum(s["completeness"] for s in scores) / len(scores)

    return {
        "n": len(scores),
        "avg_accuracy": round(avg_accuracy, 2),
        "avg_completeness": round(avg_completeness, 2),
        "scores": scores
    }

# Usage
results = run_eval_suite(test_cases=test_data, model_fn=my_rag_chain.invoke)
print(f"Avg accuracy: {results['avg_accuracy']}/5.0")

Fine-Tuning vs Prompting vs RAG

One of the most important architectural decisions in LLM engineering: when to fine-tune vs use prompting vs RAG.

ApproachBest forCostData needed
PromptingMost tasks — start here alwaysInference onlyNone
RAGGrounding answers in your documents / up-to-date dataInference + vector DBDocuments to index
Fine-tuningSpecific style, format, or domain where prompting failsTraining + inference1,000–100,000 examples
Pre-trainingSpecialized domain from scratch (medicine, law, code)Massive GPU costBillions of tokens

Rule of thumb: Always try prompting first. If the model knows the knowledge but applies it wrong — fine-tune. If the model doesn't have the knowledge — use RAG.

Fine-tuning with the OpenAI API — the complete workflow:

from openai import OpenAI
import json

client = OpenAI()

# ── Step 1: Prepare training data ────────────────────────────────────
# Fine-tuning format: JSONL with system + user + assistant turns
training_data = [
    {
        "messages": [
            {"role": "system",    "content": "You extract product data from descriptions."},
            {"role": "user",      "content": "Red Nike Air Max 90, size 42, €129.99"},
            {"role": "assistant", "content": '{"brand":"Nike","model":"Air Max 90","color":"Red","size":42,"price_eur":129.99}'}
        ]
    },
    # ... 1,000+ examples for good results
]

# Save as JSONL
with open("training_data.jsonl", "w") as f:
    for item in training_data:
        f.write(json.dumps(item) + "\n")

# ── Step 2: Upload file ───────────────────────────────────────────────
file = client.files.create(
    file=open("training_data.jsonl", "rb"),
    purpose="fine-tune"
)
print(f"File ID: {file.id}")

# ── Step 3: Create fine-tuning job ────────────────────────────────────
job = client.fine_tuning.jobs.create(
    training_file=file.id,
    model="gpt-4o-mini",       # cheapest model to fine-tune
    hyperparameters={
        "n_epochs": 3,          # 3-5 epochs is typical
        "learning_rate_multiplier": 1.8
    }
)
print(f"Job ID: {job.id}")

# ── Step 4: Monitor and use ───────────────────────────────────────────
# Check status (takes 15min–2h depending on data size)
status = client.fine_tuning.jobs.retrieve(job.id)
print(f"Status: {status.status}")  # validating_files → queued → running → succeeded

# Use the fine-tuned model — just like a normal model
response = client.chat.completions.create(
    model=status.fine_tuned_model,  # e.g. "ft:gpt-4o-mini:your-org:name:abc123"
    messages=[
        {"role": "system", "content": "You extract product data from descriptions."},
        {"role": "user",   "content": "Blue Adidas Ultraboost 22, men's UK 9, £160.00"}
    ]
)
print(response.choices[0].message.content)
# → '{"brand":"Adidas","model":"Ultraboost 22","color":"Blue","size_uk":9,"price_gbp":160.00}'
⚠️ When NOT to Fine-Tune

Fine-tuning is expensive to maintain — every time your requirements change you re-train. Avoid fine-tuning when: the base model can do the task with a good prompt (very common), your knowledge changes frequently (use RAG), you have fewer than ~200 examples (few-shot prompting will work better), or your goal is just a specific output format (use structured outputs instead). The sweet spot for fine-tuning: you have a well-defined, stable task; prompting can't reliably achieve the style/format you need; and you have 1,000+ diverse, high-quality examples.

🔍 RAG

What Is RAG and Why Does It Matter?

Retrieval-Augmented Generation (RAG) connects an LLM to an external knowledge base at inference time. Instead of relying on knowledge baked into model weights, the model retrieves relevant documents and uses them as context to answer questions.

Problems RAG solves:

  • Hallucination — the model fabricates facts. RAG grounds answers in real documents.
  • Knowledge cutoff — the model's training has a cutoff date. RAG provides current information.
  • Private data — the model has no knowledge of your internal documents. RAG gives it access.
  • Citations — RAG enables the model to cite the exact source of every claim.

RAG vs alternatives — choosing the right architecture for your use case:

Use RAG When
Knowledge Access Problem
Your documents change frequently (weekly/monthly). You need citations. You have more than 500 pages of content. You need answers to be traceable to specific sources. Your content is proprietary and can't go into training data. Example: internal knowledge base chatbot, legal document Q&A.
Don't Use RAG When
Style/Behavior Problem
The model already knows the facts but gives wrong format (use fine-tuning). You have fewer than 20 documents that never change (put them in the system prompt directly). You need ultra-low latency (RAG adds 100–500ms). Your knowledge fits in a 128K context window and never changes.
RAG + Fine-Tuning
Complementary Approaches
These are not mutually exclusive. Common pattern: fine-tune the model to follow your output format reliably + RAG to provide the factual content. The fine-tuned model learns how to structure answers; RAG gives it the specific facts for each query. Used in enterprise document Q&A and medical assistants.
Long-Context Alternative
Stuff-the-Context Pattern
If your full knowledge base fits in 100K tokens, you can skip RAG entirely and just put everything in the context window (the "stuffing" approach). Claude 3.5 Sonnet (200K) can hold a full employee handbook. Tradeoff: much higher cost per query, but simpler architecture and no retrieval errors.

What actually happens inside each pipeline stage — the decisions that determine quality:

1
Document Loading — PDFs need OCR if scanned. HTML needs cleaning to remove nav/ads. Tables in PDFs often parse badly — consider structured table extraction separately. Metadata (filename, page, section) should be preserved alongside content for citations.
2
Chunking Strategy — chunk_size 512 tokens is a common default. Too small: chunks lack context. Too large: retrieval returns less precise matches. chunk_overlap 64 prevents losing information at boundaries. RecursiveCharacterTextSplitter splits on paragraphs first, then sentences, then characters — preserving semantic boundaries where possible.
3
Embedding — every chunk is embedded using the same model you'll use at query time. You cannot mix embedding models (different vector spaces are incompatible). If you change your embedding model, you must re-index everything. Store the model name in your metadata.
4
Vector Storage — chunks stored as (vector, content, metadata) tuples. Most vector DBs support filtering by metadata fields before vector search (e.g. only retrieve from documents authored after 2023). This is called metadata filtering and dramatically improves precision for multi-domain knowledge bases.
5
Retrieval — the query is embedded with the same model and ANN search finds the top-k closest vectors. k=4 is a common default: enough context for the LLM, not so much that irrelevant chunks crowd the prompt. Every additional chunk costs tokens and risks diluting the relevant information.
6
Generation — the retrieved chunks are injected into the prompt as {context}. The most important prompt engineering decision here: instruct the model to answer ONLY from the context and say "I don't know" when the answer isn't present. Without this, the model hallucin­ates from its pre-training knowledge.
🔍 RAG

Embeddings — The Engine of RAG

An embedding is a dense numerical vector that represents the semantic meaning of a piece of text. Two texts with similar meaning will have vectors close together in high-dimensional space — regardless of exact wording.

from openai import OpenAI
import numpy as np

client = OpenAI()

def embed(text: str) -> list[float]:
    response = client.embeddings.create(
        model="text-embedding-3-small",
        input=text
    )
    return response.data[0].embedding

def cosine_similarity(a, b):
    a, b = np.array(a), np.array(b)
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

v1 = embed("What is the capital of France?")
v2 = embed("Which city serves as France's capital?")
v3 = embed("How do I bake a chocolate cake?")

print(cosine_similarity(v1, v2))  # → 0.97
print(cosine_similarity(v1, v3))  # → 0.14

Popular embedding models:

  • text-embedding-3-small (OpenAI) — 1536 dims, fast, cheap
  • text-embedding-3-large (OpenAI) — 3072 dims, higher quality
  • nomic-embed-text — open-source, runs locally
  • BGE-M3 — multilingual, strong on non-English

Embedding best practices and failure modes to avoid:

from openai import OpenAI
import numpy as np
from typing import Union

client = OpenAI()

# ── Best practice 1: Batch embedding (much cheaper) ───────────────────
def embed_batch(texts: list[str], model: str = "text-embedding-3-small") -> list[list[float]]:
    """Embed multiple texts in one API call. 1 call vs N calls."""
    # OpenAI accepts up to 2048 texts per call
    response = client.embeddings.create(model=model, input=texts)
    # Sort by index to preserve original order
    return [e.embedding for e in sorted(response.data, key=lambda x: x.index)]

texts = ["Hello world", "How are you?", "Machine learning is fascinating"]
vectors = embed_batch(texts)  # 1 API call instead of 3

# ── Best practice 2: Asymmetric embedding (query vs document) ─────────
# Some embedding models have separate modes for queries vs documents
# This improves retrieval accuracy significantly (BGE, E5 models)

from sentence_transformers import SentenceTransformer

model = SentenceTransformer("BAAI/bge-large-en-v1.5")

def embed_query(query: str) -> np.ndarray:
    """Embed a search query with query-specific prefix."""
    return model.encode(f"Represent this sentence for searching: {query}")

def embed_document(doc: str) -> np.ndarray:
    """Embed a document for storage (no prefix needed)."""
    return model.encode(doc)

# ── Best practice 3: Dimension reduction (Matryoshka embeddings) ──────
# text-embedding-3 supports truncating dimensions to save storage/cost
response = client.embeddings.create(
    model="text-embedding-3-small",
    input="Your text here",
    dimensions=256   # reduce from 1536 → 256 dims, ~85% of quality
)
small_vector = response.data[0].embedding  # only 256 floats instead of 1536

# ── Failure mode: comparing embeddings from different models ──────────
v_openai = embed("capital of France")               # OpenAI vector space
v_local  = model.encode("capital of France")        # SentenceTransformer space
# cosine_similarity(v_openai, v_local) is MEANINGLESS — different spaces!
# Always use the same embedding model for indexing and querying.
📐 Choosing Embedding Dimensions

Higher dimensions = more expressive but more storage and slower ANN search. For most RAG use cases: text-embedding-3-small at 1536 dims is the right default — fast, cheap, high quality. If you have millions of documents and need to reduce storage cost: truncate to 256–512 dims (Matryoshka embeddings preserve most quality). Only upgrade to 3-large if you've evaluated and confirmed quality is insufficient at smaller dims.

Vector Databases

A vector database stores embeddings and enables fast approximate nearest neighbor (ANN) search — finding the most semantically similar vectors in milliseconds, even across millions of documents.

  • Pinecone — fully managed, production-ready, serverless option
  • Weaviate — open-source, hybrid search (vector + keyword)
  • Chroma — lightweight, great for prototyping and local dev
  • Qdrant — open-source, high-performance, Rust-based
  • pgvector — PostgreSQL extension; vectors alongside relational data
  • FAISS (Meta) — library (not a DB), optimal for local batch search
import chromadb

client = chromadb.Client()
collection = client.create_collection("company_docs")

collection.add(

               "Technical support is available 24/7 via chat.",
               "Enterprise plans include dedicated account managers."],
    ids=["doc_1", "doc_2", "doc_3"]
)

results = collection.query(query_texts=["How do I get a refund?"], n_results=2)
# → returns the 2 most semantically relevant documents

Building a Full RAG Pipeline

Here is a production-quality RAG pipeline using LangChain — from document loading to answer generation with citations:

from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import Chroma
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.document_loaders import PyPDFLoader
from langchain.chains import RetrievalQA
from langchain.prompts import PromptTemplate

loader = PyPDFLoader("company_handbook.pdf")
docs   = loader.load()

splitter = RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=64)
chunks = splitter.split_documents(docs)

embeddings  = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma.from_documents(chunks, embeddings)

prompt = PromptTemplate.from_template("""
You are a helpful assistant. Answer ONLY using the context below.
If the answer is not in the context, say "I don't have that information."

Context:
{context}

Question: {question}
Answer:""")

qa_chain = RetrievalQA.from_chain_type(
    llm=ChatOpenAI(model="gpt-4o", temperature=0),
    retriever=vectorstore.as_retriever(search_kwargs={"k": 4}),
    chain_type_kwargs={"prompt": prompt},
    return_source_documents=True
)

result = qa_chain.invoke({"query": "What is the parental leave policy?"})
print(result["result"])
print("Sources:", [d.metadata["source"] for d in result["source_documents"]])

Production RAG with Qdrant, metadata filtering, and streaming:

from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams, Filter, FieldCondition, MatchValue
from openai import OpenAI
from datetime import datetime

openai_client = OpenAI()
qdrant = QdrantClient(url="http://localhost:6333")

# ── Indexing with rich metadata ────────────────────────────────────────
def index_documents(documents: list[dict]):
    """documents: [{text, source, department, date_iso, language}]"""

    texts = [d["text"] for d in documents]
    # Batch embedding — one API call
    vectors = openai_client.embeddings.create(
        model="text-embedding-3-small", input=texts
    ).data

    qdrant.upsert(
        collection_name="knowledge_base",
        points=[{
            "id": i,
            "vector": vectors[i].embedding,
            "payload": {                    # metadata stored alongside vector
                "text": documents[i]["text"],
                "source": documents[i]["source"],
                "department": documents[i]["department"],
                "date": documents[i]["date_iso"],
                "language": documents[i].get("language", "en")
            }
        } for i in range(len(documents))]
    )

# ── Retrieval with metadata filtering ─────────────────────────────────
def retrieve(
    query: str,
    department: str = None,
    language: str = "en",
    k: int = 5
) -> list[dict]:
    """Retrieve chunks, optionally filtered by department and language."""

    query_vector = openai_client.embeddings.create(
        model="text-embedding-3-small", input=query
    ).data[0].embedding

    # Build filter — only search within matching metadata
    filters = []
    if department:
        filters.append(FieldCondition(key="department", match=MatchValue(value=department)))
    if language:
        filters.append(FieldCondition(key="language", match=MatchValue(value=language)))

    results = qdrant.search(
        collection_name="knowledge_base",
        query_vector=query_vector,
        query_filter=Filter(must=filters) if filters else None,
        limit=k,
        with_payload=True
    )
    return [{"text": r.payload["text"], "source": r.payload["source"],
             "score": r.score} for r in results]

# ── Generation with streaming ─────────────────────────────────────────
def rag_answer_stream(query: str, department: str = None):
    chunks = retrieve(query, department=department)
    context = "\n\n".join([f"[Score: {c['score']:.2f}] {c['text']}" for c in chunks])
    sources = list(set(c["source"] for c in chunks))

    stream = openai_client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": f"""Answer ONLY from the context provided.
Context:
{context}

If the answer is not in the context, respond: "I don't have that information in the knowledge base."
Always cite sources."""},
            {"role": "user", "content": query}
        ],
        stream=True,
        temperature=0
    )

    for chunk in stream:
        if chunk.choices[0].delta.content:
            yield chunk.choices[0].delta.content  # stream tokens to UI

    yield f"\n\nSources: {', '.join(sources)}"
🔍 RAG

Advanced RAG — Beyond Basic Retrieval

Naive RAG (embed → retrieve → generate) fails in many real-world scenarios. Advanced RAG techniques address these failures:

  • Hybrid Search — combine vector similarity with BM25 keyword search for better recall
  • Re-ranking — use a cross-encoder to re-score retrieved chunks by relevance before passing to LLM
  • HyDE (Hypothetical Document Embeddings) — generate a hypothetical answer first, then embed it for retrieval
  • Query expansion — rewrite or expand the user's question to improve retrieval coverage
  • Parent-child chunking — retrieve small chunks (precise) but pass their parent context (complete) to the LLM
  • Contextual compression — extract only the relevant sentences from each retrieved chunk
  • Self-RAG — the model decides when to retrieve and reflects on whether retrieved content is relevant

The single biggest RAG improvement: re-ranking. Retrieve 20 chunks with ANN, then re-rank with a cross-encoder and keep the top 4. Consistently improves faithfulness by 15–30%.

Implementing the top 3 advanced RAG techniques — with production code:

from sentence_transformers import CrossEncoder
from openai import OpenAI
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings, ChatOpenAI

client = OpenAI()

# ── Technique 1: Re-ranking with Cross-Encoder ────────────────────────
cross_encoder = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def retrieve_and_rerank(query: str, vectorstore, initial_k: int = 20, final_k: int = 4) -> list:
    """Retrieve broad set, then re-rank for precision."""

    # Step 1: Wide retrieval — get many candidates
    initial_results = vectorstore.similarity_search(query, k=initial_k)

    # Step 2: Re-rank using cross-encoder (reads query+doc together, more accurate)
    pairs = [[query, doc.page_content] for doc in initial_results]
    scores = cross_encoder.predict(pairs)  # returns relevance score per pair

    # Step 3: Sort by cross-encoder score, keep top final_k
    ranked = sorted(zip(scores, initial_results), key=lambda x: x[0], reverse=True)
    return [doc for _, doc in ranked[:final_k]]

# ── Technique 2: HyDE — Hypothetical Document Embeddings ─────────────
def hyde_retrieve(query: str, vectorstore, k: int = 4) -> list:
    """Generate a hypothetical answer to improve retrieval of relevant chunks."""

    # Step 1: Generate a hypothetical answer (will match real documents better)
    response = client.chat.completions.create(
        model="gpt-4o-mini",  # cheap model ok for hypothesis generation
        messages=[
            {"role": "system", "content": "Write a single paragraph that would perfectly answer this question. Be specific and use technical language if needed."},
            {"role": "user", "content": query}
        ],
        max_tokens=200
    )
    hypothetical_doc = response.choices[0].message.content

    # Step 2: Embed the hypothetical answer (not the raw query)
    results = vectorstore.similarity_search(hypothetical_doc, k=k)
    return results

# ── Technique 3: Query Expansion ──────────────────────────────────────
def expand_query(query: str) -> list[str]:
    """Generate multiple query variants to improve retrieval coverage."""

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": """Generate 3 alternative versions of this search query.
Return ONLY a JSON array of strings, nothing else.
The alternatives should use different vocabulary to find the same information."""},
            {"role": "user", "content": f"Query: {query}"}
        ],
        response_format={"type": "json_object"}
    )
    import json
    data = json.loads(response.choices[0].message.content)
    queries = data.get("queries", [query])
    return [query] + queries  # original + 3 alternatives

def retrieve_with_expansion(query: str, vectorstore, k: int = 4) -> list:
    """Retrieve using multiple query variants, deduplicate results."""
    queries = expand_query(query)
    all_results = []
    seen_contents = set()

    for q in queries:
        results = vectorstore.similarity_search(q, k=k)
        for r in results:
            if r.page_content not in seen_contents:
                all_results.append(r)
                seen_contents.add(r.page_content)

    return all_results[:k * 2]  # return top 2k deduplicated results
📈 Advanced RAG Impact (Typical Results)

Baseline RAG: faithfulness ~0.75, context recall ~0.70
+ Re-ranking: faithfulness → ~0.91 (+16pts). The single highest-impact improvement.
+ Hybrid Search: context recall → ~0.85 (+15pts). Catches exact-match queries that semantic search misses.
+ Query Expansion: context recall → ~0.90. Helps with ambiguous or short queries.
Recommended order: implement these in order of impact — re-ranking first, hybrid search second, query expansion third.
