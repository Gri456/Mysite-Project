🚀 7.1 What is Deployment? — From Lab to Reality

Deployment is the moment a Machine Learning model stops being an experiment and starts being a product. A model living only in a Jupyter Notebook generates zero business value — it must be integrated into the systems, applications, and workflows where decisions are actually made.

Think of deployment as the bridge between the Data Science team and the real world. Everything built before this point — data collection, cleaning, feature engineering, modelling, evaluation — only pays off when the model is running live, serving real predictions to real users.

• REST APIs with Flask & FastAPI: Serving predictions over HTTP.
• Containerisation with Docker: Making models portable and reproducible.
• Cloud Platforms: AWS, GCP, Azure — where models actually live.
• Model Drift: Why models degrade and how to detect it early.
• Monitoring Dashboards: Tracking performance, latency, and data quality in production.
• Retraining Pipelines: Closing the loop with automated retraining.
• MLOps: The discipline that makes all of this reliable, repeatable, and scalable.
💡 The Deployment Reality Check:

Studies consistently show that 87% of ML models never make it to production. The bottleneck is rarely the model's accuracy — it's the engineering complexity of deployment and the organisational challenge of integration. This module teaches you how to be in the 13% that ships.

📦 7.2 Saving Models — The First Step to Deployment

Before a model can be deployed anywhere, it must be serialised — converted from a live Python object into a file that can be saved, transferred, and reloaded. This is the most fundamental step.

Saving with Joblib (Scikit-Learn)

import joblib
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.ensemble import RandomForestClassifier

# Build and train a full pipeline
pipeline = Pipeline([
    ('scaler', StandardScaler()),
    ('model',  RandomForestClassifier(n_estimators=100, random_state=42))
])
pipeline.fit(X_train, y_train)

# Save the complete pipeline (preprocessing + model together!)
joblib.dump(pipeline, 'churn_model_v1.pkl')
print("✅ Model saved successfully")

# Load it anywhere, anytime
loaded_pipeline = joblib.load('churn_model_v1.pkl')
predictions = loaded_pipeline.predict(new_data)

Model Versioning — Never Overwrite Production Models

Always version your saved models. If a new version degrades in production, you need to roll back instantly.

from datetime import datetime
import json, os

# Version with timestamp and metadata
version   = datetime.now().strftime("%Y%m%d_%H%M%S")
filename  = f"churn_model_{version}.pkl"
joblib.dump(pipeline, f"models/{filename}")

# Save model metadata alongside the file
metadata = {
    "version":        version,
    "filename":       filename,
    "train_f1":       0.871,
    "train_samples":  45000,
    "features":       list(X_train.columns),
    "created_by":     "data-science-team",
    "status":         "staging"   # staging | production | deprecated
}
with open(f"models/{version}_metadata.json", "w") as f:
    json.dump(metadata, f, indent=2)

print(f"✅ Model v{version} saved with metadata")

📡 7.3 REST APIs — Serving Predictions Over HTTP

The most common deployment pattern is wrapping the model in a REST API. Any application — web, mobile, internal system — can then send data to the model and receive predictions via standard HTTP requests. The model becomes a service.

Mobile App → POST /predict → API Server → loads model → Prediction → JSON response → Mobile App

Flask API — Simple and Widely Used

from flask import Flask, request, jsonify
import joblib, pandas as pd, logging

app    = Flask(__name__)
model  = joblib.load('models/churn_model_v1.pkl')
logger = logging.getLogger(__name__)

@app.route('/health', methods=['GET'])
def health():
    """Health check endpoint — used by load balancers"""
    return jsonify({"status": "healthy", "model": "churn_v1"})

@app.route('/predict', methods=['POST'])
def predict():
    try:
        # Parse incoming JSON data
        data    = request.get_json()
        df      = pd.DataFrame([data])

        # Generate prediction + probability
        pred    = model.predict(df)[0]
        proba   = model.predict_proba(df)[0][1]

        logger.info(f"Prediction made: {pred} ({proba:.3f})")

        return jsonify({
            "will_churn":        bool(pred),
            "churn_probability": round(float(proba), 4),
            "model_version":     "v1",
            "status":            "success"
        })

    except Exception as e:
        logger.error(f"Prediction error: {e}")
        return jsonify({"error": str(e), "status": "failed"}), 400

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000, debug=False)

FastAPI — The Modern Choice (Faster + Auto-Documentation)

from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
import joblib, pandas as pd

app   = FastAPI(title="Churn Prediction API", version="1.0")
model = joblib.load('models/churn_model_v1.pkl')

# Pydantic model for request validation (auto-validates input types!)
class CustomerData(BaseModel):
    account_age_days:    int
    days_since_login:    int
    monthly_spend:       float
    support_tickets:     int
    plan_type:           str

class PredictionResponse(BaseModel):
    will_churn:        bool
    churn_probability: float
    model_version:     str

@app.get("/health")
def health(): return {"status": "ok"}

@app.post("/predict", response_model=PredictionResponse)
def predict(customer: CustomerData):
    try:
        df   = pd.DataFrame([customer.dict()])
        pred = model.predict(df)[0]
        prob = model.predict_proba(df)[0][1]
        return PredictionResponse(
            will_churn=bool(pred),
            churn_probability=round(float(prob), 4),
            model_version="v1"
        )
    except Exception as e:
        raise HTTPException(status_code=400, detail=str(e))

🐳 7.4 Docker — Making Models Portable

The classic problem: "It works on my machine!". Docker solves this by packaging your model, its dependencies, and its runtime environment into a single container that runs identically on any machine — your laptop, a server, or the cloud.

Without Docker ❌
  • Works on dev machine, breaks on server
  • Dependency version conflicts
  • Manual setup on every new machine
  • "It worked last week!" mystery bugs
With Docker ✅
  • Identical environment everywhere
  • Exact dependency versions locked
  • One command to launch anywhere
  • Scales horizontally with ease

Dockerfile — The Recipe for Your Container

# Dockerfile
FROM python:3.11-slim

# Set working directory inside the container
WORKDIR /app

# Copy and install dependencies FIRST (Docker caching optimisation)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy model and application files
COPY models/ ./models/
COPY app.py .

# Expose the port the API will run on
EXPOSE 8000

# Command to start the server
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]
# requirements.txt

scikit-learn==1.3.2
pandas==2.1.3

# Build and run the container
docker build -t churn-api:v1 .
docker run -d -p 8000:8000 --name churn-service churn-api:v1

# Test it's running

Deployment Patterns — Choosing the Right Strategy

⚡ Real-time / Online Inference

Model responds to each request immediately. Best for: fraud detection during payment, product recommendations while browsing, chatbots. Requires low latency (typically < 200ms).

Tools: FastAPI + Uvicorn, Flask, TorchServe, TensorFlow Serving

🗄️ Batch Inference

Model runs on a large dataset at scheduled intervals (hourly, daily). Results stored in a database for later use. Best for: churn scores computed overnight, monthly credit risk assessments, weekly demand forecasting.

Tools: Apache Spark, AWS Batch, Airflow, scheduled Python scripts

📱 Edge Inference

Model runs directly on the device (smartphone, IoT sensor, camera) without internet. Best for: offline face recognition, on-device language translation, real-time image classification in manufacturing.

Tools: TensorFlow Lite, ONNX Runtime, Core ML (iOS), PyTorch Mobile

📉 7.6 Model Drift — Why Models Degrade in Production

A model trained on last year's data may be completely wrong about this year's world. Model Drift is the inevitable degradation of model performance as the real world changes but the model stays static. Understanding and detecting drift is one of the most critical production skills.

📊 Data Drift (Covariate Shift)

The distribution of input features changes. The model was trained on data from one population but now receives data from a different one.

Example: A credit risk model trained on pre-pandemic data sees very different spending patterns post-pandemic. Feature distributions have shifted.

🎯 Concept Drift

The relationship between features and the target changes. The same inputs now produce different outputs because the underlying concept has evolved.

Example: A fraud detection model trained in 2021 doesn't recognise new fraud techniques developed in 2024. The input data looks similar, but fraud patterns changed.

🔬 Detecting Data Drift with Statistical Tests

import numpy as np
from scipy import stats
import pandas as pd

def detect_drift(reference_data: pd.Series, current_data: pd.Series,
                 feature_name: str, threshold: float = 0.05):
    """
    Detect drift using the Kolmogorov-Smirnov test.
    p-value < threshold → distributions are significantly different → drift detected.
    """
    stat, p_value = stats.ks_2samp(reference_data, current_data)

    drift_detected = p_value < threshold
    status = "⚠️ DRIFT DETECTED" if drift_detected else "✅ No drift"

    print(f"Feature: {feature_name}")
    print(f"  KS Statistic: {stat:.4f}")
    print(f"  p-value:      {p_value:.4f}")
    print(f"  Status:       {status}")
    return drift_detected

# Compare training data distributions with current production data
training_data    = pd.read_csv('data/training_features.csv')
production_data  = pd.read_csv('data/production_last_30d.csv')

for feature in ['account_age_days', 'days_since_login', 'monthly_spend']:
    detect_drift(training_data[feature], production_data[feature], feature)
⚠️ Real-World Drift Examples:
  • E-commerce recommendation model trained in summer, deployed in winter — product preferences change seasonally
  • Demand forecasting model trained pre-COVID — supply chains and consumer behaviours completely disrupted
  • Churn prediction model — company launches new product tier, fundamentally changing the feature landscape
  • NLP sentiment model — new slang, cultural events, or political shifts change language patterns

📊 7.7 Monitoring in Production — The Model's Health Dashboard

Once deployed, a model needs constant supervision. Monitoring falls into two categories: model performance monitoring (is the model making good predictions?) and system health monitoring (is the infrastructure running correctly?).

📈 Model Performance Metrics

Prediction Accuracy (Classification)

Track F1, Precision, Recall on a rolling window of labelled data

Prediction Error (Regression)

Track rolling RMSE, MAE — alert if they exceed baseline thresholds

Prediction Distribution

Track the distribution of model outputs. Sudden shifts in average prediction score signal drift

Feature Distribution

Monitor means, std dev, null rates of input features vs. training baseline

⚙️ System Health Metrics

Latency (Response Time)

P50, P95, P99 latency. Alert if P99 exceeds SLA (e.g., 500ms)

Throughput (Requests/sec)

Track requests per second. Sudden drops or spikes indicate issues

Error Rate

% of failed predictions (exceptions, timeouts, schema errors)

Resource Usage

CPU, RAM, GPU usage. Spot memory leaks before they cause outages

🐍 Building a Simple Monitoring System

import time, logging, json
from datetime import datetime
from collections import defaultdict
import numpy as np

class ModelMonitor:
    def __init__(self, model_name: str, alert_threshold: float = 0.05):
        self.model_name        = model_name
        self.alert_threshold   = alert_threshold
        self.predictions       = []
        self.latencies         = []
        self.errors            = 0
        self.requests          = 0
        self.logger            = logging.getLogger(model_name)

    def log_prediction(self, features: dict, prediction, probability: float, latency_ms: float):
        self.requests   += 1
        self.predictions.append(probability)
        self.latencies.append(latency_ms)

        # Alert if average prediction shifts significantly
        if len(self.predictions) > 100:
            recent_avg    = np.mean(self.predictions[-100:])
            baseline_avg  = 0.15   # Expected churn rate from training
            if abs(recent_avg - baseline_avg) > self.alert_threshold:
                self.logger.warning(
                    f"⚠️ DRIFT ALERT: avg prediction {recent_avg:.3f} "
                    f"vs baseline {baseline_avg:.3f}"
                )

    def log_error(self, error: str):
        self.errors += 1
        self.logger.error(f"Prediction error: {error}")

    def get_stats(self) -> dict:
        return {
            "model":          self.model_name,
            "total_requests": self.requests,
            "error_rate":     self.errors / max(self.requests, 1),
            "avg_latency_ms": np.mean(self.latencies) if self.latencies else 0,
            "p99_latency_ms": np.percentile(self.latencies, 99) if self.latencies else 0,
            "avg_prediction": np.mean(self.predictions) if self.predictions else 0,
            "timestamp":      datetime.now().isoformat()
        }

# Use in your API
monitor = ModelMonitor("churn_model_v1")

def predict_with_monitoring(features: dict):
    start = time.time()
    try:
        df   = pd.DataFrame([features])
        pred = model.predict(df)[0]
        prob = model.predict_proba(df)[0][1]
        monitor.log_prediction(features, pred, prob, (time.time()-start)*1000)
        return pred, prob
    except Exception as e:
        monitor.log_error(str(e))
        raise

🔁 7.8 Retraining Pipelines — Keeping Models Current

Monitoring detects when performance degrades — retraining fixes it. The goal is to close the feedback loop: automatically collect new data, retrain the model, evaluate it, and — if it passes quality gates — replace the production model with zero downtime.

The Complete Model Lifecycle

📊 Collect New Data → 🛠️ Feature Engineering → 🏋️ Retrain Model → ✅ Evaluate vs Champion → 🚀 Deploy if Better → 📊 Monitor Production

↑ Loop repeats automatically when drift is detected ↑

Automated Retraining Script

import joblib, pandas as pd, logging
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import cross_val_score
from datetime import datetime

def retrain_model(new_data_path: str, champion_path: str, threshold: float = 0.02):
    """
    Retrain and promote model only if it beats the current champion.
    """
    logger = logging.getLogger("retraining")
    logger.info(f"Starting retraining pipeline at {datetime.now()}")

    # 1. Load new data
    df       = pd.read_csv(new_data_path)
    X, y     = df.drop('churned', axis=1), df['churned']

    # 2. Train challenger model
    challenger = RandomForestClassifier(n_estimators=200, random_state=42, n_jobs=-1)
    challenger_scores = cross_val_score(challenger, X, y, cv=5, scoring='f1')
    challenger_f1     = challenger_scores.mean()
    challenger.fit(X, y)

    # 3. Load champion and evaluate
    champion       = joblib.load(champion_path)
    champion_scores= cross_val_score(champion, X, y, cv=5, scoring='f1')
    champion_f1    = champion_scores.mean()

    logger.info(f"Champion F1:    {champion_f1:.4f}")
    logger.info(f"Challenger F1:  {challenger_f1:.4f}")

    # 4. Promote challenger only if meaningfully better
    if challenger_f1 > champion_f1 + threshold:
        version  = datetime.now().strftime("%Y%m%d_%H%M%S")
        new_path = f"models/churn_model_{version}.pkl"
        joblib.dump(challenger, new_path)
        logger.info(f"✅ Challenger promoted! Saved to {new_path}")
        return new_path, challenger_f1
    else:
        logger.info("⏸️ Champion retained — challenger did not improve sufficiently")
        return champion_path, champion_f1

# Run manually or via scheduler (Airflow, cron, etc.)
retrain_model("data/new_30d_data.csv", "models/churn_model_current.pkl")

When to Retrain?

Schedule-based

Retrain every week/month regardless of performance. Simple, predictable. Best for: slowly changing domains, regulatory compliance needs.

Trigger-based (Event-driven)

Retrain only when drift or performance degradation is detected. More efficient — avoids unnecessary retraining. Best for: stable domains with occasional disruptions.

🔐 7.10 Security, Scalability & A/B Testing

🔐 Security in Production

A deployed model is an exposed surface. Security must be baked in from the start, not added as an afterthought.

Authentication

API keys or OAuth 2.0 tokens on every request. Never expose a model endpoint publicly without auth.

Input Validation

Validate and sanitise all inputs before feeding to the model. Reject unexpected schemas or values.

Rate Limiting

Prevent abuse with request throttling. Protect against model inversion and data extraction attacks.

Audit Logging

Log every prediction with timestamp, user ID, inputs, and output for traceability and compliance (GDPR).

📈 Scalability Strategies

Horizontal Scaling

Run multiple identical API containers behind a load balancer. Traffic spikes handled by spinning up more instances automatically (auto-scaling groups).

Caching

Cache predictions for identical or near-identical inputs using Redis. A product recommendation for a repeat visitor doesn't need a fresh inference every time.

Model Optimisation

Use ONNX to convert models to an optimised inference format. Quantise Neural Networks (INT8) to reduce size and latency by 4x with minimal accuracy loss.

Async Processing

For non-urgent predictions, use message queues (Kafka, RabbitMQ, SQS). Client posts request, gets a job ID, polls for result — decouples traffic peaks from inference.

🧪 A/B Testing Models in Production

Before fully replacing the champion model, route a fraction of traffic to the challenger and compare real-world performance — not just offline metrics.

import random

# Traffic splitting: 90% champion, 10% challenger
def predict_with_ab_test(features: dict, challenger_pct: float = 0.10):
    use_challenger = random.random() < challenger_pct

    if use_challenger:
        model    = challenger_model
        version  = "challenger_v2"
    else:
        model    = champion_model
        version  = "champion_v1"

    df   = pd.DataFrame([features])
    pred = model.predict(df)[0]
    prob = model.predict_proba(df)[0][1]

    # Log which model was used — essential for analysis
