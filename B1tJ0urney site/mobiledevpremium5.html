Why API & Database Integration Matters

Every mobile app that goes beyond simple UI needs to communicate with external data sources:

  • Fetching product lists
  • Logging in users
  • Storing preferences
  • Syncing data between devices
  • Uploading photos
  • Sending messages
  • Real-time updates

Mobile apps depend on: APIs (backends the app communicates with), Databases (local or remote), and Synchronization mechanisms.

Typical mobile stack: UI → Business Logic → API Layer → Database

What Is an API?

API = Application Programming Interface — it defines how your app talks to a server or external service.

Most mobile apps use:

  • REST APIs (JSON) — most common
  • GraphQL APIs — flexible querying
  • gRPC — less common in mobile
  • WebSockets — real-time communication

REST APIs – The Most Common Mobile Standard

REST uses HTTP methods to perform operations:

GETRetrieve data
POSTCreate data
PUTUpdate data
DELETERemove data
// GET request
GET https://api.bitjourney.com/users/5

// Response
{
  "id": 5,
  "name": "Nuno",
  "role": "Premium"
}

Common HTTP status codes:

200 OK
Request succeeded
201 Created
Resource created
401 Unauthorized
Auth required
404 Not Found
Resource missing
500 Server Error
Backend failed
429 Rate Limited
Too many requests

GraphQL – A More Flexible API Model

GraphQL lets your app request exactly the data it needs.

# GraphQL query — request only what you need
{
  user(id: 5) {
    name
    email
    subscription
  }
}

Advantages: Reduces over-fetching · Faster on unstable networks · Excellent for mobile data efficiency

Making API Requests in Mobile Apps

Swift (iOS) – URLSession

let url = URL(string: "https://api.example.com/users")!
let task = URLSession.shared.dataTask(with: url) { data, _, _ in
    let users = try? JSONDecoder().decode([User].self, from: data!)
}
task.resume()

Kotlin (Android) – Retrofit + Coroutines

interface Api {
    @GET("users")
    suspend fun getUsers(): List<User>
}

Flutter (Dart)

final response = await http.get(Uri.parse("https://api.example.com/users"));

React Native (JavaScript)

const response = await fetch("https://api.example.com/users");

Authentication & Security

Common authentication systems for mobile apps:

1. API Keys — Simple access control, sent with each request.

2. OAuth2 — Used by Google, Apple, Facebook login.

3. JWT (JSON Web Tokens) — Stores user identity and permissions in a compact token.

// Example JWT
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

// Decoded payload
{
  "userId": 5,
  "role": "premium",
  "exp": 1735689600
}

4. Refresh Tokens — Keep users logged in without repeating credentials.

Secure token storage by platform:

  • iOS → Keychain
  • Android → Encrypted SharedPreferences
  • Flutter / React Native → SecureStorage

Local Databases – Offline Storage on Device

Mobile apps often need offline capabilities. Common local storage formats:

Key-Value (small data) → SharedPreferences (Android), UserDefaults (iOS)

Relational DB → SQLite (cross-platform), Room (Android), Core Data (iOS)

NoSQL Mobile DBs → Realm, ObjectBox — great for complex objects

Use cases: Offline mode · Caching API responses · Storing user settings · Complex object storage

Remote Databases – Data in the Cloud

Most apps sync with a remote backend:

  • Firebase Firestore — real-time NoSQL
  • Supabase — open-source Firebase alternative
  • MongoDB Atlas — cloud NoSQL
  • MySQL / PostgreSQL via REST or GraphQL
  • Strapi / Appwrite — headless CMS options

Remote databases enable: Real-time updates · Multi-device sync · Scalable storage · Authentication integration

Data Synchronization Strategies

Synchronizing data between local and remote sources is crucial for great UX:

1. Online-first — Always loads server data, stores locally as backup.

2. Offline-first — App uses local DB, syncs silently when online. Best for unreliable networks.

3. Real-time sync — Uses WebSockets or Firestore listeners. Ideal for: Chats · Notifications · Multiplayer apps · Tracking apps

Conflict resolution: Last-write-wins · Server priority · Merge logic

API Errors & Handling

Apps must handle real-world issues gracefully:

  • Network failure
  • API downtime
  • Invalid or expired tokens
  • Rate limiting (429)
  • Slow internet / timeouts

Good apps:

  • Retry automatically with exponential backoff
  • Serve cached fallback data
  • Show clear, actionable error messages
  • Refresh tokens before they expire

JSON Parsing & Data Modelling

API responses arrive as raw JSON text. The app must parse them into typed model objects before using the data in the UI.

Swift – Codable protocol:

struct User: Codable {
    let id: Int
    let name: String
    let email: String
}

// Decode JSON into model
let user = try? JSONDecoder().decode(User.self, from: data)

Kotlin – data class + Gson/Moshi:

data class User(
    val id: Int,
    val name: String,
    val email: String
)

// Retrofit handles parsing automatically

Best practices:

  • Always handle null or missing fields gracefully
  • Use optional/nullable types for fields that may not be present
  • Map API field names to clean model names using annotations
  • Validate data before saving to local database

The Repository Pattern

The Repository Pattern is the standard architectural approach for managing data access in mobile apps. It acts as a single source of truth that abstracts whether data comes from the network, local database, or cache.

How it works: The ViewModel (or Presenter) only talks to the Repository. The Repository decides whether to fetch fresh data from the API, return cached data, or read from the local database.

// Kotlin – UserRepository
class UserRepository(
    private val api: ApiService,
    private val db: UserDao
) {
    suspend fun getUser(id: Int): User {
        val cached = db.getUser(id)
        if (cached != null) return cached

        val remote = api.fetchUser(id)
        db.insert(remote) // cache it
        return remote
    }
}

Benefits: Clean separation of concerns · Easy to unit test · Swap data source without touching UI · Supports both online and offline scenarios

Pagination & Infinite Scroll

APIs rarely return all records at once. Pagination splits large datasets into pages to reduce bandwidth and improve performance.

Common pagination strategies:

  • Offset-based – GET /posts?page=2&limit=20 — simple but can miss items if data changes
  • Cursor-based – GET /posts?cursor=abc123 — more reliable for real-time feeds
  • Keyset pagination – GET /posts?after_id=100 — very efficient for ordered data
// Typical paginated API response
{
  "data": [...],
  "meta": {
    "page": 2,
    "total_pages": 10,
    "next_cursor": "eyJpZCI6MTAwfQ=="
  }
}

Android: The Paging 3 library (Jetpack) handles pagination, loading states, and error handling out of the box. iOS: Manual implementation or third-party libraries. Flutter: infinite_scroll_pagination package.

WebSockets & Real-Time Communication

WebSockets enable a persistent two-way connection between the app and server — unlike HTTP which is request/response only.

When to use WebSockets:

  • Chat applications (messages appear instantly)
  • Live sports scores or stock prices
  • Multiplayer games
  • Real-time location tracking
  • Collaborative editing tools
// React Native – WebSocket connection
const ws = new WebSocket('wss://api.example.com/chat');

ws.onopen = () => ws.send('Hello!');
ws.onmessage = (e) => console.log(e.data);
ws.onerror = (e) => console.error(e.message);
ws.onclose = () => console.log('Disconnected');

Alternatives to raw WebSockets: Socket.IO (adds reconnection and rooms), Firebase Realtime Database, Firestore listeners, Pusher, Ably — all provide managed real-time infrastructure.

Caching Strategies

Good caching makes apps feel fast and resilient even with poor connectivity. There are several levels of caching to consider:

Memory cache (in-app) – Stores data in RAM for the current session. Fastest access but lost when the app closes. Ideal for images currently on screen.

Disk cache – Persists data to local storage across sessions. Slower than memory but survives app restarts. Great for API responses, images, and files.

HTTP cache (ETag / Cache-Control) – The server can instruct the client how long to cache a response. The app re-validates only when the cache expires.

Common caching tools by platform:

  • Android – OkHttp cache (HTTP), Room (DB), Glide/Coil (images)
  • iOS – NSURLCache (HTTP), Core Data/UserDefaults, SDWebImage/Kingfisher (images)
  • Flutter – cached_network_image, Hive, dio cache interceptor
  • React Native – React Query (server state), MMKV (fast key-value), FastImage (images)

TTL (Time-to-Live) – Every cached item should have an expiry time. After TTL, the app fetches fresh data. Balances freshness with performance.

File Uploads & Media Handling

Many apps need to upload user-generated content — profile photos, documents, videos. This requires a different approach to regular API calls.

Multipart form upload (most common): Files are sent as multipart/form-data POST requests. Used for images, PDFs, and small files.

// Flutter – Upload image with http package
var request = http.MultipartRequest(
  'POST', Uri.parse('https://api.example.com/upload')
);
request.files.add(await http.MultipartFile.fromPath('photo', filePath));
final response = await request.send();

Presigned URLs (for large files): The app first requests a temporary upload URL from the backend. The file is then uploaded directly to cloud storage (S3, GCS) — bypassing the server entirely. Faster and more scalable.

  • Always compress images before uploading to reduce bandwidth
  • Show upload progress indicators for large files
  • Handle upload interruptions and support resumable uploads
  • Validate file types and sizes on both client and server

API Versioning & Backward Compatibility

Mobile apps cannot force users to update immediately. When a backend changes, older app versions must still work — this is the versioning challenge.

Common versioning approaches:

  • URL versioning – https://api.example.com/v2/users — most common, easy to see at a glance
  • Header versioning – Accept: application/vnd.api+json;version=2 — cleaner URLs but less visible
  • Query param – GET /users?api_version=2 — easy to test but less RESTful

Best practices: Never remove fields from an API response without a deprecation period. Always add new fields as optional. Send a min_supported_version flag so the app can prompt users to update when a version is sunset.

Security Best Practices for APIs

Insecure API integration is one of the most common sources of mobile app vulnerabilities. These practices are essential:

  • Always use HTTPS – Never send sensitive data over plain HTTP; enforce TLS 1.2+ on all endpoints
  • Certificate pinning – The app verifies the server's certificate matches an expected value, blocking man-in-the-middle attacks
  • Never log tokens or passwords – Sensitive data in logs can be extracted from device storage or crash reports
  • Validate tokens on every request – Don't trust the client; always verify JWT signatures server-side
  • Use short-lived access tokens – Access tokens should expire in minutes/hours, not weeks; use refresh tokens for long sessions
  • Rate limit & throttle – Protect your API from abuse; respond with 429 and a Retry-After header
  • Encrypt sensitive local data – Data stored on-device (tokens, personal info) should be encrypted at rest

Testing API Integration

Properly testing the data layer prevents runtime crashes and ensures reliability. There are several levels of testing to apply:

Unit tests for the Repository: Mock the API and database, test that the repository returns the correct data and handles errors properly.

Mocking API responses: Use tools like MockWebServer (OkHttp/Android), OHHTTPStubs (iOS), or nock (React Native/Node) to simulate API responses without a real server.

Integration tests: Test the full flow from ViewModel to API to database. Ensure that a real (or staging) API call results in the correct UI state.

Development tools:

  • Postman / Insomnia – Design and test API endpoints manually
  • Charles Proxy / Proxyman – Inspect all network traffic from the device in real time
  • Flipper – Facebook's mobile debugging tool with a network inspector plugin
  • Swagger / OpenAPI – Auto-generate API documentation and client SDKs
