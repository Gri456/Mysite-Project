Introduction: Why Mobile DevOps Is Different

Mobile CI/CD is fundamentally harder than web CI/CD. Beyond the usual code-test-deploy cycle, mobile teams must deal with code signing (certificates, provisioning profiles), app store review (Apple review takes 1–3 days), device fragmentation (thousands of Android devices), and over-the-air update limitations (you can't force users to update).

This module covers:

  • The mobile CI/CD pipeline end-to-end
  • Automated testing: unit, integration, UI/E2E
  • Code signing for iOS and Android
  • Fastlane — automating builds, signing, and store uploads
  • GitHub Actions for mobile
  • Beta distribution: TestFlight and Firebase App Distribution
  • App Store and Play Store automated deployment
  • Mobile monitoring and crash reporting

The four reasons mobile DevOps pipelines break in practice — and how to prevent each:

Failure 1
Certificate Expiry
iOS distribution certificates last 1 year. Provisioning profiles last 1 year. When they expire mid-pipeline at 3am before a release, the build fails with a cryptic codesign error. Prevention: Fastlane Match + calendar alerts at 60 and 30 days before expiry. Fastlane can auto-renew profiles.
Failure 2
Build Number Collisions
TestFlight rejects builds with a build number Apple has seen before — even if rejected or deleted. CI must auto-increment the build number on every run. Fastlane's increment_build_number uses the latest App Store Connect number as the baseline. Never hardcode build numbers.
Failure 3
Xcode Version Drift
A developer updates Xcode locally but CI still uses the old version. Swift 5.9 vs 5.10 can break builds silently. Apple mandates minimum Xcode versions for App Store submissions. Pin Xcode versions explicitly in CI with sudo xcode-select. Use .xcode-version file (Bitrise/Codemagic respect it).
Failure 4
Dependency Hell
CocoaPods, SPM, Gradle dependencies change between CI runs if not locked. A transitive dependency update can introduce a breaking API change. Lock everything: Podfile.lock committed to Git, Package.resolved committed, Gemfile.lock for Fastlane gems. Never run pod update or bundle update in CI without reviewing changes first.

🏎️ Pipeline Speed Matters — Real Time Targets

Teams shipping daily need fast feedback loops. PR check (unit tests only): under 5 minutes. Full pipeline to TestFlight: under 15 minutes. Nightly E2E suite: under 30 minutes. Anything slower discourages frequent commits. Key optimizations: cache DerivedData (iOS) and ~/.gradle (Android) between runs. Parallelize unit and UI test targets. Use incremental builds with derived data caching.

Automated Testing Strategy

A robust mobile test suite has three layers — the testing pyramid applied to mobile:

  • Unit tests — test pure business logic and ViewModels/Presenters in isolation. Fast, hundreds of them. No device needed. (XCTest, JUnit, Jest for RN)
  • Integration tests — test interactions between components: ViewModel + Repository + DB. Still fast, no full UI. (XCTest, Robolectric, Mockito)
  • UI / E2E tests — drive the real app on a device or simulator. Slow, expensive, keep them focused on critical flows only. (XCUITest, Espresso, Detox for RN, Maestro)

Unit and integration test patterns in production — iOS XCTest and Android JUnit:

// ── iOS: Testing a ViewModel with XCTest ─────────────────────────────
import XCTest
@testable import MyApp

class ProductViewModelTests: XCTestCase {

    var sut: ProductViewModel!
    var mockRepository: MockProductRepository!

    override func setUp() {
        super.setUp()
        mockRepository = MockProductRepository()
        sut = ProductViewModel(repository: mockRepository)
    }

    // ✓ Test success path
    func testFetchProductsSuccess() async {
        // Given
        mockRepository.stubbedProducts = [
            Product(id: "1", name: "Widget", price: 9.99, inStock: true),
            Product(id: "2", name: "Gadget", price: 19.99, inStock: false)
        ]

        // When
        await sut.loadProducts()

        // Then
        XCTAssertEqual(sut.state, .loaded)
        XCTAssertEqual(sut.products.count, 2)
        XCTAssertEqual(sut.products.first?.name, "Widget")
    }

    // ✓ Test that only in-stock items are shown when filter is active
    func testFilterInStockOnly() async {
        mockRepository.stubbedProducts = [
            Product(id: "1", name: "Widget", price: 9.99, inStock: true),
            Product(id: "2", name: "Gadget", price: 19.99, inStock: false)
        ]
        await sut.loadProducts()

        sut.showInStockOnly = true   // toggle filter

        XCTAssertEqual(sut.filteredProducts.count, 1)
        XCTAssertEqual(sut.filteredProducts.first?.name, "Widget")
    }

    // ✓ Test error handling
    func testFetchProductsNetworkError() async {
        mockRepository.shouldThrowError = true

        await sut.loadProducts()

        XCTAssertEqual(sut.state, .error("Network unavailable"))
        XCTAssertTrue(sut.products.isEmpty)
    }
}

// ── Android: Testing a ViewModel with JUnit + Coroutines ─────────────
@RunWith(MockitoJUnitRunner::class)
class ProductViewModelTest {

    @get:Rule val coroutineRule = MainCoroutineRule()

    private val repository = mock(ProductRepository::class.java)
    private lateinit var viewModel: ProductViewModel

    @Before fun setUp() { viewModel = ProductViewModel(repository) }

    @Test fun `fetch products emits success state`() = coroutineRule.runTest {
        // Given
        val products = listOf(Product("1", "Widget", 9.99, inStock = true))
        whenever(repository.getProducts()).thenReturn(Result.success(products))

        // When
        viewModel.loadProducts()

        // Then
        val state = viewModel.uiState.value
        assertThat(state).isInstanceOf(UiState.Success::class.java)
        assertThat((state as UiState.Success).products).hasSize(1)
    }

    @Test fun `network error emits error state`() = coroutineRule.runTest {
        whenever(repository.getProducts())
            .thenReturn(Result.failure(IOException("Network error")))

        viewModel.loadProducts()

        assertThat(viewModel.uiState.value).isInstanceOf(UiState.Error::class.java)
    }
}

iOS Code Signing — The Hard Part

iOS code signing requires three things working together perfectly:

  • Certificate — identifies the developer/team. Issued by Apple. Has an expiry date (1 year). Stored in the macOS Keychain.
  • Provisioning Profile — binds the certificate + App ID + device list + entitlements (push notifications, iCloud, etc.) together. Created on developer.apple.com.
  • Keychain — the CI machine must have the certificate installed and trusted in its keychain before building.
# Manual keychain setup in CI (GitHub Actions)
- name: Install Apple Certificate
  env:
    BUILD_CERTIFICATE_BASE64: ${{ secrets.BUILD_CERTIFICATE_BASE64 }}
    P12_PASSWORD: ${{ secrets.P12_PASSWORD }}
    KEYCHAIN_PASSWORD: ${{ secrets.KEYCHAIN_PASSWORD }}
  run: |
    echo "$BUILD_CERTIFICATE_BASE64" | base64 --decode > certificate.p12
    security create-keychain -p "$KEYCHAIN_PASSWORD" build.keychain
    security set-keychain-settings -lut 21600 build.keychain
    security unlock-keychain -p "$KEYCHAIN_PASSWORD" build.keychain
    security import certificate.p12 -P "$P12_PASSWORD" \
      -A -t cert -f pkcs12 -k build.keychain
    security list-keychain -d user -s build.keychain

Better approach: use Fastlane Match to manage all certificates and profiles in an encrypted Git repo. One command syncs everything to any CI machine automatically.

Fastlane Match — complete setup for a team managing multiple apps and environments:

# ── Setting up Fastlane Match from scratch ───────────────────────────

# 1. Create a private Git repo for certificates (e.g. github.com/myorg/ios-certs)
# 2. Run match init — creates Matchfile
fastlane match init

# Matchfile
git_url("git@github.com:myorg/ios-certs.git")
storage_mode("git")
type("appstore")   # or "adhoc", "development", "enterprise"

# 3. Generate and store certificates for each environment
fastlane match development   # dev certs for team members
fastlane match adhoc         # for Firebase App Distribution / Ad Hoc
fastlane match appstore      # for TestFlight + App Store

# This encrypts and commits to your private repo:
# certs/development/com.myapp.cer
# certs/development/com.myapp.p12   (password protected)
# profiles/development/AppStore_com.myapp.mobileprovision

# 4. In Fastfile — use match before every build
lane :beta do
  match(
    type: "appstore",
    readonly: true,                         # CI never modifies certs
    app_identifier: "com.myapp.ios",
    git_basic_authorization: ENV["MATCH_GIT_TOKEN"]
  )
  increment_build_number(
    build_number: latest_testflight_build_number + 1
  )
  build_app(
    scheme: "MyApp",
    export_method: "app-store",
    xcargs: "-allowProvisioningUpdates"
  )
  upload_to_testflight
end

# 5. Adding a new team member — one command installs everything
fastlane match development --readonly
# → downloads + installs all dev certs and profiles
# → team member can build immediately
Profile Type
Development
Run on registered devices during development. Required for team members' personal iPhones. Expires after 1 year. Match generates one per developer or a shared profile for the team.
Profile Type
Ad Hoc
Install on up to 100 specific registered devices. Used for Firebase App Distribution and internal QA. Must register every tester's device UDID on developer.apple.com before generating the profile.
Profile Type
App Store
Required for TestFlight upload and App Store submission. Cannot install directly on devices without TestFlight. This is what CI uses for beta and production builds.
Profile Type
Enterprise
For companies with Apple Developer Enterprise Program ($299/year). Distribute apps internally to all employees without App Store. Requires Apple approval. Not for external distribution.

Fastlane — The Mobile Automation Backbone

Fastlane is the industry-standard open-source tool for automating iOS and Android workflows. It wraps complex Xcode, Gradle, and App Store API calls into simple Ruby-based actions.

# Fastfile — defines "lanes" (automated workflows)
default_platform(:ios)

platform :ios do

  lane :test do
    run_tests(
      scheme: "MyApp",
      devices: ["iPhone 15 Pro"],
      code_coverage: true
    )
  end

  lane :beta do
    match(type: "appstore")
    increment_build_number
    build_app(scheme: "MyApp")
    upload_to_testflight(skip_waiting_for_build_processing: true)
    slack(message: "🚀 New beta build uploaded to TestFlight!")
  end

  lane :release do
    match(type: "appstore")
    build_app(scheme: "MyApp")
    upload_to_app_store(
      submit_for_review: true,
      automatic_release: false,
      metadata_path: "./metadata"
    )
  end

end

Key Fastlane tools:

  • match — sync certificates and profiles from encrypted Git repo
  • gym — build iOS .ipa files
  • pilot — upload to TestFlight
  • deliver — upload metadata + binary to App Store
  • supply — upload APK/AAB + metadata to Play Store
  • scan — run automated tests

Advanced Fastlane patterns used by production teams — multi-target apps, changelog automation, and error handling:

# ── Production-Grade Fastfile Patterns ───────────────────────────────

platform :ios do

  # Error handling — notify team if any lane fails
  error do |lane, exception, options|
    slack(
      message: "❌ Lane '#{lane}' failed: #{exception.message}",
      channel: "#mobile-alerts",
      success: false,
      payload: { "Lane" => lane.to_s, "Git Branch" => git_branch }
    )
  end

  # Multi-environment lane — same code, different bundle IDs
  lane :beta_staging do
    ENV["APP_ENVIRONMENT"] = "staging"
    build_and_distribute(
      scheme: "MyApp-Staging",
      bundle_id: "com.myapp.staging",
      testflight_group: "Internal QA"
    )
  end

  lane :beta_production do
    ENV["APP_ENVIRONMENT"] = "production"
    build_and_distribute(
      scheme: "MyApp",
      bundle_id: "com.myapp.ios",
      testflight_group: "Beta Testers"
    )
  end

  # Shared private action — avoids code duplication
  private_lane :build_and_distribute do |options|
    match(type: "appstore", app_identifier: options[:bundle_id])

    # Auto-generate release notes from Git history
    changelog = changelog_from_git_commits(
      commits_count: 20,
      pretty: "- %s",
      date_format: "short",
      merge_commit_filtering: "exclude_merges"
    )

    increment_build_number(
      build_number: latest_testflight_build_number(
        app_identifier: options[:bundle_id]
      ) + 1
    )

    build_app(
      scheme: options[:scheme],
      export_method: "app-store",
      output_name: "#{options[:scheme]}_#{get_build_number}.ipa"
    )

    upload_to_testflight(
      app_identifier: options[:bundle_id],
      groups: [options[:testflight_group]],
      changelog: changelog,
      skip_waiting_for_build_processing: true
    )
  end

  # App Extensions — build main app + Today Widget + Notification Extension
  lane :beta_with_extensions do
    match(type: "appstore", app_identifier: [
      "com.myapp.ios",
      "com.myapp.ios.widget",
      "com.myapp.ios.notification-extension"
    ])
    build_app(scheme: "MyApp")   # Xcode builds all targets in scheme
    upload_to_testflight
  end

end

GitHub Actions for Mobile

GitHub Actions runs mobile CI/CD natively — macOS runners for iOS (Xcode available), Ubuntu runners for Android. The YAML workflow defines triggers, jobs, and steps.

# .github/workflows/ios-ci.yml
name: iOS CI/CD

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: macos-14
    steps:
      - uses: actions/checkout@v4

      - name: Select Xcode version
        run: sudo xcode-select -s /Applications/Xcode_15.2.app

      - name: Install dependencies
        run: bundle install

      - name: Run unit tests
        run: bundle exec fastlane test

  deploy-beta:
    runs-on: macos-14
    needs: test
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - name: Install dependencies
        run: bundle install

      - name: Deploy to TestFlight
        env:
          APP_STORE_CONNECT_API_KEY_KEY_ID: ${{ secrets.ASC_KEY_ID }}
          APP_STORE_CONNECT_API_KEY_ISSUER_ID: ${{ secrets.ASC_ISSUER_ID }}
          APP_STORE_CONNECT_API_KEY_KEY: ${{ secrets.ASC_PRIVATE_KEY }}
          MATCH_PASSWORD: ${{ secrets.MATCH_PASSWORD }}
          MATCH_GIT_BASIC_AUTHORIZATION: ${{ secrets.MATCH_GIT_TOKEN }}
        run: bundle exec fastlane beta

Caching strategy — cutting iOS build time from 15 minutes to under 7 minutes:

# .github/workflows/ios-ci-optimized.yml — with aggressive caching

jobs:
  build:
    runs-on: macos-14
    steps:
      - uses: actions/checkout@v4

      # ── Cache 1: Ruby gems (Fastlane + plugins) ─────────────────────
      - name: Cache Ruby gems
        uses: actions/cache@v4
        with:
          path: vendor/bundle
          key: ${{ runner.os }}-gems-${{ hashFiles('Gemfile.lock') }}
          restore-keys: ${{ runner.os }}-gems-

      - name: Install gems
        run: bundle install --path vendor/bundle

      # ── Cache 2: CocoaPods ────────────────────────────────────────────
      - name: Cache CocoaPods
        uses: actions/cache@v4
        with:
          path: |
            Pods
            ~/.cocoapods
          key: ${{ runner.os }}-pods-${{ hashFiles('Podfile.lock') }}
          restore-keys: ${{ runner.os }}-pods-

      - name: Install CocoaPods
        run: bundle exec pod install --repo-update
        if: steps.cache-pods.outputs.cache-hit != 'true'

      # ── Cache 3: SPM packages ─────────────────────────────────────────
      - name: Cache SPM packages
        uses: actions/cache@v4
        with:
          path: ~/Library/Developer/Xcode/DerivedData/**/SourcePackages
          key: ${{ runner.os }}-spm-${{ hashFiles('MyApp.xcodeproj/project.xcworkspace/xcshareddata/swiftpm/Package.resolved') }}

      # ── Cache 4: DerivedData (incremental builds) ─────────────────────
      - name: Cache DerivedData
        uses: actions/cache@v4
        with:
          path: ~/Library/Developer/Xcode/DerivedData
          key: ${{ runner.os }}-xcode-${{ hashFiles('**/*.swift', '**/*.m', '**/*.h') }}
          restore-keys: ${{ runner.os }}-xcode-

      - name: Build and test
        run: bundle exec fastlane test

# Result: First run ~14min. Subsequent runs with cache hits: ~6min 30s

Android CI/CD — Build & Sign

Android signing uses a keystore file — a single file that holds your signing key. Unlike iOS, Android signing is simpler but losing the keystore means you can never update your app on the Play Store.

# GitHub Actions — Android build and sign
jobs:
  android-build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with: { java-version: '17', distribution: 'temurin' }

      - name: Decode keystore
        run: |
          echo "${{ secrets.KEYSTORE_BASE64 }}" | base64 --decode \
            > app/release.keystore

      - name: Build release AAB
        run: ./gradlew bundleRelease

      - name: Sign AAB
        uses: r0adkll/sign-android-release@v1
        with:
          releaseDirectory: app/build/outputs/bundle/release
          signingKeyBase64: ${{ secrets.KEYSTORE_BASE64 }}
          alias: ${{ secrets.KEY_ALIAS }}
          keyStorePassword: ${{ secrets.KEYSTORE_PASSWORD }}
          keyPassword: ${{ secrets.KEY_PASSWORD }}

      - name: Upload to Play Store (internal track)
        uses: r0adkll/upload-google-play@v1
        with:
          serviceAccountJsonPlainText: ${{ secrets.SERVICE_ACCOUNT_JSON }}
          packageName: com.myapp.android
          releaseFiles: app/build/outputs/bundle/release/*.aab
          track: internal

Android build variants — managing dev/staging/prod from a single codebase:

// ── build.gradle (app) — multi-environment configuration ─────────────

android {
    compileSdk 34

    defaultConfig {
        applicationId "com.myapp.android"
        minSdk 24; targetSdk 34
        versionCode 1; versionName "1.0"
    }

    // Signing configurations (values from environment variables in CI)
    signingConfigs {
        release {
            storeFile file(System.getenv("KEYSTORE_PATH") ?: "release.keystore")
            storePassword System.getenv("KEYSTORE_PASSWORD") ?: ""
            keyAlias System.getenv("KEY_ALIAS") ?: ""
            keyPassword System.getenv("KEY_PASSWORD") ?: ""
        }
    }

    // Build types
    buildTypes {
        debug {
            applicationIdSuffix ".debug"
            debuggable true
            buildConfigField "String", "API_BASE_URL", '"https://api-dev.myapp.com"'
        }
        release {
            minifyEnabled true
            shrinkResources true
            signingConfig signingConfigs.release
            proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'), 'proguard-rules.pro'
            buildConfigField "String", "API_BASE_URL", '"https://api.myapp.com"'
        }
    }

    // Product flavors — separate staging and production App IDs
    flavorDimensions += "environment"
    productFlavors {
        staging {
            dimension "environment"
            applicationIdSuffix ".staging"
            versionNameSuffix "-staging"
            buildConfigField "String", "API_BASE_URL", '"https://api-staging.myapp.com"'
            resValue "string", "app_name", "MyApp Staging"
        }
        production {
            dimension "environment"
            // applicationId stays as defaultConfig — no suffix
            buildConfigField "String", "API_BASE_URL", '"https://api.myapp.com"'
            resValue "string", "app_name", "MyApp"
        }
    }
}

// This produces 4 build variants:
// stagingDebug   → com.myapp.android.staging.debug  (local dev)
// stagingRelease → com.myapp.android.staging         (QA / Firebase)
// productionDebug → com.myapp.android.debug          (rarely used)
// productionRelease → com.myapp.android              (Play Store)

Complete beta testing workflow — from QA to external testers to production:

1
Internal QA (Day 1–2). Firebase App Distribution to 5–10 QA engineers. No App Store involvement — instant distribution. QA runs test cases, records bugs in Jira/Linear. Crash reports flow to Crashlytics in real time. Goal: zero P0/P1 bugs before external beta.
2
Internal TestFlight / Play Internal (Day 2–3). Same build uploaded to TestFlight (iOS) and Play Store internal track (Android). Product managers, executives, and CS team install and validate from user perspective. Apple processing: ~15 min. Play Store: instant.
3
External Beta (Day 3–7). TestFlight external group (up to 10,000 users) or Play Store open beta. Collect feedback via in-app prompts. Monitor D1 retention and crash-free rate. A crash-free rate below 99.5% triggers a hold and a new build.
4
Production (Staged Rollout). Submit App Store for review (1–2 days). Play Store: immediate release at 1% staged rollout. Monitor crash rate and ratings at each percentage increase (1% → 5% → 10% → 25% → 50% → 100%). Automated rollout halt if crash rate spikes above threshold.

App Store & Play Store Deployment

Automated store submission eliminates the manual upload → fill metadata → submit cycle. Both stores expose APIs that Fastlane and CI tools use:

  • App Store Connect API — Apple's REST API for submitting builds, updating metadata, and managing TestFlight. Use JWT-based API Keys (never username/password in CI).
  • Google Play Developer API — upload AABs to tracks (internal → alpha → beta → production) with staged rollouts.
  • Staged rollouts — release to 1% → 5% → 10% → 50% → 100% of users on Android. Catch crashes before full release.
  • Phased release — Apple's equivalent: 1% per day over 7 days automatically.
# Fastlane — upload to Play Store with staged rollout

    track: 'production',
    rollout: '0.1',
    aab: 'app/build/outputs/bundle/release/app-release.aab',
    json_key_data: ENV['PLAY_STORE_JSON_KEY'],
    metadata_path: './fastlane/metadata/android'
  )
end

Mobile Monitoring & Crash Reporting

Shipping is not the end — monitoring production apps is critical. The tools every mobile team needs:

  • Firebase Crashlytics — real-time crash reporting for iOS and Android. Groups similar crashes, shows stack traces, user impact. Free.
  • Sentry — crash reporting + performance monitoring + session replay. Works across iOS, Android, React Native, Flutter.
  • Firebase Performance Monitoring — app start time, network request latency, custom traces.
  • App Store Connect Analytics — downloads, sessions, crashes, retention from Apple.
  • OTA updates (Expo / CodePush) — for React Native: push JavaScript bundle updates without app store submission. Critical for hotfixes.

Crash-free rate goal: industry standard is >99.5% crash-free sessions. Alert on anything below 99%. A drop below 98% is a P0 incident — halt the rollout immediately.

Sentry for mobile — custom performance traces that measure what matters to users:

// ── iOS: Sentry Performance Monitoring with Custom Traces ─────────────
import Sentry

class AppDelegate: UIResponder, UIApplicationDelegate {
    func application(_ app: UIApplication,
                     didFinishLaunchingWithOptions options: [UIApplication.LaunchOptionsKey: Any]?) -> Bool {
        SentrySDK.start { options in
            options.dsn = "https://your-dsn@sentry.io/project"
            options.tracesSampleRate = 0.2    // sample 20% of transactions
            options.profilesSampleRate = 0.1  // profile 10% of sampled transactions
            options.enableAutoPerformanceTracing = true
            options.enableUIViewControllerTracing = true
            options.enableNetworkTracking = true
            options.enableFileIOTracing = true
        }
        return true
    }
}

// Custom transaction — measure checkout flow end-to-end
class CheckoutViewController: UIViewController {

    func startCheckout() {
        // Create a transaction for the entire checkout flow
        let transaction = SentrySDK.startTransaction(
            name: "checkout-flow",
            operation: "navigation"
        )

        // Span 1: Load cart items
        let cartSpan = transaction.startChild(
            operation: "db.query",
            description: "Load cart items from database"
        )
        let items = cartRepository.loadItems()
        cartSpan.finish()

        // Span 2: Validate payment method
        let paymentSpan = transaction.startChild(
            operation: "http.client",
            description: "Validate payment method with Stripe"
        )
        paymentService.validate { result in
            paymentSpan.finish(status: result.isValid ? .ok : .invalidArgument)

            // Span 3: Process payment
            if result.isValid {
                let processSpan = transaction.startChild(
                    operation: "http.client",
                    description: "POST /payments/charge"
                )
                self.paymentService.charge(amount: self.total) { chargeResult in
                    processSpan.finish()
                    transaction.finish()   // ← marks transaction complete
                }
            }
        }
    }
}

// Android: Sentry performance monitoring equivalent
class CheckoutViewModel : ViewModel() {
    fun processCheckout() {
        val transaction = Sentry.startTransaction("checkout-flow", "navigation")

        viewModelScope.launch {
            val cartSpan = transaction.startChild("db.query", "Load cart items")
            val items = cartRepository.getItems()
            cartSpan.finish()

            val paymentSpan = transaction.startChild("http.client", "Validate payment")
            val isValid = paymentService.validate()
            paymentSpan.finish(if (isValid) SpanStatus.OK else SpanStatus.INVALID_ARGUMENT)

            transaction.finish()
        }
    }
}

Secrets & Environment Management

Mobile CI pipelines are full of secrets: signing keys, API keys, service account JSONs. Managing them securely is critical:

  • Never commit secrets to Git — keystore files, .p12 certificates, API keys. Use base64-encoded GitHub Secrets or CI platform secret stores.
  • Environment-specific configs — separate dev, staging, and prod configurations. Use .xcconfig for iOS, build variants for Android, .env files for React Native.
  • Fastlane Match — stores all iOS certificates in an encrypted private Git repo. One password unlocks everything, rotatable without changing CI configs.
  • Secret rotation — App Store Connect API keys and service accounts should be rotated annually and use minimum required permissions.
# Encode keystore to base64 for GitHub Secrets
base64 -i release.keystore | pbcopy   # macOS — copies to clipboard

# In CI, decode it back:
echo "$KEYSTORE_BASE64" | base64 --decode > release.keystore

# ❌ Never hardcode secrets:
storePassword = "MyPassword123"

# ✅ Always use environment variables:
storePassword = System.getenv("KEYSTORE_PASSWORD") ?: ""
