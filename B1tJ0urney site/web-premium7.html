🚀 7.1 What is DevOps?

Modern software development doesn't end when the code is written. Before DevOps existed, there was a wall between Development teams (who write code) and Operations teams (who manage servers and deployments). This created slow releases, blame-throwing when things broke, and painful manual processes.


DevOps is a culture, a set of practices, and a toolchain that unites these two worlds. The goal is simple: ship software faster, more reliably, and with less risk.

⚡
Speed

Deploy multiple times per day instead of once per month

🛡️
Reliability

Automated tests and rollbacks prevent broken releases reaching users

📈
Scale

Infrastructure scales automatically with user demand

🤝
Collaboration

Dev and Ops share responsibility for the full lifecycle of the application

🔄 The DevOps Infinity Loop:
Plan → Code → Build → Test → Release → Deploy → Operate → Monitor → (back to Plan)

This continuous loop replaces the old "waterfall" model where code was thrown over the wall to operations after months of development.

🌿 7.2 Git — Version Control in Depth

Git is the foundation of every modern DevOps workflow. It's a distributed version control system that records the full history of your codebase, enables collaboration between teams, and makes it safe to experiment without breaking things.

ESSENTIAL GIT COMMANDS

# Initial setup
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# Start a project
git init                    # Initialize a new repo
git clone <url>             # Clone an existing repo from GitHub

# Daily workflow
git status                  # See what changed
git add .                   # Stage all changes
git add src/app.js          # Stage a specific file
git commit -m "feat: add login page"   # Save a snapshot

# Remote (GitHub/GitLab)
git push origin main        # Send commits to remote
git pull origin main        # Get latest changes from remote

# Inspect history
git log --oneline           # Compact commit history
git diff                    # See unstaged changes
🌳 Branching Strategy:

Branches let you work on features, fixes, or experiments in isolation without touching the stable main branch.

# Create and switch to a new branch
git checkout -b feature/user-authentication

# Make changes, commit...
git add .
git commit -m "feat: implement JWT authentication"

# Merge back into main
git checkout main
git merge feature/user-authentication

# Delete the branch after merge
git branch -d feature/user-authentication

# See all branches
git branch -a
📝 Commit Message Convention

Use the Conventional Commits standard for clean, readable history:

feat: add shopping cart
fix: resolve login redirect bug
docs: update README setup steps
refactor: extract auth helpers
test: add unit tests for Cart
chore: update dependencies
⚠️ Git Conflicts

A merge conflict happens when two branches edit the same line. Git marks the conflict in the file:

<<<<<<< HEAD
console.log("Hello Dev");
=======
console.log("Hello Ops");
>>>>>>> feature/ops-branch
# Edit file to resolve, then git add + commit

Workflow Best For How it works

GitHub Flow Continuous delivery, web apps main + short-lived feature branches, PR to merge
Trunk-Based Large teams, CI/CD heavy Everyone commits to main (with feature flags)

🔄 7.3 CI/CD — Automating the Entire Pipeline

CI/CD is the practice of automating everything between writing code and having it running in production. The goal is to make releasing software as boring and routine as possible — eliminating the human drama and manual steps.

🔧 CI — Continuous Integration

Every time a developer pushes code, an automated system:

  • → Pulls the new code
  • → Installs dependencies (npm install)
  • → Runs linting (code quality check)
  • → Runs all unit and integration tests
  • → Builds the application
  • → Reports ✅ pass or ❌ fail to the team

🚀 CD — Continuous Delivery / Deployment

After CI passes, CD takes over:

  • → Continuous Delivery: Code is always ready to deploy (human triggers it)
  • → Continuous Deployment: Code is deployed automatically if all tests pass
  • → Sends the build to staging or production
  • → Runs smoke tests on the deployed version
  • → Notifies the team (Slack, email)
⚙️ GitHub Actions — The most popular CI/CD tool:

.github/workflows/deploy.yml

name: CI/CD Pipeline

on:
  push:
    branches: [main]        # Trigger on push to main

jobs:
  build-and-test:
    runs-on: ubuntu-latest  # Run on a fresh Linux machine

    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'

      - name: Install dependencies
        run: npm ci

      - name: Run linting
        run: npm run lint

      - name: Run tests
        run: npm test

      - name: Build application
        run: npm run build

      - name: Deploy to Vercel
        uses: amondnet/vercel-action@v20
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.ORG_ID }}
          vercel-project-id: ${{ secrets.PROJECT_ID }}

🐳 7.4 Docker — Containerization

The classic problem: "It works on my machine!". Docker solves this by packaging your application along with everything it needs to run — the runtime, dependencies, configuration — into a portable unit called a container.


A container is like a lightweight, isolated virtual machine. It runs identically on your laptop, your colleague's machine, and any cloud server in the world.

😩 Without Docker
  • → "Works on my machine but not on the server"
  • → Different Node versions across developers
  • → Manual server setup for each environment
  • → Hours of debugging environment issues
😊 With Docker
  • → Same container runs on every machine
  • → All dependencies locked inside the image
  • → New developer onboards with one command
  • → Scales horizontally with ease

Dockerfile — Node.js Application

# Start from an official Node.js image
FROM node:18-alpine

# Set the working directory inside the container
WORKDIR /app

# Copy dependency files first (for layer caching)
COPY package*.json ./

# Install dependencies
RUN npm ci --production

# Copy the rest of the application code
COPY . .

# Build the application
RUN npm run build

# Expose port 3000 to the outside world
EXPOSE 3000

# Command to start the app
CMD ["node", "dist/server.js"]

ESSENTIAL DOCKER COMMANDS

# Build an image from Dockerfile
docker build -t my-app:1.0 .

# Run a container from the image
docker run -p 3000:3000 my-app:1.0

# Run in background (detached)
docker run -d -p 3000:3000 --name my-container my-app:1.0

# List running containers
docker ps

# Stop and remove a container
docker stop my-container
docker rm my-container

# List local images
docker images

# Remove an image
docker rmi my-app:1.0

# View logs
docker logs my-container
🔑 Key Docker Concepts:
🖼️ Image

A read-only template/blueprint for creating containers. Like a class in OOP.

📦 Container

A running instance of an image. Like an object created from a class.

📋 Dockerfile

A script of instructions to build an image from scratch.

🏪 Docker Hub

A public registry of images (like npm but for Docker). Pull official images: node, postgres, nginx.

🎼 7.5 Docker Compose — Multi-Container Apps

Real applications are not just one container. They have a backend API, a database, a cache, perhaps a reverse proxy. Docker Compose lets you define and run all of them together in a single file.

docker-compose.yml — Full Stack App (Node + PostgreSQL + Redis)

version: '3.8'

services:
  # Your Node.js backend API
  api:
    build: ./backend
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://user:pass@db:5432/myapp
      REDIS_URL: redis://cache:6379
    depends_on:
      - db
      - cache

  # PostgreSQL Database
  db:
    image: postgres:15-alpine
    volumes:
      - postgres_data:/var/lib/postgresql/data
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: myapp

  # Redis Cache
  cache:
    image: redis:7-alpine
    ports:
      - "6379:6379"

volumes:
  postgres_data:  # Persist database data between restarts

DOCKER COMPOSE COMMANDS

docker compose up -d        # Start all services in background
docker compose down         # Stop and remove all containers
docker compose logs -f api  # Follow logs of the 'api' service
docker compose ps           # Status of all services
docker compose exec api sh  # Open a shell inside the api container

☁️ 7.6 Cloud Hosting and Deployment Platforms

Where your application lives in production matters. The right platform depends on the type of app, team size, and technical needs.

▲ Vercel

Optimized for frontend and Next.js. Zero-config deployments from GitHub. Built-in CDN, preview deployments per PR, Edge Functions. Free tier very generous.

Best for: React, Vue, Next.js, Nuxt
◈ Netlify

Git-connected deployments, serverless functions, form handling, identity. Similar to Vercel, slightly broader feature set for JAMstack.

Best for: Static sites, JAMstack, SPAs
🚂 Render / Railway

Full-stack platforms. Deploy Node, Python, Go, Docker, PostgreSQL, Redis. Simpler than AWS, more powerful than Vercel for backends.

Best for: Full-stack, APIs, databases
☁️ AWS / Azure / GCP

Enterprise cloud providers. Hundreds of services. Complete control but high complexity. Used by Netflix, Airbnb, every large company.

Best for: Enterprise, high scale, complex infra

🌍 What is a CDN?

A Content Delivery Network is a global network of servers. When a user in Tokyo visits your site, files are served from the nearest server (e.g., Singapore), not from your origin server in the US. This dramatically reduces latency. Vercel and Netlify have built-in CDNs. For custom setups, use Cloudflare (free tier is excellent).

🔐 7.7 Environment Variables and Secrets Management

Your application needs different configurations in different environments (development vs production). It also needs to store sensitive values like API keys, database passwords, and JWT secrets — but never in your code repository.

📄 .env file (local only)

# .env — NEVER commit this!
NODE_ENV=development
PORT=3000

DATABASE_URL=postgres://localhost/mydb
JWT_SECRET=my-super-secret-key-change-in-prod

STRIPE_SECRET_KEY=sk_test_abc123...
SMTP_PASSWORD=email-password-here

# Different per environment
API_BASE_URL=http://localhost:3000

📋 .env.example (commit this)

# .env.example — Safe to commit!
# Copy to .env and fill in real values
NODE_ENV=
PORT=

DATABASE_URL=
JWT_SECRET=

STRIPE_SECRET_KEY=
SMTP_PASSWORD=

API_BASE_URL=

.gitignore — protect secrets

.env
.env.local
.env.production
node_modules/
dist/
🚨 Never do this: 
// BAD — hardcoded secret in source code
const stripe = new Stripe("sk_live_REAL_KEY_HERE"); // EXPOSED TO THE WORLD!

// GOOD — use environment variable
const stripe = new Stripe(process.env.STRIPE_SECRET_KEY);

GitHub scans public repos for exposed secrets and automatically alerts you — but the damage may already be done. Attackers scrape GitHub constantly for leaked keys.

🔒 Production Secrets Management:

In production, use platform-specific secret stores — never .env files on servers:

🛡️ 7.8 Security — DevSecOps

DevSecOps means integrating security into every stage of the development pipeline — not bolting it on at the end. Security is everyone's responsibility, from developer to operations.

🔒 HTTPS and TLS

All production apps must use HTTPS. TLS encrypts data in transit between browser and server. Modern platforms (Vercel, Netlify, Render) provide free certificates via Let's Encrypt automatically.

# Force HTTPS redirect in Nginx
server {
  listen 80;
  return 301 https://$host$request_uri;
}

# HTTP Strict Transport Security
Strict-Transport-Security: max-age=31536000
🧪 Dependency Scanning

Your node_modules folder contains thousands of packages, any of which could have a known vulnerability. Scan regularly:

# Built-in npm audit
npm audit
npm audit fix        # Auto-fix vulnerabilities

# Snyk — advanced scanning
npx snyk test

# GitHub Dependabot — automatic PRs
# for security updates (enable in repo settings)
🚫 OWASP Top 10 — Know the Most Common Web Vulnerabilities:
# Vulnerability Prevention
A01 Broken Access Control Validate permissions on every request
A02 Cryptographic Failures Encrypt sensitive data, use HTTPS, bcrypt passwords
A03 Injection (SQL, XSS) Use parameterized queries, sanitize user inputs
A05 Security Misconfiguration Disable debug mode in production, remove default credentials
A07 Identification & Auth Failures Rate limiting, MFA, secure session management
🔑 Rate Limiting

Prevent brute-force attacks and API abuse by limiting how many requests a client can make:

// Express + express-rate-limit
import rateLimit from 'express-rate-limit';
const limiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 min
  max: 100 // max 100 requests/window
});
app.use('/api/', limiter);
🧹 Input Validation and Sanitization

Never trust user input. Always validate on the server side:

// Zod schema validation
import { z } from 'zod';
const UserSchema = z.object({
  email: z.string().email(),
  password: z.string().min(8),
  age: z.number().int().min(18)
});
// Throws if invalid

📊 7.9 Monitoring and Observability

You cannot manage what you cannot measure. Once your app is in production, you need to know what it's doing — is it responding fast? Are there errors? Is the server running out of memory? Monitoring tells you before users start complaining.

📋
Logs

Chronological record of events. What happened and when? Use structured JSON logs.

📈
Metrics

Numerical measurements over time: CPU%, memory, request rate, error rate, response time.

🔍
Traces

Follow a single request through all services to find bottlenecks. Essential for microservices.

🚨 Sentry

Real-time error tracking. Captures exceptions with full stack traces, user context, and release info. Free tier available. Integrates with every major framework in minutes.

import * as Sentry from "@sentry/node";
Sentry.init({ dsn: process.env.SENTRY_DSN });
📊 Grafana + Prometheus

The industry standard for metrics. Prometheus scrapes metrics from your services; Grafana visualizes them in beautiful dashboards. Widely used in enterprise DevOps.

🔍 Datadog / New Relic

All-in-one observability platforms (logs + metrics + traces). More turnkey than the Grafana stack but paid. Very common in mid-large companies.

⏱️ Uptime Monitoring

Tools like UptimeRobot or Better Uptime ping your URL every minute and alert you via SMS/email/Slack if your site goes down. Free tiers available.

⚡ 7.10 Performance Optimization in Production

A fast application improves user experience, SEO rankings, and conversion rates. Performance optimization happens at every layer — from the browser to the database to the network.

Frontend Performance:
  • → Code Splitting: Split JS bundles per route
  • → Tree Shaking: Remove unused code from bundle
  • → Image Optimization: WebP format, lazy loading, correct sizes
  • → Minification: Remove whitespace/comments from JS/CSS
  • → Compression: gzip/Brotli encoding for all assets
  • → Preloading: <link rel="preload"> for critical resources
Backend Performance:
  • → Database Indexing: Speed up slow queries with indexes
  • → Redis Caching: Store frequent query results in memory
  • → Connection Pooling: Reuse database connections
  • → Async Processing: Move slow tasks to background queues
  • → Load Balancing: Distribute traffic across multiple servers
  • → Horizontal Scaling: Add more instances, not just bigger ones

REDIS CACHING EXAMPLE (Node.js)

import { createClient } from 'redis';
const redis = createClient({ url: process.env.REDIS_URL });

async function getProduct(id) {
  const cacheKey = `product:${id}`;
  
  // 1. Check cache first
  const cached = await redis.get(cacheKey);
  if (cached) {
    console.log('Cache HIT');
    return JSON.parse(cached);
  }
  
  // 2. Cache miss — fetch from database
  console.log('Cache MISS');
  const product = await db.query('SELECT * FROM products WHERE id = $1', [id]);
  
  // 3. Store in cache for 1 hour
  await redis.setEx(cacheKey, 3600, JSON.stringify(product));
  
  return product;
}

🎯 7.11 Deploy Strategies — Zero Downtime

How you deploy matters as much as what you deploy. A bad deploy strategy can take your entire application down for thousands of users. Here are the main approaches:

🔴 Big Bang (Risky)

Take the server down, deploy new version, bring it back up. Causes downtime. Never do this in production.

🔵🟢 Blue/Green

Run two identical environments (Blue = current, Green = new). Switch traffic to Green once tested. Instant rollback: just switch back to Blue.

🐤 Canary Release

Deploy new version to 5% of users first. Monitor for errors. If stable, gradually roll out to 25%, 50%, 100%. If issues: rollback instantly.

🔄 Rolling Update

Replace old containers/servers one at a time while others keep serving traffic. Default strategy in Kubernetes.

🔙 Rollback Strategy:

Always have a rollback plan before deploying. If something goes wrong in production, you need to be able to revert in seconds, not hours:

# Vercel — instant rollback via CLI or dashboard
vercel rollback [deployment-url]

# Docker — tag images with git commit hash
docker build -t my-app:$(git rev-parse --short HEAD) .
# To rollback: just run the previous image tag

# Git — revert a bad commit
git revert HEAD         # Create a new "undo" commit
git push origin main    # Push the fix — triggers new deployment

🏗️ 7.12 Infrastructure as Code (IaC)

Instead of clicking through cloud dashboards to create servers, databases, and networks, you define your entire infrastructure in code files. This makes it reproducible, version-controlled, and automated.

🌍 Terraform

The most popular IaC tool. Works with AWS, Azure, GCP, and 1000+ providers. Declarative: you describe what you want, Terraform figures out how to build it.

# main.tf
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
  tags = {
    Name = "WebServer"
  }
}

# terraform init
# terraform plan
# terraform apply
🎭 Ansible

Automates server configuration (installing software, setting up files, running commands) across many servers simultaneously using YAML playbooks.

☸️ Kubernetes (K8s)

Orchestrates Docker containers at scale. Handles automatic scaling, self-healing (restarts crashed containers), load balancing, and rolling updates. The industry standard for large deployments.
